# Part 7. 백엔드 개발

## 7-3. 확장성과 부하 분산 — 서버를 늘리는 것(수평 확장)이 항상 답이 아닌 이유

> **학습 목표**
> - 수직 확장과 수평 확장의 차이와, 각각이 어떤 지점에서 한계에 부딪히는지 설명할 수 있다.
> - 로드밸런서가 여러 서버로 트래픽을 나누는 원리와 대표 알고리즘을 설명할 수 있다.
> - 캐싱과 샤딩이 왜 "서버만 늘려서는 풀리지 않는" 병목을 해결하는지 설명할 수 있다.

> **개요** 트래픽이 늘어날 때 "서버를 늘리면 된다"는 직관은 절반만 맞습니다. 확장의 두 방식(수직·수평)을 비교하고, 수평 확장이 부딪히는 근본적인 한계와, 이를 보완하는 로드밸런싱·캐싱·샤딩을 살펴봅니다.

---

## 1. 확장성이란 무엇인가 — 수직 확장과 수평 확장

> **개요** 서버 처리 능력을 늘리는 두 가지 방향인 수직 확장(Scale-up)과 수평 확장(Scale-out)의 원리와 한계를 비교합니다.

![수직 확장과 수평 확장 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part7_백엔드_개발/7-3_확장성과_부하_분산/assets/diagram1_scaling.svg)

<u>**수직 확장(Vertical Scaling, Scale-up)**</u>은 한 대의 서버 사양(CPU, 메모리, 디스크)을 높이는 방식입니다. <u>**수평 확장(Horizontal Scaling, Scale-out)**</u>은 같은 사양의 서버를 여러 대로 늘려 트래픽을 나누는 방식입니다.

| 구분 | 수직 확장 | 수평 확장 |
|---|---|---|
| 방법 | 서버 1대의 사양을 올린다 | 서버 대수를 늘린다 |
| 한계 | 하드웨어의 물리적 최대 사양 | 상태를 가진 자원(DB 등)의 병목 |
| 장애 대응 | 그 서버가 죽으면 전체가 멈춘다(SPOF) | 한 대가 죽어도 나머지가 처리한다 |
| 적용 난이도 | 낮음(재부팅·사양 교체만으로 가능) | 높음(무상태 설계, 로드밸런싱 필요) |

> **Q.** 그러면 수평 확장이 항상 더 나은 선택 아닌가요?
> **A.** 아닙니다. 애플리케이션 서버처럼 요청을 처리하고 끝나는(상태가 없는) 계층은 대수를 늘릴수록 처리량이 거의 선형으로 늘어납니다. 하지만 데이터베이스처럼 <u>**모든 서버가 결국 같이 바라봐야 하는 상태(state)를 가진 자원**</u>은 단순히 서버를 늘린다고 해결되지 않습니다. 애플리케이션 서버를 10대로 늘려도, 그 10대가 모두 같은 DB에 쓰기 작업을 몰아넣는다면 병목은 사라진 것이 아니라 DB 앞으로 옮겨간 것뿐입니다.

> **원칙** 이 현상은 병렬 처리 이론에서 이미 정식화되어 있습니다. 암달의 법칙(Amdahl's Law, 1967)은 프로그램(또는 시스템)에서 병렬화할 수 없는 직렬 구간이 존재하면, 아무리 자원을 늘려도 전체 성능 향상은 그 직렬 구간의 비중에 의해 상한이 정해진다고 말합니다. 여러 대의 서버가 공유하는 하나의 DB, 하나의 세션 저장소는 바로 이 "직렬 구간"에 해당하며, <u>**수평 확장은 무상태(stateless) 구간에서만 선형에 가까운 효과를 내고, 상태를 가진 구간은 별도의 전략(캐싱·샤딩·복제)이 필요합니다.**</u>

## 2. 로드밸런싱 — 여러 서버로 트래픽을 나누는 방법

> **개요** 수평 확장이 실제로 동작하려면 요청을 여러 서버에 고르게 나눠줄 장치가 필요합니다. 로드밸런서의 역할과 대표 알고리즘을 다룹니다.

![로드밸런서를 통한 수평 확장 구조 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part7_백엔드_개발/7-3_확장성과_부하_분산/assets/diagram2_load_balancer.svg)

<u>**로드밸런서(Load Balancer)**</u>는 클라이언트의 요청을 받아, 뒤에 있는 여러 서버 중 하나로 트래픽을 분배하는 중간 장치입니다. 클라이언트 입장에서는 로드밸런서 하나만 바라보므로, 뒤에서 서버가 몇 대든 몇 대가 늘거나 줄든 신경 쓸 필요가 없습니다.

대표적인 분배 알고리즘은 다음과 같습니다.

- <u>**라운드 로빈(Round Robin)**</u>: 서버에 요청을 순서대로 하나씩 고르게 배분합니다. 서버 성능이 균일할 때 단순하고 효과적입니다.
- <u>**최소 연결(Least Connections)**</u>: 현재 처리 중인 연결이 가장 적은 서버로 보냅니다. 요청마다 처리 시간이 크게 다를 때 유리합니다.
- <u>**IP 해시(IP Hash)**</u>: 클라이언트 IP를 해시해 항상 같은 서버로 보냅니다. 특정 서버에 세션 상태가 남아있어야 하는 sticky session 상황에 사용합니다.

> **참고** 로드밸런서는 단순히 트래픽만 나누지 않고, 주기적으로 각 서버에 헬스 체크(health check)를 보내 응답이 없는 서버는 배분 대상에서 자동으로 제외합니다. 이 덕분에 서버 한 대가 갑자기 죽어도 전체 서비스는 멈추지 않습니다. 수평 확장이 수직 확장보다 장애에 강하다고 말하는 근거가 바로 이 부분입니다.

## 3. 캐싱 — 같은 요청을 반복해서 처리하지 않기

> **개요** 서버를 늘려도 DB가 병목이 되는 문제를, DB에 가는 요청 자체를 줄여서 우회하는 캐싱의 원리를 다룹니다.

<u>**캐싱(Caching)**</u>은 자주 조회되지만 자주 바뀌지는 않는 데이터를, 원본 저장소(DB)보다 훨씬 빠르게 접근할 수 있는 곳(메모리)에 미리 복사해두는 기법입니다. 흔히 쓰이는 방식은 <u>**캐시 어사이드(Cache-aside) 패턴**</u>입니다. 서버는 데이터를 요청받으면 먼저 캐시를 확인하고, 캐시에 있으면(캐시 히트) 그대로 반환하며, 없으면(캐시 미스) DB를 조회한 뒤 그 결과를 캐시에 저장하고 반환합니다.

| 캐시 종류 | 위치 | 예시 |
|---|---|---|
| 클라이언트 캐시 | 브라우저 | 정적 이미지, JS 파일을 재요청 없이 재사용 |
| CDN | 사용자와 가까운 네트워크 거점 | 이미지·동영상 등 정적 자원 분산 제공 |
| 서버 캐시 | 애플리케이션과 DB 사이 | Redis·Memcached에 자주 조회되는 상품 정보 저장 |

> **팁** 캐싱이 확장성에 기여하는 이유는 서버를 늘리는 것과 방향이 다릅니다. 서버를 늘리는 것은 "같은 일을 처리할 일꾼을 늘리는 것"이고, 캐싱은 "애초에 DB까지 가야 할 일 자체를 줄이는 것"입니다. 병목이 DB에 있다면, 서버를 아무리 늘려도 소용없지만 캐시 히트율을 높이면 DB에 도달하는 요청 자체가 줄어 병목이 완화됩니다.

> **참고** 캐시는 원본 데이터가 바뀌어도 자동으로 갱신되지 않기 때문에, 오래된 데이터를 보여줄 위험(캐시 정합성 문제)이 항상 따라옵니다. 그래서 캐시에는 만료 시간(TTL)을 두거나, 원본이 바뀔 때 캐시를 함께 무효화하는 전략이 함께 설계됩니다.

## 4. 샤딩 — 데이터를 나누어 병목을 분산하기

> **개요** 캐싱으로도 흡수되지 않는 쓰기 트래픽과 데이터 규모 문제를, 하나의 DB를 여러 대로 쪼개 분산하는 샤딩으로 해결하는 원리를 다룹니다.

<u>**샤딩(Sharding)**</u>은 하나의 큰 데이터베이스를 여러 개의 물리적으로 독립된 데이터베이스(샤드)로 쪼개, 데이터를 특정 기준(샤드 키)에 따라 나누어 저장하는 기법입니다. 이는 데이터베이스를 <u>**수평 파티셔닝(Horizontal Partitioning)**</u>하는 방식으로, 각 샤드는 전체 데이터의 일부만 담당하므로 하나의 서버가 감당하던 저장 용량과 쓰기 트래픽이 여러 대로 나뉩니다.

- 샤드 키 예시: `user_id`의 범위(0~999는 Shard A, 1000~1999는 Shard B)로 나누거나, 해시값을 기준으로 고르게 분산합니다.
- 장점: 각 샤드의 데이터 규모와 트래픽이 줄어, 단일 DB로는 감당할 수 없던 규모까지 확장할 수 있습니다.
- 단점: 서로 다른 샤드에 걸친 데이터를 한 번에 조인하기 어렵고, 샤드 키 선택이 잘못되면 특정 샤드에 트래픽이 몰리는 핫스팟이 생기며, 운영·백업·마이그레이션 복잡도가 크게 늘어납니다.

> **Q.** 캐싱만으로는 왜 부족할까요?
> **A.** 캐싱은 "읽기(read)" 요청에는 효과적이지만, 새로 쓰는(write) 데이터나 매번 값이 달라지는 요청에는 적용하기 어렵습니다. 주문 생성, 결제처럼 반드시 DB에 새로 기록해야 하는 쓰기 트래픽은 캐시로 우회할 수 없으므로, 데이터 자체를 여러 DB로 분산하는 샤딩이 필요합니다. 캐싱이 읽기 병목을, 샤딩이 쓰기·저장 병목을 각각 다른 축에서 해결한다고 볼 수 있습니다.

## 5. 실생활 예시 — 블랙프라이데이 세일 때 서버가 늘어나는 과정

> **해석** 아래 표는 평소 트래픽의 수십 배가 몰리는 대규모 세일 상황에서, 지금까지 배운 개념들이 실제로 어떤 순서로 동작하는지 보여줍니다.

| 시점 | 트래픽 상태 | 실제로 벌어지는 일 |
|---|---|---|
| 평소 | 기준 트래픽 | 서버 3대가 로드밸런서 뒤에서 트래픽을 나누어 처리 |
| 세일 시작 직전 | 트래픽 급증 예고 | 운영팀이 서버 대수를 미리 늘려두거나, 오토스케일링 임계치를 조정 |
| 세일 시작 직후 | 트래픽 폭증 | 로드밸런서가 늘어난 요청을 감지하고, 오토스케일링이 서버를 추가로 기동해 자동으로 풀에 등록 |
| 상품 목록 조회 급증 | 동일한 상품 정보에 반복 조회 | 캐시가 대부분의 조회 요청을 흡수해 DB까지 가는 요청을 크게 줄임 |
| 결제·주문 폭증 | 쓰기 트래픽 급증 | 캐시로 우회할 수 없는 쓰기 요청이 샤딩된 DB에 분산 기록됨 |
| 세일 종료 후 | 트래픽 하락 | 오토스케일링이 여분의 서버를 다시 줄여 비용을 절감 |

> **참고** 이 과정에서 "서버를 늘리는 것"은 로드밸런서 뒤의 애플리케이션 서버 계층에서만 일어납니다. 상태를 가진 계층(DB)은 서버를 늘리는 방식이 아니라 캐싱과 샤딩이라는, 이번 강의에서 다룬 별도의 전략으로 대응한다는 점이 핵심입니다.

## 요약

> **핵심**
> - 수직 확장은 한 대의 사양을 높이는 방식으로 물리적 한계와 단일 장애점을 가지며, 수평 확장은 여러 대로 나누는 방식으로 무상태 계층에서는 선형에 가까운 효과를 냅니다.
> - <u>**서버를 늘리는 것이 항상 답이 아닌 이유는, 상태를 가진 자원(DB 등)은 단순히 서버 대수를 늘려서는 병목이 사라지지 않고 다른 곳으로 옮겨갈 뿐이기 때문**</u>입니다(암달의 법칙).
> - 로드밸런서는 여러 서버로 트래픽을 나누고 장애 서버를 자동으로 배제하며, 캐싱은 읽기 병목을, 샤딩은 쓰기·저장 병목을 각각 분산해 해결합니다.
