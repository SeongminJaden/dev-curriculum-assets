# Part 7. 백엔드 개발

## 7-2. 인증과 인가 — 세션 방식과 JWT 방식은 "상태를 어디에 두는가"로 어떻게 갈리는가

> **학습 목표**
> - 인증(Authentication)과 인가(Authorization)의 차이를 구분해 설명할 수 있다.
> - 세션 기반 인증과 토큰(JWT) 기반 인증이 "로그인 상태를 어디에 저장하는가"에 따라 어떻게 갈리는지 설명할 수 있다.
> - JWT의 구조(헤더.페이로드.서명)와 서명 검증 원리, OAuth 2.0의 위임 인증 개념을 설명할 수 있다.

> **개요** HTTP는 원래 이전 요청을 기억하지 못하는 프로토콜입니다. 로그인 이후 "이 요청을 보낸 사람이 누구인가"를 서버가 매 요청마다 무엇을 근거로 판단하는지, 그 판단 근거를 서버가 들고 있는 방식(세션)과 클라이언트가 들고 있는 방식(토큰)을 비교합니다.

---

## 1. 인증과 인가는 서로 다른 질문입니다

> **개요** "누구인가"를 확인하는 인증과 "무엇을 할 수 있는가"를 확인하는 인가를 구분합니다.

> **Q.** 로그인에 성공하면 인증과 인가가 동시에 끝난 것 아닌가요?
> **A.** 아닙니다. <u>**인증(Authentication)**</u>은 "이 사람이 실제로 본인이 맞는가"를 확인하는 절차이고, <u>**인가(Authorization)**</u>는 "본인이 맞다고 확인된 사람이 이 자원에 접근할 권한이 있는가"를 확인하는 절차입니다. 로그인은 인증에서 끝나고, 이후 "이 사용자가 관리자 페이지에 들어갈 수 있는가"를 판단하는 것은 매번 별도의 인가 절차입니다.

| 구분 | 질문 | 예시 |
|---|---|---|
| 인증 | 너는 누구인가 | 아이디·비밀번호로 로그인 |
| 인가 | 너는 이것을 할 수 있는가 | 일반 회원은 못 보는 관리자 페이지 접근 차단 |

> **참고** 인증에 성공했다고 모든 인가가 자동으로 허용되는 것은 아닙니다. 로그인한 일반 사용자도 다른 사람의 주문 내역을 조회하려 하면 인가 단계에서 차단되어야 합니다. 이 둘을 하나로 섞어 설계하면, 인증 로직 안에 권한 판단이 뒤섞여 나중에 권한 체계를 바꾸기 어려워집니다.

## 2. HTTP는 왜 로그인 상태를 "기억"하지 못할까요?

> **개요** HTTP가 상태를 유지하지 않는 프로토콜(stateless)로 설계된 이유와, 그 때문에 로그인 유지를 위한 별도 장치가 필요한 이유를 다룹니다.

HTTP는 설계 단계부터 <u>**무상태(stateless)**</u> 프로토콜로 만들어졌습니다. 서버는 요청 하나를 처리하고 나면 그 요청을 보낸 클라이언트가 누구였는지 기억하지 않습니다. 이는 결함이 아니라 의도된 설계입니다. 서버가 모든 클라이언트의 상태를 계속 들고 있지 않아도 되므로, 요청을 어느 서버가 처리하든 상관없이 확장하기 쉽습니다.

> **원칙** 문제는 "로그인 유지"처럼 여러 요청에 걸쳐 상태를 이어가야 하는 기능은 HTTP 자체가 해결해주지 않는다는 점입니다. 이 문제를 <u>**어디에 상태를 둘 것인가**</u>로 풀어낸 두 가지 대표 방식이 세션 기반 인증과 토큰 기반 인증입니다.

---

## 3. 세션 기반 인증 — 상태를 서버에 둔다

> **개요** 세션 기반 인증이 로그인 상태를 서버의 저장소에 두고, 클라이언트에는 그 상태를 가리키는 열쇠만 건네주는 원리를 다룹니다.

![세션 기반 인증 흐름 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part7_백엔드_개발/7-2_인증과_인가/assets/diagram1_session_flow.svg)

로그인에 성공하면 서버는 <u>**세션(Session)**</u>이라는 저장 공간을 만들어 그 안에 "누가 로그인했는지, 언제 로그인했는지" 같은 상태를 저장합니다. 그리고 이 세션을 구분할 수 있는 무작위 문자열인 세션 ID를 쿠키에 담아 클라이언트에 돌려줍니다. 이후 클라이언트는 요청마다 이 쿠키를 자동으로 함께 보내고, 서버는 쿠키의 세션 ID로 저장소를 조회해 "누구의 요청인가"를 판단합니다.

### 실생활 예시 — 로그인 유지가 세션 쿠키로 되는 원리

> **해석** 아래 표는 웹사이트에 로그인한 뒤, 페이지를 이동해도 로그아웃되지 않는 이유를 단계별로 보여줍니다.

| 단계 | 실제로 벌어지는 일 |
|---|---|
| 로그인 | 서버가 세션 저장소에 `{session_id: "abc123", user: "roboticsmaster"}`를 저장하고, 브라우저에 `Set-Cookie: session_id=abc123`을 응답 |
| 다른 페이지로 이동 | 브라우저가 같은 도메인으로 가는 모든 요청에 쿠키를 자동으로 첨부 |
| 서버의 확인 | 서버가 쿠키의 `session_id=abc123`으로 저장소를 조회해 "roboticsmaster"임을 확인 |
| 로그아웃 | 서버가 세션 저장소에서 해당 항목을 즉시 삭제 → 이후 같은 쿠키로 요청해도 무효 |

> **참고** 세션 기반 인증의 강점 중 하나가 바로 이 "즉시 로그아웃"입니다. 상태가 서버에 있으므로, 서버가 저장소에서 항목을 지우는 순간 그 세션은 더 이상 유효하지 않습니다.

> **팁** 반대로 세션 기반 인증의 약점은 확장성입니다. 서버를 여러 대로 늘리면, A 서버에서 로그인한 사용자가 B 서버로 요청을 보냈을 때도 같은 세션을 조회할 수 있어야 하므로, 모든 서버가 세션 저장소를 공유하도록(Redis 등 공용 저장소, 또는 sticky session) 별도로 설계해야 합니다.

## 4. 토큰 기반 인증(JWT) — 상태를 클라이언트에 둔다

> **개요** JWT가 상태를 서버 저장소가 아니라 토큰 자체에 담아, 서버가 별도 조회 없이 서명 검증만으로 사용자를 확인하는 원리를 다룹니다.

![JWT 기반 인증 구조 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part7_백엔드_개발/7-2_인증과_인가/assets/diagram2_jwt_flow.svg)

<u>**JWT(JSON Web Token)**</u>는 세션과 반대되는 접근을 취합니다. 서버는 로그인 상태를 저장소에 두지 않고, "이 사용자는 user123이고, 이 정보는 몇 시에 만료된다"라는 내용 자체를 토큰 안에 담아 클라이언트에 건네줍니다. 이후 클라이언트는 매 요청마다 이 토큰을 `Authorization: Bearer <token>` 헤더에 담아 보내고, 서버는 토큰이 위조되지 않았는지만 확인하면 됩니다. 세션 저장소를 조회하는 과정 자체가 없습니다.

### JWT의 구조 — 헤더.페이로드.서명

JWT는 점(.)으로 구분된 세 부분으로 이루어지며, 각 부분은 Base64URL로 인코딩됩니다.

- <u>**헤더(Header)**</u>: 서명에 사용할 알고리즘과 토큰 타입을 담습니다. 예: `{"alg":"HS256","typ":"JWT"}`
- <u>**페이로드(Payload)**</u>: 사용자 식별자(`sub`), 만료 시각(`exp`), 권한(`role`) 같은 claim(주장하는 정보)을 담습니다.
- <u>**서명(Signature)**</u>: 헤더와 페이로드를 이어붙인 문자열을 비밀키(또는 개인키)로 해시한 값입니다.

> **참고** 헤더와 페이로드는 암호화된 것이 아니라 Base64URL로 인코딩만 된 것입니다. 누구나 디코딩해서 내용을 읽을 수 있습니다. 따라서 <u>**JWT의 페이로드에는 비밀번호처럼 노출되면 안 되는 정보를 담아서는 안 됩니다.**</u> JWT가 지키는 것은 "내용을 숨기는 것"이 아니라 "내용이 위·변조되지 않았음을 증명하는 것"입니다.

서명 검증 원리는 다음과 같습니다. 서버는 토큰을 받으면 헤더와 페이로드를 다시 같은 알고리즘과 비밀키로 해시해 새 서명값을 계산하고, 이를 토큰에 붙어 있던 서명값과 비교합니다. 둘이 같으면 "이 토큰은 발급 이후 변조되지 않았다"고 판단합니다. 대칭키 방식(HMAC-SHA256)은 서명과 검증에 같은 비밀키를 쓰므로 발급자와 검증자가 같은 조직일 때 적합하고, 비대칭키 방식(RSA, ECDSA)은 개인키로 서명하고 공개키로 검증하므로 여러 서비스가 하나의 발급자를 신뢰해야 하는 마이크로서비스·연합 인증 환경에 적합합니다.

> **Q.** 페이로드의 사용자 정보를 공격자가 몰래 바꾸면 어떻게 될까요?
> **A.** 서명이 맞지 않게 됩니다. 서명은 헤더와 페이로드의 내용을 근거로 계산되므로, 페이로드를 한 글자라도 바꾸면 서버가 다시 계산한 서명값과 토큰에 붙어 있던 서명값이 달라집니다. 서버는 이 불일치를 근거로 요청을 거부합니다. 이것이 "암호화 없이도 위·변조를 막을 수 있는" 서명의 원리입니다.

> **팁** JWT의 대표적인 약점은 "즉시 폐기"가 어렵다는 점입니다. 세션은 서버 저장소에서 항목을 지우면 그 즉시 무효가 되지만, JWT는 서버가 들고 있는 상태가 없으므로 이미 발급된 토큰을 중간에 무효화할 방법이 마땅치 않습니다. 실무에서는 만료 시간(`exp`)을 짧게 잡고, 만료된 토큰을 재발급하는 별도의 리프레시 토큰을 함께 운영해 이 약점을 보완합니다.

### 세션 vs JWT — 상태의 위치로 갈리는 특성

| 비교 항목 | 세션 기반 | JWT 기반 |
|---|---|---|
| 상태를 두는 곳 | 서버(세션 저장소) | 클라이언트(토큰 자체) |
| 서버가 매 요청마다 하는 일 | 저장소 조회 | 서명 검증(연산만, 조회 없음) |
| 서버 수평 확장 | 세션 저장소 공유 설계 필요 | 별도 조율 없이 확장 용이 |
| 즉시 로그아웃(폐기) | 쉬움(저장소에서 삭제) | 어려움(만료 시간에 의존) |
| 전송 데이터 크기 | 작음(세션 ID만) | 상대적으로 큼(claim 전체 포함) |

> **핵심**
> - 세션 기반과 JWT 기반의 차이는 결국 <u>**로그인 상태를 서버에 둘 것인가, 클라이언트가 들고 다니는 토큰에 둘 것인가**</u>의 차이입니다.
> - 상태를 서버에 두면 즉시 폐기가 쉬운 대신 서버를 늘릴 때 상태 공유 문제가 생기고, 상태를 토큰에 두면 확장은 쉬운 대신 즉시 폐기가 어렵습니다.

## 5. OAuth 2.0 — 위임 인증의 개념

> **개요** "네이버로 로그인" 같은 제3자 로그인이 비밀번호를 넘기지 않고도 가능한 이유인 OAuth 2.0의 권한 위임 개념을 다룹니다.

<u>**OAuth 2.0**</u>은 인증 그 자체를 위한 프로토콜이 아니라, "한 서비스가 다른 서비스의 자원에 접근할 권한을 사용자 대신 위임받는" 절차를 표준화한 프로토콜입니다. 어떤 앱이 사용자의 캘린더에 접근하려 할 때, 사용자의 비밀번호를 직접 받지 않고도 "이 사용자가 캘린더 접근을 허용했다"는 사실만 증명받을 수 있게 해줍니다.

핵심 흐름은 다음과 같습니다.

1. 사용자가 앱에서 "네이버로 로그인"을 누르면, 네이버의 인증 화면으로 이동합니다.
2. 사용자가 네이버에 직접 로그인하고, 앱에 어떤 권한(예: 이메일 조회)을 허용할지 동의합니다.
3. 네이버는 앱에 "인가 코드(Authorization Code)"를 전달합니다.
4. 앱은 이 코드를 네이버 서버에 제시해 <u>**액세스 토큰(Access Token)**</u>을 발급받습니다.
5. 앱은 이 액세스 토큰으로 네이버 API를 호출해 필요한 정보만 조회합니다.

> **참고** 이 과정에서 앱은 사용자의 네이버 비밀번호를 한 번도 전달받지 않습니다. 이것이 OAuth 2.0이 풀어낸 핵심 문제, 즉 "신뢰하지 않는(또는 유출 위험이 있는) 제3자에게 비밀번호를 직접 주지 않고도 제한된 권한만 위임하는 것"입니다. 액세스 토큰이 만료 시간과 범위(scope)를 갖는 것도 이 위임 범위를 최소화하기 위한 설계입니다.

> **Q.** OAuth 2.0과 JWT는 같은 것인가요?
> **A.** 아닙니다. OAuth 2.0은 "권한을 어떻게 위임하고 발급받는가"를 정의하는 흐름(프로토콜)이고, JWT는 "토큰을 어떤 형식으로 표현하는가"를 정의하는 데이터 포맷입니다. OAuth 2.0의 액세스 토큰은 JWT 형식일 수도, 서버만 해석할 수 있는 임의의 문자열(opaque token)일 수도 있습니다. 다만 OAuth 2.0 위에 사용자 인증 절차를 표준화해 얹은 OpenID Connect(OIDC)는 ID 토큰의 형식을 JWT로 명시적으로 규정하고 있습니다.

> - HTTP는 원래 상태를 기억하지 않는(stateless) 프로토콜이므로, 로그인 상태를 유지하려면 별도의 장치가 필요합니다.
> - 세션 기반 인증은 <u>**상태를 서버에**</u> 두고 클라이언트는 세션 ID만 들고 다니며, JWT 기반 인증은 <u>**상태를 토큰 자체에**</u> 담아 서버가 서명 검증만으로 판단합니다.
> - JWT는 헤더.페이로드.서명 구조이며, 서명은 내용을 숨기는 것이 아니라 위·변조 여부를 증명하는 역할을 합니다.
> - OAuth 2.0은 비밀번호를 직접 넘기지 않고 제한된 권한만 위임하는 절차를 표준화한 프로토콜입니다.
