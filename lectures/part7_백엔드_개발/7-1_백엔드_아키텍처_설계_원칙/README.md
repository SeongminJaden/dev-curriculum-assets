# Part 7. 백엔드 개발

## 7-1. 백엔드 아키텍처 설계 원칙 — 계층을 분리하는 설계는 어떤 변경 비용을 줄이기 위함인가

> **학습 목표**
> - 계층형 아키텍처(프레젠테이션/비즈니스 로직/데이터 접근)의 각 계층이 맡는 책임을 구분해 설명할 수 있다.
> - 관심사 분리(Separation of Concerns) 원칙이 왜 변경 비용을 줄이는지, 학문적 근거와 함께 설명할 수 있다.
> - 결합도와 응집도 개념으로 잘 나뉜 계층 설계와 그렇지 않은 설계를 구분할 수 있다.

> **개요** 계층형 아키텍처가 왜 대부분의 백엔드 시스템의 기본 골격인지, 계층을 나누는 설계가 정확히 어떤 비용을 줄이기 위한 것인지를 살펴봅니다.

---

## 1. 왜 계층을 나눌까요?

> **개요** 계층을 나누지 않은 코드에서 어떤 문제가 생기는지 먼저 살펴본 뒤, 계층형 아키텍처가 이를 어떻게 해결하는지 다룹니다.

> **Q.** 코드 한 파일에 화면 출력, 할인 계산, DB 쿼리를 다 넣어도 프로그램은 똑같이 동작하는데, 왜 굳이 계층을 나눌까요?
> **A.** 동작 자체는 문제가 없습니다. 문제는 "동작 이후, 무언가 바뀔 때" 생깁니다. 화면 디자인을 바꾸는 요청인데 할인 계산 로직까지 다시 읽고 건드려야 한다면, 변경 하나의 파급 범위가 시스템 전체로 번집니다. 계층을 나누는 이유는 하나의 변경이 관련 없는 다른 부분에 영향을 주지 않도록, <u>**변경의 파급 범위를 격리**</u>하기 위함입니다.

이 아이디어는 소프트웨어공학에서 이미 오래전에 정리된 원칙입니다. 데이비드 파나스(David Parnas)는 1972년 논문 "On the Criteria to Be Used in Decomposing Systems into Modules"에서, 모듈은 기능 단위가 아니라 <u>**변경될 가능성이 있는 설계 결정(design decision) 단위**</u>로 나누어야 한다고 주장했습니다. 에츠허르 데이크스트라(Edsger Dijkstra)는 1974년 "On the Role of Scientific Thought"에서 <u>**관심사의 분리(Separation of Concerns)**</u>라는 표현을 통해, 한 번에 하나의 관심사만 고려할 수 있도록 시스템을 나누는 것이 인간의 인지 한계를 다루는 유일한 방법이라고 설명했습니다.

> **원칙** 계층을 나누는 근본 목적은 코드를 보기 좋게 정리하는 것이 아니라, <u>**서로 다른 이유로 변경되는 코드를 물리적으로 분리해, 한 이유의 변경이 다른 이유의 코드에 영향을 주지 않게 만드는 것**</u>입니다.

## 2. 계층형 아키텍처의 3계층

> **개요** 백엔드에서 가장 널리 쓰이는 3계층(프레젠테이션 / 비즈니스 로직 / 데이터 접근) 구조와 각 계층의 책임, 그리고 계층 사이의 의존 규칙을 다룹니다.

![3계층 백엔드 아키텍처 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part7_백엔드_개발/7-1_백엔드_아키텍처_설계_원칙/assets/diagram1_layers.svg)

<u>**계층형 아키텍처(Layered Architecture)**</u>는 시스템을 수평의 층으로 나누고, 각 계층이 오직 바로 아래(또는 인접) 계층에만 의존하도록 강제하는 설계 패턴입니다.

| 계층 | 책임 | 바뀌는 이유 |
|---|---|---|
| 프레젠테이션 계층 | HTTP 요청·응답 형식 변환, 입력 검증 — "무엇을 원하는가"만 해석 | API 형식, 화면 요구사항이 바뀔 때 |
| 비즈니스 로직 계층 | 할인 정책, 재고 검증, 결제 승인 조건 등 도메인 규칙 처리 | 정책·규칙이 바뀔 때 |
| 데이터 접근 계층 | SQL 쿼리 작성, ORM 매핑, 트랜잭션 관리 | 저장 방식·DB 종류가 바뀔 때 |

> **참고** 실제 현업에서는 계층을 컨트롤러·서비스·리포지토리·엔티티처럼 더 세분화하기도 합니다. 이름은 프레임워크마다 다르지만, "표현 - 로직 - 데이터"라는 3단 구조의 변형이라는 점은 동일합니다.

> **팁** 실무에서 어떤 코드를 어느 계층에 둘지 판단하는 기준은 하나입니다. "이 코드가 바뀌는 이유가 몇 가지인가?" 화면 요구사항 변경, 정책 변경, DB 스키마 변경은 서로 완전히 다른 이유이므로 다른 계층에 둡니다. 이는 SRP(Single Responsibility Principle, 단일 책임 원칙)를 계층 단위로 확장한 것이라고 볼 수 있습니다.

## 3. 계층 분리가 줄이는 변경 비용 — 결제 수단 추가 시나리오

> **개요** "간편결제 수단을 추가해 달라"는 동일한 요청이, 계층이 없는 시스템과 계층형 시스템에서 각각 어떤 파급 범위를 일으키는지 비교합니다.

![계층 유무에 따른 변경 범위 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part7_백엔드_개발/7-1_백엔드_아키텍처_설계_원칙/assets/diagram2_change_cost.svg)

> **해석** 왼쪽처럼 계층이 없으면, 결제 정책 하나를 바꾸는 일인데도 화면 출력 코드와 DB 쿼리 코드가 한 파일에 뒤섞여 있어 전체를 다시 읽고 검증해야 합니다. 오른쪽처럼 계층이 나뉘어 있으면, 같은 요청이 비즈니스 로직 계층 한 곳만 건드리고 끝납니다. 프레젠테이션 계층과 데이터 접근 계층은 인터페이스(메서드 시그니처)가 바뀌지 않는 한 재검증할 필요가 없습니다.

| 항목 | 계층 없는 시스템 | 계층형 시스템 |
|---|---|---|
| 수정 위치 | 파일 전체에 흩어진 코드 | 비즈니스 로직 계층 한 곳 |
| 회귀 테스트 범위 | 시스템 전체 | 해당 계층 + 그 계층의 인터페이스 |
| 부수 효과 위험 | 화면·DB 코드까지 실수로 건드릴 위험 | 다른 계층은 영향받지 않음 |

> **Q.** 계층을 나누면 코드량이 늘고 파일도 더 많아지는데, 그래도 이득일까요?
> **A.** 네. 결합도(coupling)를 낮추는 대신 치르는 비용은 파일 개수 증가와, 계층 간 인터페이스를 미리 설계해야 하는 수고입니다. 반면 절약하는 비용은 변경이 생길 때마다 관련 없는 코드까지 다시 이해해야 하는 시간입니다. 시스템이 커지고 오래 유지보수될수록 후자의 절약분이 압도적으로 커지기 때문에, 대부분의 실무 백엔드는 이 트레이드오프를 계층 분리 쪽으로 선택합니다.

## 4. 결합도와 응집도로 설계 품질 판단하기

> **개요** 계층을 "잘" 나눴는지 판단하는 정량적 기준인 결합도(coupling)와 응집도(cohesion) 개념을 다룹니다.

- <u>**응집도(Cohesion)**</u>: 한 모듈 내부의 요소들이 얼마나 하나의 목적에 집중되어 있는가를 나타냅니다. 높을수록 좋습니다.
- <u>**결합도(Coupling)**</u>: 모듈들이 서로 얼마나 의존하는가를 나타냅니다. 낮을수록 좋습니다.

좋은 설계의 목표는 항상 같습니다. <u>**높은 응집도와 낮은 결합도**</u>를 동시에 만족시키는 것입니다. 계층형 아키텍처는 이 목표를 달성하는 구체적인 실행 방법 중 하나입니다. 계층 내부는 같은 종류의 책임(예: 도메인 규칙)만 모아 응집도를 높이고, 계층 사이는 정해진 인터페이스로만 통신하게 해 결합도를 낮춥니다.

> **원칙** 좋은 계층 설계는 "계층 내부는 응집도를 높이고, 계층 사이는 결합도를 낮추는" 방향으로 이루어집니다. 계층 사이의 결합도를 낮추는 대표적 방법이 구체적인 구현이 아니라 추상화된 인터페이스에 의존하는 것이며, 이는 이후 심화 자료에서 다루는 의존성 역전 원칙(Dependency Inversion Principle)과 직결됩니다.

## 요약

> **핵심**
> - 계층형 아키텍처는 프레젠테이션·비즈니스 로직·데이터 접근 3계층으로 나뉘며, 각 계층은 오직 바로 아래 계층에만 의존합니다(단방향 의존성).
> - 계층을 나누는 근본 목적은 <u>**서로 다른 이유로 변경되는 코드를 물리적으로 분리해, 한 계층의 변경 비용이 다른 계층으로 파급되지 않게 하는 것**</u>입니다(Parnas 1972, Dijkstra 1974).
> - 좋은 계층 설계는 계층 내부의 응집도를 높이고, 계층 사이의 결합도를 낮추는 방향으로 판단합니다.
