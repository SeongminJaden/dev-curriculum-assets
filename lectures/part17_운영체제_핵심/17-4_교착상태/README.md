# Part 17. 운영체제 핵심

## 17-4. 교착상태 — 아무도 잘못하지 않았는데 모두가 멈추는 이유

> **학습 목표**
> - 교착상태의 4대 필요조건을 열거하고, 각 조건이 왜 필요조건인지 설명할 수 있다.
> - 예방·회피·탐지·회복이라는 네 가지 대응 전략을 구분하고, 은행원 알고리즘이 안전 상태를 판정하는 절차를 설명할 수 있다.
> - 자원 할당 그래프에서 순환을 찾아 교착 여부를 판정하고, 락 순서 규약으로 실무 코드의 교착을 예방할 수 있다.

> **개요** 두 스레드 모두 올바른 코드를 실행하고 있고, 어느 쪽도 무한 루프에 빠지지 않았습니다. 그런데 프로그램이 영원히 멈춰 있습니다. 이 강의는 개별적으로는 모두 정상인 동작들이 어떻게 결합해 전체를 마비시키는지, 그리고 운영체제와 데이터베이스가 이 상황에 어떤 네 가지 태도로 대응하는지를 다룹니다.

## 1. 교착상태란 무엇인가요?

> **Q.** 스레드가 멈춰 있으면 그냥 기다리다 풀리는 것 아닌가요?
> **A.** 풀릴 수 있는 대기와 절대 풀리지 않는 대기가 다릅니다. 어떤 스레드가 락을 기다리는데, 그 락을 쥔 스레드가 **이 스레드가 쥔 락**을 기다리고 있다면, 두 스레드는 서로가 먼저 놓기를 기다리며 영원히 멈춥니다. 외부에서 개입하지 않는 한 시간이 아무리 흘러도 상태가 변하지 않습니다.

<u>**교착상태(Deadlock)**</u>는 둘 이상의 프로세스나 스레드가 서로가 점유한 자원을 기다리며 어느 쪽도 진행하지 못하는 상태입니다. 흔히 <u>**기아(Starvation)**</u>와 혼동되는데, 기아는 "가능성은 있지만 계속 밀리는" 상태라 운이 좋으면 풀리는 반면, 교착상태는 **원리적으로 절대 풀리지 않습니다.**

```python
# 전형적인 교착상태 — 두 스레드가 락을 반대 순서로 잡습니다
def thread_1():
    with lock_A:          # A 획득
        with lock_B:      # B 대기 → thread_2가 쥐고 있음
            transfer()

def thread_2():
    with lock_B:          # B 획득
        with lock_A:      # A 대기 → thread_1이 쥐고 있음
            transfer()
```

## 2. 교착상태의 4대 필요조건

![교착상태의 4대 필요조건과 자원 할당 그래프](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part17_운영체제_핵심/17-4_교착상태/assets/diagram1_deadlock_conditions.svg)

> **해석** 1971년 에드워드 코프만(Edward G. Coffman Jr.) 연구진이 정리한 네 조건은 교착상태의 **필요조건**입니다. 즉 교착이 발생했다면 네 조건이 모두 성립하고 있으며, 뒤집어 말하면 **네 조건 중 단 하나만 깨뜨려도 교착은 발생할 수 없습니다.** 이것이 이후 모든 대응 전략의 논리적 근거입니다.

| 조건 | 의미 | 이것이 없으면 |
|---|---|---|
| <u>**상호 배제(Mutual Exclusion)**</u> | 자원을 한 번에 하나의 프로세스만 쓸 수 있다 | 기다릴 이유 자체가 없음 |
| <u>**점유와 대기(Hold and Wait)**</u> | 자원을 쥔 채로 다른 자원을 기다린다 | 대기 중인 프로세스가 아무것도 붙잡고 있지 않음 |
| <u>**비선점(No Preemption)**</u> | 쥔 자원을 강제로 빼앗을 수 없다 | 필요하면 뺏어서 진행시킬 수 있음 |
| <u>**순환 대기(Circular Wait)**</u> | 대기 관계가 원을 이룬다 | 대기 사슬의 끝에 있는 누군가는 반드시 진행 가능 |

> **해석** 그림 오른쪽의 <u>**자원 할당 그래프(Resource Allocation Graph)**</u>는 프로세스와 자원을 노드로, "할당됨"과 "요청함"을 방향 있는 화살표로 그린 그래프입니다. 자원마다 인스턴스가 하나뿐이라면 **그래프에 순환이 존재하는 것과 교착상태인 것이 정확히 동치**입니다. 인스턴스가 여럿인 자원이 섞여 있으면 순환은 교착의 필요조건이지만 충분조건은 아니어서, 추가 판정이 필요합니다.

## 3. 네 가지 대응 전략

![교착상태 대응 전략 네 가지](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part17_운영체제_핵심/17-4_교착상태/assets/diagram2_deadlock_strategies.svg)

> **해석** 네 전략은 "언제 개입할 것인가"라는 시간 축 위에 놓입니다. **예방**은 시스템 설계 시점에 구조적으로 불가능하게 만들고, **회피**는 자원을 할당하는 매 순간 안전한지 검사하며, **탐지와 회복**은 일단 일어나게 두고 사후에 처리하고, **무시**는 그냥 놔둡니다. 앞으로 갈수록 안전하지만 비용과 제약이 크고, 뒤로 갈수록 값싸지만 위험합니다.

### 3-1. 예방(Prevention) — 네 조건 중 하나를 원천 봉쇄

| 깨뜨릴 조건 | 방법 | 현실적 한계 |
|---|---|---|
| 상호 배제 | 자원을 공유 가능하게 설계(읽기 전용 데이터, 불변 객체) | 프린터·쓰기 락처럼 본질적으로 배타적인 자원에는 불가 |
| 점유와 대기 | 필요한 자원을 처음에 **전부** 한꺼번에 요청 | 자원 이용률이 크게 떨어지고, 미리 다 알기 어려움 |
| 비선점 | 요청이 거절되면 이미 쥔 자원을 모두 반납하고 재시도 | CPU·메모리와 달리 프린터·파일 락은 중간 상태 복구가 어려움 |
| 순환 대기 | 모든 자원에 전역 번호를 매기고 **오름차순으로만** 획득 | 실무에서 가장 널리 쓰이는 유일하게 실용적인 방법 |

> **팁** 앞의 교착 예제는 순환 대기를 깨는 것만으로 완전히 해결됩니다. 모든 스레드가 `lock_A` → `lock_B` 순서로만 잡도록 규약을 정하면, `lock_B`를 쥔 스레드는 `lock_A`를 절대 기다리지 않으므로 순환이 만들어질 수 없습니다. 계좌 이체 코드처럼 두 객체의 락을 잡아야 할 때는 **계좌 ID가 작은 쪽을 먼저 잠그는** 식으로 전역 순서를 정하는 것이 표준 관행입니다.

```python
# 락 순서 규약 — 항상 ID가 작은 계좌부터 잠급니다
def transfer(src, dst, amount):
    first, second = (src, dst) if src.id < dst.id else (dst, src)
    with first.lock:
        with second.lock:
            src.balance -= amount
            dst.balance += amount
```

### 3-2. 회피(Avoidance) — 은행원 알고리즘

<u>**회피(Avoidance)**</u>는 자원 요청이 들어올 때마다 "이 요청을 들어주면 시스템이 안전 상태에 머무는가"를 검사해, 위험하면 요청을 미루는 방식입니다. 다익스트라가 1965년 제안한 <u>**은행원 알고리즘(Banker's Algorithm)**</u>이 대표적입니다. 은행이 모든 고객의 신용 한도를 알고 있을 때, 어떤 순서로든 모든 고객의 대출을 갚게 할 수 있는 상태만 유지한다는 발상에서 이름이 왔습니다.

각 프로세스가 앞으로 최대 얼마나 요청할지(`Max`)를 미리 선언한다는 전제 아래, 알고리즘은 다음을 확인합니다.

$$\text{Need}_i = \text{Max}_i - \text{Allocation}_i$$

> **해석** <u>**안전 상태(Safe State)**</u>란 "모든 프로세스를 어떤 순서로든 끝까지 완료시킬 수 있는 실행 순서가 최소 하나 존재하는 상태"입니다. 판정 절차는 이렇습니다. 현재 남은 자원(`Available`)으로 `Need`를 충족시킬 수 있는 프로세스를 하나 찾습니다. 찾았다면 그 프로세스가 끝나서 자원을 모두 반납한다고 가정하고 `Available`에 더합니다. 이 과정을 반복해 모든 프로세스를 소진하면 안전 상태이고, 도중에 어느 프로세스의 `Need`도 충족시킬 수 없게 되면 불안전 상태입니다. 여기서 주의할 점은 **불안전 상태가 곧 교착은 아니라는 것**입니다. 불안전 상태는 "교착이 될 가능성이 있는 상태"이며, 회피 전략은 그 가능성조차 허용하지 않는 보수적인 태도를 취합니다.

> **참고** 은행원 알고리즘은 개념적으로 아름답지만 범용 운영체제에서는 거의 쓰이지 않습니다. 프로세스가 앞으로 필요한 최대 자원량을 미리 선언해야 하는데 현실에서는 알 수 없고, 요청마다 $O(n^2 m)$ 규모의 검사를 수행하는 비용이 크며, 프로세스 수가 실행 중 계속 변하기 때문입니다.

### 3-3. 탐지와 회복(Detection & Recovery)

주기적으로 자원 할당 그래프를 만들어 순환을 찾고, 교착이 발견되면 회복 조치를 취합니다. 회복 방법은 두 가지입니다. 하나는 교착에 얽힌 프로세스를 종료시키는 것(전부 종료하거나, 하나씩 종료하며 순환이 풀리는지 확인)이고, 다른 하나는 <u>**롤백(Rollback)**</u>으로 특정 프로세스를 이전 체크포인트 상태로 되돌려 자원을 빼앗는 것입니다.

> **참고** 이 전략을 가장 잘 보여주는 실무 사례가 관계형 데이터베이스입니다. PostgreSQL은 `deadlock_timeout`(기본 1초) 동안 락을 얻지 못하면 대기 그래프를 검사해 순환을 찾고, 교착이 확인되면 트랜잭션 하나를 골라 강제 종료하며 `40P01 deadlock detected` 오류를 반환합니다. MySQL InnoDB도 유사하게 동작하며, 되돌리는 비용이 가장 적은(변경한 행 수가 적은) 트랜잭션을 희생자로 고릅니다. 애플리케이션은 이 오류를 받으면 트랜잭션을 재시도하도록 작성하는 것이 표준 패턴입니다.

### 3-4. 무시 — 타조 알고리즘

리눅스와 윈도우를 포함한 대부분의 범용 운영체제는 커널 차원의 일반적 교착 처리를 하지 않습니다. 교착이 드물게 발생하고, 발생하더라도 사용자가 프로세스를 강제 종료하거나 재부팅하면 되며, 예방·회피의 상시 비용이 그 손실보다 크다는 판단 때문입니다. 이를 위험을 못 본 척한다는 뜻으로 <u>**타조 알고리즘(Ostrich Algorithm)**</u>이라 부릅니다.

> **해석** 이것은 게으름이 아니라 명시적인 공학적 트레이드오프입니다. 교착 방지의 상시 비용이 교착의 기대 손실보다 크다면 무시가 합리적 선택입니다. 다만 이 계산은 도메인에 따라 완전히 달라집니다. 항공 제어나 원자로 제어처럼 한 번의 정지가 치명적인 시스템에서는 예방 전략에 큰 비용을 들이는 것이 옳습니다.

## 4. 실무에서 교착을 피하는 습관

| 습관 | 근거 조건 | 설명 |
|---|---|---|
| 전역 락 순서 규약 | 순환 대기 차단 | 잡아야 할 락이 여럿이면 항상 정해진 순서로 |
| 락 획득에 타임아웃 부여 | 비선점 완화 | `tryLock(500ms)` 실패 시 전부 놓고 재시도 |
| 락 보유 중 외부 호출 금지 | 점유와 대기 축소 | 락 안에서 네트워크·다른 모듈 콜백 호출 금지 |
| 락 개수 자체를 줄이기 | 조건 성립 기회 축소 | 불변 객체·메시지 큐로 공유 상태를 없앰 |
| 데이터베이스 접근 순서 통일 | 순환 대기 차단 | 여러 행을 갱신할 때 항상 기본키 오름차순으로 |

> **팁** 자바에서는 `jstack`이, 파이썬에서는 `faulthandler.dump_traceback()`이 멈춘 프로세스의 모든 스레드 스택을 덤프해줍니다. 자바 스레드 덤프는 순환을 감지하면 `Found one Java-level deadlock:` 이라는 문구와 함께 어떤 스레드가 어떤 락을 기다리는지 직접 알려주므로, 응답이 멈춘 서버를 만나면 가장 먼저 확인할 도구입니다.

## 요약

> **핵심**
> - 교착상태는 **상호 배제·점유와 대기·비선점·순환 대기** 네 조건이 모두 성립할 때만 발생합니다(1971년 코프만 조건). 따라서 **하나만 깨뜨려도 교착은 불가능**합니다.
> - 대응 전략은 개입 시점에 따라 **예방·회피·탐지와 회복·무시**로 나뉩니다. 회피의 대표인 **은행원 알고리즘**은 안전 상태를 유지하지만 최대 요구량을 미리 알아야 해 범용 OS에서는 쓰이지 않고, 리눅스·윈도우는 비용 판단에 따라 **타조 알고리즘**(무시)을 택합니다.
> - 실무에서 유일하게 널리 쓰이는 예방책은 **순환 대기를 깨는 전역 락 순서 규약**이며, 데이터베이스는 **탐지 후 희생자 트랜잭션 롤백**(PostgreSQL의 `deadlock_timeout`)으로 대응하므로 애플리케이션은 재시도 로직을 갖춰야 합니다.
