# 17-4. 교착상태 — 실습

> **참고** 이 실습은 제출물이 아니라, 교착상태를 의도적으로 만들어 관찰하고 락 순서 규약으로 직접 고쳐보는 과정입니다.

## 실습 1 — 교착상태 재현하기

- [ ] 락 두 개(`lock_A`, `lock_B`)를 만들고, 스레드 1은 A→B 순서로, 스레드 2는 B→A 순서로 잡는 프로그램을 작성하세요.
- [ ] 각 락 획득 사이에 `sleep(0.1)`을 넣어 교착이 거의 매번 재현되게 만들고, 프로그램이 실제로 멈추는지 확인하세요.
- [ ] 멈춘 상태에서 스레드 덤프를 뜨고(`jstack`, `py-spy dump`, 또는 `faulthandler`), 어떤 스레드가 어떤 락을 기다리는지 출력에서 찾아 적어보세요.

## 실습 2 — 자원 할당 그래프 그려보기

- [ ] 실습 1의 상황을 프로세스 노드 2개와 자원 노드 2개로 그리고, 할당 화살표와 요청 화살표를 방향까지 정확히 표시하세요.
- [ ] 그래프에서 순환을 찾아 색으로 표시하고, 그 순환이 4대 필요조건 중 어느 것에 해당하는지 적으세요.
- [ ] 스레드 2의 락 획득 순서만 A→B로 바꾼 그래프를 다시 그리고, 순환이 사라지는지 확인하세요.

## 실습 3 — 락 순서 규약으로 고치기

- [ ] 계좌 객체 100개와 각 계좌의 락을 만들고, 무작위 두 계좌 사이의 이체를 수행하는 스레드 8개를 실행해 교착이 발생하는지 확인하세요.
- [ ] 이체 함수를 "계좌 ID가 작은 쪽을 먼저 잠근다"는 규약으로 수정하고, 같은 부하로 60초간 실행해 한 번도 멈추지 않는지 확인하세요.
- [ ] 수정 전후의 초당 이체 처리 건수를 측정해 비교하고, 규약 도입이 성능에 비용을 유발했는지 판단해 적으세요.

## 실습 4 — 데이터베이스 교착 탐지 확인하기

- [ ] PostgreSQL 또는 MySQL에서 터미널 두 개를 열고 각각 트랜잭션을 시작한 뒤, 같은 두 행을 서로 반대 순서로 `UPDATE`하세요.
- [ ] 몇 초 뒤 한쪽 세션이 받는 오류 메시지 전문을 기록하고, 오류 코드(PostgreSQL의 `40P01` 등)를 확인하세요.
- [ ] 살아남은 트랜잭션과 강제 종료된 트랜잭션 중 어느 쪽이 변경한 행이 더 많았는지 확인하고, DB가 희생자를 어떤 기준으로 골랐을지 추론해 적어보세요.

> **팁** 실습 3에서 규약 적용 후에도 멈춘다면 락을 잡는 코드 경로가 하나 더 있는지 확인하세요. 락 순서 규약은 프로그램 안의 **모든** 경로가 지켜야만 효과가 있으며, 단 하나의 예외가 순환을 되살립니다. 그래서 규약은 코드 리뷰 항목으로 명문화해두는 것이 실무 관행입니다.
