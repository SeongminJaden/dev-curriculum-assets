# 17-1. 프로세스와 스레드 — 실습

> **참고** 이 실습은 제출물이 아니라, 커널이 유지하는 프로세스 정보를 직접 들여다보고 스레드 공유 자원을 손으로 확인해보는 과정입니다.

## 실습 1 — 내 컴퓨터의 프로세스 관찰하기

- [ ] 터미널에서 `ps aux` (윈도우는 `Get-Process`)를 실행하고, 출력된 프로세스 개수를 세어 CPU 코어 수와 비교해 적어보세요.
- [ ] `ps -eo pid,ppid,stat,comm`으로 PID·PPID·상태를 출력하고, STAT 열이 `S`인 프로세스와 `R`인 프로세스의 개수를 각각 세어보세요.
- [ ] 리눅스라면 `cat /proc/self/status`를 실행해 State, Threads, VmSize 값을 기록하고, 각 값이 PCB의 어느 항목에 대응하는지 옆에 적어보세요.

## 실습 2 — 주소 공간의 네 영역을 주소로 확인하기

- [ ] C 또는 원하는 언어로 전역 변수 1개, `malloc`(또는 동적 할당)으로 얻은 포인터 1개, 지역 변수 1개, 함수 포인터 1개를 만들고 각각의 주소를 16진수로 출력하세요.
- [ ] 출력된 네 주소를 크기순으로 정렬하고, 강의의 주소 공간 그림(코드 < 데이터 < 힙 < 스택)과 순서가 일치하는지 확인하세요.
- [ ] 동적 할당을 두 번 더 하고 주소가 증가하는지, 함수를 한 단계 더 깊게 호출한 뒤 지역 변수 주소가 감소하는지 각각 확인해 힙과 스택의 성장 방향을 기록하세요.

## 실습 3 — 스레드가 정말 메모리를 공유하는지 검증하기

- [ ] 전역 변수 `counter = 0`을 두고, 스레드 2개가 각각 `counter`를 100,000번 증가시키는 프로그램을 작성해 실행하세요.
- [ ] 최종 `counter` 값을 10회 반복 실행해 기록하고, 200,000이 나오지 않은 횟수를 세어보세요.
- [ ] 같은 로직을 스레드 대신 프로세스 2개(`fork` 또는 `multiprocessing`)로 바꿔 실행하고, 각 프로세스가 출력한 `counter` 값이 100,000인지 확인해 메모리가 격리되었음을 증명하세요.

## 실습 4 — 생성 비용 측정하기

- [ ] 스레드 1,000개를 만들고 즉시 종료(join)하는 데 걸린 총 시간을 측정하세요.
- [ ] 프로세스 1,000개에 대해 같은 측정을 하고, 두 시간의 배율을 계산해 적어보세요.
- [ ] 측정 중 `ps` 또는 작업 관리자로 메모리 사용량을 함께 관찰하고, 프로세스 방식이 왜 더 많은 메모리를 쓰는지 강의의 주소 공간 개념으로 설명해보세요.

> **팁** 실습 3에서 200,000이 아닌 값이 나왔다면 그것이 바로 경쟁 상태(race condition)입니다. 실행할 때마다 결과가 달라진다는 점, 그리고 스레드 수를 늘리면 오차가 커진다는 점을 꼭 확인해두세요. 이 현상을 어떤 도구로 막는지는 17-3에서 다룹니다.
