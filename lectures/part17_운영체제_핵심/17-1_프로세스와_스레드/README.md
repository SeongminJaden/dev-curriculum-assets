# Part 17. 운영체제 핵심

## 17-1. 프로세스와 스레드 — 실행 중인 프로그램은 커널에게 무엇으로 보이는가

> **학습 목표**
> - 프로그램과 프로세스의 차이를 설명하고, 커널이 프로세스를 관리하기 위해 유지하는 PCB의 구성 요소를 열거할 수 있다.
> - 프로세스 상태 전이도(생성·준비·실행·대기·종료)를 그리고, 각 전이가 어떤 사건에 의해 일어나는지 설명할 수 있다.
> - 프로세스 주소 공간의 네 영역과 스레드가 공유하는 자원/독립적으로 갖는 자원을 구분하고, 멀티프로세스와 멀티스레드의 트레이드오프를 근거를 들어 비교할 수 있다.

> **개요** 디스크에 있는 실행 파일을 더블클릭하면 무슨 일이 벌어질까요? 커널은 그 순간 파일을 메모리에 올리고, 그 실행 흐름을 추적하기 위한 자료구조를 하나 만듭니다. 이 강의는 그 자료구조가 무엇인지, 왜 하나의 프로세스 안에 여러 실행 흐름(스레드)을 두는 방식이 등장했는지, 그리고 그 선택이 어떤 대가를 요구하는지를 다룹니다.

---

## 1. 프로그램과 프로세스는 무엇이 다른가요?

> **Q.** `app.exe` 파일 하나를 두 번 실행하면 프로세스도 두 개가 됩니다. 같은 파일인데 왜 별개로 취급될까요?
> **A.** 프로그램은 디스크에 저장된 **명령어의 집합**, 즉 정적인 데이터일 뿐이기 때문입니다. 반면 프로세스는 그 명령어가 메모리에 적재되어 **실제로 실행되고 있는 상태 전체**를 가리킵니다. 같은 악보를 두 연주자가 동시에 연주하면 두 개의 연주가 되는 것과 같습니다. 두 프로세스는 같은 코드를 실행하지만 각자의 변수 값, 각자의 프로그램 카운터, 각자의 스택을 갖습니다. 커널 입장에서 이 둘은 완전히 별개의 관리 대상이며, 서로의 메모리를 들여다볼 수도 없습니다.

<u>**프로세스(Process)**</u>는 실행 중인 프로그램의 인스턴스이며, 커널이 자원을 할당하는 기본 단위입니다. 커널은 각 프로세스에 고유한 <u>**PID(Process Identifier)**</u>를 부여하고, 그 프로세스가 사용하는 메모리·열린 파일·권한을 모두 PID에 묶어 추적합니다. 리눅스에서 `ps` 명령이 보여주는 목록이 바로 커널이 유지하는 이 프로세스 테이블의 일부입니다.

## 2. PCB — 커널이 프로세스에 대해 기억하는 모든 것

> **개요** 커널은 프로세스마다 하나씩 제어 블록을 만들어 그 프로세스의 상태를 기록합니다. 이 자료구조가 프로세스라는 추상의 실체입니다.

<u>**PCB(Process Control Block)**</u>는 커널이 프로세스 하나를 관리하기 위해 유지하는 자료구조입니다. 리눅스 커널에서는 `task_struct`라는 이름의 C 구조체로 구현되어 있습니다.

| 항목 | 내용 | 왜 필요한가 |
|---|---|---|
| PID / PPID | 프로세스 식별자와 부모 식별자 | 프로세스를 지목하고 계층을 추적하기 위해 |
| 프로세스 상태 | 준비·실행·대기 등 | 스케줄러가 다음에 실행할 후보를 고르기 위해 |
| 프로그램 카운터 | 다음에 실행할 명령어 주소 | 중단된 지점부터 정확히 재개하기 위해 |
| CPU 레지스터 값 | 범용 레지스터·스택 포인터 등 | 컨텍스트 스위칭 시 복원하기 위해 |
| 메모리 관리 정보 | 페이지 테이블 베이스 주소 등 | 가상 주소를 물리 주소로 변환하기 위해 |
| 열린 파일 테이블 | 파일 디스크립터 목록 | 프로세스가 접근 중인 파일을 추적하기 위해 |
| 스케줄링 정보 | 우선순위, 누적 실행 시간 | 어느 프로세스에 CPU를 줄지 결정하기 위해 |

> **해석** PCB의 항목들은 두 부류로 나뉩니다. 프로그램 카운터와 레지스터 값은 **"중단된 실행을 되살리기 위한"** 정보이고, 메모리 정보와 파일 테이블은 **"이 프로세스가 무엇을 소유하고 있는지"**에 관한 정보입니다. 앞의 것을 통틀어 <u>**컨텍스트(Context)**</u>라 부르며, 이것을 저장하고 복원하는 작업이 다음 강의에서 다룰 컨텍스트 스위칭입니다.

## 3. 프로세스 상태 전이 — 실행 중이 아닌 프로세스는 무엇을 하고 있는가

> **개요** 실행 중인 프로세스는 CPU 코어 수만큼만 존재할 수 있습니다. 나머지 수백 개의 프로세스는 어떤 상태로 대기하고 있는지 살펴봅니다.

![프로세스 상태 전이도](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part17_운영체제_핵심/17-1_프로세스와_스레드/assets/diagram1_process_states.svg)

> **해석** 여기서 반드시 구분해야 할 두 상태가 <u>**준비(Ready)**</u>와 <u>**대기(Waiting/Blocked)**</u>입니다. 준비 상태는 "CPU만 주면 지금 당장 실행할 수 있는" 상태이고, 대기 상태는 "CPU를 줘도 할 일이 없는" 상태입니다. 예를 들어 디스크에서 파일을 읽는 중인 프로세스는 데이터가 도착하기 전까지 아무것도 계산할 수 없으므로 대기 상태로 내려갑니다. 그래서 대기 상태의 프로세스는 스케줄러의 후보 목록에서 아예 제외되고, 입출력이 끝났다는 인터럽트가 도착해야 비로소 준비 상태로 되돌아옵니다. 실행에서 준비로 내려가는 화살표는 시간 할당량 소진(선점)에 의한 것이고, 실행에서 대기로 내려가는 화살표는 프로세스 스스로의 입출력 요청에 의한 것이라는 점이 결정적인 차이입니다.

> **참고** 리눅스에서 `ps aux`의 STAT 열에 보이는 `R`은 실행 또는 준비, `S`는 인터럽트 가능한 대기, `D`는 인터럽트 불가능한 대기(주로 디스크 입출력), `Z`는 종료했으나 부모가 아직 종료 코드를 회수하지 않은 좀비 상태입니다. `D` 상태 프로세스가 쌓여 있다면 디스크나 네트워크 스토리지에 병목이 있다는 강한 신호입니다.

## 4. 프로세스 주소 공간 — 코드·데이터·힙·스택

> **개요** 커널은 각 프로세스에 자기만의 가상 주소 공간을 통째로 부여합니다. 그 안이 어떻게 나뉘어 있는지 살펴봅니다.

![프로세스 주소 공간과 스레드 공유 범위](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part17_운영체제_핵심/17-1_프로세스와_스레드/assets/diagram2_address_space.svg)

> **해석** 주소 공간은 네 영역으로 나뉩니다. <u>**코드 영역(Text Segment)**</u>은 기계어 명령어가 담기며 실행 중 변하지 않으므로 읽기 전용으로 보호됩니다. <u>**데이터 영역(Data Segment)**</u>에는 전역 변수와 정적 변수가 놓입니다. <u>**힙(Heap)**</u>은 실행 중 동적으로 할당되는 영역으로 낮은 주소에서 높은 주소 방향으로 자랍니다. <u>**스택(Stack)**</u>은 함수 호출 시 지역 변수와 복귀 주소가 쌓이는 영역으로, 힙과 반대 방향인 높은 주소에서 낮은 주소로 자랍니다. 두 영역을 서로 마주 보게 배치한 이유는 어느 쪽이 많이 쓰일지 미리 알 수 없기 때문에, 가운데 빈 공간을 양쪽이 필요에 따라 나눠 쓰게 하려는 설계입니다. 재귀 호출이 끝없이 깊어져 스택이 힙 영역까지 침범하면 발생하는 오류가 바로 스택 오버플로입니다.

> **참고** 각 프로세스가 보는 주소는 실제 물리 메모리 주소가 아니라 커널이 만들어준 <u>**가상 주소(Virtual Address)**</u>입니다. 그래서 두 프로세스가 똑같이 `0x400000` 번지를 읽어도 서로 다른 물리 메모리를 보게 되며, 이것이 프로세스 간 메모리 격리의 근거입니다. 가상 주소를 물리 주소로 바꾸는 페이징 메커니즘과 캐시 계층에 관해서는 `2-3 메모리 계층 구조`를 참고하세요.

## 5. 스레드 — 하나의 주소 공간을 나눠 쓰는 실행 흐름

> **Q.** 프로세스가 이미 있는데 왜 스레드라는 개념을 또 만들었을까요?
> **A.** 프로세스는 격리가 강력한 대신 만들고 전환하는 비용이 비싸고, 무엇보다 서로 데이터를 주고받기가 어렵기 때문입니다. 웹 서버가 요청 1만 개를 동시에 처리하려고 프로세스 1만 개를 만들면, 각 프로세스마다 주소 공간을 통째로 복사해야 하고 공유 캐시 하나 읽으려 해도 프로세스 간 통신을 거쳐야 합니다. <u>**스레드(Thread)**</u>는 이 문제를 "주소 공간은 하나로 두되 실행 흐름만 여럿 두자"는 방식으로 해결합니다. 그래서 스레드는 CPU 할당의 기본 단위이고, 프로세스는 자원 할당의 기본 단위라고 구분해 표현합니다.

| 자원 | 스레드 간 공유 여부 | 이유 |
|---|---|---|
| 코드 영역 | 공유 | 같은 프로그램의 명령어를 함께 실행 |
| 데이터 영역(전역·정적 변수) | 공유 | 프로세스에 하나만 존재 |
| 힙 | 공유 | 동적 할당 영역은 프로세스 단위로 관리 |
| 열린 파일·소켓 | 공유 | 파일 디스크립터 테이블은 프로세스 소유 |
| 스택 | **독립** | 각 스레드가 독자적 함수 호출 이력을 가짐 |
| 레지스터·프로그램 카운터 | **독립** | 각 스레드가 서로 다른 명령어를 실행 중 |

> **해석** 이 표의 위쪽 네 줄이 멀티스레드의 **장점**이자 동시에 **위험**입니다. 전역 변수와 힙을 공유하기 때문에 데이터를 넘기는 비용이 사실상 0이지만, 바로 그 이유로 두 스레드가 같은 변수를 동시에 수정하는 경쟁 상태가 발생할 수 있습니다. 이 문제를 커널이 어떤 도구로 막는지가 `17-3 동기화` 강의의 주제입니다. 반대로 아래 두 줄이 독립이기에 각 스레드가 서로 다른 함수를 서로 다른 깊이로 호출하며 동시에 진행할 수 있습니다.

## 6. 멀티프로세스 vs 멀티스레드 — 무엇을 대가로 무엇을 얻는가

| 기준 | 멀티프로세스 | 멀티스레드 |
|---|---|---|
| 메모리 격리 | 완전 격리 — 한쪽이 죽어도 다른 쪽 무사 | 공유 — 한 스레드의 잘못된 메모리 접근이 프로세스 전체를 종료 |
| 생성 비용 | 높음(주소 공간·페이지 테이블 준비) | 낮음(스택과 레지스터 세트만 준비) |
| 컨텍스트 스위칭 비용 | 높음(페이지 테이블 교체 → TLB 무효화) | 낮음(주소 공간 유지, TLB 보존) |
| 데이터 공유 | IPC 필요(파이프·공유 메모리·소켓) | 전역·힙 직접 접근 |
| 동기화 부담 | 낮음 | 높음(락·원자 연산 필수) |
| 디버깅 난이도 | 상대적으로 쉬움 | 재현이 어려운 경쟁 상태 발생 |

> **해석** 이 표는 하나의 축을 두고 저울질한 결과입니다. 그 축은 **격리 대 공유**입니다. 격리를 택하면 안정성과 디버깅 편의를 얻고 성능과 통신 편의를 잃습니다. 공유를 택하면 그 반대입니다. 따라서 "멀티스레드가 더 좋다"는 명제는 성립하지 않으며, 장애 격리가 더 중요한지 처리량이 더 중요한지에 따라 답이 갈립니다.

> **참고** 실제 시스템은 이 선택을 명시적으로 드러냅니다. 크롬 브라우저는 2008년 출시 때부터 탭마다 별도 프로세스를 두는 <u>**사이트 격리(Site Isolation)**</u> 구조를 채택했는데, 탭 하나가 크래시해도 브라우저 전체가 죽지 않게 하고 악성 사이트가 다른 탭의 메모리를 읽지 못하게 하려는 의도적인 성능 희생입니다. 반면 Nginx는 코어 수만큼의 워커 프로세스를 띄우고 각 워커 안에서는 이벤트 기반으로 수만 개의 연결을 처리하여, 두 모델을 계층적으로 조합합니다.

> **팁** 파이썬에서 CPU 연산이 많은 작업을 `threading`으로 병렬화해도 속도가 오르지 않는 경우가 흔한데, CPython의 <u>**GIL(Global Interpreter Lock)**</u> 때문에 한 시점에 하나의 스레드만 바이트코드를 실행할 수 있어서입니다. 이런 작업은 `multiprocessing`으로 프로세스를 분리해야 실제로 여러 코어를 씁니다. 반대로 네트워크 대기가 많은 작업은 GIL이 입출력 대기 중 해제되므로 스레드로도 이득을 봅니다.

## 요약

> **핵심**
> - 프로그램은 디스크의 정적 명령어 집합이고, 프로세스는 그것이 메모리에 적재되어 실행 중인 인스턴스입니다. 커널은 프로세스마다 **PCB**(리눅스의 `task_struct`)를 만들어 상태·레지스터·메모리 정보·파일 테이블을 추적합니다.
> - 프로세스는 생성·준비·실행·대기·종료 상태를 오가며, **준비**(CPU만 있으면 실행 가능)와 **대기**(입출력 완료를 기다리는 중)를 구분하는 것이 스케줄링 이해의 출발점입니다.
> - 주소 공간은 코드·데이터·힙·스택으로 나뉘고, 스레드는 이 중 **스택과 레지스터만 독립**으로 갖고 나머지를 공유합니다. 이 공유가 성능 이득과 경쟁 상태 위험을 동시에 만들어내며, 멀티프로세스와 멀티스레드의 선택은 결국 **격리와 공유 사이의 트레이드오프**입니다.
