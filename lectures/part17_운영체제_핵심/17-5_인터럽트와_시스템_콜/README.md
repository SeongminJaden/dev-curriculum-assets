# Part 17. 운영체제 핵심

## 17-5. 인터럽트와 시스템 콜 — 사용자 코드는 어떻게 커널의 힘을 빌리는가

> **학습 목표**
> - 인터럽트와 폴링의 차이를 설명하고, 하드웨어 인터럽트와 소프트웨어 인터럽트를 구분할 수 있다.
> - IVT와 ISR의 역할을 설명하고, 인터럽트가 발생한 순간부터 원래 코드로 복귀할 때까지의 처리 흐름을 순서대로 서술할 수 있다.
> - 사용자 모드와 커널 모드의 구분 이유를 설명하고, `malloc`이 언제 커널을 호출하고 언제 호출하지 않는지 설명할 수 있다.

> **개요** 응용 프로그램은 디스크에 직접 명령을 내릴 수 없고, 다른 프로세스의 메모리를 읽을 수도 없습니다. 그런데도 파일을 저장하고 네트워크로 데이터를 보냅니다. 이 강의는 그 사이를 잇는 두 개의 문, 즉 하드웨어가 CPU를 부르는 **인터럽트**와 사용자 코드가 커널을 부르는 **시스템 콜**을 다룹니다.

## 1. 폴링과 인터럽트 — 누가 누구를 부르는가

> **Q.** 키보드가 눌렸는지 CPU가 계속 확인하면 되지 않나요?
> **A.** 그 방식이 <u>**폴링(Polling)**</u>이며, 실제로 그렇게 만들 수 있습니다. 문제는 낭비입니다. 사람이 1초에 5번 타이핑한다면, 초당 수십억 번 명령을 실행하는 CPU가 "눌렸나?"를 확인하는 대부분의 시도는 헛수고입니다. 게다가 확인 주기를 늘리면 반응이 느려지고, 줄이면 CPU를 더 태웁니다. <u>**인터럽트(Interrupt)**</u>는 이 관계를 뒤집습니다. CPU는 자기 일을 하고 있다가, 사건이 실제로 일어났을 때 장치가 CPU에게 전기 신호로 알립니다.

| 구분 | 폴링 | 인터럽트 |
|---|---|---|
| 주도권 | CPU가 장치에 반복적으로 물어봄 | 장치가 CPU에게 알림 |
| CPU 낭비 | 사건이 드물수록 심함 | 사건이 있을 때만 소모 |
| 지연 시간 | 확인 주기에 비례 | 거의 즉시 |
| 코드 복잡도 | 단순 | 핸들러·동시성 고려 필요 |
| 유리한 상황 | 사건이 매우 잦고 예측 가능할 때 | 사건이 드물고 불규칙할 때 |

> **참고** 폴링이 언제나 열등한 것은 아닙니다. 초당 수백만 패킷이 들어오는 고성능 네트워크 카드에서는 패킷마다 인터럽트를 걸면 인터럽트 처리 비용이 시스템을 마비시키는 <u>**인터럽트 폭풍(Interrupt Storm)**</u>이 발생합니다. 그래서 리눅스 네트워크 스택은 NAPI라는 방식으로, 트래픽이 적을 때는 인터럽트를 쓰다가 폭주하면 인터럽트를 끄고 폴링으로 전환합니다.

## 2. 인터럽트의 종류

<u>**하드웨어 인터럽트(Hardware Interrupt)**</u>는 CPU 외부의 장치가 전기 신호로 발생시킵니다. 타이머 칩의 주기적 신호, 키보드 입력, 디스크 읽기 완료, 네트워크 패킷 도착이 여기 속합니다. 이들은 현재 실행 중인 명령과 아무 인과 관계 없이 **비동기적으로** 발생합니다.

<u>**소프트웨어 인터럽트(Software Interrupt)**</u>는 CPU 내부에서, 실행 중인 명령 자체 때문에 발생합니다. 크게 두 갈래로 나뉩니다.

| 유형 | 원인 | 예시 | 복귀 지점 |
|---|---|---|---|
| <u>**예외(Exception)/폴트(Fault)**</u> | 명령 실행 중 문제 발생 | 0으로 나누기, 페이지 폴트, 잘못된 메모리 접근 | 문제를 해결했다면 **같은 명령을 재실행** |
| <u>**트랩(Trap)**</u> | 프로그램이 의도적으로 발생시킴 | 시스템 콜, 디버거 중단점 | **다음 명령**부터 계속 |

> **해석** 페이지 폴트가 "실패"가 아니라 정상적인 동작 흐름이라는 점이 중요합니다. 요청한 페이지가 물리 메모리에 없을 때 CPU가 폴트를 일으키면, 커널이 디스크에서 페이지를 읽어 올리고 **바로 그 명령을 다시 실행**시킵니다. 사용자 프로그램 입장에서는 아무 일도 없었던 것처럼 보입니다. 가상 메모리라는 추상 전체가 이 메커니즘 위에 서 있습니다(관련 메모리 계층 이야기는 `2-3 메모리 계층 구조` 참고).

## 3. IVT와 ISR — 인터럽트가 처리되는 흐름

![인터럽트 처리 흐름](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part17_운영체제_핵심/17-5_인터럽트와_시스템_콜/assets/diagram1_interrupt_flow.svg)

> **해석** 인터럽트 신호에는 "몇 번 인터럽트인가"를 뜻하는 번호가 실려 옵니다. CPU는 이 번호를 색인으로 <u>**IVT(Interrupt Vector Table)**</u>, 즉 인터럽트 번호별 처리 함수 주소를 모아둔 표를 조회해 실행할 함수의 주소를 얻습니다. x86-64에서는 이 표를 IDT(Interrupt Descriptor Table)라 부르며, `IDTR` 레지스터가 그 위치를 가리킵니다. 표에서 찾아낸 함수가 <u>**ISR(Interrupt Service Routine)**</u> 또는 인터럽트 핸들러입니다. 순서를 정리하면 ① 현재 명령 완료 → ② 프로그램 카운터와 상태 레지스터 저장 → ③ 커널 모드 전환 → ④ IVT에서 ISR 주소 조회 → ⑤ ISR 실행 → ⑥ 저장된 상태 복원 → ⑦ 원래 코드로 복귀입니다.

> **참고** 인터럽트는 파이프라인에도 부담을 줍니다. 인터럽트가 들어오면 아직 완료되지 않은 후속 명령들을 정리해야 하므로 파이프라인이 비워지고, 다시 채워질 때까지 성능이 떨어집니다. 파이프라인 구조 자체에 관해서는 `2-2 CPU 명령어 실행 사이클과 파이프라이닝`을 참고하세요.

> **팁** ISR은 가능한 한 짧아야 합니다. ISR 실행 중에는 같은 종류의 인터럽트가 차단되므로, 여기서 오래 머무르면 후속 사건을 놓칩니다. 그래서 리눅스는 처리를 두 단계로 나눕니다. <u>**상반부(Top Half)**</u>는 인터럽트를 즉시 인지하고 최소한의 일만 한 뒤 곧바로 반환하고, 시간이 걸리는 작업은 <u>**하반부(Bottom Half)**</u>로 미뤄 인터럽트가 다시 허용된 상태에서 처리합니다. `softirq`, `tasklet`, `workqueue`가 그 구현체입니다.

## 4. 사용자 모드와 커널 모드

> **Q.** 왜 응용 프로그램이 하드웨어를 직접 다루면 안 되나요?
> **A.** 한 프로그램의 실수나 악의가 시스템 전체를 무너뜨릴 수 있기 때문입니다. 아무 프로그램이나 디스크에 직접 쓸 수 있다면 파일 시스템이 즉시 깨질 것이고, 아무 프로그램이나 임의의 물리 메모리를 읽을 수 있다면 다른 프로세스의 비밀번호를 훔쳐볼 수 있습니다. 그래서 CPU는 하드웨어 차원에서 **권한 수준**을 나눕니다.

<u>**사용자 모드(User Mode)**</u>에서는 특권 명령(입출력 포트 접근, 페이지 테이블 변경, 인터럽트 비활성화 등)을 실행할 수 없습니다. 시도하면 CPU가 예외를 일으켜 커널이 개입합니다. <u>**커널 모드(Kernel Mode)**</u>에서는 모든 명령과 모든 메모리에 접근할 수 있습니다. x86 계열은 링 0(커널)부터 링 3(사용자)까지 네 단계 권한 레벨을 제공하지만, 실제 운영체제는 대개 링 0과 링 3만 사용합니다.

## 5. 시스템 콜 — 문을 두드리는 유일한 방법

![시스템 콜과 모드 전환](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part17_운영체제_핵심/17-5_인터럽트와_시스템_콜/assets/diagram2_syscall_modes.svg)

> **해석** <u>**시스템 콜(System Call)**</u>은 사용자 프로그램이 커널의 기능을 요청하는 공식 통로입니다. 핵심은 이것이 **단순한 함수 호출이 아니라는 점**입니다. 일반 함수 호출은 프로그램 카운터를 다른 주소로 옮길 뿐이지만, 시스템 콜은 권한 수준 자체를 바꿔야 하므로 반드시 CPU의 특수 명령을 거쳐야 합니다. 그림처럼 ① 시스템 콜 번호를 약속된 레지스터(x86-64 리눅스에서는 `rax`)에 넣고 ② `syscall` 명령을 실행하면 ③ CPU가 커널 모드로 전환하며 커널의 진입점으로 점프하고 ④ 커널이 번호로 시스템 콜 테이블을 조회해 해당 처리 함수를 실행한 뒤 ⑤ 결과를 레지스터에 담아 사용자 모드로 복귀합니다. 사용자가 임의의 커널 주소로 뛰어들 수 없고 오직 정해진 진입점을 통과해야 한다는 점이 이 설계의 안전성 근거입니다.

| 분류 | 대표 시스템 콜 |
|---|---|
| 프로세스 제어 | `fork`, `execve`, `exit`, `wait` |
| 파일 조작 | `open`, `read`, `write`, `close` |
| 장치 관리 | `ioctl`, `mmap` |
| 정보 관리 | `getpid`, `time` |
| 통신 | `socket`, `bind`, `send`, `recv` |

> **참고** 우리가 쓰는 `printf`나 `open()`은 시스템 콜 자체가 아니라 C 표준 라이브러리(glibc)의 <u>**래퍼 함수(Wrapper Function)**</u>입니다. 래퍼는 인자를 레지스터 규약에 맞게 배치하고 `syscall` 명령을 실행한 뒤, 반환값이 음수면 `errno`를 설정해주는 일을 합니다. 리눅스에서 `strace ./program`을 실행하면 프로그램이 실제로 호출한 시스템 콜 목록을 순서대로 볼 수 있습니다.

> **팁** 모드 전환에는 비용이 듭니다. 그래서 성능이 중요한 코드에서는 시스템 콜 횟수 자체를 줄이는 것이 핵심 최적화입니다. 1바이트씩 1만 번 `write`하는 대신 버퍼에 모아 한 번에 쓰는 표준 입출력 라이브러리의 버퍼링, 소켓마다 상태를 묻는 대신 준비된 것만 한 번에 받아오는 `epoll`, 시스템 콜 요청을 큐에 모아 처리하는 `io_uring`이 모두 같은 동기에서 나온 설계입니다. 참고로 `gettimeofday` 같은 일부 호출은 vDSO 메커니즘 덕분에 모드 전환 없이 사용자 공간에서 처리됩니다.

## 6. malloc이 커널과 만나는 지점

> **Q.** `malloc(100)`을 호출하면 매번 시스템 콜이 일어나나요?
> **A.** 대부분은 일어나지 않습니다. `malloc`은 커널 함수가 아니라 C 표준 라이브러리가 제공하는 <u>**사용자 공간 메모리 할당자(Allocator)**</u>입니다. 할당자는 커널에게서 큰 덩어리의 메모리를 미리 받아두고, 프로그램이 요청할 때마다 그 덩어리를 잘라 나눠줍니다. `free`로 반환된 조각은 커널에 돌려주지 않고 자유 리스트에 보관했다가 다음 요청에 재사용합니다. 커널을 부르는 것은 **가진 덩어리가 부족할 때뿐**입니다.

| 계층 | 하는 일 |
|---|---|
| 응용 코드 | `malloc(100)` / `free(p)` 호출 |
| 라이브러리 할당자(glibc `ptmalloc` 등) | 자유 리스트에서 적당한 조각을 찾아 반환. 없으면 아래로 요청 |
| 시스템 콜 | 힙 경계를 옮기는 `brk`/`sbrk`, 또는 새 영역을 매핑하는 `mmap` |
| 커널 가상 메모리 관리자 | 가상 주소 영역을 등록. 물리 페이지는 아직 붙이지 않음 |
| 페이지 폴트 처리 | 그 주소에 **실제로 접근하는 순간** 물리 페이지를 할당 |

> **해석** 마지막 두 줄이 중요합니다. 커널은 `mmap` 시점에 물리 메모리를 주지 않고 "이 주소 범위를 쓸 수 있다"고 장부에만 적어둡니다. 프로그램이 그 주소를 처음 건드리는 순간 페이지 폴트가 발생하고, 그제서야 물리 페이지가 붙습니다. 이 <u>**요구 페이징(Demand Paging)**</u> 덕분에 큰 메모리를 할당해두고 일부만 쓰는 프로그램이 실제 메모리를 낭비하지 않습니다. 여기서 3절의 페이지 폴트와 6절의 메모리 할당이 하나로 이어집니다 — 동적 할당은 결국 인터럽트 메커니즘 위에서 완성됩니다.

> **참고** glibc의 `malloc`은 요청 크기가 임계값(기본 128KB)을 넘으면 `brk` 대신 `mmap`으로 별도 영역을 잡고, `free` 시 그 영역을 즉시 커널에 반납합니다. 작은 할당은 힙에서 재사용해 시스템 콜을 아끼고, 큰 할당은 즉시 반납해 메모리 파편화를 줄이려는 절충입니다.

## 요약

> **핵심**
> - **폴링**은 CPU가 반복해서 묻는 방식, **인터럽트**는 장치가 CPU를 부르는 방식입니다. 인터럽트는 드물고 불규칙한 사건에 유리하지만, 사건이 폭주하면 폴링으로 전환하는 편이 낫습니다(리눅스 NAPI).
> - 인터럽트 번호로 **IVT**를 조회해 **ISR**을 실행하고 원래 위치로 복귀합니다. ISR은 짧아야 하므로 리눅스는 **상반부와 하반부**로 처리를 나눕니다. 페이지 폴트처럼 문제를 해결한 뒤 **같은 명령을 재실행**하는 예외와, 다음 명령으로 진행하는 **트랩**은 복귀 지점이 다릅니다.
> - 사용자 모드는 특권 명령을 쓸 수 없고, **시스템 콜**만이 커널로 들어가는 정해진 문입니다. 모드 전환에 비용이 들기에 버퍼링·`epoll`·`io_uring` 같은 호출 횟수 절감 기법이 존재합니다. **`malloc`은 대부분 라이브러리 안에서 끝나고**, 덩어리가 부족할 때만 `brk`/`mmap`으로 커널을 부르며, 물리 페이지는 실제 접근 시 페이지 폴트로 비로소 붙습니다.
