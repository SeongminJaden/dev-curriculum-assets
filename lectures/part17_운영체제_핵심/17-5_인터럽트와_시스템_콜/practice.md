# 17-5. 인터럽트와 시스템 콜 — 실습

> **참고** 이 실습은 제출물이 아니라, 프로그램이 실제로 호출하는 시스템 콜을 눈으로 확인하고 모드 전환 비용을 측정해보는 과정입니다.

## 실습 1 — 시스템 콜을 눈으로 보기

- [ ] "Hello"를 출력하는 최소 프로그램을 작성하고 `strace ./program`(macOS는 `dtruss`)으로 실행해, 출력된 시스템 콜 전체 목록을 파일로 저장하세요.
- [ ] 목록에서 `write` 호출을 찾아 인자(파일 디스크립터 번호, 버퍼, 길이)를 적고, 실제 출력에 관여한 호출이 몇 개인지 세어보세요.
- [ ] `strace -c ./program`으로 시스템 콜별 호출 횟수 요약을 뽑고, 프로그램 시작에만 몇 개의 호출이 쓰이는지 확인하세요.

## 실습 2 — 시스템 콜 횟수가 성능에 미치는 영향 측정하기

- [ ] 1MB 데이터를 1바이트씩 100만 번 `write`로 파일에 쓰는 프로그램을 작성하고 실행 시간을 측정하세요.
- [ ] 같은 1MB를 64KB 버퍼에 모아 16번의 `write`로 쓰는 버전을 작성해 실행 시간을 측정하고, 두 값의 배율을 계산하세요.
- [ ] 두 버전을 각각 `strace -c`로 실행해 `write` 호출 횟수를 확인하고, 시간 차이가 호출 횟수 차이와 어떤 관계인지 한 문단으로 정리하세요.

## 실습 3 — 페이지 폴트와 요구 페이징 관찰하기

- [ ] 1GB를 한 번에 할당하되 아무 데이터도 쓰지 않는 프로그램을 작성하고, 할당 직후 `/proc/self/status`의 `VmSize`와 `VmRSS`를 출력하세요.
- [ ] 할당한 영역 전체를 0으로 채운 뒤 같은 두 값을 다시 출력하고, 어느 값이 얼마나 늘었는지 비교하세요.
- [ ] `/usr/bin/time -v ./program`으로 마이너 페이지 폴트 횟수를 확인하고, 그 값이 (할당 크기 ÷ 페이지 크기 4KB)와 비슷한지 계산해 확인하세요.

## 실습 4 — malloc이 언제 커널을 부르는지 확인하기

- [ ] 100바이트를 10,000번 `malloc`하는 프로그램을 `strace -e trace=brk,mmap`으로 실행하고, 발생한 `brk`/`mmap` 호출 횟수를 세어보세요.
- [ ] 이번에는 1MB를 10번 `malloc`하는 프로그램을 같은 방식으로 실행하고, `mmap` 호출이 매번 발생하는지 확인하세요.
- [ ] 두 결과를 비교해 "작은 할당은 라이브러리가, 큰 할당은 커널이 처리한다"는 강의 내용이 실제로 관찰되는지 결론을 적고, glibc의 임계값(기본 128KB)을 기준으로 설명하세요.

> **팁** 실습 3, 4는 리눅스에서 가장 잘 관찰됩니다. macOS나 윈도우 환경이라면 도커 컨테이너나 가상 머신 안의 리눅스에서 수행하세요. `/proc` 파일 시스템 자체가 "커널의 내부 상태를 파일처럼 읽게 해주는" 리눅스 고유의 인터페이스라는 점도 함께 확인해보면 좋습니다.
