# Part 17. 운영체제 핵심

## 17-3. 동기화 — 뮤텍스와 세마포어는 무엇이 다른가

> **학습 목표**
> - 경쟁 상태가 발생하는 원리를 기계어 수준에서 설명하고, 임계 구역 해법이 만족해야 할 세 조건을 열거할 수 있다.
> - 뮤텍스와 세마포어의 차이를 소유권 개념을 근거로 설명하고, 이진 세마포어가 뮤텍스와 같지 않은 이유를 제시할 수 있다.
> - 모니터와 조건 변수의 역할을 설명하고, CAS 같은 원자적 연산이 락의 토대가 되는 방식을 설명할 수 있다.

> **개요** 두 스레드가 `count = count + 1`을 각각 10만 번 실행했는데 결과가 20만이 아닙니다. 어디서 값이 사라졌을까요? 이 강의는 그 손실이 CPU 명령어 수준에서 왜 필연적인지 밝히고, 커널과 라이브러리가 제공하는 동기화 도구들이 각각 어떤 문제를 겨냥해 만들어졌는지 다룹니다. 여기서 다루는 것은 애플리케이션의 비동기 문법이 아니라, 여러 실행 흐름이 물리적으로 같은 메모리를 건드릴 때 커널과 하드웨어가 제공하는 보호 장치입니다.

## 1. 경쟁 상태 — 값은 어디서 사라지는가

> **Q.** `count++`는 코드 한 줄인데 왜 중간에 끼어들 수 있나요?
> **A.** 한 줄의 고급 언어 문장이 CPU에서는 여러 개의 기계어 명령으로 번역되기 때문입니다. `count++`는 대개 ① 메모리에서 `count` 값을 레지스터로 읽고(load), ② 레지스터 값을 1 증가시키고(add), ③ 레지스터 값을 메모리에 되쓰는(store) 세 단계로 나뉩니다. 선점형 스케줄러는 이 세 단계 **사이 어디서든** 타이머 인터럽트로 스레드를 교체할 수 있습니다.

```text
초기값 count = 10

스레드 A                  스레드 B
1) load  count → 10
                          2) load  count → 10
                          3) add   → 11
                          4) store count = 11
5) add   → 11
6) store count = 11        ← B의 증가가 통째로 사라짐

두 번 증가시켰지만 결과는 11
```

<u>**경쟁 상태(Race Condition)**</u>는 여러 실행 흐름이 공유 데이터에 동시에 접근할 때, 접근 순서(타이밍)에 따라 결과가 달라지는 상황입니다. 그리고 이렇게 공유 데이터를 건드리는 코드 구간을 <u>**임계 구역(Critical Section)**</u>이라 부릅니다.

> **참고** 경쟁 상태의 가장 고약한 성질은 **재현되지 않는다**는 점입니다. 위 인터리빙은 스케줄러가 하필 그 지점에서 전환할 때만 발생하므로, 개발 환경에서 1만 번 돌려도 안 나타나던 버그가 부하가 높은 운영 환경에서 하루에 몇 번씩 터지곤 합니다.

## 2. 임계 구역 해법이 만족해야 할 세 조건

1948년 이후 여러 해법이 제안되었지만, 올바른 해법이라면 반드시 다음 세 조건을 모두 만족해야 합니다.

| 조건 | 의미 | 위반하면 |
|---|---|---|
| <u>**상호 배제(Mutual Exclusion)**</u> | 한 스레드가 임계 구역에 있으면 다른 스레드는 들어갈 수 없다 | 경쟁 상태 발생 — 데이터가 깨짐 |
| <u>**진행(Progress)**</u> | 임계 구역이 비어 있으면 들어가려는 스레드 중 하나는 반드시 들어갈 수 있다 | 아무도 못 들어가는 교착 |
| <u>**한정 대기(Bounded Waiting)**</u> | 어떤 스레드도 유한한 횟수 안에 반드시 차례를 얻는다 | 특정 스레드의 기아 |

> **해석** 세 조건은 각각 다른 실패 양상을 막습니다. 상호 배제만 만족시키는 것은 쉽습니다 — 아무도 들여보내지 않으면 됩니다. 하지만 그러면 진행 조건이 깨집니다. 진행까지 만족시켜도, 운 나쁜 한 스레드가 계속 밀려나면 한정 대기가 깨집니다. 세 조건을 동시에 만족시키는 것이 어려워서 동기화가 어려운 주제인 것입니다. 1981년 게리 피터슨(Gary Peterson)이 제시한 피터슨 알고리즘은 두 프로세스에 대해 세 조건을 모두 만족하는 순수 소프트웨어 해법을 제시했지만, 프로세스가 셋 이상이면 복잡해지고 현대 CPU의 명령어 재배치 때문에 그대로는 동작하지 않습니다.

## 3. 하드웨어의 도움 — 원자적 연산과 CAS

> **Q.** 소프트웨어만으로 해결이 어렵다면 무엇에 기대야 하나요?
> **A.** "읽기와 쓰기가 중간에 끊기지 않는" 명령어를 CPU가 직접 제공해줘야 합니다. 이런 명령을 <u>**원자적 연산(Atomic Operation)**</u>이라 하며, 실행 도중 인터럽트나 다른 코어의 접근이 끼어들 수 없음을 하드웨어가 보장합니다.

<u>**CAS(Compare-And-Swap)**</u>는 가장 널리 쓰이는 원자적 연산입니다. 의미상 다음 세 줄을 통째로 원자적으로 수행합니다.

```c
// 이 함수 전체가 CPU 명령어 하나로 끊김 없이 실행됩니다
bool CAS(int *addr, int expected, int new_value) {
    if (*addr != expected) return false;   // 내가 본 값이 아니면 포기
    *addr = new_value;                     // 맞으면 교체
    return true;
}
```

> **해석** CAS의 논리는 "내가 값을 읽은 뒤 아무도 바꾸지 않았다면 내 값으로 갱신하고, 누가 바꿨다면 실패를 알려라"입니다. 실패하면 다시 읽고 재시도하는 루프를 도는데, 이 패턴이 <u>**락-프리(Lock-free)**</u> 자료구조의 기본 골격입니다. x86-64에서는 `LOCK CMPXCHG` 명령이, ARM에서는 `LDREX`/`STREX` 명령 쌍이 이 역할을 합니다. 뮤텍스와 세마포어를 포함한 거의 모든 상위 동기화 도구가 내부적으로는 이런 원자적 명령 위에 세워져 있습니다.

> **참고** 락을 얻을 때까지 CAS 루프를 계속 도는 방식을 <u>**스핀락(Spinlock)**</u>이라 합니다. 대기 중에 CPU를 계속 태우므로 낭비 같지만, 락을 아주 짧게(수십 나노초) 잡는 커널 내부에서는 잠들었다 깨는 컨텍스트 스위칭 비용보다 오히려 싸기 때문에 널리 쓰입니다. 반대로 락 보유 시간이 길다면 스핀락은 최악의 선택입니다.

## 4. 뮤텍스와 세마포어

![뮤텍스와 세마포어의 차이](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part17_운영체제_핵심/17-3_동기화_뮤텍스와_세마포어/assets/diagram1_mutex_vs_semaphore.svg)

> **해석** 두 도구의 차이는 개수가 아니라 <u>**소유권(Ownership)**</u>에 있습니다. <u>**뮤텍스(Mutex)**</u>는 "이 자물쇠를 잠근 스레드만 열 수 있다"는 소유권 규칙을 갖습니다. 그래서 뮤텍스는 상호 배제, 즉 "이 코드 구간은 한 번에 한 스레드만"이라는 목적에 정확히 맞습니다. <u>**세마포어(Semaphore)**</u>는 소유자를 기록하지 않는 **정수 카운터**입니다. `wait`(P 연산)은 카운터를 1 줄이고 0 미만이면 대기시키며, `signal`(V 연산)은 카운터를 1 늘리고 대기자 하나를 깨웁니다. 누가 줄였는지 기억하지 않으므로, **A 스레드가 줄인 것을 B 스레드가 늘릴 수 있습니다.** 이 성질이 결함이 아니라 기능이라는 점이 핵심입니다.

| 구분 | 뮤텍스 | 세마포어 |
|---|---|---|
| 본질 | 잠금/해제 상태를 가진 자물쇠 | 사용 가능 자원 개수를 세는 정수 카운터 |
| 소유권 | 있음 — 잠근 스레드만 해제 가능 | 없음 — 아무 스레드나 signal 가능 |
| 주 용도 | 상호 배제(임계 구역 보호) | 자원 개수 제한, 실행 순서 신호 전달 |
| 우선순위 상속 | 지원하는 구현이 많음 | 소유자를 모르므로 불가 |
| 대표 실수 | 락 해제 누락으로 인한 교착 | 잘못된 signal 남발로 카운터 붕괴 |

> **Q.** 그렇다면 초기값이 1인 이진 세마포어는 뮤텍스와 같은 것 아닌가요?
> **A.** 동작은 비슷해 보이지만 같지 않습니다. 이진 세마포어는 여전히 소유권이 없어서, 락을 잡지 않은 제3의 스레드가 `signal`을 호출해 임계 구역을 열어버릴 수 있습니다. 또 뮤텍스는 소유자를 알기에 <u>**우선순위 역전(Priority Inversion)**</u>이 발생했을 때 락 보유자의 우선순위를 일시적으로 올리는 우선순위 상속을 구현할 수 있지만, 세마포어는 누가 들고 있는지 모르므로 그럴 수 없습니다. 이 차이는 실제 사고로도 증명되었습니다. 1997년 화성 탐사선 <u>**패스파인더(Mars Pathfinder)**</u>가 착륙 후 반복적으로 재부팅된 원인이 바로 우선순위 역전이었고, JPL 연구진은 원격으로 뮤텍스의 우선순위 상속 옵션을 켜서 해결했습니다.

> **참고** 세마포어는 1965년 네덜란드의 에츠허르 다익스트라(Edsger W. Dijkstra)가 `Cooperating Sequential Processes`에서 제안했습니다. `P`와 `V`라는 이름은 각각 네덜란드어 `proberen`(시도하다)과 `verhogen`(증가시키다)에서 온 것으로 알려져 있습니다. 다익스트라는 같은 글에서 식사하는 철학자 문제를 비롯한 고전적 동기화 문제들도 함께 정식화했습니다.

> **팁** 실무 기준은 간단합니다. **"한 번에 하나만"이면 뮤텍스, "한 번에 N개까지"이면 카운팅 세마포어**입니다. 데이터베이스 커넥션 풀에서 동시 연결을 20개로 제한하는 것, 다운로드 동시 실행을 4개로 묶는 것이 카운팅 세마포어의 전형적 용례입니다. 생산자-소비자 패턴에서 "버퍼에 데이터가 하나 들어왔다"는 신호를 보내는 것도 소유권이 없는 세마포어라서 가능한 일입니다.

## 5. 모니터와 조건 변수 — 락을 언어가 대신 관리해주기

![모니터와 조건 변수의 동작](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part17_운영체제_핵심/17-3_동기화_뮤텍스와_세마포어/assets/diagram2_monitor_condvar.svg)

> **해석** 뮤텍스와 세마포어는 강력하지만 프로그래머가 락 획득과 해제를 손으로 짝맞춰야 해서 실수가 잦습니다. <u>**모니터(Monitor)**</u>는 공유 데이터와 그것을 다루는 연산들을 하나의 단위로 묶고, 그 단위 안에는 항상 한 스레드만 들어갈 수 있도록 언어나 런타임이 자동으로 보장하는 고급 동기화 구조입니다. 그림의 왼쪽 <u>**진입 큐(Entry Queue)**</u>는 모니터에 들어가려고 기다리는 스레드들이고, 오른쪽 <u>**조건 변수(Condition Variable)**</u>의 대기 큐는 이미 들어왔지만 "버퍼가 빌 때까지" 같은 조건이 충족되지 않아 자발적으로 멈춘 스레드들입니다.

조건 변수는 두 연산을 제공합니다. `wait()`는 **락을 놓으면서 동시에** 잠드는 원자적 연산이고, `signal()`은 대기 중인 스레드 하나를 깨웁니다. `wait()`가 락을 놓지 않고 잠들면 아무도 조건을 바꿔줄 수 없어 영원히 깨어나지 못하므로, 이 두 동작이 원자적이어야 한다는 점이 조건 변수 설계의 핵심입니다.

> **팁** 조건 변수를 쓸 때는 반드시 `if`가 아니라 `while`로 조건을 검사해야 합니다. 깨어난 스레드가 락을 다시 얻는 사이 다른 스레드가 조건을 무너뜨렸을 수 있고, 아무도 signal하지 않았는데 깨어나는 <u>**허위 각성(Spurious Wakeup)**</u>도 POSIX 규격상 허용되기 때문입니다.

```python
# 올바른 패턴 — 깨어난 뒤 조건을 다시 확인합니다
with condition:
    while not buffer:        # if 가 아니라 while
        condition.wait()
    item = buffer.pop()
```

> **참고** 모니터는 1974년 C.A.R. 호어(C.A.R. Hoare)와 페르 브린치 한센(Per Brinch Hansen)이 각각 정식화한 개념입니다. 오늘날 자바의 `synchronized` 키워드와 `wait()`/`notify()`, 파이썬의 `threading.Condition`, C#의 `lock` 구문이 모두 모니터 개념의 직계 후손입니다.

## 6. 무엇을 언제 쓰는가

| 상황 | 적절한 도구 |
|---|---|
| 공유 변수 하나를 여러 스레드가 갱신 | 뮤텍스, 또는 원자적 정수 타입 |
| 동시 접근 수를 N개로 제한 | 카운팅 세마포어 |
| "데이터가 준비될 때까지 대기" | 조건 변수(또는 모니터) |
| 읽기가 압도적으로 많고 쓰기는 드묾 | 읽기-쓰기 락(ReadWriteLock) |
| 락을 수십 나노초만 잡는 커널 내부 | 스핀락 |
| 카운터 증감처럼 단순한 연산 | CAS 기반 원자 연산(락 없이) |

> **팁** 동기화에서 가장 흔한 실무 실수는 **락의 범위를 너무 넓게 잡는 것**입니다. 임계 구역 안에 네트워크 호출이나 파일 입출력을 넣으면 그 시간 동안 모든 스레드가 멈춰 서서, 멀티스레드로 얻으려던 이득이 통째로 사라집니다. 락은 공유 데이터를 실제로 건드리는 최소 구간에만 걸고, 무거운 작업은 락 밖에서 수행하세요.

> **핵심**
> - `count++`가 load·add·store 세 명령으로 쪼개지기 때문에 그 사이에 선점이 일어나면 갱신이 통째로 사라집니다. 이것이 **경쟁 상태**이며, 해법은 **상호 배제·진행·한정 대기** 세 조건을 모두 만족해야 합니다.
> - 실용적 해법은 하드웨어의 **원자적 연산(CAS)** 위에 세워집니다. 그 위에 **뮤텍스**(소유권 있음, 한 번에 하나)와 **세마포어**(소유권 없음, 한 번에 N개·신호 전달)가 서로 다른 목적으로 존재합니다.
> - **모니터와 조건 변수**는 락 관리를 언어에 맡기고 "조건이 충족될 때까지 대기"를 표현하게 해주며, 허위 각성 때문에 조건 검사는 반드시 `while` 루프로 해야 합니다. 세마포어는 1965년 다익스트라가, 모니터는 1974년 호어와 브린치 한센이 제안했습니다.
