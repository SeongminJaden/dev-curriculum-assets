# 17-3. 동기화 — 뮤텍스와 세마포어 — 실습

> **참고** 이 실습은 제출물이 아니라, 경쟁 상태를 직접 재현하고 각 동기화 도구로 하나씩 고쳐보는 과정입니다.

## 실습 1 — 경쟁 상태 재현하고 원인 확인하기

- [ ] 전역 카운터를 스레드 4개가 각각 200,000번 증가시키는 프로그램을 작성하고, 20회 반복 실행해 최종값을 모두 기록하세요.
- [ ] 기대값 800,000과 일치한 횟수를 세고, 실제로 손실된 최대 증가량을 계산하세요.
- [ ] 증가 코드를 `tmp = counter; sleep(0); counter = tmp + 1`처럼 세 줄로 풀어 쓴 뒤 다시 실행해, 손실량이 얼마나 커지는지 비교하고 그 이유를 강의의 load·add·store 설명으로 적어보세요.

## 실습 2 — 뮤텍스로 고치고 비용 측정하기

- [ ] 실습 1의 증가 코드를 뮤텍스로 감싸고, 20회 실행 결과가 모두 800,000이 나오는지 확인하세요.
- [ ] 뮤텍스 적용 전후의 실행 시간을 각각 측정하고 배율을 계산해, 정확성을 얻는 대가로 지불한 비용을 숫자로 적어보세요.
- [ ] 락 범위를 "증가 연산만" 감싼 버전과 "반복문 전체"를 감싼 버전 두 가지로 만들어 실행 시간을 비교하고, 어느 쪽이 왜 빠른지 또는 느린지 결론을 적으세요.

## 실습 3 — 카운팅 세마포어로 동시 실행 수 제한하기

- [ ] 1초씩 걸리는 가짜 작업 20개를 스레드로 동시에 실행하고 총 소요 시간을 측정하세요.
- [ ] 초기값 4인 카운팅 세마포어를 도입해 동시에 4개만 실행되도록 제한하고, 총 소요 시간을 다시 측정하세요.
- [ ] 세마포어 초기값을 1, 2, 4, 8, 20으로 바꿔가며 총 소요 시간을 표로 기록하고, 값이 커질수록 시간이 줄어드는 양상이 선형인지 확인하세요.

## 실습 4 — 조건 변수로 생산자-소비자 만들기

- [ ] 최대 크기 5인 버퍼를 두고, 생산자 스레드 1개와 소비자 스레드 1개가 조건 변수로 협력하는 프로그램을 작성하세요.
- [ ] 대기 조건을 `if not buffer: wait()`로 작성한 버전과 `while not buffer: wait()`로 작성한 버전을 각각 만들고, 소비자를 3개로 늘려 실행하며 `if` 버전에서 오류(빈 버퍼 접근)가 나는지 확인하세요.
- [ ] 버퍼가 가득 찼을 때 생산자가 대기하는지 확인하기 위해 소비자를 3초간 멈추고, 생산자가 실제로 멈춰 있는지 로그로 검증하세요.

> **팁** 실습 1에서 손실이 잘 재현되지 않는다면 스레드 수를 코어 수보다 많게 늘리고, 증가 횟수를 키우세요. 재현이 어렵다는 사실 자체가 경쟁 상태 버그가 운영 환경에서만 터지는 이유이며, 그래서 "테스트로 못 잡았으니 안전하다"는 추론이 위험하다는 점을 기억하세요.
