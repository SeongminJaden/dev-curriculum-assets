# 17-2. 컨텍스트 스위칭과 스케줄링 — 실습

> **참고** 이 실습은 제출물이 아니라, 컨텍스트 스위칭 비용을 실제로 측정하고 스케줄링 알고리즘을 손으로 돌려보는 과정입니다.

## 실습 1 — 컨텍스트 스위칭 횟수 관찰하기

- [ ] `vmstat 1`을 10초간 실행해 `cs`(context switch) 열의 값을 기록하고, 평균 초당 전환 횟수를 계산하세요.
- [ ] 무한 루프를 도는 프로세스를 코어 수의 4배만큼 띄운 뒤 다시 `vmstat 1`을 측정하고, `cs` 값이 어떻게 변했는지 비교하세요.
- [ ] 이번에는 짧은 `sleep(0.001)`을 반복하는 프로세스를 같은 수만큼 띄우고 `cs` 값을 측정해, CPU 바운드 작업과 입출력 바운드 작업 중 어느 쪽이 전환을 더 많이 유발하는지 결론을 적어보세요.

## 실습 2 — 전환 비용 직접 측정하기

- [ ] 파이프 두 개로 연결된 두 프로세스(또는 두 스레드)가 1바이트를 1,000,000번 주고받는 핑퐁 프로그램을 작성하세요.
- [ ] 전체 소요 시간을 왕복 횟수로 나눠 왕복 1회당 시간을 구하고, 이를 2로 나눠 전환 1회의 대략적 비용(마이크로초)을 계산하세요.
- [ ] 같은 실험을 스레드 버전과 프로세스 버전으로 각각 수행하고 두 값을 비교해, 주소 공간 교체가 있는 쪽이 얼마나 더 비싼지 배율로 적어보세요.

## 실습 3 — 스케줄링 알고리즘 손으로 돌려보기

- [ ] 도착 시각과 실행 시간이 각각 (0, 8), (1, 4), (2, 9), (3, 5)인 프로세스 4개를 종이에 적으세요.
- [ ] FCFS, SJF(비선점), 라운드 로빈(quantum = 3)에 대해 각각 실행 순서를 간트 차트로 그리고, 프로세스별 대기 시간과 평균 대기 시간을 계산하세요.
- [ ] 세 결과를 표로 정리하고, 어떤 알고리즘이 평균 대기 시간이 가장 짧은지와 어떤 알고리즘이 가장 늦게 시작하는 프로세스의 대기 시간이 가장 짧은지를 각각 표시하세요.

## 실습 4 — 우선순위와 기아 실험하기

- [ ] CPU를 100% 쓰는 계산 프로그램을 두 개 띄우되, 하나는 `nice -n 0`, 다른 하나는 `nice -n 19`로 실행하세요.
- [ ] `top` 또는 `htop`으로 두 프로세스의 CPU 점유율을 30초간 관찰해 기록하고, 두 값의 비율을 계산하세요.
- [ ] `nice -n 19` 프로세스가 완전히 0%가 되지 않고 일정 비율은 확보하는지 확인하고, 이 결과가 CFS의 "가장 적게 실행된 작업을 먼저" 원리와 어떻게 연결되는지 두세 문장으로 적어보세요.

> **팁** 실습 2의 측정값은 CPU 모델과 커널 버전에 따라 수 마이크로초 단위로 흔들립니다. 절대값보다 "스레드 대 프로세스의 배율"과 "측정값이 레지스터 저장 시간만으로는 설명되지 않는다"는 사실에 주목하세요. 그 차이가 곧 캐시·TLB 오염의 간접 비용입니다.
