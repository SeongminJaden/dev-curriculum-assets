# Part 17. 운영체제 핵심

## 17-2. 컨텍스트 스위칭과 스케줄링 — 코어 8개로 프로세스 500개를 돌리는 방법

> **학습 목표**
> - 컨텍스트 스위칭의 절차를 순서대로 설명하고, 직접 비용과 간접 비용(TLB·캐시 오염)을 구분해 서술할 수 있다.
> - FCFS·SJF·우선순위·라운드 로빈·다단계 피드백 큐의 동작 원리와 장단점을 비교하고, 주어진 작업 집합에 대해 평균 대기 시간을 계산할 수 있다.
> - 선점형과 비선점형 스케줄링의 차이를 설명하고, 기아 현상이 왜 발생하며 에이징이 이를 어떻게 완화하는지 설명할 수 있다.

> **개요** 코어가 8개인데 실행 중인 프로세스는 500개입니다. 사용자는 모든 프로그램이 동시에 돌아가는 것처럼 느낍니다. 이 착시는 커널이 매우 짧은 간격으로 CPU를 이 프로세스에서 저 프로세스로 옮겨 주기 때문에 생깁니다. 이 강의는 그 "옮기는 작업"이 정확히 무엇이고 얼마나 비싼지, 그리고 누구에게 다음 차례를 줄지 결정하는 규칙들을 다룹니다.

---

## 1. 컨텍스트 스위칭이란 정확히 무엇을 옮기는 작업인가요?

> **Q.** 프로세스를 바꾼다는 게 구체적으로 어떤 데이터를 어디로 옮기는 일인가요?
> **A.** CPU 안의 레지스터 값들을 현재 프로세스의 PCB로 **저장**하고, 다음 프로세스의 PCB에서 레지스터 값들을 CPU로 **복원**하는 일입니다. CPU는 근본적으로 "프로그램 카운터가 가리키는 명령어를 읽어 실행하고 다음으로 넘어가는" 기계일 뿐이라, 프로그램 카운터와 나머지 레지스터를 통째로 다른 값으로 바꿔치기하면 그 순간부터 완전히 다른 프로세스를 실행하게 됩니다. 여기에 더해 메모리 관리 레지스터(x86-64의 `CR3`)를 교체해 주소 공간까지 바꾸면 전환이 완료됩니다.

<u>**컨텍스트 스위칭(Context Switching)**</u>은 CPU를 한 실행 흐름에서 다른 실행 흐름으로 넘기기 위해 컨텍스트를 저장하고 복원하는 커널 작업입니다. 이 작업이 진행되는 동안 CPU는 어떤 사용자 코드도 실행하지 못하므로, 컨텍스트 스위칭은 그 자체로 순수한 오버헤드입니다.

## 2. 컨텍스트 스위칭의 실제 비용 — 눈에 보이는 것과 보이지 않는 것

![컨텍스트 스위칭의 직접 비용과 간접 비용](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part17_운영체제_핵심/17-2_컨텍스트_스위칭과_스케줄링/assets/diagram1_context_switch_cost.svg)

> **해석** 그림의 왼쪽이 <u>**직접 비용(Direct Cost)**</u>입니다. 레지스터 수십 개를 메모리에 쓰고 다시 읽어오는 작업으로, 현대 CPU에서 보통 수백 나노초 규모입니다. 그런데 실제로 더 아픈 것은 오른쪽의 <u>**간접 비용(Indirect Cost)**</u>입니다. 새 프로세스가 실행되기 시작하면 CPU 캐시에는 이전 프로세스의 데이터만 들어 있어 거의 모든 접근이 캐시 미스가 되고, 새로 들어온 프로세스가 자기 데이터로 캐시를 다시 채우는 동안 수 마이크로초를 낭비합니다. 이 현상을 <u>**캐시 오염(Cache Pollution)**</u>이라 부릅니다. 주소 공간까지 바뀌는 프로세스 간 전환이라면 <u>**TLB(Translation Lookaside Buffer)**</u>, 즉 가상 주소를 물리 주소로 바꾼 결과를 캐싱해 둔 표까지 무효화되어 페이지 테이블 조회를 처음부터 다시 해야 합니다. 그래서 실측 총비용은 직접 비용의 수 배에서 수십 배에 이릅니다.

> **참고** 같은 프로세스 안의 스레드끼리 전환할 때는 주소 공간이 그대로이므로 `CR3`을 건드릴 필요가 없고 TLB도 유지됩니다. 이것이 스레드 전환이 프로세스 전환보다 싼 핵심 이유입니다. 또한 x86-64에서는 각 TLB 항목에 프로세스 식별자를 붙이는 <u>**PCID(Process-Context Identifier)**</u> 기능으로, 프로세스 전환 시에도 TLB를 통째로 비우지 않고 재사용할 수 있습니다. 캐시와 메모리 계층 자체에 관해서는 `2-3 메모리 계층 구조`를 참고하세요.

> **팁** 컨텍스트 스위칭 횟수는 `vmstat 1`의 `cs` 열이나 `pidstat -w`로 직접 볼 수 있습니다. CPU 사용률은 낮은데 `cs` 값이 초당 수십만을 넘는다면, 스레드를 지나치게 많이 만들었거나 락 경합으로 스레드가 계속 잠들고 깨는 상황을 의심해야 합니다.

## 3. 스케줄링의 목표는 하나가 아닙니다

<u>**CPU 스케줄링(CPU Scheduling)**</u>은 준비 상태의 프로세스 중 다음에 CPU를 줄 대상을 고르는 일입니다. 문제는 "좋은 스케줄링"의 기준이 여럿이고 서로 충돌한다는 점입니다.

| 지표 | 정의 | 누가 중요하게 여기는가 |
|---|---|---|
| 처리량(Throughput) | 단위 시간당 완료한 작업 수 | 배치 처리 서버 |
| 응답 시간(Response Time) | 요청 후 첫 반응까지의 시간 | 대화형 데스크톱·게임 |
| 반환 시간(Turnaround Time) | 제출부터 완료까지 총 시간 | 일괄 계산 작업 |
| 대기 시간(Waiting Time) | 준비 큐에서 보낸 총 시간 | 공정성 평가 |
| 공정성(Fairness) | 자원이 고르게 분배되는 정도 | 다중 사용자 시스템 |

> **해석** 처리량과 응답 시간은 정면으로 충돌합니다. 처리량을 극대화하려면 컨텍스트 스위칭을 최소화하고 한 작업을 오래 붙잡아야 하는데, 그러면 새로 들어온 대화형 작업이 오래 기다리게 되어 응답 시간이 나빠집니다. 반대로 시간 할당량을 아주 짧게 잡으면 모두가 빠르게 반응하지만, 전환 오버헤드가 전체 처리량을 갉아먹습니다. 이후 살펴볼 알고리즘들은 이 저울 위에서 어느 쪽에 무게를 둘지 선택한 결과물입니다.

## 4. 대표 스케줄링 알고리즘

![주요 스케줄링 알고리즘 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part17_운영체제_핵심/17-2_컨텍스트_스위칭과_스케줄링/assets/diagram2_scheduling_algorithms.svg)

> **해석** 다섯 알고리즘은 "무엇을 기준으로 다음 차례를 정하는가"에서 갈립니다. <u>**FCFS(First-Come, First-Served)**</u>는 도착 순서만 봅니다. 구현이 가장 단순하지만, 앞에 긴 작업이 하나 서 있으면 뒤의 짧은 작업들이 통째로 발목 잡히는 <u>**호위 효과(Convoy Effect)**</u>가 생깁니다. <u>**SJF(Shortest Job First)**</u>는 실행 시간이 짧은 작업을 먼저 처리해 평균 대기 시간을 최소화하며, 이것이 최적임은 수학적으로 증명되어 있습니다. 그러나 각 작업의 실행 시간을 미리 알아야 한다는 치명적 전제 때문에 실제 범용 OS에서는 그대로 쓰이지 못하고, 과거 실행 이력을 지수 이동 평균으로 예측하는 근사 방식으로 응용됩니다. <u>**우선순위 스케줄링(Priority Scheduling)**</u>은 중요도가 높은 작업을 먼저 처리합니다. <u>**라운드 로빈(Round Robin)**</u>은 모든 작업에 동일한 <u>**시간 할당량(Time Quantum)**</u>을 주고 돌아가며 실행해 응답 시간을 균등하게 만듭니다.

$$\text{평균 대기 시간} = \frac{1}{n}\sum_{i=1}^{n} \left( \text{프로세스 } i \text{의 시작 시각} - \text{도착 시각} \right)$$

> **개요** 실제 계산으로 확인해봅시다. 작업 A(24ms), B(3ms), C(3ms)가 시각 0에 동시 도착했다고 가정합니다.

| 알고리즘 | 실행 순서 | 각 작업 대기 시간 | 평균 대기 시간 |
|---|---|---|---|
| FCFS | A → B → C | A=0, B=24, C=27 | $(0+24+27)/3 = 17$ ms |
| SJF | B → C → A | B=0, C=3, A=6 | $(0+3+6)/3 = 3$ ms |
| RR (quantum 4ms) | A→B→C→A→A… | A=6, B=4, C=7 | $(6+4+7)/3 \approx 5.67$ ms |

> **해석** 같은 세 작업인데 순서만 바꿨을 뿐 평균 대기 시간이 17ms에서 3ms로 줄었습니다. 스케줄링이 성능 문제인 이유가 여기 있습니다. SJF가 가장 좋아 보이지만 A(24ms 작업)는 항상 맨 뒤로 밀리며, 짧은 작업이 계속 들어오면 A는 영원히 실행되지 못할 수도 있습니다. 라운드 로빈은 평균에서 SJF에 지지만, 어떤 작업도 4ms 이상 완전히 방치되지 않는다는 응답성 보장을 제공합니다.

## 5. 선점과 비선점, 그리고 다단계 피드백 큐

<u>**비선점형(Non-preemptive)**</u> 스케줄링에서는 프로세스가 스스로 CPU를 놓기 전까지(종료하거나 입출력 대기로 들어가기 전까지) 커널이 뺏지 않습니다. <u>**선점형(Preemptive)**</u> 스케줄링에서는 타이머 인터럽트가 발생하면 커널이 실행 중인 프로세스에게서 CPU를 강제로 회수합니다.

| 구분 | 비선점형 | 선점형 |
|---|---|---|
| CPU 회수 시점 | 프로세스가 자발적으로 놓을 때 | 시간 할당량 소진, 더 높은 우선순위 작업 도착 |
| 컨텍스트 스위칭 빈도 | 낮음 | 높음 |
| 응답성 | 나쁨 — 폭주 프로세스가 시스템을 독점 가능 | 좋음 — 상한이 보장됨 |
| 커널 자료구조 보호 | 상대적으로 단순 | 임계 구역 보호 필수 |
| 대표 사례 | 초기 macOS(협조적 멀티태스킹), 코루틴 | 리눅스, 윈도우 NT 이후 모든 범용 OS |

> **해석** 오늘날 모든 범용 운영체제가 선점형인 이유는 단 하나, **신뢰할 수 없는 프로그램으로부터 시스템을 지키기 위해서**입니다. 비선점형에서는 무한 루프에 빠진 프로그램 하나가 컴퓨터 전체를 멈춰 세울 수 있습니다. 선점형은 하드웨어 타이머라는, 어떤 사용자 코드도 무시할 수 없는 장치에 기대어 커널이 항상 통제권을 되찾을 수 있게 보장합니다.

<u>**다단계 피드백 큐(MLFQ, Multi-Level Feedback Queue)**</u>는 서로 다른 우선순위의 큐를 여러 개 두고, 프로세스의 **행동을 관찰해** 큐 사이를 옮기는 방식입니다. 규칙은 대체로 이렇습니다. 새 프로세스는 가장 높은 우선순위 큐에 들어갑니다. 시간 할당량을 다 쓰고도 끝나지 않으면 한 단계 낮은 큐로 강등됩니다. 반대로 할당량을 다 쓰기 전에 입출력 대기로 들어가면 우선순위를 유지합니다.

> **해석** 이 규칙의 아름다움은 **실행 시간을 미리 물어보지 않고도 SJF에 가까운 효과를 낸다**는 데 있습니다. 짧은 작업은 강등되기 전에 끝나므로 자연히 먼저 처리되고, 키보드 입력을 기다리는 대화형 작업은 늘 입출력으로 빠지므로 높은 우선순위를 유지해 응답이 빠릅니다. 반면 CPU만 붙잡고 도는 계산 작업은 점점 아래 큐로 내려가 배경에서 돌게 됩니다. MLFQ는 1962년 MIT의 CTSS 시스템에서 페르난도 코르바토(Fernando Corbató) 연구진이 도입했으며, 코르바토는 시분할 시스템 연구로 1990년 튜링상을 받았습니다.

## 6. 기아 현상과 에이징

> **Q.** 우선순위 스케줄링을 쓰면 낮은 우선순위 작업은 어떻게 되나요?
> **A.** 높은 우선순위 작업이 끊임없이 들어오는 한, 낮은 우선순위 작업은 영원히 실행되지 못할 수 있습니다. 이를 <u>**기아(Starvation)**</u> 또는 무기한 연기라고 합니다. 이 문제의 고전적 해법이 <u>**에이징(Aging)**</u>으로, 준비 큐에서 오래 기다린 프로세스의 우선순위를 시간이 지남에 따라 조금씩 올려 주는 기법입니다. 예를 들어 15초마다 우선순위를 1씩 올리면, 아무리 낮은 우선순위로 시작한 작업이라도 유한한 시간 안에 반드시 실행 차례를 얻습니다. MLFQ 역시 주기적으로 모든 프로세스를 최상위 큐로 되돌리는 <u>**우선순위 부스팅(Priority Boost)**</u> 규칙을 두어 같은 문제를 해결합니다.

> **참고** 리눅스의 기본 스케줄러였던 <u>**CFS(Completely Fair Scheduler)**</u>는 2007년 커널 2.6.23에서 잉고 몰나르(Ingo Molnár)가 도입했으며, 큐 대신 각 작업의 누적 실행 시간(`vruntime`)을 키로 하는 레드-블랙 트리를 사용해 항상 가장 적게 실행된 작업을 $O(\log n)$에 골라냅니다. "가장 굶주린 작업을 먼저"라는 원리 자체가 기아 방지를 구조적으로 내장한 설계입니다. 참고로 리눅스 6.6부터는 후속 스케줄러인 EEVDF가 기본으로 채택되었습니다.

> **팁** 리눅스에서 `nice` 값(-20~19)을 조정하면 프로세스 우선순위를 바꿀 수 있습니다. 백업 스크립트처럼 급하지 않지만 CPU를 많이 쓰는 작업은 `nice -n 19 ./backup.sh`로 실행해 대화형 작업의 응답성을 지키는 것이 실무의 기본 습관입니다.

## 요약

> **핵심**
> - 컨텍스트 스위칭은 레지스터를 PCB에 저장하고 복원하는 작업이며, 직접 비용(수백 ns)보다 **캐시 오염과 TLB 무효화라는 간접 비용**이 훨씬 큽니다. 스레드 간 전환이 싼 이유는 주소 공간이 유지되어 TLB를 버리지 않기 때문입니다.
> - 스케줄링의 목표는 처리량·응답 시간·공정성 등 여러 개이고 서로 충돌하므로, **FCFS(단순하지만 호위 효과), SJF(평균 대기 최적이지만 실행 시간을 알아야 함), 라운드 로빈(응답성 보장)** 은 각각 다른 선택지입니다.
> - 현대 OS는 타이머 인터럽트에 기반한 **선점형**을 채택해 폭주 프로세스로부터 시스템을 지키고, **다단계 피드백 큐**로 실행 시간을 묻지 않고도 짧은 작업과 대화형 작업을 우대합니다. 그 부작용인 **기아**는 **에이징 또는 우선순위 부스팅**으로 막습니다.
