# Part 8. 프론트엔드 개발

## 8-2. 상태 관리와 컴포넌트 기반 설계 — 재사용 가능한 화면 만들기

> **학습 목표**
> - 화면을 컴포넌트 단위로 쪼개는 설계가 무엇을 재사용하기 위한 것인지 설명할 수 있다.
> - 단방향 데이터 흐름(unidirectional data flow)의 구조와, 이 구조가 양방향 바인딩보다 디버깅에 유리한 이유를 설명할 수 있다.
> - 로컬 상태와 전역 상태를 구분하는 기준을 실제 화면 요소에 적용할 수 있다.

> **개요** 화면을 만드는 방법은 하나의 거대한 HTML 덩어리를 작성하는 것부터, 재사용 가능한 조각(컴포넌트)을 조합하는 것까지 다양합니다. 이 강의는 왜 현업이 후자를 선택했는지, 그리고 그 조각들 사이에서 데이터가 어떤 규칙으로 움직여야 하는지를 다룹니다.

---

## 1. 왜 화면을 컴포넌트로 쪼갤까요?

> **Q.** 상품 카드 하나를 화면에 그리는 데 필요한 HTML·CSS·동작을 그냥 페이지마다 복사해서 쓰면 안 될까요?
> **A.** 안 되는 것은 아니지만, 복사한 코드는 서로 독립된 존재가 됩니다. 상품 카드에 "품절" 배지를 추가해야 한다면, 복사해 둔 모든 페이지를 일일이 찾아 똑같이 고쳐야 합니다. <u>**컴포넌트로 쪼갠다는 것은 화면의 조각(마크업+스타일+동작)을 하나의 정의로 묶어, 그 정의를 여러 곳에서 참조하도록 만드는 것**</u>입니다. 정의를 한 곳에서 고치면 그 컴포넌트를 쓰는 모든 화면에 즉시 반영됩니다. 이는 소프트웨어 공학의 오래된 원칙인 <u>**모듈화(높은 응집도, 낮은 결합도)**</u>를 화면 단위에 적용한 것입니다.

> **참고** 이 아이디어는 프레임워크만의 발명이 아닙니다. W3C Web Components 표준(Custom Elements, Shadow DOM)은 브라우저가 네이티브로 "재사용 가능한 커스텀 태그"를 만들 수 있도록 정의합니다. React·Vue 같은 프레임워크의 컴포넌트 개념은 이 플랫폼 차원의 아이디어를 각자의 방식으로 구현한 것이라고 볼 수 있습니다.

---

## 2. 컴포넌트 트리 구조

> **개요** 컴포넌트는 홀로 존재하지 않고, DOM이 트리인 것처럼 서로 부모-자식 관계로 중첩되어 하나의 트리를 이룹니다. 이 절은 그 트리 구조와, 트리 안에서 "무엇을 재사용하는가"를 다룹니다.

![컴포넌트 트리 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part8_프론트엔드_개발/8-2_상태_관리와_컴포넌트_기반_설계/assets/diagram1_component_tree.svg)

> **해석** `App`이 최상위 부모이고, 그 아래 `Header`, `ProductList`, `Cart`가 자식으로 연결됩니다. 주목할 부분은 `ProductList` 아래 `ProductCard`가 **하나의 정의로 여러 번 인스턴스화**된다는 점입니다. 상품이 30개면 `ProductCard` 정의는 하나인 채로, 그 정의가 30번 서로 다른 데이터(가격, 이름, 이미지)를 받아 렌더링됩니다.

> **Q.** 그러면 정확히 "무엇"을 재사용하는 걸까요? 데이터를 재사용하는 건 아니지 않나요?
> **A.** 맞습니다. 재사용되는 것은 데이터가 아니라 <u>**마크업 구조, 스타일, 이벤트 처리 로직이 결합된 "정의"**</u>입니다. `ProductCard`는 "가격을 어느 위치에 어떤 폰트로 보여줄지", "장바구니 버튼을 눌렀을 때 무슨 일이 일어날지"를 한 번만 정의해 두고, 실제로 보여줄 값(props)만 매번 다르게 주입받습니다. 이는 프로그래밍 언어에서 함수를 정의해 두고 인자만 바꿔가며 여러 번 호출하는 것과 정확히 같은 구조입니다.

## 3. 단방향 데이터 흐름 (Unidirectional Data Flow)

> **개요** 컴포넌트 트리 안에서 데이터가 어느 방향으로 움직여야 하는지에 대한 규칙을 다룹니다. 이 규칙이 왜 대규모 화면에서 디버깅 비용을 줄이는지 살펴봅니다.

![단방향 데이터 흐름 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part8_프론트엔드_개발/8-2_상태_관리와_컴포넌트_기반_설계/assets/diagram2_data_flow.svg)

컴포넌트 기반 프레임워크 대부분이 채택하는 규칙은 다음과 같습니다.

- **데이터는 위에서 아래로만 내려간다 (props)**: 부모가 가진 상태를 자식에게 읽기 전용 값으로 전달합니다.
- **자식은 부모의 상태를 직접 바꾸지 않는다**: 대신 부모가 자식에게 내려준 콜백 함수(이벤트 핸들러)를 호출해, "이 상태를 바꿔달라"고 요청만 합니다.
- **실제 상태 변경은 그 상태를 소유한 컴포넌트에서만 일어난다**: 콜백이 호출되면 부모가 자신의 상태를 갱신하고, 갱신된 값이 다시 props로 자식에게 내려갑니다.

> **원칙** 이 구조를 <u>**단방향 데이터 흐름**</u>이라고 부릅니다. Facebook(현 Meta)이 React와 함께 제안한 Flux 아키텍처 문서는 이 원칙이 필요한 이유를 명확히 밝힙니다. 데이터가 양방향으로 흐르면(자식이 부모 상태를 직접 수정할 수 있으면), 화면에 이상한 값이 보일 때 "어느 컴포넌트가 이 값을 바꿨는지" 추적하려면 트리 전체를 뒤져야 합니다. 반면 단방향 흐름에서는 상태 변경이 항상 그 상태의 소유자 한 곳에서만 일어나므로, 문제가 생겨도 원인을 소유자 컴포넌트로 좁혀 추적할 수 있습니다.

> **참고** 순수 HTML의 `<input>` 요소나 일부 프레임워크의 양방향 바인딩(two-way binding)은 사용자 입력을 곧바로 상태에 반영해 코드량을 줄여주는 편의 기능입니다. 다만 이는 "누가 상태를 바꿨는지"의 추적 지점을 흐리게 만들 수 있으므로, 대규모 애플리케이션에서는 양방향 바인딩도 내부적으로는 단방향 흐름(값 전달 + 변경 이벤트 콜백)으로 풀어서 구현되는 경우가 많습니다.

## 4. 로컬 상태와 전역 상태

> **개요** 컴포넌트가 다루는 데이터를 "그 컴포넌트만 알면 되는 데이터"와 "여러 컴포넌트가 함께 알아야 하는 데이터"로 구분하는 기준을 다룹니다.

모든 상태를 최상위 컴포넌트 하나에 몰아넣으면 관리가 쉬울 것 같지만, 실제로는 그렇지 않습니다. 관련 없는 상태가 바뀔 때마다 최상위 컴포넌트가 다시 렌더링되고, 그 아래 전체 트리가 영향을 받을 수 있기 때문입니다. 그래서 상태는 아래 기준으로 어디에 둘지 구분합니다.

| 구분 | 정의 | 판단 기준 | 예시 |
|---|---|---|---|
| 로컬 상태 (local state) | 하나의 컴포넌트(또는 그 하위 트리) 안에서만 필요한 상태 | 이 값이 바뀌었을 때, 화면의 다른 먼 영역도 함께 바뀌어야 하는가? → 아니오 | 검색창에 입력 중인 텍스트, 드롭다운의 열림/닫힘 여부 |
| 전역 상태 (global state) | 서로 멀리 떨어진 여러 컴포넌트가 동시에 참조해야 하는 상태 | → 예, 트리 곳곳에서 같은 값을 봐야 한다 | 로그인한 사용자 정보, 장바구니에 담긴 상품 개수 |

> **팁** 상태를 어디에 둘지 정할 때는 "이 값이 필요한 컴포넌트들의 가장 가까운 공통 조상이 어디인가"부터 찾는 것이 실무 원칙입니다. React 공식 문서는 이를 "상태 끌어올리기(lifting state up)"라고 부르며, 공통 조상을 넘어서까지 props를 여러 단계 전달해야 한다면(prop drilling) 그때 Context나 별도의 전역 상태 관리 라이브러리(Redux, Zustand 등) 도입을 고려하라고 안내합니다. 처음부터 모든 상태를 전역으로 두면, 로컬로 두었을 때보다 더 많은 컴포넌트가 불필요하게 다시 렌더링됩니다.

### 실생활 예시 — 쇼핑몰 화면의 상태 구분

> **해석** 아래 표는 쇼핑몰 상품 목록 화면을 구성하는 세 가지 요소가 각각 어떤 종류의 상태를 갖는지 보여줍니다.

| 화면 요소 | 다루는 데이터 | 상태 종류 | 이유 |
|---|---|---|---|
| 검색어 입력창 | 사용자가 입력 중인 글자 | 로컬 상태 | 입력창 자신과 그 아래 자동완성 목록만 알면 됨 |
| 상단 내비게이션의 장바구니 아이콘 배지 | 장바구니에 담긴 상품 개수 | 전역 상태 | 상품 목록, 상세 페이지, 헤더 등 트리 곳곳에서 "담기" 버튼을 누르면 헤더의 배지 숫자가 즉시 바뀌어야 함 |
| 상품 카드의 "찜하기" 버튼 눌림 여부 | 이 카드 하나의 하트 아이콘 색 | 로컬 상태(단, 서버에도 저장) | 화면상 즉시 반응은 그 카드 하나에만 영향 |

> **핵심**
> - 컴포넌트로 잘게 쪼개는 설계는 <u>**마크업·스타일·동작이 결합된 "정의"를 재사용**</u>하기 위함이며, 데이터가 아니라 구조를 재사용한다는 점이 핵심입니다.
> - <u>**단방향 데이터 흐름**</u>은 상태 변경이 항상 그 상태의 소유자에서만 일어나도록 강제해, 대규모 화면에서도 문제의 원인을 한곳으로 좁혀 추적할 수 있게 합니다.
> - 상태는 <u>**그 값을 필요로 하는 컴포넌트들의 가장 가까운 공통 조상**</u>에 두는 것이 원칙이며, 이 범위를 기준으로 로컬 상태와 전역 상태를 구분합니다.
