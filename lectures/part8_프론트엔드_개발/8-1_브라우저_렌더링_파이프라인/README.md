# Part 8. 프론트엔드 개발

## 8-1. 브라우저 렌더링 파이프라인 — HTML에서 픽셀까지

> **학습 목표**
> - HTML·CSS·JS가 화면의 픽셀로 바뀌기까지 브라우저가 거치는 6단계(파싱→DOM/CSSOM→렌더 트리→레이아웃→페인트→컴포지트)를 순서대로 설명할 수 있다.
> - 리플로우(reflow)와 리페인트(repaint)가 왜 서로 다른 비용을 가지는지, 어떤 CSS 속성이 각각을 유발하는지 구분할 수 있다.
> - React 등 현업 프레임워크가 왜 렌더링 비용을 줄이는 방향으로 설계되었는지 파이프라인 구조에 근거해 설명할 수 있다.

> **개요** 브라우저는 텍스트로 된 HTML·CSS·JS 파일을 받아, 이를 실제 모니터에 찍히는 픽셀로 바꾸는 일련의 변환 파이프라인을 거칩니다. 이 강의는 그 파이프라인의 각 단계를 W3C·WHATWG 표준과 브라우저 렌더링 엔진(예: Chromium의 Blink) 동작 원리에 근거해 살펴보고, 왜 특정 CSS 속성을 바꾸면 화면이 버벅이는지 그 비용 구조를 다룹니다.

---

## 1. 왜 렌더링 파이프라인을 알아야 할까요?

> **Q.** 프론트엔드 개발자는 React나 Vue 같은 프레임워크의 사용법만 알면 되지 않을까요? 그 아래에서 브라우저가 무엇을 하는지까지 알아야 할까요?
> **A.** 프레임워크는 결국 이 파이프라인 위에서 동작하는 얇은 층일 뿐입니다. "리스트에 아이템을 하나 추가했을 뿐인데 스크롤이 버벅인다", "애니메이션이 끊긴다" 같은 문제는 프레임워크 문법을 아무리 뜯어봐도 원인이 보이지 않습니다. 이런 문제의 원인은 대부분 이 강의에서 다루는 <u>**레이아웃(layout) 재계산 비용**</u>에 있으며, 원인을 정확히 짚으려면 파이프라인의 어느 단계가 다시 실행되었는지를 알아야 합니다.

## 2. 전체 파이프라인 한눈에 보기

> **개요** 브라우저가 HTML·CSS 텍스트를 픽셀로 바꾸는 과정은 크게 6단계로 나뉩니다. 이후 절에서 각 단계를 자세히 다루기 전에, 전체 흐름을 먼저 조망합니다.

![브라우저 렌더링 파이프라인 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part8_프론트엔드_개발/8-1_브라우저_렌더링_파이프라인/assets/diagram1_pipeline.svg)

> **해석** HTML 파싱과 CSS 파싱은 서로 독립적인 트리(DOM, CSSOM)를 만들며 병렬로 진행됩니다. 이 두 트리가 합쳐져야만 "실제로 화면에 무엇을 그릴지"를 결정하는 렌더 트리가 만들어지고, 이후 레이아웃 → 페인트 → 컴포지트 순서로 이어집니다. 화살표 방향이 곧 데이터가 반드시 지나야 하는 순서이며, 이 순서를 건너뛸 수 없다는 점이 이후 성능 최적화(8-3강)의 근거가 됩니다.

이 여섯 단계는 WHATWG HTML Living Standard의 파싱 알고리즘과 W3C CSSOM(CSS Object Model) 명세, 그리고 각 브라우저 렌더링 엔진(Blink, Gecko, WebKit)의 렌더링 파이프라인 문서가 공통으로 기술하는 구조입니다. 엔진마다 세부 구현은 다르지만, 이 6단계의 순서와 존재 이유는 동일합니다.

## 3. HTML 파싱과 CSS 파싱 — DOM과 CSSOM

> **개요** 브라우저가 텍스트 파일 두 개(HTML, CSS)를 각각 어떤 자료구조로 바꾸는지, 그리고 그 과정에서 자바스크립트가 왜 끼어들 수 있는지를 다룹니다.

### 3.1 HTML 파싱 → DOM

브라우저는 서버에서 받은 HTML 바이트 스트림을 다음 순서로 처리합니다.

1. **바이트 → 문자**: 지정된 인코딩(예: UTF-8)에 따라 바이트를 문자로 변환합니다.
2. **문자 → 토큰**: `<`, `div`, `>` 같은 문자열을 HTML 토크나이저가 `StartTag`, `EndTag` 같은 토큰으로 분리합니다.
3. **토큰 → 노드 → DOM**: 토큰을 객체(노드)로 만들고, 태그의 중첩 관계에 따라 트리 구조(Document Object Model)로 연결합니다.

> **참고** 이 트리 구조를 DOM이라고 부르는 이유는, 이것이 단순한 데이터 구조가 아니라 자바스크립트가 `document.querySelector()` 등으로 실시간으로 읽고 쓸 수 있는 **살아있는 객체 모델**이기 때문입니다. W3C DOM 명세가 이 상호작용 인터페이스를 정의합니다.

### 3.2 CSS 파싱 → CSSOM

CSS도 동일한 원리로 처리되지만 결과물은 다릅니다. CSS 텍스트는 토큰화된 뒤 **CSSOM(CSS Object Model)** 이라는 별도의 트리로 구성됩니다. CSSOM은 각 노드가 최종적으로 어떤 스타일 값을 가지는지(상속과 캐스케이딩을 모두 계산한 결과)를 담고 있는 트리입니다.

> **Q.** DOM은 한 번 훑고 그리기 시작하면서, 왜 CSSOM은 스타일시트를 끝까지 다 읽어야 화면을 그릴 수 있을까요?
> **A.** CSS는 <u>**나중에 나온 규칙이 앞선 규칙을 덮어쓸 수 있는(캐스케이딩)**</u> 언어이기 때문입니다. 스타일시트 중간까지만 읽고 화면을 그리면, 이후 등장하는 규칙 때문에 이미 그린 화면이 틀린 상태가 됩니다. 그래서 CSSOM은 **렌더 차단(render-blocking)** 리소스로 취급되며, 브라우저는 CSSOM 생성이 끝나기 전까지 화면을 그리지 않습니다.

> **참고** 자바스크립트의 `<script>` 태그는 기본적으로 **동기적으로 실행**되며, 실행되는 동안 HTML 파서를 일시 정지시킵니다. 스크립트가 `document.write()`처럼 DOM 구조 자체를 바꿀 가능성이 있기 때문입니다. `async`나 `defer` 속성을 붙이면 파서를 멈추지 않고 다운로드·실행 시점만 조정할 수 있습니다. 이것이 실무에서 `<script>` 태그를 `</body>` 직전에 두거나 `defer`를 쓰라고 권장하는 이유입니다.

## 4. 렌더 트리 결합과 레이아웃(Layout)

> **개요** DOM과 CSSOM이 합쳐져 "실제로 그릴 것"만 남은 렌더 트리가 되는 과정과, 그 트리의 각 요소가 화면 위 정확히 어디에 몇 픽셀 크기로 위치할지 계산하는 레이아웃 단계를 다룹니다.

렌더 트리(Render Tree)는 DOM과 CSSOM을 결합해 만들어지는데, 이때 두 가지 종류의 노드는 제외됩니다.

- `<head>`, `<script>`처럼 애초에 화면에 그려지지 않는 노드
- `display: none`이 적용된 노드 (단, `visibility: hidden`은 공간은 차지하므로 렌더 트리에 남습니다)

렌더 트리가 만들어지면, 브라우저는 **레이아웃(Layout, 또는 리플로우reflow)** 단계에서 각 노드가 뷰포트 안에서 정확히 몇 픽셀의 위치와 크기를 가지는지 계산합니다. 같은 `<div>`라도 뷰포트 너비가 360px인 스마트폰과 1920px인 모니터에서는 다른 위치·크기 값을 갖게 되므로, 이 계산은 실제 화면(또는 화면 크기 정보)이 있어야만 수행할 수 있습니다.

> **원칙** 레이아웃은 **한 노드의 크기 변화가 부모·형제·자식 노드의 위치에 연쇄적으로 영향을 줄 수 있다**는 전제 위에 설계되어 있습니다. 그래서 `width`, `height`, `top`, `left`, `font-size`, `display`처럼 <u>**기하(geometry)에 관여하는 속성**</u>이 바뀌면, 브라우저는 영향을 받는 범위 전체의 레이아웃을 다시 계산해야 합니다. 이것이 "리플로우"라는 이름이 붙은 이유이며, 6단계 중 가장 비용이 큰 단계입니다.

## 5. 페인트(Paint)와 컴포지트(Composite) — 리플로우와 리페인트의 비용 차이

> **개요** 레이아웃이 끝난 요소에 실제 색과 픽셀을 채우는 페인트 단계, 그리고 여러 레이어를 합성해 최종 화면을 만드는 컴포지트 단계를 다루고, 세 종류의 CSS 변경이 왜 서로 다른 비용을 유발하는지 비교합니다.

- **페인트(Paint)**: 레이아웃이 확정된 각 노드에 배경색, 테두리, 그림자, 텍스트 같은 실제 픽셀 값을 채워 넣는 단계입니다. 위치·크기는 이미 정해졌으므로, 색상만 바뀌는 경우에는 레이아웃을 다시 계산할 필요가 없습니다.
- **컴포지트(Composite)**: 브라우저는 성능을 위해 일부 요소를 별도의 **레이어(layer)** 로 분리해 GPU에서 독립적으로 처리합니다. 컴포지트 단계는 이 레이어들을 화면에 보일 순서대로 겹쳐 최종 이미지를 완성하는 단계로, 별도의 컴포지터 스레드(compositor thread)에서 메인 스레드와 분리되어 실행될 수 있습니다.

![리플로우·리페인트·컴포지트 비용 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part8_프론트엔드_개발/8-1_브라우저_렌더링_파이프라인/assets/diagram2_cost.svg)

> **해석** 세 막대는 같은 "화면이 바뀐다"는 결과를 만들어내는 세 가지 변경이, 파이프라인상에서 얼마나 뒤(왼쪽) 단계부터 다시 실행해야 하는지를 색의 짙기로 나타냅니다. 색이 짙을수록 더 이른 단계(레이아웃)부터 다시 실행해야 하므로 비용이 큽니다.

| 바꾸는 CSS 속성 예시 | 다시 실행되는 단계 | 비용 |
|---|---|---|
| `width`, `height`, `top`, `left`, `display`, `font-size` | 레이아웃 → 페인트 → 컴포지트 전체 | 가장 큼 (리플로우) |
| `background-color`, `color`, `box-shadow`, `visibility` | 페인트 → 컴포지트 | 보통 (리페인트) |
| `transform`, `opacity` | 컴포지트만 | 가장 작음 |

> **팁** `transform`과 `opacity`만으로 애니메이션을 구현하면 레이아웃과 페인트를 건너뛰고 컴포지트 단계만 GPU에서 처리되기 때문에 메인 스레드의 자바스크립트 실행과 충돌하지 않고 부드럽게 움직입니다. `will-change: transform` 같은 힌트를 미리 주면 브라우저가 해당 요소를 별도 레이어로 미리 승격시켜 둘 수 있습니다. 반대로 애니메이션에 `top`, `left`, `width`를 쓰면 매 프레임마다 레이아웃 전체를 다시 계산하므로 프레임이 떨어지기 쉽습니다.

> **참고** React 같은 프레임워크가 <u>**가상 DOM(virtual DOM)과 배치(batching) 업데이트**</u>를 쓰는 이유도 이 비용 구조 때문입니다. 상태가 바뀔 때마다 실제 DOM을 그때그때 수정하면 매번 레이아웃·페인트가 발생할 수 있지만, 여러 상태 변경을 한 번의 렌더 사이클로 묶어 실제 DOM에는 최종 결과만 한 번에 반영하면 불필요한 리플로우 횟수를 줄일 수 있습니다. 이는 프레임워크가 브라우저 파이프라인의 물리적 제약을 우회하는 것이 아니라, 그 제약에 맞춰 호출 횟수를 최소화하도록 설계된 것입니다.

## 6. 실생활 예시 — 무한 스크롤 상품 목록에서 화면이 버벅이는 이유

> **해석** 아래 표는 쇼핑몰 상품 목록 페이지에서 스크롤을 내릴 때마다 상품 카드가 추가로 로드되는 "무한 스크롤" 기능이, 구현 방식에 따라 왜 버벅임이 다른지를 파이프라인 관점에서 비교합니다.

| 구현 방식 | 실제로 벌어지는 일 | 파이프라인 영향 |
|---|---|---|
| 상품 카드의 이미지 높이가 로드 전까지 미확정(`height` 미지정) | 이미지가 로드될 때마다 카드 높이가 바뀌고, 그 아래 모든 카드 위치가 밀림 | 매 이미지 로드마다 레이아웃 전체 재계산 (리플로우 반복) |
| 카드에 `aspect-ratio`나 고정 `height`를 미리 지정 | 이미지 로드 전에도 카드 공간이 확보되어 있어 위치가 바뀌지 않음 | 이미지 로드 시 페인트만 발생, 레이아웃은 재계산되지 않음 |
| 스크롤에 따라 카드에 페이드인 애니메이션 적용 시 `opacity`만 사용 | 등장 효과가 컴포지트 단계에서만 처리됨 | 메인 스레드 부담 없이 GPU에서 처리 |

> **핵심**
> - 브라우저는 HTML·CSS를 <u>**파싱 → DOM/CSSOM 생성 → 렌더 트리 결합 → 레이아웃 → 페인트 → 컴포지트**</u>의 고정된 순서로 처리하며, 이 순서는 W3C·WHATWG 표준과 렌더링 엔진 구현이 공통으로 따르는 구조입니다.
> - <u>**리플로우(레이아웃 재계산)는 리페인트보다 항상 비용이 크며, 컴포지트만 다시 실행되는 경우가 가장 저렴합니다.**</u> `width`·`top` 같은 기하 속성 대신 `transform`·`opacity`를 쓰면 이 비용 차이를 실제로 줄일 수 있습니다.
> - React의 가상 DOM과 배치 업데이트는 마법이 아니라, 이 파이프라인의 물리적 비용 구조를 이해하고 실제 DOM 조작 횟수를 줄이도록 설계된 결과입니다.
