# Part 8. 프론트엔드 개발

## 8-3. 프론트엔드 성능 최적화 — 체감 속도를 만드는 것들

> **학습 목표**
> - 코드 스플리팅과 지연 로딩이 "무엇을 언제 내려받는가"를 바꿔 체감 로딩 속도를 개선하는 원리를 설명할 수 있다.
> - 불필요한 리렌더링이 왜 비용인지, 8-1강의 렌더링 파이프라인 지식에 근거해 설명할 수 있다.
> - Core Web Vitals의 세 지표(LCP, INP, CLS)가 각각 무엇을 측정하는지 말할 수 있다.

> **개요** 같은 정보를 보여주는 두 웹페이지라도 사용자가 느끼는 속도는 크게 다를 수 있습니다. 이 강의는 실제 전송량이나 연산량을 줄이는 기법(코드 스플리팅, 지연 로딩, 리렌더링 최소화)과, 그 결과를 정량적으로 측정하는 기준(Core Web Vitals)을 다룹니다.

---

## 1. 왜 "로딩 속도"는 하나의 숫자가 아닐까요?

> **Q.** 페이지가 다 로드되는 데 걸리는 시간을 하나의 숫자로 재면 되지 않을까요?
> **A.** "다 로드된 시점"이 사용자가 느끼는 시점과 다르기 때문입니다. 전체 리소스가 0.1초 늦게 완전히 끝나더라도, 사용자 눈에 <u>**가장 먼저 보이는 콘텐츠(예: 상품 이미지)가 빨리 뜨면 체감 속도는 빠르게 느껴집니다.**</u> 반대로 전체 로드는 빨리 끝나도 화면이 계속 깜빡이며 버튼 위치가 바뀌면 "느리고 불안정하다"고 느낍니다. 그래서 실무에서는 하나의 총 로딩 시간 대신, <u>**사용자가 실제로 경험하는 순간들을 각각 다른 지표로 나눠 측정**</u>합니다. 이 지표들이 이 강의 마지막에 다룰 Core Web Vitals입니다.

## 2. 번들 크기와 코드 스플리팅

> **개요** 프론트엔드 코드가 왜 하나의 거대한 파일로 뭉쳐지는지, 그리고 이를 필요한 시점에만 나눠 내려받게 하는 코드 스플리팅의 원리를 다룹니다.

React·Vue 같은 프레임워크로 만든 애플리케이션은 보통 Webpack, Vite 같은 번들러(bundler)를 통해 여러 개의 자바스크립트 파일을 하나(또는 소수)의 파일로 합칩니다. 파일 하나로 합치는 이유는 브라우저가 파일마다 별도의 HTTP 요청을 보내는 오버헤드를 줄이기 위함입니다. 하지만 애플리케이션이 커질수록 이 하나의 번들 파일도 커지고, 사용자는 "지금 보는 화면에 필요 없는 코드"까지 포함된 파일을 전부 내려받은 뒤에야 화면을 볼 수 있습니다.

![코드 스플리팅 전후 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part8_프론트엔드_개발/8-3_프론트엔드_성능_최적화/assets/diagram1_code_splitting.svg)

> **해석** 위쪽은 결제 페이지, 관리자 페이지, 리뷰 작성 페이지의 코드가 모두 하나의 번들에 뭉쳐 있어, 사용자가 상품 목록만 보려 해도 전체를 내려받아야 하는 상태를 나타냅니다. 아래쪽은 <u>**코드 스플리팅(code splitting)**</u>을 적용해 경로(라우트)별로 번들을 나눈 상태로, 상품 목록 페이지에 접속한 사용자는 그에 필요한 조각만 내려받습니다.

> **원칙** 코드 스플리팅은 자바스크립트 표준의 **동적 임포트(dynamic import, `import()`)** 문법을 기반으로 합니다. 정적 `import` 문은 파일을 빌드 시점에 하나로 합치지만, 동적 `import()`는 함수처럼 호출되는 시점에 해당 모듈을 별도 파일로 내려받도록 번들러에게 지시합니다. React의 `React.lazy()`나 Vue의 비동기 컴포넌트는 모두 이 표준 문법 위에 만들어진 편의 기능입니다.

## 3. 지연 로딩(Lazy Loading)

> **개요** 코드뿐 아니라 이미지·비디오 같은 리소스도 "지금 화면에 보이지 않는다면 아직 받지 않는다"는 원칙을 적용할 수 있습니다. 이를 가능하게 하는 브라우저 표준 기능을 다룹니다.

지연 로딩은 뷰포트(사용자에게 실제로 보이는 화면 영역) 밖에 있는 리소스의 다운로드를 뒤로 미루는 기법입니다. 이미지에 대해서는 두 가지 표준적인 방법이 있습니다.

- **`<img loading="lazy">`**: HTML 표준 속성으로, 브라우저가 자체적으로 이미지가 뷰포트에 가까워질 때까지 다운로드를 미룹니다.
- **Intersection Observer API**: W3C 표준 API로, 특정 요소가 뷰포트(또는 지정한 영역)에 들어오고 나가는 시점을 콜백으로 감지합니다. `loading="lazy"`보다 더 세밀한 제어(예: 몇 픽셀 전에 미리 로드를 시작할지)가 필요할 때 사용합니다.

> **참고** Intersection Observer 이전에는 `scroll` 이벤트마다 각 요소의 위치를 `getBoundingClientRect()`로 직접 계산해 뷰포트 진입 여부를 확인했습니다. 이 방식은 스크롤이 일어날 때마다 레이아웃 정보를 강제로 다시 읽어오는 **강제 동기 레이아웃(forced synchronous layout)** 을 유발해 8-1강에서 다룬 리플로우 비용을 반복적으로 발생시켰습니다. Intersection Observer는 이 계산을 브라우저 엔진 내부에서 비동기적으로 처리하도록 표준화해 이 문제를 해결합니다.

### 실생활 예시 — 쇼핑몰 상품 목록 페이지의 이미지 지연 로딩

> **해석** 아래 표는 상품 300개가 나열된 목록 페이지에서, 이미지 로딩 방식에 따라 사용자가 체감하는 과정이 어떻게 달라지는지 비교합니다.

| 구현 방식 | 첫 화면 진입 시 벌어지는 일 | 체감 속도 |
|---|---|---|
| 모든 이미지를 한 번에 로드 | 화면에 보이지 않는 270여 개 이미지까지 동시에 네트워크 요청이 발생, 실제로 보이는 30개 이미지도 대역폭을 나눠 써서 늦게 뜸 | 첫 화면조차 늦게 완성됨 |
| `loading="lazy"` 적용 | 처음 보이는 상품 30개 이미지만 즉시 요청, 나머지는 스크롤해서 가까워질 때 요청 | 첫 화면이 빠르게 채워지고, 스크롤에 맞춰 자연스럽게 이어짐 |
| 지연 로딩 + 이미지 자리에 `aspect-ratio`로 고정 크기 확보 | 이미지 도착 전에도 카드 크기가 흔들리지 않음 | 8-1강에서 다룬 레이아웃 밀림(리플로우 반복) 현상까지 함께 방지 |

## 4. 리렌더링 최소화

> **개요** 프레임워크가 상태 변화를 화면에 반영하는 과정에서, 실제로는 바뀌지 않은 부분까지 다시 계산하는 낭비가 왜 발생하는지, 그리고 이를 줄이는 방법을 다룹니다.

React 같은 프레임워크는 상태가 바뀌면 해당 컴포넌트와 그 자식 트리 전체를 다시 렌더링(가상 DOM을 다시 계산)하는 것이 기본 동작입니다. 가상 DOM 계산 자체는 실제 DOM 조작보다 가볍지만, 트리가 커지면 이 계산도 공짜가 아닙니다. 더 큰 문제는 가상 DOM 비교(diffing) 결과 실제로 DOM이 바뀌어야 한다고 판단되면, 결국 8-1강에서 다룬 레이아웃·페인트·컴포지트가 실제로 실행된다는 점입니다.

> **Q.** 상태가 바뀐 컴포넌트만 다시 그리면 되지, 왜 자식들까지 전부 다시 계산할까요?
> **A.** 부모의 상태가 자식에게 props로 전달되는 구조(8-2강)이기 때문에, 프레임워크 입장에서는 "부모가 바뀌었으니 자식에게 내려가는 props도 바뀌었을 가능성이 있다"고 보수적으로 가정하는 것이 기본값입니다. 이 가정 자체는 안전하지만, 실제로는 바뀌지 않은 자식까지 매번 다시 계산하는 비용을 치릅니다.

이 낭비를 줄이는 대표적인 방법은 다음과 같습니다.

- **컴포넌트 메모이제이션** (`React.memo` 등): props가 실제로 바뀌지 않았다면 해당 컴포넌트의 재계산을 건너뜁니다.
- **값·함수 메모이제이션** (`useMemo`, `useCallback` 등): 매 렌더링마다 새로 생성되는 객체·함수 때문에 자식이 "props가 바뀌었다"고 오판하는 것을 방지합니다.
- **리스트의 `key` 속성**: 배열을 렌더링할 때 각 항목에 안정적인 `key`를 지정하지 않으면, 프레임워크가 항목 순서 변경을 "전부 삭제 후 재생성"으로 오인해 불필요한 DOM 조작이 늘어납니다.

> **팁** 메모이제이션은 공짜가 아닙니다. 이전 값과 새 값을 비교하는 연산 자체에도 비용이 들기 때문에, 렌더링 비용이 이미 작은 컴포넌트에 무분별하게 적용하면 오히려 손해일 수 있습니다. 리스트처럼 반복되거나, 렌더링 비용이 눈에 띄게 큰 컴포넌트에 선택적으로 적용하는 것이 원칙입니다.

## 5. Core Web Vitals — 체감 속도를 측정하는 표준 지표

> **개요** Google이 제안하고 W3C Web Performance Working Group의 표준 API 위에서 측정되는 세 가지 지표를 통해, 지금까지 다룬 최적화 기법들이 실제로 무엇을 개선하는지 정리합니다.

![Core Web Vitals 세 지표 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part8_프론트엔드_개발/8-3_프론트엔드_성능_최적화/assets/diagram2_core_web_vitals.svg)

- **LCP (Largest Contentful Paint)**: 뷰포트 안에서 가장 큰 콘텐츠 요소(대개 대표 이미지나 제목)가 화면에 그려지기까지 걸린 시간입니다. "핵심 콘텐츠가 눈에 보이기까지 얼마나 걸리는가"를 측정하며, 코드 스플리팅과 지연 로딩이 직접적으로 개선하는 지표입니다.
- **INP (Interaction to Next Paint)**: 사용자가 클릭·탭 등으로 상호작용한 뒤, 그 반응이 화면에 그려지기까지 걸린 시간입니다. 리렌더링 비용이 크면 이 지표가 나빠지며, 예전에는 첫 입력 지연만 측정하는 FID(First Input Delay)가 쓰였으나 모든 상호작용을 포괄하도록 INP로 대체되었습니다.
- **CLS (Cumulative Layout Shift)**: 페이지가 로드되는 동안 요소들이 예상치 못하게 위치를 이동한 총량입니다. 이미지 자리에 크기를 미리 확보하지 않아 발생하는 레이아웃 밀림(8-1강, 8-3강 2절 표 참고)이 이 지표를 악화시킵니다.

> **참고** 세 지표 모두 W3C Web Performance Working Group이 정의한 `PerformanceObserver`, `LargestContentfulPaint`, `LayoutShift` 등의 브라우저 표준 API를 통해 실제 사용자 브라우저에서 측정할 수 있습니다. Google은 각 지표에 대해 "양호(good)" 구간의 기준값을 공개하고 있으며(LCP는 일정 초 이내, INP는 일정 밀리초 이내, CLS는 점수 0.1 이하), 이 기준값은 실제 서비스 개선 목표로 널리 쓰입니다. 구체적인 기준값은 측정 방법론과 함께 계속 갱신되므로, 최신 수치는 web.dev의 공식 문서를 확인하는 것을 권장합니다.

> **핵심**
> - 코드 스플리팅과 지연 로딩은 <u>**"지금 화면에 필요한 것만 먼저 받는다"**</u>는 하나의 원칙을 각각 자바스크립트 코드와 이미지·리소스에 적용한 것입니다.
> - 리렌더링 최소화는 프레임워크의 보수적인 기본 동작(부모가 바뀌면 자식도 다시 계산)이 낳는 낭비를, <u>**실제로 값이 바뀌었는지 확인한 뒤에만 계산하도록**</u> 줄이는 작업입니다.
> - Core Web Vitals의 <u>**LCP·INP·CLS는 각각 "보이는 속도", "반응하는 속도", "안정감"을 측정**</u>하며, 이 강의에서 다룬 최적화 기법들은 하나의 총 로딩 시간이 아니라 이 세 축 중 하나를 겨냥해 이뤄집니다.

실습은 `practice.md`를 참고하세요.
