# Part 16. 알고리즘 심화

## 16-3. 탐색 — 선형 탐색과 이진 탐색 — 왜 이진 탐색은 짜기 어려운가

> **학습 목표**
> - 선형 탐색과 이진 탐색의 전제 조건과 복잡도 차이를 설명하고, 정렬 비용까지 포함한 손익분기점을 계산할 수 있다.
> - 오버플로와 경계 조건을 피한 이진 탐색을 정확히 구현하고, lower_bound와 upper_bound를 구분해 작성할 수 있다.
> - 최적화 문제를 결정 문제로 바꿔 답 자체를 이진 탐색하는 파라메트릭 서치를 적용할 수 있다.

> **개요** 이진 탐색은 아이디어가 세 줄이면 설명되는데도 정확히 구현하기로 악명 높은 알고리즘입니다. 1946년에 처음 언급되었지만 버그 없는 구현이 발표된 것은 1962년이었고, 자바 표준 라이브러리의 이진 탐색에서 오버플로 버그가 발견된 것은 2006년입니다. 이 강의는 그 함정들을 하나씩 짚고, "정렬된 배열에서 값 찾기"를 훌쩍 넘어서는 이진 탐색의 진짜 쓰임인 파라메트릭 서치까지 나아갑니다.

---

## 1. 선형 탐색 — 언제 이것이 정답인가

> **Q.** 이진 탐색이 $O(\log n)$인데 왜 아직도 선형 탐색을 쓸까요?
> **A.** 이진 탐색은 **정렬되어 있어야 한다**는 전제를 요구하고, 그 정렬 자체가 $O(n \log n)$이기 때문입니다. 한 번만 찾을 것이라면 정렬($n \log n$) + 이진 탐색($\log n$)보다 그냥 선형 탐색($n$)이 압도적으로 빠릅니다. 또 선형 탐색은 임의 접근이 불가능한 연결 리스트나 스트림에서도 동작하고, 상수 계수가 극도로 작아 캐시에 잘 맞는 작은 배열에서는 실측이 더 빠른 경우도 흔합니다.

<u>**선형 탐색(linear search)**</u>은 처음부터 끝까지 순서대로 비교합니다. 최선 $\Theta(1)$, 평균·최악 $\Theta(n)$입니다.

정렬 후 이진 탐색이 이득이 되는 지점을 계산해 봅시다. 탐색을 $q$번 반복한다면,

$$\underbrace{n \log n + q \log n}_{\text{정렬 + 이진 탐색}} \ < \ \underbrace{q \cdot n}_{\text{선형 탐색}}$$

> **해석** $q$가 충분히 크면 오른쪽이 훨씬 빨리 증가하므로 정렬이 이깁니다. 대략 $q \gtrsim \log n$ 정도만 되어도 정렬에 투자하는 편이 유리해집니다. 즉 **정렬은 반복될 탐색을 위한 선투자**입니다. 한 번 쓰고 버릴 데이터라면 투자하지 마세요.

> **참고** 데이터가 계속 바뀐다면 정렬 상태를 유지하는 비용도 따져야 합니다. 삽입이 잦다면 정렬 배열보다 균형 이진 탐색 트리나 해시 테이블이 유리합니다. 자료구조 자체의 선택 기준은 Part 15를 참고하세요.

## 2. 이진 탐색의 정확한 구현 — 세 가지 함정

> **개요** 이진 탐색은 불변식(invariant)을 명확히 정하지 않으면 반드시 틀립니다. 대표적인 세 함정을 봅니다.

**함정 1 — 정수 오버플로.** 중간 인덱스를 `(lo + hi) / 2`로 계산하면 `lo`와 `hi`가 모두 클 때 합이 자료형 범위를 넘습니다. 2006년 조슈아 블로크(Joshua Bloch)가 Google 블로그에서 이 버그가 JDK의 `Arrays.binarySearch`에 20년 가까이 숨어 있었다고 공개해 널리 알려졌습니다.

```python
mid = (lo + hi) // 2        # 위험: lo + hi가 오버플로할 수 있음
mid = lo + (hi - lo) // 2   # 안전: 차이는 항상 범위 안
```

**함정 2 — 무한 루프.** 구간을 줄일 때 `hi = mid`처럼 `mid`를 그대로 남기면, `lo`와 `hi`가 인접했을 때 `mid == lo`가 되어 구간이 줄지 않습니다. 반드시 **매 반복마다 구간이 최소 1 이상 줄어드는지** 확인해야 합니다.

**함정 3 — 구간 표현의 혼용.** `[lo, hi]`(닫힌 구간)와 `[lo, hi)`(반열린 구간) 중 하나를 정하고 끝까지 일관되게 써야 합니다. 종료 조건과 갱신식이 구간 표현에 종속되기 때문입니다.

```python
def binary_search(a, target):
    lo, hi = 0, len(a) - 1          # 닫힌 구간 [lo, hi]
    while lo <= hi:                  # 구간이 비지 않은 동안
        mid = lo + (hi - lo) // 2
        if a[mid] == target:
            return mid
        elif a[mid] < target:
            lo = mid + 1             # mid를 제외해야 구간이 반드시 줄어듦
        else:
            hi = mid - 1
    return -1
```

![이진 탐색의 구간 축소와 경계 조건](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part16_알고리즘_심화/16-3_탐색_선형_탐색과_이진_탐색/assets/diagram1_binary_search_steps.svg)

> **해석** 그림은 16개 원소에서 값을 찾는 과정입니다. 매 단계 구간이 절반으로 줄어 4번 만에 끝납니다. 일반적으로 $n$개에서 필요한 비교 횟수는 $\lfloor \log_2 n \rfloor + 1$입니다. 아래쪽 표시는 `lo = mid + 1`과 `hi = mid - 1`에서 왜 $\pm 1$이 필요한지 보여줍니다. `mid`는 이미 검사해서 정답이 아님이 확인되었으므로 다음 구간에서 제외해야 하고, 이 제외가 곧 **종료 보장**입니다.

> **팁** 이진 탐색을 짤 때는 코드를 먼저 쓰지 말고 "루프 불변식"을 한 문장으로 적으세요. 예: "정답이 존재한다면 반드시 `[lo, hi]` 안에 있다." 이 문장이 모든 갱신식 뒤에도 참인지 확인하면 경계 버그가 거의 사라집니다.

## 3. lower_bound와 upper_bound — 값이 없거나 여러 개일 때

> **Q.** 정렬된 배열에 같은 값이 여러 개 있으면 이진 탐색은 그중 어느 것을 돌려줄까요?
> **A.** **정해져 있지 않습니다.** 위의 기본 구현은 우연히 먼저 만난 인덱스를 돌려줍니다. 그래서 실무에서는 값 하나를 찾는 함수보다 "경계"를 찾는 두 함수를 훨씬 자주 씁니다.

- <u>**lower_bound**</u>: `target` **이상**인 첫 원소의 인덱스. 즉 `target`을 삽입해도 정렬이 유지되는 가장 왼쪽 위치.
- <u>**upper_bound**</u>: `target`을 **초과**하는 첫 원소의 인덱스. 즉 삽입 가능한 가장 오른쪽 위치.

```python
def lower_bound(a, target):
    lo, hi = 0, len(a)               # 반열린 구간 [lo, hi)
    while lo < hi:
        mid = lo + (hi - lo) // 2
        if a[mid] < target:          # mid는 조건 미달 → 버림
            lo = mid + 1
        else:                        # mid가 후보 → 남김
            hi = mid
    return lo

def upper_bound(a, target):
    lo, hi = 0, len(a)
    while lo < hi:
        mid = lo + (hi - lo) // 2
        if a[mid] <= target:         # 부등호만 다름
            lo = mid + 1
        else:
            hi = mid
    return lo
```

> **해석** 두 함수의 차이는 부등호 `<`와 `<=` 하나뿐입니다. 그리고 이 구현에서는 `hi = mid`(빼기 1 없음)가 올바른데, 반열린 구간이라 `hi`는 애초에 "포함되지 않는 경계"이기 때문입니다. 앞 절의 함정 3이 말한 "구간 표현을 정하고 일관되게"가 실제로 어떤 차이를 만드는지 보여주는 예입니다.

| 표현 | Python | C++ | Java |
|---|---|---|---|
| lower_bound | `bisect.bisect_left` | `std::lower_bound` | (직접 구현) |
| upper_bound | `bisect.bisect_right` | `std::upper_bound` | (직접 구현) |

이 둘로 파생되는 유용한 연산들이 있습니다.

- **`target`의 개수**: `upper_bound(a, t) - lower_bound(a, t)`
- **존재 여부**: `lower_bound(a, t) < len(a) and a[lower_bound(a, t)] == t`
- **$t$ 미만인 원소 수**: `lower_bound(a, t)`
- **정렬 유지 삽입 위치**: `lower_bound` 또는 `upper_bound` 결과

> **참고** Python의 `bisect.insort`는 `bisect_left`로 위치를 찾은 뒤 리스트에 삽입합니다. 위치 찾기는 $O(\log n)$이지만 삽입 자체가 $O(n)$이므로 전체는 $O(n)$입니다. "이진 탐색을 썼으니 $O(\log n)$이겠지"라고 넘겨짚는 것이 흔한 실수입니다.

## 4. 파라메트릭 서치 — 최적화를 결정 문제로 바꾸기

> **개요** 이진 탐색의 진짜 위력은 배열이 아니라 **답의 후보 범위** 위에서 탐색할 때 드러납니다.

<u>**파라메트릭 서치(parametric search)**</u>는 "최댓값/최솟값을 구하라"는 최적화 문제를 "값 $x$가 가능한가?"라는 예/아니오 <u>**결정 문제(decision problem)**</u>로 바꾼 뒤, 그 답 $x$를 이진 탐색하는 기법입니다.

적용 조건은 딱 하나입니다. 결정 함수 $f(x)$가 <u>**단조성(monotonicity)**</u>을 가져야 합니다. 즉 어떤 경계 $x^*$가 있어서 그보다 작으면 전부 참, 크면 전부 거짓(또는 그 반대)이어야 합니다.

![파라메트릭 서치 — 답 위에서 이진 탐색하기](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part16_알고리즘_심화/16-3_탐색_선형_탐색과_이진_탐색/assets/diagram2_parametric_search.svg)

> **해석** 그림의 아래쪽 띠가 결정 함수 $f(x)$의 값입니다. 왼쪽은 전부 "가능", 오른쪽은 전부 "불가능"으로 딱 한 번만 바뀝니다. 이 경계가 곧 우리가 찾는 최적해이고, 정렬된 배열에서 lower_bound를 찾는 것과 **구조적으로 완전히 같은 문제**입니다. 다만 배열 원소를 읽는 대신 $f(x)$를 계산하므로, 전체 복잡도는 $O(f\text{의 비용} \times \log(\text{범위}))$가 됩니다.

**예제 — 랜선 자르기.** 길이가 제각각인 랜선 $N$개를 잘라 같은 길이의 랜선 $K$개를 만들 때, 가능한 최대 길이는?

```python
def can_make(lines, length, k):
    if length == 0:
        return True
    return sum(x // length for x in lines) >= k

def max_length(lines, k):
    lo, hi = 1, max(lines)            # 답의 후보 범위
    ans = 0
    while lo <= hi:
        mid = lo + (hi - lo) // 2
        if can_make(lines, mid, k):   # 가능하면 더 길게 시도
            ans = mid
            lo = mid + 1
        else:
            hi = mid - 1
    return ans
```

> **해석** `can_make`는 명백히 단조입니다. 길이 10으로 $K$개를 만들 수 있다면 길이 9로도 당연히 만들 수 있습니다. 그래서 "가능/불가능"의 경계가 정확히 한 곳이고 이진 탐색이 성립합니다. 복잡도는 `can_make`가 $O(N)$이고 범위가 최대 $2^{31}$이므로 $O(N \log(\max))$, 즉 $N$이 10만이어도 300만 번 정도의 연산으로 끝납니다. 후보를 1부터 하나씩 다 시도하는 $O(N \cdot \max)$와 비교하면 차원이 다릅니다.

| 문제 유형 | 결정 문제로의 변환 |
|---|---|
| 최소 길이의 최댓값 | "길이 $x$로 조건을 만족할 수 있는가?" |
| 최대 비용의 최솟값 | "비용 상한을 $x$로 두면 배치가 가능한가?" |
| $K$번째 수 찾기 | "$x$ 이하인 수가 $K$개 이상인가?" |
| 실수 답(정밀도) | 고정 횟수(예: 100회) 반복으로 구간을 줄임 |

> **팁** 파라메트릭 서치를 적용할 수 있는지 판별하는 가장 빠른 방법은 이 질문입니다. **"답이 $x$일 때 가능하다면, $x$보다 조금 더 나쁜 값일 때도 반드시 가능한가?"** 이 질문에 "예"라고 답할 수 있으면 단조성이 있고, 이진 탐색이 됩니다.

## 5. 이진 탐색이 통하지 않는 경우

> **개요** 조건을 확인하지 않고 쓰면 조용히 틀린 답을 내는 것이 이진 탐색의 위험입니다.

- **정렬되어 있지 않은 배열**: 당연하지만 가장 흔한 실수입니다. 특히 정렬 기준과 탐색 기준이 다른 경우(이름순 정렬 배열에서 나이로 이진 탐색) 코드는 정상 실행되고 답만 틀립니다.
- **단조성이 없는 결정 함수**: $f$가 참·거짓을 여러 번 오가면 이진 탐색은 그중 아무 경계나 찾아 줍니다. 파라메트릭 서치의 실패는 대부분 여기서 옵니다.
- **회전된 정렬 배열**: `[4,5,6,7,0,1,2]`처럼 회전된 배열에서는 기본형이 실패합니다. 다만 "어느 절반이 정렬되어 있는가"를 먼저 판정하면 여전히 $O(\log n)$에 풀 수 있습니다.
- **실수 범위**: 부동소수점에서 `lo <= hi`로 종료를 기다리면 정밀도 때문에 무한 루프에 빠질 수 있습니다. 대신 **반복 횟수를 고정**(예: 100회)하는 것이 표준 관행입니다.

> **참고** 균등하게 분포한 데이터에서는 중간이 아니라 값의 비율로 위치를 추정하는 <u>**보간 탐색(interpolation search)**</u>이 평균 $O(\log \log n)$을 냅니다. 다만 분포가 치우치면 최악 $O(n)$으로 무너지고 계산 비용도 크기 때문에, 전화번호부처럼 분포가 확실히 균등한 특수 상황이 아니면 실무에서는 그냥 이진 탐색을 씁니다.

## 요약

> **핵심**
> - 이진 탐색은 **정렬**이라는 전제를 요구하고 그 정렬은 $O(n \log n)$입니다. 탐색을 몇 번 반복할 것인지에 따라 선형 탐색이 정답일 수 있으며, 정렬은 반복될 탐색을 위한 선투자로 이해해야 합니다.
> - 정확한 구현의 관건은 세 가지입니다. `lo + (hi-lo)//2`로 오버플로 회피, `mid ± 1`로 구간 축소 보장, 닫힌 구간과 반열린 구간 중 하나를 골라 일관되게 유지하기.
> - 값이 중복될 수 있는 실무에서는 단일 값 탐색보다 **lower_bound / upper_bound**가 훨씬 유용합니다. 두 함수의 차이는 부등호 하나이며, 이 둘로 개수·존재 여부·삽입 위치를 모두 얻습니다.
> - 파라메트릭 서치는 최적화 문제를 단조적인 결정 문제로 바꿔 **답의 범위 위에서** 이진 탐색합니다. 적용 가능 여부는 "조금 더 나쁜 값에서도 여전히 가능한가"라는 단조성 질문으로 판별합니다.
