# Part 16. 알고리즘 심화

## 16-2. 정렬 알고리즘 — 왜 표준 라이브러리는 교과서 정렬을 그대로 쓰지 않는가

> **학습 목표**
> - 버블·선택·삽입·병합·퀵·힙·계수 정렬의 동작 원리와 시간·공간복잡도를 각각 설명할 수 있다.
> - 안정성(stable)과 제자리(in-place) 여부를 기준으로 정렬 알고리즘을 분류하고, 실무 상황에 맞는 정렬을 선택할 수 있다.
> - 비교 기반 정렬의 $\Omega(n \log n)$ 하한이 왜 성립하는지 결정 트리로 설명하고, Timsort·introsort 같은 하이브리드가 왜 등장했는지 설명할 수 있다.

> **개요** 정렬은 이미 모든 언어의 표준 라이브러리에 들어 있어서 직접 구현할 일이 거의 없습니다. 그런데도 정렬을 깊이 배우는 이유는, 정렬이 알고리즘 설계의 거의 모든 기법(분할 정복, 자료구조 활용, 하한 증명, 하이브리드 전략)이 한자리에 모이는 교과서이기 때문입니다. 이 강의는 일곱 가지 정렬을 훑고, "비교로는 $n \log n$보다 빠를 수 없다"는 하한을 증명한 뒤, 실제 표준 라이브러리가 그 이론 위에서 어떤 타협을 했는지 봅니다.

## 1. 정렬을 평가하는 네 가지 축

> **Q.** 시간복잡도만 보면 되지 않나요? 왜 안정성이니 제자리니 하는 이야기가 따라붙을까요?
> **A.** 같은 $O(n \log n)$ 안에서도 실무의 선택을 가르는 요소가 따로 있기 때문입니다. 예를 들어 주문 목록을 "날짜순으로 정렬한 뒤 다시 고객명순으로 정렬"하면, 안정 정렬을 썼을 때만 같은 고객 안에서 날짜 순서가 보존됩니다. 또 메모리가 빠듯한 임베디드 환경에서는 추가 배열 $n$개를 잡는 병합 정렬을 쓸 수 없습니다.

- <u>**시간복잡도(time complexity)**</u>: 최선·평균·최악 각각을 봅니다.
- <u>**공간복잡도(space complexity)**</u>: 보조 공간이 $O(1)$이면 <u>**제자리 정렬(in-place sort)**</u>입니다.
- <u>**안정성(stability)**</u>: 값이 같은 원소들의 원래 상대 순서가 보존되면 <u>**안정 정렬(stable sort)**</u>입니다.
- **적응성(adaptivity)**: 이미 정렬에 가까운 입력에서 더 빨라지면 적응적입니다.

![정렬 알고리즘 7종 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part16_알고리즘_심화/16-2_정렬_알고리즘/assets/diagram1_sort_comparison.svg)

> **해석** 그림에서 위쪽 세 개($O(n^2)$ 군)는 구현이 짧고 작은 입력에서 상수 계수가 작다는 장점이 있습니다. 가운데 세 개($O(n \log n)$ 군)는 대규모 데이터의 실전 주력입니다. 맨 아래 계수 정렬은 비교를 전혀 하지 않기 때문에 하한 $\Omega(n \log n)$의 적용을 받지 않고 $O(n + k)$에 끝나지만, 값의 범위 $k$가 작아야 한다는 강한 제약이 붙습니다. **어떤 정렬이 우월한 것이 아니라, 각자 다른 제약 조건을 대가로 다른 성능을 삽니다.**

## 2. $O(n^2)$ 삼총사 — 버블·선택·삽입

> **개요** 세 알고리즘 모두 평균 $\Theta(n^2)$이지만 내부 동작과 실무적 가치는 서로 크게 다릅니다.

**버블 정렬(bubble sort)** 은 인접한 두 원소를 비교해 순서가 어긋나면 교환하는 과정을 반복합니다. 한 번 훑을 때마다 가장 큰 값이 거품처럼 끝으로 올라간다고 해서 붙은 이름입니다. 교환이 한 번도 없었다면 이미 정렬된 것이므로 즉시 종료하는 최적화를 넣으면 최선의 경우 $\Theta(n)$이 됩니다. 안정적이고 제자리입니다.

**선택 정렬(selection sort)** 은 남은 구간에서 최솟값을 찾아 맨 앞과 교환합니다. 비교는 항상 $\Theta(n^2)$번 하지만 **교환은 정확히 $n-1$번**뿐이라, 원소 하나를 옮기는 비용이 극도로 비싼 상황(예: 거대한 레코드를 직접 스왑)에서 의미가 있습니다. 멀리 떨어진 원소를 건너뛰어 교환하므로 기본형은 **불안정**합니다.

**삽입 정렬(insertion sort)** 은 왼쪽 정렬된 구간에 새 원소를 적절한 자리에 끼워 넣습니다. 손에 쥔 카드를 정렬하는 방식 그대로입니다. 이미 정렬된 입력에서는 비교 한 번씩만 하고 끝나 $\Theta(n)$이며, 뒤집힌 쌍(inversion)의 수 $I$에 대해 $\Theta(n + I)$이라는 **적응적** 성질을 가집니다. 안정적이고 제자리입니다.

```python
def insertion_sort(a):
    for i in range(1, len(a)):
        key = a[i]
        j = i - 1
        while j >= 0 and a[j] > key:   # > 이므로 같은 값은 넘지 않음 → 안정
            a[j + 1] = a[j]
            j -= 1
        a[j + 1] = key
```

> **해석** 위 코드의 `a[j] > key` 조건에서 부등호가 `>=`가 아니라 `>`라는 점이 안정성의 전부입니다. 같은 값을 만나면 멈추므로 원래 앞에 있던 동일 값이 계속 앞에 남습니다. 부등호 하나로 안정성이 깨지는 것이 정렬 구현의 대표적인 함정입니다.

> **팁** 삽입 정렬은 "느린 정렬"로 배우지만 실제로는 모든 고성능 정렬 라이브러리 안에 반드시 들어 있습니다. 입력이 대략 16~32개 이하로 작아지면 재귀 오버헤드가 없는 삽입 정렬이 퀵 정렬보다 빠르기 때문에, 재귀의 바닥에서 삽입 정렬로 전환하는 것이 표준 관행입니다.

## 3. 병합 정렬 — 안정성과 최악 보장을 사는 대가

> **개요** 1945년 존 폰 노이만(John von Neumann)이 고안한 병합 정렬은 분할 정복의 원형입니다.

배열을 절반으로 나누고, 각각을 재귀적으로 정렬한 뒤, 정렬된 두 배열을 하나로 <u>**병합(merge)**</u>합니다. 병합은 양쪽 맨 앞을 비교해 작은 쪽을 꺼내는 선형 작업입니다.

$$T(n) = 2T(n/2) + \Theta(n) \ \Rightarrow \ T(n) = \Theta(n \log n)$$

> **해석** 마스터 정리 경우 2에 해당합니다(16-1 참고). 재귀 트리로 보면 층마다 총 $n$개의 원소를 한 번씩 훑고, 층수가 $\log_2 n$이므로 $n \log n$입니다. 중요한 점은 이 값이 **최선·평균·최악 모두 동일**하다는 것입니다. 입력이 어떻게 생겼든 항상 절반으로 나뉘기 때문입니다.

병합할 때 양쪽 값이 같으면 **왼쪽 것을 먼저 꺼내도록** 하면 안정 정렬이 됩니다. 대신 병합용 보조 배열 $\Theta(n)$이 필요해 제자리가 아닙니다.

| 성질 | 병합 정렬 |
|---|---|
| 시간(최선/평균/최악) | $\Theta(n \log n)$ / $\Theta(n \log n)$ / $\Theta(n \log n)$ |
| 보조 공간 | $\Theta(n)$ |
| 안정성 | 안정 |
| 강점 | 최악 보장, 안정성, 연결 리스트와 외부 정렬에 적합 |

> **참고** 데이터가 메모리에 다 안 들어가 디스크에 나눠 두고 정렬해야 하는 <u>**외부 정렬(external sort)**</u>에서는 병합 정렬이 사실상 유일한 선택입니다. 순차 읽기만으로 병합이 가능해 디스크 접근 패턴이 최적이기 때문입니다.

## 4. 퀵 정렬 — 평균이 가장 빠른 이유와 최악이 터지는 이유

> **Q.** 퀵 정렬은 최악이 $\Theta(n^2)$인데 왜 "가장 빠른 정렬"로 불릴까요?
> **A.** 평균 $\Theta(n \log n)$이면서 **상수 계수가 작고** 보조 공간이 재귀 스택뿐이기 때문입니다. 병합 정렬은 매번 보조 배열에 값을 복사해야 하지만 퀵 정렬은 배열 안에서 교환만 합니다. 이 차이가 캐시 지역성에서 큰 이득을 만듭니다.

1959년 토니 호어(C. A. R. Hoare)가 고안한 퀵 정렬은 <u>**피벗(pivot)**</u> 하나를 고르고, 배열을 "피벗보다 작은 것 / 큰 것" 두 구역으로 <u>**분할(partition)**</u>한 뒤 각 구역을 재귀적으로 정렬합니다. 병합 정렬과 반대로 **분할이 어렵고 병합이 공짜**인 구조입니다.

![퀵 정렬의 분할 과정과 최악의 경우](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part16_알고리즘_심화/16-2_정렬_알고리즘/assets/diagram2_quicksort_partition.svg)

> **해석** 그림 왼쪽은 피벗이 매번 중앙값 근처를 잡아 구간이 절반씩 줄어드는 이상적인 경우로, 재귀 깊이가 $\log n$이라 $\Theta(n \log n)$입니다. 오른쪽은 이미 정렬된 배열에서 첫 원소를 피벗으로 골랐을 때로, 분할 결과가 "빈 구간 + $n-1$개 구간"이 되어 깊이가 $n$까지 늘어납니다. 이때 비용은 $n + (n-1) + \cdots + 1 = \Theta(n^2)$이 되고, 재귀 스택도 $\Theta(n)$까지 커져 스택 오버플로 위험이 생깁니다.

최악을 피하는 실무 기법은 세 가지입니다.

- **무작위 피벗**: 피벗을 난수로 고르면 특정 입력이 최악을 유발할 수 없습니다. 다만 확률적 보장일 뿐 최악 자체는 남습니다.
- **중앙값 3 선택(median-of-three)**: 첫·중간·끝 세 값의 중앙값을 피벗으로 씁니다. 정렬된 입력에 특히 효과적입니다.
- **깊이 제한 후 전환**: 재귀 깊이가 $2\log n$을 넘으면 힙 정렬로 갈아탑니다. 이것이 introsort입니다.

> **참고** 퀵 정렬은 기본적으로 **불안정**합니다. 분할 과정에서 멀리 떨어진 원소를 교환하기 때문입니다. C++ 표준은 `std::sort`에 안정성을 요구하지 않고, 안정성이 필요하면 별도의 `std::stable_sort`를 쓰도록 API를 나눠 두었습니다.

## 5. 힙 정렬과 계수 정렬 — 자료구조와 비교 회피

**힙 정렬(heap sort)** 은 1964년 J. W. J. 윌리엄스(Williams)가 힙 자료구조와 함께 발표했습니다. 배열을 최대 힙으로 만든 뒤(이 과정이 $\Theta(n)$), 루트의 최댓값을 맨 뒤와 교환하고 힙 크기를 줄이는 일을 $n$번 반복합니다. 각 반복의 재정렬이 $O(\log n)$이므로 전체 $\Theta(n \log n)$입니다.

힙 정렬의 결정적 장점은 **최악에도 $\Theta(n \log n)$이면서 보조 공간이 $\Theta(1)$**이라는 것입니다. 병합 정렬의 최악 보장과 퀵 정렬의 제자리 성질을 동시에 가집니다. 그런데도 주력이 되지 못한 이유는 배열 여기저기를 뛰어다니는 접근 패턴 때문에 **캐시 지역성이 나빠** 실측 속도가 퀵 정렬보다 느리기 때문입니다. 그래서 introsort에서 "안전망" 역할로 쓰입니다. 힙 구조 자체의 자세한 내용은 Part 15를 참고하세요.

**계수 정렬(counting sort)** 은 값을 비교하지 않습니다. 값의 범위가 $0 \sim k$일 때 각 값의 개수를 세어 누적합을 구한 뒤, 그 위치로 원소를 직접 배치합니다.

```python
def counting_sort(a, k):
    count = [0] * (k + 1)
    for x in a:
        count[x] += 1
    for i in range(1, k + 1):
        count[i] += count[i - 1]          # 누적합 = 각 값의 끝 위치
    out = [0] * len(a)
    for x in reversed(a):                  # 뒤에서부터 채워야 안정성 유지
        count[x] -= 1
        out[count[x]] = x
    return out
```

> **해석** 시간은 $\Theta(n + k)$, 공간은 $\Theta(n + k)$입니다. $k$가 $n$과 비슷하면 사실상 선형이지만, 값의 범위가 32비트 정수 전체라면 $k \approx 43$억이 되어 쓸 수 없습니다. **원소를 뒤에서부터 배치하는 것**이 안정성의 조건이며, 이 안정성 덕분에 계수 정렬을 자릿수마다 반복하는 기수 정렬(radix sort)이 성립합니다.

## 6. 비교 기반 정렬의 하한 — $\Omega(n \log n)$은 왜 넘을 수 없는가

> **개요** "더 똑똑한 비교 정렬을 만들면 $O(n)$이 가능하지 않을까?"라는 질문에 수학이 내린 최종 답입니다.

비교 정렬은 실행 과정을 <u>**결정 트리(decision tree)**</u>로 그릴 수 있습니다. 각 내부 노드는 "$a_i < a_j$?"라는 비교 하나이고, 두 갈래로 갈라집니다. 잎 노드는 최종적으로 확정된 하나의 순열입니다.

$n$개 원소의 가능한 순열은 $n!$가지이고, 알고리즘은 이 중 어느 것이든 구분해 낼 수 있어야 하므로 **잎이 최소 $n!$개** 필요합니다. 높이 $h$인 이진 트리의 잎은 최대 $2^h$개이므로,

$$2^h \ge n! \ \Rightarrow \ h \ge \log_2(n!)$$

스털링 근사 $\log_2(n!) = \Theta(n \log n)$을 적용하면 결론이 나옵니다.

$$h = \Omega(n \log n)$$

> **해석** $h$는 최악의 경우 비교 횟수, 즉 트리의 높이입니다. 따라서 **어떤 비교 기반 정렬도 최악의 경우 $\Omega(n \log n)$번의 비교를 피할 수 없습니다.** 이는 특정 알고리즘의 한계가 아니라 "비교만으로 정보를 얻는다"는 모델 자체의 한계입니다. 병합 정렬과 힙 정렬은 이 하한에 도달하므로 **점근적으로 최적**입니다.

> **참고** 계수 정렬·기수 정렬·버킷 정렬이 $O(n)$에 도달하는 것은 이 하한을 깬 것이 아닙니다. 이들은 비교가 아니라 **값 자체를 인덱스로 사용**하므로 애초에 결정 트리 모델 밖에 있습니다. 그 대신 "값이 정수여야 하고 범위가 제한적이어야 한다"는 조건을 지불합니다. 하한을 우회하려면 문제에 대한 추가 정보를 알고리즘에 넣어야 한다는 일반 원리의 사례입니다.

## 7. 실무의 답 — Timsort와 introsort

> **개요** 이론적으로 최적인 알고리즘과 실제로 빠른 구현은 다릅니다. 현대 표준 라이브러리는 모두 하이브리드입니다.

**Timsort**는 2002년 팀 피터스(Tim Peters)가 CPython을 위해 만든 정렬로, 현재 Python의 `list.sort()`·`sorted()`, Java의 객체 배열 `Arrays.sort()`, Android 런타임, Rust의 `sort()` 등에 쓰입니다. 핵심 아이디어는 **실제 데이터에는 이미 정렬된 구간(run)이 많다**는 관찰입니다.

1. 배열을 훑으며 자연적으로 증가/감소하는 구간(run)을 찾습니다. 감소 구간은 뒤집어 증가 구간으로 만듭니다.
2. run이 너무 짧으면 삽입 정렬로 최소 길이(보통 32~64)까지 늘립니다.
3. run들을 스택에 쌓으며 크기 균형 규칙에 따라 병합합니다.
4. 갤로핑 모드(galloping)로, 한쪽이 계속 이기면 이진 탐색으로 건너뛰어 병합을 가속합니다.

결과적으로 이미 정렬된 입력에서 $\Theta(n)$, 최악에서도 $\Theta(n \log n)$이며 **안정 정렬**입니다.

**introsort**는 1997년 데이비드 무서(David Musser)가 발표했으며 C++ `std::sort`의 표준 구현입니다. 퀵 정렬로 시작하되, 재귀 깊이가 $2\lfloor \log_2 n \rfloor$를 넘으면 힙 정렬로 전환하고, 구간이 충분히 작아지면 삽입 정렬로 마무리합니다. 퀵 정렬의 평균 속도, 힙 정렬의 최악 보장, 삽입 정렬의 작은 입력 효율을 모두 챙깁니다.

| 언어/런타임 | 기본 정렬 | 안정성 | 근간 |
|---|---|---|---|
| Python `sorted()` | Timsort | 안정 | 병합 + 삽입 |
| Java `Arrays.sort(Object[])` | Timsort | 안정 | 병합 + 삽입 |
| Java `Arrays.sort(int[])` | 듀얼 피벗 퀵 정렬 | 불안정 | 퀵 |
| C++ `std::sort` | introsort | 불안정 | 퀵 + 힙 + 삽입 |
| C++ `std::stable_sort` | 병합 계열 | 안정 | 병합 |
| Rust `slice::sort` | Timsort 계열 | 안정 | 병합 + 삽입 |

> **해석** Java가 원시 타입 배열(`int[]`)과 객체 배열(`Object[]`)에 **서로 다른 정렬**을 쓰는 것이 이 표에서 가장 흥미로운 대목입니다. `int` 값은 서로 구별할 수 없으므로 안정성이 의미가 없어 더 빠른 퀵 정렬을 쓰고, 객체는 같은 키를 가진 서로 다른 객체가 존재하므로 안정성이 필요해 Timsort를 씁니다. **안정성이 필요한지 여부가 알고리즘 선택을 갈랐다**는 실무 사례입니다.

> **팁** 실무에서 정렬을 직접 구현할 일은 거의 없지만, "정렬 키를 무엇으로 잡을 것인가"는 매일 하는 결정입니다. 안정 정렬을 쓴다면 여러 기준으로 정렬할 때 **우선순위가 낮은 키부터 차례로 정렬**하면 다중 키 정렬이 자연스럽게 완성됩니다.

## 요약

> **핵심**
> - 정렬은 시간복잡도만이 아니라 **안정성·제자리 여부·적응성**의 네 축으로 평가합니다. 같은 $\Theta(n \log n)$ 안에서도 이 축들이 실무의 선택을 가릅니다.
> - 병합 정렬은 최악 보장과 안정성을 $\Theta(n)$ 공간으로 사고, 퀵 정렬은 작은 상수 계수와 제자리 성질을 $\Theta(n^2)$ 최악 위험으로 사며, 힙 정렬은 둘 다 가지지만 캐시 지역성을 잃습니다.
> - 비교 기반 정렬은 결정 트리의 잎이 $n!$개 필요하다는 논증에서 최악의 경우 $\Omega(n \log n)$ 하한을 가집니다. 계수·기수 정렬이 $O(n)$인 것은 하한을 깬 것이 아니라 **비교가 아닌 모델**을 쓰기 때문입니다.
> - 현대 표준 라이브러리는 모두 하이브리드입니다. Timsort는 실제 데이터의 정렬된 run을 활용해 안정성과 적응성을, introsort는 깊이 제한 전환으로 퀵 정렬의 평균 속도와 최악 보장을 동시에 얻습니다.
