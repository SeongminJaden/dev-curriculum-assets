# Part 16. 알고리즘 심화

## 16-6. 동적 계획법과 백트래킹 — 어떤 문제가 "기억"으로 풀리고, 어떤 문제가 "되돌리기"로 풀리는가

> **학습 목표**
> - 동적 계획법이 성립하는 두 조건(중복 부분 문제, 최적 부분 구조)을 판별하고, 어떤 문제가 DP로 풀리는지 스스로 진단할 수 있다.
> - 메모이제이션(top-down)과 타뷸레이션(bottom-up)의 차이를 설명하고, 같은 점화식을 두 방식으로 각각 구현할 수 있다.
> - 백트래킹이 완전탐색과 다른 지점(가지치기로 유망하지 않은 경로를 조기 차단)을 이해하고, 상태 공간 트리로 문제를 모델링할 수 있다.

> **개요** 16-5에서 그리디는 되돌리지 않아 빠르지만 최적을 놓칠 수 있고, 완전탐색은 최적을 보장하지만 느리다는 것을 봤습니다. 이 강의는 그 사이의 두 전략을 다룹니다. 동적 계획법은 "한 번 푼 부분 문제의 답을 기억해 두어" 완전탐색의 지수 시간을 다항 시간으로 줄이고, 백트래킹은 "가망 없는 경로를 일찍 포기해" 완전탐색의 탐색 공간을 쳐냅니다. 두 기법 모두 문제를 **더 작은 부분 문제로 쪼개는 것**에서 출발합니다.

## 1. 동적 계획법은 언제 쓸 수 있는가 — 두 가지 조건

> **Q.** 어떤 문제를 보고 "아, 이건 DP로 풀 수 있겠다"를 어떻게 알아보나요?
> **A.** 두 조건을 모두 만족하는지 확인하면 됩니다. 첫째, **중복 부분 문제** — 같은 작은 문제를 여러 번 반복해서 풀게 되는가. 둘째, **최적 부분 구조** — 큰 문제의 최적해가 작은 부분 문제의 최적해로부터 조립되는가. 이 두 가지가 성립하면, 부분 문제의 답을 한 번만 계산해 저장해 두는 것으로 전체를 효율적으로 풀 수 있습니다.

<u>**동적 계획법(Dynamic Programming, DP)**</u>은 1950년대 리처드 벨만(Richard Bellman)이 정립한 기법입니다. "Dynamic Programming"이라는 이름은 사실 수학적 의미가 아니라, 당시 연구 예산을 심사하던 관료를 설득하려고 멋있게 지은 이름이었다는 벨만 본인의 회고가 유명합니다. 이름과 달리 핵심 아이디어는 단순합니다 — **같은 계산을 두 번 하지 마라.**

![중복 부분 문제: 피보나치 재귀 트리와 메모이제이션](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part16_알고리즘_심화/16-6_동적_계획법과_백트래킹/assets/diagram1_overlapping_subproblems.svg)

> **해석** 위 그림은 피보나치 수 $F(5)$를 단순 재귀로 계산할 때 만들어지는 호출 트리입니다. $F(3)$이 두 번, $F(2)$가 세 번 중복 계산됩니다. $n$이 커지면 이 중복이 지수적으로 늘어나 순수 재귀는 $O(2^n)$이 됩니다. 오른쪽처럼 한 번 계산한 값을 표에 저장해 두고 재사용하면, 각 $F(k)$를 정확히 한 번씩만 계산하므로 $O(n)$으로 줄어듭니다. **이것이 DP의 전부입니다** — 중복 부분 문제가 존재하고(피보나치는 그렇습니다), 큰 답이 작은 답의 합으로 조립되면(최적 부분 구조), 저장이 곧 속도가 됩니다.

> **참고** 최적 부분 구조가 없으면 DP가 성립하지 않습니다. 예를 들어 "그래프에서 가장 긴 단순 경로"는 부분 경로의 최적해를 이어붙여도 전체 최적해가 되지 않아(정점 재방문 제약 때문에) DP로 풀리지 않으며, 실제로 NP-난해 문제입니다. 반면 "가장 짧은 경로"는 최적 부분 구조가 성립해 다익스트라·벨만-포드 같은 DP 계열 알고리즘으로 풀립니다(16-7 참고).

## 2. 점화식 세우기 — DP의 90%는 여기서 결정된다

> **개요** DP 문제를 푸는 실제 과정은 코드를 짜는 것이 아니라 **점화식(recurrence relation)** 을 세우는 것입니다. 점화식만 정확하면 구현은 기계적입니다.

점화식을 세우는 절차는 항상 같습니다.

1. **상태 정의**: $dp[i]$가 "무엇의 답"인지 한 문장으로 정확히 정의합니다. 이 정의가 애매하면 반드시 실패합니다.
2. **점화 관계**: $dp[i]$를 자기보다 작은 $dp[j]$들로 표현합니다.
3. **기저 조건**: 재귀가 멈추는 가장 작은 경우의 값을 정합니다.
4. **계산 순서**: 작은 부분 문제가 큰 부분 문제보다 먼저 계산되도록 순서를 정합니다.

대표적인 <u>**0/1 배낭 문제(0/1 Knapsack)**</u>로 이 절차를 밟아봅시다. 무게 한도 $W$인 배낭에, 각각 무게 $w_i$·가치 $v_i$인 물건들을 넣어 가치 합을 최대로 만드는 문제입니다.

- **상태**: $dp[i][c]$ = 앞의 $i$개 물건만 고려하고 남은 용량이 $c$일 때 얻을 수 있는 최대 가치
- **점화**: 각 물건은 넣거나 안 넣거나 둘 중 하나입니다.

$$dp[i][c] = \max\big(\underbrace{dp[i-1][c]}_{\text{안 넣음}},\ \underbrace{dp[i-1][c-w_i] + v_i}_{\text{넣음 (}c \geq w_i\text{일 때)}}\big)$$

- **기저**: $dp[0][c] = 0$ (물건이 없으면 가치도 0)

> **해석** 이 한 줄의 점화식이 문제 전체를 담고 있습니다. "$i$번째 물건을 넣는다면 그 이전 상태는 용량이 $w_i$만큼 남아 있던 $dp[i-1][c-w_i]$였을 것"이라는 논리가 핵심입니다. 완전탐색이라면 $2^n$가지 부분집합을 모두 시도하지만, DP는 $(i, c)$ 조합이 겹치는 것을 이용해 $O(nW)$로 끝냅니다.

> **팁** 문제를 만났을 때 "상태를 무엇으로 잡을까"가 막히면, **"내가 지금까지 내린 결정을 요약하는 데 필요한 최소한의 정보가 무엇인가"**를 물어보세요. 그것이 곧 상태입니다. 배낭 문제에서는 "몇 번째 물건까지 봤고, 용량이 얼마 남았는가"가 그 최소 정보입니다.

## 3. Top-down과 Bottom-up — 같은 점화식, 두 가지 구현

> **개요** 점화식을 코드로 옮기는 방법은 두 가지입니다. 큰 문제에서 시작해 재귀로 내려가며 답을 저장하는 방식과, 가장 작은 문제부터 표를 채워 올라가는 방식입니다.

![Top-down 메모이제이션과 Bottom-up 타뷸레이션 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part16_알고리즘_심화/16-6_동적_계획법과_백트래킹/assets/diagram2_topdown_bottomup.svg)

<u>**메모이제이션(Memoization, Top-down)**</u>은 재귀를 그대로 두되, 계산한 값을 캐시에 저장해 두 번째 호출부터는 즉시 반환합니다.

```python
from functools import lru_cache

@lru_cache(maxsize=None)      # 계산 결과를 자동으로 기억
def fib(n):
    if n < 2:
        return n              # 기저 조건
    return fib(n - 1) + fib(n - 2)
```

<u>**타뷸레이션(Tabulation, Bottom-up)**</u>은 재귀 없이 가장 작은 부분 문제부터 표를 채워 나갑니다.

```python
def fib(n):
    dp = [0, 1]
    for i in range(2, n + 1):
        dp.append(dp[i - 1] + dp[i - 2])   # 작은 것부터 위로
    return dp[n]
```

> **해석** 두 방식은 같은 점화식의 서로 다른 얼굴입니다. 차이를 표로 정리하면 다음과 같습니다.

| 구분 | Top-down (메모이제이션) | Bottom-up (타뷸레이션) |
|---|---|---|
| 방향 | 큰 문제 → 작은 문제 (재귀) | 작은 문제 → 큰 문제 (반복) |
| 계산 범위 | **필요한 부분 문제만** 계산 | 표의 모든 칸을 계산 |
| 오버헤드 | 재귀 호출·스택 사용 | 반복문, 스택 없음 |
| 위험 | 깊이가 크면 스택 오버플로 | 없음 |
| 장점 | 점화식을 거의 그대로 옮겨 직관적 | 빠르고, 표 재사용으로 공간 최적화 쉬움 |

> **팁** 실무에서는 **먼저 top-down으로 점화식이 맞는지 빠르게 검증**한 뒤, 성능이나 스택 깊이가 문제되면 bottom-up으로 옮기는 순서가 편합니다. bottom-up에서는 배낭 문제의 $dp[i][c]$가 바로 앞 행 $dp[i-1]$만 참조하므로, 2차원 표를 1차원으로 줄이는 <u>**공간 최적화(rolling array)**</u>도 자주 씁니다.

## 4. 대표 문제 하나 더 — 최장 공통 부분 수열(LCS)

> **개요** 배낭이 "넣느냐 마느냐"의 DP라면, LCS는 "문자를 맞추느냐 건너뛰느냐"의 DP입니다. 두 문자열을 다루는 DP의 원형입니다.

<u>**최장 공통 부분 수열(Longest Common Subsequence, LCS)**</u>은 두 문자열에서 순서를 유지하되 연속일 필요는 없는 가장 긴 공통 부분 수열의 길이입니다. `ABCBDAB`와 `BDCAB`의 LCS는 `BCAB`로 길이 4입니다.

- **상태**: $dp[i][j]$ = 첫 문자열의 앞 $i$글자와 둘째 문자열의 앞 $j$글자의 LCS 길이
- **점화**:

$$dp[i][j] = \begin{cases} dp[i-1][j-1] + 1 & \text{if } A_i = B_j \\ \max(dp[i-1][j],\ dp[i][j-1]) & \text{if } A_i \neq B_j \end{cases}$$

> **해석** 두 문자열의 마지막 글자가 같으면, 그 글자를 LCS에 포함시키고 두 문자열을 각각 한 칸씩 줄인 부분 문제로 넘어갑니다($+1$). 다르면, 둘 중 하나의 마지막 글자를 버린 두 경우 중 더 나은 쪽을 취합니다. 시간복잡도는 두 문자열 길이 $m, n$에 대해 $O(mn)$입니다.

> **참고** LCS는 학술적 연습이 아닙니다. `git diff`와 유닉스 `diff` 명령이 두 파일의 차이를 계산하는 핵심이 바로 LCS의 변형(최소 편집 스크립트)이며, 생명정보학에서 DNA 서열을 정렬하는 문제도 같은 DP 골격을 씁니다.

## 5. 백트래킹 — 되돌릴 수 있는 완전탐색

> **개요** DP가 "부분 문제의 답을 저장해 재사용"하는 기법이라면, 백트래킹은 "선택을 하나씩 시도하다가 막히면 되돌아가는" 기법입니다. 둘은 자주 함께 등장하지만 접근이 다릅니다.

<u>**백트래킹(Backtracking)**</u>은 해를 한 조각씩 만들어 나가다가, 현재까지의 부분 해가 <u>**유망하지 않다(non-promising)**</u>고 판단되는 순간 그 경로를 포기하고 직전 갈림길로 돌아가는 기법입니다. 16-5의 가지치기를 상태 공간 트리 전체에 체계적으로 적용한 것입니다.

고전 예제인 <u>**N-Queens 문제**</u>(N×N 체스판에 서로 공격하지 않는 퀸 N개 놓기)로 보겠습니다.

```python
def solve(row, cols, diag1, diag2, n):
    if row == n:
        return 1                      # 모든 행에 배치 성공 → 해 1개
    count = 0
    for col in range(n):
        d1, d2 = row + col, row - col
        if col in cols or d1 in diag1 or d2 in diag2:
            continue                  # 유망하지 않음 → 이 열은 건너뜀 (가지치기)
        cols.add(col); diag1.add(d1); diag2.add(d2)
        count += solve(row + 1, cols, diag1, diag2, n)
        cols.remove(col); diag1.remove(d1); diag2.remove(d2)   # 되돌리기(backtrack)
    return count
```

> **해석** 핵심은 두 줄입니다. `continue`로 **유망하지 않은 열을 건너뛰는 것**(가지치기)과, 재귀에서 돌아온 뒤 `remove`로 **선택을 원상복구하는 것**(백트래킹)입니다. 이 되돌리기 덕분에 하나의 자료구조를 재사용하며 모든 경우를 탐색할 수 있습니다. 만약 가지치기 없이 모든 배치를 다 만든 뒤 검사하면 $N^N$이지만, 열·대각선 충돌을 조기에 걸러내면 실제 탐색량이 극적으로 줄어듭니다.

> **팁** 백트래킹 문제는 세 가지만 정하면 뼈대가 완성됩니다. ① **선택(choice)**: 지금 무엇을 고를 수 있는가, ② **제약(constraint)**: 어떤 선택이 유망하지 않아 가지치기 대상인가, ③ **목표(goal)**: 언제 완성된 해인가. N-Queens에서는 각각 "어느 열에 놓을까", "열·대각선이 겹치면 안 됨", "N개 행을 모두 채우면 완성"입니다.

## 6. DP와 백트래킹은 어떻게 갈라지는가

> **개요** 두 기법은 모두 부분 문제로 쪼개지만, 목적과 조건이 다릅니다. 언제 무엇을 쓸지 정리합니다.

| 관점 | 동적 계획법(DP) | 백트래킹 |
|---|---|---|
| 주로 답하는 질문 | "최적값은 얼마인가", "경우의 수는 몇 개인가" | "가능한 모든 해를 나열하라", "조건을 만족하는 해가 있는가" |
| 핵심 무기 | 중복 부분 문제의 결과 **저장·재사용** | 유망하지 않은 경로 **조기 차단** |
| 성립 조건 | 최적 부분 구조 + 중복 부분 문제 | 부분 해로 유망성을 판정할 수 있을 것 |
| 전형적 복잡도 | 다항 시간 (예: $O(nW)$, $O(mn)$) | 최악은 지수, 가지치기로 실질 감소 |

> **해석** 둘은 경쟁 관계가 아니라 상황에 따른 도구 선택입니다. "가장 큰 가치는?"처럼 **하나의 최적값**을 묻고 부분 문제가 겹치면 DP가 압도적으로 빠릅니다. "가능한 모든 배치를 출력하라"처럼 **해 자체를 구성**해야 하고 겹치는 부분 문제가 없다면 백트래킹이 맞습니다. 어떤 문제(예: 부분집합 합)는 백트래킹으로도, DP로도 풀 수 있으며, "저장할 부분 문제가 많은가 vs 가지치기가 잘 되는가"로 선택합니다.

## 요약

> **핵심**
> - **DP는 두 조건이 모두 성립할 때만 쓴다** — 중복 부분 문제(같은 작은 문제를 반복해 풀게 됨)와 최적 부분 구조(큰 답이 작은 답으로 조립됨). 이 둘이 있으면 부분 문제를 한 번만 계산해 저장하는 것으로 지수 시간을 다항 시간으로 줄인다.
> - **점화식이 DP의 90%다** — 상태를 한 문장으로 정확히 정의하고, 자기보다 작은 상태들로 관계식을 세우고, 기저 조건과 계산 순서를 정하면 구현은 기계적이다. Top-down(메모이제이션)은 필요한 부분 문제만 재귀로 풀어 직관적이고, Bottom-up(타뷸레이션)은 표를 채워 올라가 빠르고 공간 최적화가 쉽다.
> - **백트래킹은 되돌릴 수 있는 완전탐색이다** — 해를 한 조각씩 만들며 유망하지 않은 경로를 조기 차단(가지치기)하고, 막히면 직전 선택을 원상복구해 되돌아간다. 선택·제약·목표 세 가지만 정하면 뼈대가 완성된다.
