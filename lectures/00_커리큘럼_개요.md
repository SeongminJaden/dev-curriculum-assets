# 개발 전체 커리큘럼 — 목차 및 로드맵

> **학습 목표**
> - 소프트웨어가 동작하는 원리를 하드웨어(반도체·컴퓨터 구조)부터 애플리케이션(백엔드·프론트엔드)까지 하나의 흐름으로 이해한다.
> - 각 계층의 개념이 실무에서 왜 그렇게 설계되었는지, 학문적 근거와 산업 사례를 함께 설명한다.

> **개요** 이 문서는 전체 14개 파트, 45개 강의로 구성된 개발 커리큘럼의 목차입니다. "왜 이 순서로 배우는가"를 먼저 정리한 뒤, 각 파트별 강의 목록과 핵심 질문을 제시합니다.

## 1. 왜 이 순서로 배우는가

> **Q.** 프론트엔드나 백엔드를 바로 배우지 않고 컴퓨터 구조부터 시작하는 이유는 무엇일까요?
> **A.** 상위 계층(언어, 프레임워크)의 설계 결정은 대부분 하위 계층(하드웨어, 운영체제)의 제약에서 비롯됩니다. 예를 들어 Python이 느린 이유, Rust가 GC 없이 메모리를 관리하는 이유, 마이크로서비스가 필요한 이유는 모두 하드웨어와 네트워크의 물리적 한계에서 출발합니다. 아래에서 위로(bottom-up) 쌓아 올려야 각 설계 결정을 "암기"가 아니라 "이해"로 받아들일 수 있습니다.

전체 흐름은 다음과 같습니다.

1. **개발 기초** — 코드를 다루는 데 필요한 최소한의 개념(패러다임, 자료구조, 버전 관리)
2. **컴퓨터 구조론** — 코드가 실제로 실행되는 물리적 기계의 원리
3. **서버와 시스템 아키텍처** — 여러 대의 컴퓨터가 네트워크로 연결되었을 때의 설계 문제
4. **프로세서 아키텍처 심화(x86 vs ARM, 통합 메모리 아키텍처)** — 같은 원리가 실제 반도체마다 어떻게 다르게 구현되는가
5. **언어별 심화** — 하드웨어 위에서 언어가 메모리와 실행을 다루는 방식의 차이
6. **기능별 심화** — 언어와 무관하게 모든 애플리케이션이 공통으로 마주치는 문제(DB, 동시성, API)
7. **백엔드 개발** — 서버 측 애플리케이션을 설계하는 구체적 방법론
8. **프론트엔드 개발** — 사용자 접점을 설계하는 구체적 방법론
9. **보안** — 앞의 모든 계층을 관통하는 위협과 방어 원리
10. **DevOps와 CI/CD** — 만든 코드를 안전하고 반복 가능하게 운영에 올리는 방법
11. **테스트와 코드 품질** — 변경을 두려워하지 않고 계속 개선할 수 있는 안전망
12. **모바일 개발** — 웹과 다른 제약(배터리, 네트워크, 생명주기) 위에서 앱을 설계하는 방법
13. **AI/ML 엔지니어링 기초** — 모델을 실제 서비스에 올리는 데 필요한 엔지니어링
14. **대규모 시스템 설계 사례** — 앞서 배운 원리들을 실제 설계 문제에 통합 적용

## 2. 파트별 목차

### Part 1. 개발 기초

> **개요** 이후 모든 파트에서 전제로 삼는 최소한의 개발 공통 언어(패러다임, 복잡도, 협업 도구)를 정리합니다.

| 강 | 제목 | 핵심 질문 |
|---|---|---|
| 1-1 | 소프트웨어 개발의 전체 그림 | 요구사항이 어떻게 실행 파일이 되어 사용자에게 도달하는가? |
| 1-2 | 프로그래밍 패러다임 | 절차적·객체지향·함수형은 같은 문제를 왜 다르게 푸는가? |
| 1-3 | 자료구조와 알고리즘 복잡도 | Big-O 표기법은 왜 "느리다/빠르다"보다 정확한 언어인가? |
| 1-4 | 버전 관리와 협업 워크플로우 | Git은 왜 단일 파일 백업 방식과 근본적으로 다른가? |

### Part 2. 컴퓨터 구조론

> **개요** 코드가 실행 파일로 바뀐 뒤 실제 하드웨어에서 어떻게 처리되는지, 폰노이만 구조부터 메모리 계층·병렬 처리까지 다룹니다.

| 강 | 제목 | 핵심 질문 |
|---|---|---|
| 2-1 | 폰노이만 구조와 컴퓨터의 5대 구성요소 | CPU와 메모리를 분리한 설계는 어떤 문제를 낳았는가? |
| 2-2 | CPU 명령어 실행 사이클과 파이프라이닝 | 명령어 하나를 처리하는 데 왜 여러 단계가 필요한가? |
| 2-3 | 메모리 계층 구조 | 캐시는 왜 "더 빠른데 더 작은" 메모리를 여러 겹으로 두는가? |
| 2-4 | 병렬 처리와 멀티코어 | 코어를 늘려도 프로그램이 항상 빨라지지 않는 이유는 무엇인가? |

### Part 3. 서버와 시스템 아키텍처

| 강 | 제목 | 핵심 질문 |
|---|---|---|
| 3-1 | 클라이언트-서버 모델과 네트워크 기초 | HTTP 요청 하나가 오가는 동안 어떤 계층을 거치는가? |
| 3-2 | 시스템 아키텍처 패턴 | 모놀리식에서 마이크로서비스로 옮겨가는 기준은 무엇인가? |
| 3-3 | 가상화와 컨테이너 | Docker는 가상 머신과 무엇이 다르고, 왜 더 가벼운가? |
| 3-4 | 클라우드 컴퓨팅 모델 | IaaS·PaaS·SaaS의 경계는 "누가 무엇을 관리하는가"로 어떻게 나뉘는가? |

### Part 4. 프로세서 아키텍처 심화 — x86 vs ARM & 통합 메모리 아키텍처

> **개요** 같은 폰노이만 원리를 서로 다른 철학으로 구현한 x86(CISC)과 ARM(RISC)을 비교하고, Apple Silicon·Jetson 등에 쓰이는 통합 메모리 아키텍처(UMA)의 원리를 다룹니다.

| 강 | 제목 | 핵심 질문 |
|---|---|---|
| 4-1 | CISC와 RISC — x86과 ARM 명령어 집합 구조 | 명령어를 "적고 단순하게" 두는 설계는 왜 저전력에 유리한가? |
| 4-2 | 반도체 설계 철학과 전력·성능 트레이드오프 | 같은 공정 노드에서도 설계에 따라 전력 효율이 갈리는 이유는 무엇인가? |
| 4-3 | 통합 메모리 아키텍처(UMA)의 원리 | CPU와 GPU가 메모리를 공유하면 왜 복사 비용이 사라지는가? |

### Part 5. 프로그래밍 언어별 심화

| 5-2 | 정적 타입과 동적 타입 | 타입을 컴파일 시점에 고정하는 것은 무엇을 방지하기 위함인가? |
