# Part 2. 컴퓨터 구조론

## 2-4. 병렬 처리와 멀티코어 — 코어를 늘려도 프로그램이 항상 빨라지지 않는 이유는 무엇인가?

> **학습 목표**
> - 암달의 법칙 공식을 직접 유도하고, 각 기호가 의미하는 바를 설명할 수 있다.
> - 병렬화 가능 비율(P)이 전체 속도 향상의 상한을 어떻게 결정하는지 계산할 수 있다.
> - 플린의 분류(SISD/SIMD/MISD/MIMD)를 기준으로 CPU와 GPU의 병렬 처리 방식이 어떻게 다른지 구분할 수 있다.

> **개요** 2-1강에서 2-3강까지는 CPU 코어 하나가 빨라지는 방법(파이프라이닝, 캐시)을 다뤘습니다. 이번 강의는 코어 자체를 여러 개로 늘리는 방법, 즉 멀티코어와 병렬 처리를 다룹니다. "코어가 4개면 4배 빨라지지 않을까"라는 직관이 왜 틀렸는지, 암달의 법칙을 통해 수식으로 확인합니다.

---

## 1. 왜 코어를 늘려도 배수만큼 빨라지지 않을까요?

> **Q.** 코어가 2개면 2배, 4개면 4배 빨라져야 할 것 같은데, 실제로는 왜 그렇지 않을까요?
> **A.** 프로그램의 모든 부분이 동시에 나눠서 처리할 수 있는 것은 아니기 때문입니다. 예를 들어 "재료를 준비하고 → 4명이 나눠서 볶고 → 한 접시에 담아 서빙한다"는 요리 과정에서, 볶는 단계는 4명이 나눠 할 수 있지만 재료 준비와 마지막 플레이팅은 한 사람(또는 정해진 순서)이 해야 합니다. 아무리 볶는 인원을 늘려도 준비와 플레이팅에 걸리는 시간은 줄어들지 않습니다. 프로그램도 마찬가지로 <u>**반드시 순서대로(직렬로) 실행해야 하는 부분**</u>이 있고, 이 부분이 전체 속도 향상의 한계를 결정합니다.

이 직관을 정확한 수식으로 정리한 것이 컴퓨터 구조학자 진 암달(Gene Amdahl)이 1967년에 발표한 <u>**암달의 법칙(Amdahl's Law)**</u>입니다.

---

## 2. 암달의 법칙 — 공식 유도

> **개요** "속도 향상 = 기존 실행 시간 ÷ 개선된 실행 시간"이라는 정의에서 출발해, 암달의 법칙 공식을 한 단계씩 유도합니다.

### 2.1 기호 정의

- **T** — 프로그램 전체를 1개의 프로세서로 실행했을 때 걸리는 총 실행 시간. 계산을 단순화하기 위해 **T = 1**로 정규화합니다.
- **P** — 전체 실행 시간 중 병렬화가 가능한 부분의 비율 (0 ≤ P ≤ 1)
- **1 − P** — 반드시 순서대로 실행해야 하는(병렬화할 수 없는) 직렬 부분의 비율
- **N** — 사용하는 프로세서(코어)의 개수
- **S(N)** — 프로세서 N개를 사용했을 때의 속도 향상(Speedup)

### 2.2 단계별 유도

1. 전체 실행 시간 T = 1을 두 부분으로 나눕니다. 직렬 부분의 시간은 `(1 − P)`이고, 병렬화 가능한 부분의 시간은 `P`입니다. (`(1-P) + P = 1`)
2. 프로세서를 N개 사용하면, 직렬 부분은 여전히 한 프로세서가 순서대로 처리해야 하므로 시간이 그대로 `(1 − P)`입니다. 반면 병렬화 가능한 부분은 이상적으로 N개 프로세서에 균등하게 나뉘므로 시간이 `P / N`으로 줄어듭니다.
3. 따라서 N개 프로세서를 사용했을 때의 총 실행 시간은 다음과 같습니다.

```
T(N) = (1 − P) + P / N
```

4. 속도 향상은 정의상 "기존 시간 ÷ 개선된 시간"이므로, T(1) = 1을 T(N)으로 나눕니다.

```
S(N) = T(1) / T(N) = 1 / [ (1 − P) + P / N ]
```

이것이 <u>**암달의 법칙**</u>입니다. 이 하나의 식이 "코어를 늘리는 것이 왜 무한정한 이득을 주지 않는지"를 정확히 설명합니다.

> **참고** N이 무한히 커지면 `P / N`은 0에 가까워지므로, 속도 향상의 이론적 상한은 다음과 같이 수렴합니다.
> ```
> lim(N→∞) S(N) = 1 / (1 − P)
> ```
> 즉 <u>**직렬 부분의 비율(1 − P)이 전체 속도 향상의 한계를 결정**</u>합니다. 병렬화 가능한 부분이 아무리 커도, 직렬 부분이 조금만 남아 있어도 속도 향상은 특정 값 이상 올라가지 못합니다.

![암달의 법칙 그래프](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part2_컴퓨터구조론/2-4_병렬_처리와_멀티코어/assets/diagram1_amdahl_graph.svg)

> **해석** 위 그래프는 병렬화 가능 비율 P를 0.5, 0.75, 0.9, 0.95로 고정하고, 프로세서 수 N을 1부터 64까지 늘렸을 때의 속도 향상을 그린 것입니다. P=0.5(전체의 절반만 병렬화 가능)인 경우 코어를 아무리 늘려도 속도 향상은 2배 부근에서 평평해집니다. 반면 P=0.95인 경우에는 N=64에서도 여전히 가파르게 상승하지만, 이 곡선도 결국 1/(1-0.95)=20배에서 수렴합니다. <u>**곡선이 평평해지는 지점이 "코어를 더 늘려도 소용없는 지점"**</u>입니다.

### 2.3 직접 계산해보기

예를 들어 어떤 프로그램의 90%가 병렬화 가능하다면(P = 0.9), 코어 10개를 쓸 때의 속도 향상은 다음과 같이 계산됩니다.

```
S(10) = 1 / [ (1 − 0.9) + 0.9 / 10 ] = 1 / [ 0.1 + 0.09 ] = 1 / 0.19 ≈ 5.26배
```

코어를 10개 썼지만 10배가 아니라 약 5.26배 빨라졌습니다. 코어 수를 아무리 늘려도 이 프로그램의 속도 향상은 `1 / (1 − 0.9) = 10배`를 절대 넘을 수 없습니다.

> **Q.** 실무에서 암달의 법칙이 왜 중요할까요?
> **A.** 서버를 스케일업(코어 추가)할지 결정할 때, "이 작업의 몇 퍼센트가 병렬화 가능한가"를 먼저 파악하지 않으면 예산만 쓰고 성능은 기대만큼 오르지 않는 상황이 생기기 때문입니다. 데이터베이스 트랜잭션 로그처럼 순서를 반드시 지켜야 하는 작업이 섞여 있는 시스템은, 코어를 아무리 늘려도 그 직렬 구간이 전체 처리량의 병목으로 남습니다. 암달의 법칙은 "코어를 늘리는 투자가 어디까지 의미 있는가"를 수치로 판단하게 해줍니다.

## 3. 병렬 처리의 두 축 — SIMD와 MIMD

> **개요** 같은 "병렬 처리"라도 무엇을 여러 개로 늘리느냐에 따라 방식이 다릅니다. 컴퓨터 구조학자 마이클 플린(Michael Flynn)이 1966년 제안한 분류 기준으로 이를 정리합니다.

플린의 분류(Flynn's Taxonomy)는 "동시에 처리하는 명령어 스트림이 몇 개인가"와 "동시에 처리하는 데이터 스트림이 몇 개인가"라는 두 기준으로 컴퓨터의 처리 방식을 네 가지로 나눕니다.

![플린의 분류 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part2_컴퓨터구조론/2-4_병렬_처리와_멀티코어/assets/diagram2_flynn_taxonomy.svg)

- **SISD (Single Instruction, Single Data)** — 전통적인 단일 코어 프로세서가 명령어를 하나씩 순서대로 실행하는 방식입니다. 2-2강에서 배운 파이프라이닝조차 결국 SISD를 더 빠르게 만드는 기법입니다.
- **SIMD (Single Instruction, Multiple Data)** — 하나의 명령어로 여러 데이터를 동시에 처리합니다. 이미지의 모든 픽셀에 같은 밝기 보정을 적용하는 연산처럼, "같은 연산을 서로 다른 데이터에 반복"하는 작업에 적합합니다. GPU가 그래픽·AI 연산에 강한 이유가 바로 수천 개의 연산 유닛으로 SIMD를 대규모로 수행하기 때문이며, CPU도 AVX와 같은 벡터 명령어 집합으로 이를 부분 지원합니다.
- **MISD (Multiple Instruction, Single Data)** — 여러 명령어가 같은 데이터를 각기 다른 방식으로 동시에 처리합니다. 범용 컴퓨팅에서는 거의 쓰이지 않고, 같은 데이터를 여러 유닛이 독립적으로 계산해 결과를 대조하는 결함 허용(fault-tolerant) 시스템 등 특수한 분야에서만 제한적으로 쓰입니다.
- **MIMD (Multiple Instruction, Multiple Data)** — 여러 코어가 각자 다른 명령어로 다른 데이터를 독립적으로 처리합니다. 오늘날의 멀티코어 CPU, 그리고 여러 대의 서버로 구성된 분산 시스템이 MIMD에 해당합니다.

> **Q.** 그렇다면 멀티코어 CPU와 GPU는 왜 이렇게 다르게 설계되었을까요?
> **A.** 둘이 잘하는 병렬성의 "종류"가 다르기 때문입니다. CPU의 코어들은 각자 독립적인 제어장치를 가진 MIMD 구조로, 서로 전혀 다른 작업(웹 서버 요청 처리, 파일 압축, UI 렌더링)을 동시에 처리하는 데 강합니다. GPU는 하나의 명령어를 수천 개의 단순한 연산 유닛에 동시에 뿌리는 SIMD 구조로, 동일한 연산을 대량의 데이터에 반복해야 하는 행렬 연산이나 그래픽 처리에 압도적으로 강합니다. 인공지능 모델 학습에 GPU를 쓰는 이유도 신경망 연산 대부분이 동일한 행렬 곱셈을 대량으로 반복하는 SIMD 친화적 작업이기 때문입니다.

### 실생활 예시 — 이삿짐 나르기

> **해석** 아래 표는 암달의 법칙과 SIMD/MIMD 개념을 이삿짐을 나르는 상황에 빗댄 것입니다.

| 상황 | 대응 개념 | 설명 |
|---|---|---|
| 트럭에 짐을 미리 순서대로 싣고, 마지막에 확인 서명을 한 사람이 해야 한다 | 암달의 법칙의 직렬 구간 | 사람을 아무리 늘려도 이 단계는 줄어들지 않는다 |
| 박스 20개를 나르는 일을 인부 4명이 5개씩 나눠 든다 | 암달의 법칙의 병렬 구간 | 인부 수에 비례해 이 구간의 시간이 줄어든다 |
| 인부 4명이 각자 다른 방(주방, 침실, 거실, 서재)의 짐을 동시에 챙긴다 | MIMD | 각자 다른 종류의 일을 동시에 한다 |
| 인부 4명이 같은 종류의 박스 20개를 동시에 같은 방식으로 포장한다 | SIMD | 같은 동작을 여러 대상에 동시에 반복한다 |

> **팁** 실무에서 "왜 이 작업은 서버를 늘려도 안 빨라지나요"라는 질문을 받으면, 먼저 그 작업에 반드시 순서를 지켜야 하는 구간(잠금, 트랜잭션, 순차 로그 기록)이 얼마나 되는지부터 확인해야 합니다. 그 구간을 줄이지 않는 한 코어나 서버를 늘리는 것은 암달의 법칙이 정한 상한 근처에서 효과가 급격히 줄어듭니다.

> **핵심**
> - 암달의 법칙 **S(N) = 1 / [(1 − P) + P / N]** 은 병렬화 가능 비율 P와 프로세서 수 N으로 속도 향상의 상한을 계산하는 공식이며, N이 무한대로 가도 속도 향상은 **1 / (1 − P)**를 넘지 못합니다.
> - 속도 향상의 한계를 결정하는 것은 병렬화 가능한 부분이 아니라 <u>**반드시 직렬로 처리해야 하는 부분의 비율**</u>입니다.
> - 플린의 분류에서 **SIMD**는 하나의 명령어로 다량의 데이터를 동시에 처리(GPU에 적합), **MIMD**는 여러 코어가 각자 다른 명령어와 데이터를 독립적으로 처리(멀티코어 CPU에 적합)하는 방식입니다.

## 요약

> **핵심**
> - **암달의 법칙(S(N) = 1 / [(1 − P) + P / N])**은 프로그램의 병렬화 가능 비율(P)이 코어 수(N)를 늘렸을 때의 속도 향상 상한을 결정한다는 것을 수식으로 보여줍니다.
> - N을 무한히 늘려도 속도 향상은 **1 / (1 − P)**를 넘지 못하며, 이는 직렬 구간의 비율이 성능의 진짜 병목임을 의미합니다.
> - **플린의 분류(SISD/SIMD/MISD/MIMD)**는 병렬 처리의 종류를 구분하는 표준 기준이며, GPU의 SIMD와 멀티코어 CPU의 MIMD는 서로 다른 종류의 작업에 최적화되어 있습니다.
