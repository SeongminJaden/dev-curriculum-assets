# Part 2. 컴퓨터 구조론

## 2-2. CPU 명령어 실행 사이클과 파이프라이닝 — 명령어 하나를 처리하는 데 왜 여러 단계가 필요한가?

> **학습 목표**
> - 명령어 실행 사이클(Fetch-Decode-Execute)의 각 단계가 무엇을 하는지 설명할 수 있다.
> - 5단계 파이프라인이 무엇이며, 왜 처리량(throughput)을 늘리는 방향으로 설계되었는지 설명할 수 있다.
> - 파이프라인 해저드가 무엇이고, 이것이 왜 "이상적인 속도 향상"을 방해하는지 설명할 수 있다.

> **개요** 2-1강에서 폰노이만 병목, 즉 CPU가 기억장치를 기다리며 시간을 허비하는 문제를 배웠습니다. 이번 강의는 CPU가 그 기다리는 시간을 어떻게 활용하는지, 즉 명령어 하나를 처리하는 과정을 여러 단계로 쪼개고 그 단계들을 겹쳐 실행하는 파이프라이닝을 다룹니다.

---

## 1. 명령어 하나는 사실 "여러 개의 작은 일"이다

> **Q.** "명령어 하나를 실행한다"는 것은 CPU 입장에서 정확히 어떤 뜻일까요?
> **A.** 우리가 코드에서 `a = b + c` 한 줄을 쓰면, CPU는 이를 하나의 동작이 아니라 여러 개의 물리적 단계로 나누어 처리합니다. 명령어를 기억장치에서 읽어오고, 그 명령어가 무슨 뜻인지 해석하고, 실제 덧셈을 수행하고, 결과를 저장하는 과정이 모두 별도의 회로에서 순차적으로 일어납니다. 이 단계를 통틀어 <u>**명령어 실행 사이클(Instruction Cycle)**</u>이라 부릅니다.

![명령어 실행 사이클 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part2_컴퓨터구조론/2-2_CPU_명령어_실행_사이클과_파이프라이닝/assets/diagram1_fde_cycle.svg)

가장 단순화된 형태는 3단계, 세부적으로는 4단계로 나뉩니다.

1. **인출 (Fetch)** — 프로그램 카운터(Program Counter, PC)가 가리키는 기억장치 주소에서 다음 명령어를 읽어옵니다.
2. **해독 (Decode)** — 읽어온 명령어(비트 패턴)가 어떤 연산이고, 어떤 레지스터를 사용하는지 제어장치가 해석합니다.
3. **실행 (Execute)** — 해독된 내용에 따라 ALU가 실제 연산을 수행합니다.
4. **저장 (Writeback)** — 연산 결과를 레지스터나 기억장치에 반영하고, PC를 다음 명령어 주소로 갱신합니다.

> **참고** 이 4단계는 매 클럭 사이클(clock cycle)마다 반복됩니다. CPU의 클럭 속도(예: 3.5GHz)란 이 사이클을 구성하는 각 단계를 얼마나 촘촘한 시간 간격으로 진행할 수 있는지를 나타내는 지표입니다.

---

## 2. 왜 이 단계를 겹쳐 실행할까 — 파이프라이닝

> **개요** 4단계를 하나씩 순서대로 끝내고 다음 명령어로 넘어가는 방식이 왜 비효율적인지, 그리고 이를 해결하는 파이프라이닝의 원리를 다룹니다.

만약 CPU가 명령어 하나의 4단계를 모두 끝낸 뒤에야 다음 명령어의 인출을 시작한다면, 인출 단계를 처리하는 회로는 나머지 3단계가 진행되는 동안 그냥 놀고 있게 됩니다. <u>**파이프라이닝(Pipelining)**</u>은 이 유휴 시간을 없애기 위해, 한 명령어가 해독 단계로 넘어가는 순간 다음 명령어의 인출을 동시에 시작하는 방식입니다.

현대 CPU가 표준적으로 다루는 형태는 5단계 파이프라인입니다.

- **IF (Instruction Fetch)** — 명령어 인출
- **ID (Instruction Decode)** — 명령어 해독 및 레지스터 값 읽기
- **EX (Execute)** — ALU 연산 수행
- **MEM (Memory Access)** — 필요시 기억장치 접근(적재/저장)
- **WB (Write Back)** — 결과를 레지스터에 기록

![5단계 파이프라인 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part2_컴퓨터구조론/2-2_CPU_명령어_실행_사이클과_파이프라이닝/assets/diagram2_pipeline.svg)

> **해석** 위 다이어그램의 대각선 배치를 보면, 클럭 사이클 5에서 I1은 WB 단계, I2는 MEM 단계, I3는 EX 단계, I4는 ID 단계, I5는 IF 단계를 동시에 진행하고 있습니다. 다섯 개의 서로 다른 회로(IF용, ID용, EX용, MEM용, WB용 회로)가 동시에 다섯 개의 서로 다른 명령어를 처리하는 것입니다. 이것이 파이프라이닝의 핵심입니다. <u>**한 명령어의 지연시간(latency)은 줄지 않지만, 단위 시간당 처리하는 명령어 수(처리량, throughput)는 늘어납니다.**</u>

### 처리량 공식

파이프라인이 없는 경우, n개의 명령어를 처리하는 데 걸리는 사이클은 `n × k`입니다(k는 단계 수). 이상적인 k단계 파이프라인이 있는 경우, n개의 명령어를 처리하는 데 걸리는 사이클은 다음과 같습니다.

```
사이클 수 = k + (n - 1)
```

여기서 `k`는 처음 한 명령어가 전 단계를 채우기까지 걸리는 사이클(파이프라인을 "채우는" 시간), `n - 1`은 이후 각 명령어가 한 사이클마다 하나씩 완료되는 구간입니다. n이 충분히 크면 이 식은 사실상 `n` 사이클에 수렴하므로, 이상적인 속도 향상(speedup)은 파이프라인 단계 수 k에 가까워집니다.

> **Q.** 그렇다면 단계를 더 잘게 쪼갤수록(파이프라인을 더 깊게 만들수록) 계속 빨라질까요?
> **A.** 이론적으로는 그렇지만, 실제로는 두 가지 제약에 부딪힙니다. 첫째, 각 단계를 나누는 데도 회로적 오버헤드(단계 사이의 레지스터, 제어 로직)가 있어 무한히 쪼갤 수 없습니다. 둘째, 아래에서 다루는 해저드가 파이프라인이 깊어질수록 더 자주, 더 큰 손실로 발생합니다. 2000년대 초 일부 CPU가 20단계가 넘는 파이프라인(Pentium 4의 초장 파이프라인)을 시도했지만, 해저드로 인한 손실과 발열 문제로 이후 세대에서는 오히려 단계를 줄이는 방향으로 돌아섰습니다.

## 3. 파이프라인 해저드 — 이상적인 속도 향상을 방해하는 것

> **개요** 파이프라인이 항상 이상적으로 흐르지는 않습니다. 명령어들이 서로 얽혀 있을 때 발생하는 세 가지 해저드를 살펴봅니다.

<u>**해저드(Hazard)**</u>란 파이프라인의 다음 명령어가 정상적인 사이클에 자신의 단계를 진행하지 못하고 멈추거나(stall), 잘못된 값을 사용할 위험이 있는 상황을 말합니다. 표준적으로 세 가지로 분류합니다.

| 해저드 종류 | 발생 원인 | 대응 방식 |
|---|---|---|
| 구조적 해저드 (Structural Hazard) | 두 명령어가 동시에 같은 하드웨어 자원(예: 같은 메모리 포트)을 필요로 함 | 자원을 이중화하거나 한쪽을 대기시킴 |
| 데이터 해저드 (Data Hazard) | 뒤 명령어가 앞 명령어의 연산 결과를 아직 저장되지 않은 상태에서 필요로 함 | 포워딩(forwarding)으로 결과를 미리 전달하거나, 버블(bubble)을 삽입해 대기 |
| 제어 해저드 (Control Hazard) | 분기(if, 반복문) 명령어의 결과가 나오기 전까지 다음에 인출할 명령어를 알 수 없음 | 분기 예측(branch prediction)으로 미리 추측해 진행 |

![파이프라인 해저드 예시](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part2_컴퓨터구조론/2-2_CPU_명령어_실행_사이클과_파이프라이닝/assets/diagram2_pipeline.svg)

> **해석** 다이어그램 하단의 Ia, Ib 예시는 데이터 해저드를 보여줍니다. Ib의 EX 단계가 Ia의 연산 결과를 필요로 하는데, 그 결과는 Ia의 WB 단계에서야 레지스터에 기록됩니다. 이 격차를 메우기 위해 하드웨어는 Ib의 EX 단계 진입을 한 사이클 늦추는 "버블"을 끼워 넣습니다. 버블 한 칸은 곧 손실된 처리량 한 칸입니다.

> **참고** 포워딩(forwarding, 혹은 bypassing)은 WB 단계까지 기다리지 않고, ALU가 연산 결과를 만들어내는 즉시 그 값을 필요로 하는 다음 명령어의 EX 단계로 직접 전달하는 회로적 지름길입니다. 이를 통해 다수의 데이터 해저드는 버블 없이도 해결되며, 실제 CPU에서는 버블이 필요한 경우가 이론상 최악의 경우보다 훨씬 적습니다.

### 실생활 예시 — 세탁소의 빨래 처리 과정

> **해석** 아래 표는 컴퓨터 구조론 교재에서 파이프라이닝을 설명할 때 자주 쓰이는 세탁 비유를 표로 정리한 것입니다. 세탁(Wash) → 건조(Dry) → 개기(Fold) → 정리(Store)라는 4단계를 4벌의 빨래에 적용합니다.

| 방식 | 처리 방식 | 4벌을 마치는 데 걸리는 시간(각 단계 30분 가정) |
|---|---|---|
| 파이프라인 없음 | 1벌의 세탁-건조-개기-정리를 모두 끝낸 뒤 다음 벌 시작 | 4벌 × 4단계 × 30분 = 480분 |
| 파이프라인 적용 | 1벌이 건조기로 넘어가는 즉시 2벌째 세탁 시작 | 4단계 채우는 시간 + 나머지 3벌 각 30분 = (4+3) × 30분 = 210분 |

> **팁** 이 비유에서 "세탁기가 고장 나서 건조기가 한 대뿐이라 순서를 기다려야 한다"는 상황이 구조적 해저드에 해당하고, "3번째 빨래를 개키는 사람이 2번째 빨래가 완전히 마르기를 기다려야 한다"는 상황이 데이터 해저드에 해당합니다. 실무에서 컴파일러 최적화나 CPU의 비순차 실행(out-of-order execution)은 결국 이 대기 시간을 최대한 줄이려는 시도입니다.

> **핵심**
> - 명령어 실행 사이클은 인출(IF) → 해독(ID) → 실행(EX) → 메모리 접근(MEM) → 저장(WB)의 단계로 이루어지며, 표준적인 파이프라인은 이 5단계를 겹쳐 실행합니다.
> - 파이프라이닝은 명령어 하나의 지연시간을 줄이지 않고 <u>**처리량(단위 시간당 처리하는 명령어 수)을 늘리는**</u> 설계이며, 이상적인 속도 향상은 파이프라인 단계 수에 수렴합니다.
> - 구조적·데이터·제어 해저드는 이 이상적인 속도 향상을 갉아먹으며, 포워딩과 분기 예측 같은 기법은 모두 해저드로 인한 손실을 줄이기 위한 하드웨어적 장치입니다.

## 요약

> **핵심**
> - CPU는 명령어 하나를 인출·해독·실행·메모리 접근·저장의 여러 단계로 나누어 처리하며, 이를 **명령어 실행 사이클**이라 부릅니다.
> - **파이프라이닝**은 이 단계들을 여러 명령어에 걸쳐 겹쳐 실행함으로써, 지연시간이 아니라 **처리량**을 늘리는 설계입니다.
> - 구조적·데이터·제어 **해저드**는 파이프라인이 항상 이상적으로 흐르지 못하게 만드는 원인이며, 포워딩과 분기 예측은 그 손실을 최소화하는 대응책입니다.
