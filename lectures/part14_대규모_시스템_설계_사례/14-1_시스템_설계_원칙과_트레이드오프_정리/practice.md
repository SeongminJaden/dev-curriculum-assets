# 14-1 실습 — 시스템 설계 원칙과 트레이드오프 정리

> **참고** 제출물이 아니라, 오늘 배운 다섯 단계 절차와 판단 틀을 스스로 적용해보는 실습입니다.

## 실습 1. 서비스 하나를 골라 다섯 단계를 따라가보기

- [ ] 본인이 최근에 사용한 서비스를 하나 골라주세요. (예: 대학교 수강신청 시스템, 학교 커뮤니티 게시판)
- [ ] 그 서비스가 해야 할 기능 요구사항을 두 가지만 문장으로 적어주세요.
- [ ] 그 서비스의 비기능 요구사항(응답 속도, 가용성, 일관성 요구 수준)을 한 문장씩 적어주세요.
- [ ] 아래 표를 채워 용량을 어림 계산해보세요.
  - [ ] 일일 활성 사용자 수를 가정해보세요.
  - [ ] 사용자당 하루 평균 요청 횟수를 가정해보세요.
  - [ ] 평균 QPS와 피크 QPS(피크 배수 2~3배)를 계산해보세요.
- [ ] 계산된 QPS를 근거로, 캐싱이나 샤딩이 필요한지 아닌지 스스로 판단해보고 이유를 적어주세요.

## 실습 2. 판단 틀 적용하기

- [ ] 실습 1에서 고른 서비스에 대해 아래 네 가지 질문에 답해보세요.
  - [ ] 읽기와 쓰기 중 무엇이 더 많을까요?
  - [ ] 일관성과 가용성 중 무엇을 더 우선해야 할까요? (CAP 이론 관점에서 CP·AP 중 선택)
  - [ ] 트래픽이 몰릴 때 어떤 방식으로 나눌 수 있을까요?
  - [ ] 서비스 내부에서 즉시 응답이 필요 없는 작업(알림, 통계 집계 등)이 있다면 무엇일까요?
- [ ] 네 가지 답변이 서로 모순되지 않는지 확인해보세요. (예: 가용성을 우선한다고 답했는데 캐시 TTL을 매우 짧게 잡는 것은 모순일 수 있습니다.)

## 생각해볼 질문

- [ ] 용량 추정 단계를 건너뛰고 바로 상세 설계로 들어가면 어떤 문제가 생길까요?
- [ ] "좋은 설계"란 트레이드오프가 없는 설계일까요, 아니면 다른 무엇일까요? 본인의 언어로 정리해보세요.
