# 14-2 실습 — 사례 연구: URL 단축기·뉴스피드 설계

> **참고** 제출물이 아니라, 오늘 배운 두 사례를 스스로 다시 설계해보는 실습입니다.

## 실습 1. URL 단축기 용량 추정 다시 해보기

- [ ] 월간 신규 URL 생성 건수를 강의와 다른 값(예: 5억 건)으로 바꿔서 하루 평균 쓰기 QPS를 다시 계산해보세요.
- [ ] 읽기:쓰기 비율을 100:1이 아니라 500:1로 바꾸면 평균 읽기 QPS가 얼마가 되는지 계산해보세요.
- [ ] 바뀐 QPS 수치를 근거로, 캐시 서버를 몇 대 이상 두어야 할지 스스로 근거를 들어 판단해보세요.

## 실습 2. 단축 코드 생성 방식 비교하기

- [ ] 자동 증가 ID + Base62 인코딩 방식과 해시 기반 방식 중, 하나를 골라 그 이유를 두 문장으로 적어보세요.
- [ ] 여러 대의 서버가 동시에 단축 코드를 발급해야 하는 상황에서, 자동 증가 ID 방식이 겪을 수 있는 문제와 해결 방법(Snowflake 등)을 찾아 정리해보세요.
- [ ] 리디렉션 응답을 301로 할지 302로 할지, 서비스에 클릭 통계 기능이 있다고 가정하고 선택해보세요.

## 실습 3. 뉴스피드 팬아웃 전략 설계하기

- [ ] 팔로워 수가 각각 50명, 5,000명, 500만 명인 세 계정에 대해 팬아웃 온 라이트를 적용했을 때 게시물 1건당 팬아웃 쓰기 건수를 계산해보세요.
- [ ] 위 세 계정 중 어디까지 팬아웃 온 라이트를, 어디부터 팬아웃 온 리드를 적용할지 임계치를 정하고 이유를 적어보세요.
- [ ] 하이브리드 전략을 적용했을 때, 팔로우 대상 중 유명 계정과 일반 계정이 섞여 있는 사용자의 피드는 어떻게 조합해서 보여줘야 할지 생각해보세요.

## 생각해볼 질문

- [ ] URL 단축기와 뉴스피드 중, Part 6-1의 CAP 이론 관점에서 더 강한 일관성(C)을 요구하는 쪽은 어디일까요? 이유도 함께 적어보세요.
- [ ] 만약 뉴스피드에 "실시간 좋아요 수"까지 함께 보여줘야 한다면, 이 강의에서 다룬 캐싱·팬아웃 전략에 어떤 추가 고려사항이 생길까요?
