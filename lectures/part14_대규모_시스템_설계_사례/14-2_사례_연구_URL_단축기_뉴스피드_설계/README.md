# Part 14. 대규모 시스템 설계 사례

## 14-2. 사례 연구 — URL 단축기·뉴스피드 설계

> **학습 목표**
> - URL 단축기를 요구사항 파악부터 고수준 설계까지 직접 따라가며, 해시(단축 코드) 생성 방식과 캐싱 전략을 설명할 수 있다.
> - 뉴스피드 설계에서 팬아웃 온 라이트(fan-out-on-write)와 팬아웃 온 리드(fan-out-on-read)의 차이와 각각의 비용을 설명할 수 있다.
> - "읽기가 압도적으로 많다"는 같은 특성을 가진 두 서비스가 왜 서로 다른 설계로 귀결되는지, 데이터 구조의 차이로 설명할 수 있다.

> **개요** 14-1에서 정리한 다섯 단계(요구사항 파악 → 용량 추정 → 고수준 설계 → 상세 설계 → 트레이드오프 논의)를 URL 단축기와 뉴스피드라는 두 서비스에 실제로 적용해봅니다. 두 서비스 모두 "쓰기보다 읽기가 훨씬 많다"는 같은 트래픽 패턴을 갖지만, 데이터 사이의 관계 구조가 다르기 때문에 전혀 다른 설계로 이어집니다.

---

## 1. 왜 같은 "많이 읽고 가끔 쓰는" 패턴이 다른 설계로 이어질까요?

> **Q.** URL 단축기와 뉴스피드는 둘 다 읽기:쓰기 비율이 100:1에 가깝습니다. 그런데도 왜 설계가 완전히 달라질까요?
> **A.** 읽기·쓰기 비율은 같아도 "읽는 대상"의 성격이 다르기 때문입니다. URL 단축기의 읽기는 단축 코드 하나에 원본 URL 하나가 대응하는 <u>**단순 키-값 조회**</u>이고, 데이터 사이에 관계가 없습니다. 반면 뉴스피드의 읽기는 "내가 팔로우하는 모든 사람의 최신 게시물을 시간순으로 모아서" 보여주는 <u>**관계 기반 집계**</u>입니다. 같은 읽기 병목이라도 전자는 캐시 하나로 거의 해결되지만, 후자는 팔로우 관계 그래프 전체를 어떻게 미리 계산해둘지를 별도로 설계해야 합니다.

이 강의는 두 서비스를 각각 14-1의 절차대로 짚어봅니다.

- **사례 1 — URL 단축기**: 단순 키-값 조회 문제. 해시(단축 코드) 생성 방식과 캐싱이 핵심입니다.
- **사례 2 — 뉴스피드**: 관계 기반 집계 문제. 팬아웃 온 라이트와 팬아웃 온 리드의 트레이드오프가 핵심입니다.

## 2. 사례 1 — URL 단축기

> **개요** 긴 URL을 짧은 코드로 바꾸고, 그 코드로 접속하면 원본으로 리디렉션하는 서비스를 설계합니다.

### 2.1 요구사항 파악

- **기능 요구사항**: 사용자는 긴 URL을 제출해 짧은 URL을 받을 수 있어야 하고, 짧은 URL로 접속하면 원본 URL로 리디렉션되어야 합니다.
- **비기능 요구사항**: 리디렉션은 매우 낮은 지연 시간(수십 ms) 안에 응답해야 하고, 짧은 URL은 한 번 생성되면 영구적으로 유지되어야 합니다.

### 2.2 용량 추정

> **해석** 월간 신규 URL 생성이 1억 건, 읽기(리디렉션)가 쓰기(생성)보다 100배 많다고 가정했을 때의 계산입니다.

| 항목 | 가정 | 계산 | 결과 |
|---|---|---|---|
| 월간 신규 URL | — | — | 100,000,000건 |
| 하루 신규 URL | 30일 기준 | 100,000,000 ÷ 30 | 약 3,300,000건/일 |
| 평균 쓰기 QPS | 하루 86,400초에 분산 | 3,300,000 ÷ 86,400 | 약 38 QPS |
| 평균 읽기 QPS | 읽기:쓰기 = 100:1 | 38 × 100 | 약 3,800 QPS |
| URL 1건 저장 용량 | 단축 코드 7자 + 원본 URL 평균 100바이트 | — | 약 100바이트 |
| 5년 누적 저장 용량 | 60개월 누적 | 100,000,000 × 60 × 100바이트 | 약 600GB |

> **팁** 쓰기 QPS(약 38)는 그리 크지 않지만 읽기 QPS(약 3,800)는 무시할 수 없는 규모입니다. 이 격차가 "쓰기 경로는 단순하게, 읽기 경로는 캐시로 방어한다"는 이후 설계 방향을 결정합니다. 저장 용량도 5년치를 다 모아도 600GB 수준이라 단일 DB로 충분히 감당할 수 있다는 것을 미리 확인할 수 있습니다.

### 2.3 고수준 설계 — 쓰기 경로와 읽기 경로

![URL 단축기 쓰기 경로와 읽기 경로 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part14_대규모_시스템_설계_사례/14-2_사례_연구_URL_단축기_뉴스피드_설계/assets/diagram1_url_shortener.svg)

> **해석** 쓰기 경로는 원본 URL을 받아 고유 ID를 발급받고, 그 ID를 짧은 문자열로 인코딩해 DB에 매핑을 저장합니다. 읽기 경로는 짧은 URL로 접속했을 때 먼저 캐시를 확인하고, 있으면(히트) 즉시 리디렉션하며, 없으면(미스) DB를 조회한 뒤 캐시를 채우고 리디렉션합니다.

### 2.4 상세 설계 — 단축 코드는 어떻게 만들어질까요

> **Q.** 원본 URL을 그냥 MD5나 SHA 같은 해시 함수에 넣어 앞 7자리만 잘라 쓰면 안 될까요?
> **A.** 가능은 하지만 <u>**해시 충돌(Hash Collision)**</u> 문제가 남습니다. 서로 다른 두 URL이 같은 7자리 해시값을 만들어낼 가능성이 있고, 이 경우 충돌을 감지해 다시 만드는 절차가 필요해 로직이 복잡해집니다. 그래서 실무에서는 아래 두 방식을 더 흔히 씁니다.

| 방식 | 원리 | 장점 | 단점 |
|---|---|---|---|
| 자동 증가 ID + Base62 인코딩 | DB가 발급한 정수 ID(1, 2, 3…)를 62진법(0-9, a-z, A-Z) 문자열로 변환 | 충돌이 원천적으로 없음, 구현이 단순 | 단일 ID 발급 지점이 병목이 될 수 있음(분산 ID 생성기로 완화) |
| 해시 + 충돌 처리 | 원본 URL을 해시한 뒤 앞자리를 자르고, 이미 쓰인 값이면 salt를 더해 재해시 | ID 발급 서버 없이 각 노드가 독립적으로 생성 가능 | 충돌 감지·재시도 로직이 추가로 필요 |

<u>**Base62 인코딩**</u>은 숫자를 0~9, a~z, A~Z로 이루어진 62개의 문자로 표현하는 방식입니다. 정수 하나를 62진법으로 변환하면 10진법보다 훨씬 짧은 문자열이 되므로, 7자리만으로도 62^7(약 3.5조) 개의 서로 다른 코드를 표현할 수 있어 당분간 코드가 고갈될 걱정이 없습니다.

> **참고** 자동 증가 ID 방식에서 ID 발급 지점이 단일 병목이 되는 문제는, 여러 서버가 서로 겹치지 않는 ID 범위를 미리 나눠 갖거나(range 할당), 트위터의 Snowflake처럼 타임스탬프와 서버 번호를 조합해 각 서버가 독립적으로 고유 ID를 생성하는 방식으로 완화합니다.

### 2.5 캐싱 전략과 트레이드오프

읽기 QPS(약 3,800)가 쓰기 QPS(약 38)보다 100배 많다는 용량 추정 결과는, Part 7-3에서 다룬 <u>**캐시 어사이드(Cache-aside) 패턴**</u>을 그대로 적용할 근거가 됩니다. 전체 URL 중에서도 실제로는 소수의 인기 URL(뉴스 기사, 이벤트 링크 등)에 접속이 몰리는 경향(파레토 분포)이 있으므로, 캐시 크기가 전체 데이터보다 훨씬 작아도 높은 히트율을 기대할 수 있습니다.

리디렉션 응답에는 HTTP 301(영구 이동)과 302(임시 이동) 중 하나를 선택해야 하는데, 이 선택 자체가 트레이드오프입니다.

| 상태 코드 | 브라우저 동작 | 서버 측 영향 |
|---|---|---|
| 301 (Moved Permanently) | 브라우저가 결과를 캐시해, 다음부터는 서버를 거치지 않고 바로 원본으로 이동 | 서버 트래픽은 줄지만, 클릭 수를 집계하는 분석 기능을 쓸 수 없음 |
| 302 (Found) | 브라우저가 매번 서버에 다시 확인 | 클릭 수 집계가 가능하지만 서버가 모든 리디렉션 요청을 계속 받아야 함 |

> **원칙** URL 단축 서비스가 클릭 통계를 제공한다면 302를 선택해 모든 요청이 서버를 거치도록 해야 하고, 순수하게 리디렉션 속도와 서버 부하 절감이 우선이라면 301을 선택합니다. 이는 Part 14-1에서 다룬 "트레이드오프는 요구사항에 따라 선택하는 것"이라는 원칙이 HTTP 상태 코드 하나를 고르는 수준까지 내려온 사례입니다.

## 3. 사례 2 — 뉴스피드

> **개요** 사용자가 팔로우한 대상들의 게시물을 시간순으로 모아 보여주는 뉴스피드를 설계합니다.

### 3.1 요구사항 파악

- **기능 요구사항**: 사용자는 글을 게시할 수 있고, 다른 사용자를 팔로우할 수 있으며, 자신이 팔로우한 사람들의 게시물을 최신순으로 모아 볼 수 있어야 합니다.
- **비기능 요구사항**: 피드 조회는 빠르게 응답해야 하고(체감 지연이 곧 서비스 이탈로 이어짐), 새 게시물이 팔로워의 피드에 반영되는 데는 다소 지연이 있어도 괜찮습니다(Part 6-1의 CAP 이론 관점에서 AP에 가까운 요구사항).

### 3.2 용량 추정

> **해석** DAU 1,000만 명, 사용자당 팔로워 평균 200명, 하루 평균 게시물 0.5건, 피드 조회 20회를 가정했을 때의 계산입니다.

| 항목 | 가정 | 계산 | 결과 |
|---|---|---|---|
| 일일 활성 사용자(DAU) | — | — | 10,000,000명 |
| 하루 신규 게시물 | 사용자당 0.5건/일 | 10,000,000 × 0.5 | 5,000,000건/일 |
| 평균 쓰기 QPS | 하루 86,400초에 분산 | 5,000,000 ÷ 86,400 | 약 58 QPS |
| 하루 피드 조회 | 사용자당 20회/일 | 10,000,000 × 20 | 200,000,000회/일 |
| 평균 읽기 QPS | 하루 86,400초에 분산 | 200,000,000 ÷ 86,400 | 약 2,315 QPS |
| 평균 팔로워 수 | — | — | 200명 |
| 팬아웃 쓰기(하루) | 게시물 수 × 평균 팔로워 수 | 5,000,000 × 200 | 1,000,000,000건/일 |
| 팬아웃 쓰기 QPS | 하루 86,400초에 분산 | 1,000,000,000 ÷ 86,400 | 약 11,600 QPS |

> **팁** 이 표에서 가장 눈에 띄는 숫자는 마지막 줄입니다. 게시물 자체의 쓰기 QPS는 58에 불과하지만, 그 게시물을 팔로워 200명의 피드함에 각각 복사하는 순간 쓰기 QPS가 11,600으로 200배 뛰어오릅니다. 이 하나의 계산이 다음 절의 팬아웃 방식 논쟁이 왜 존재하는지를 설명합니다.

### 3.3 팬아웃 온 라이트 vs 팬아웃 온 리드

<u>**팬아웃(Fan-out)**</u>이란 게시물 하나가 여러 사용자의 피드에 전달되는 과정을 뜻합니다. 이를 언제 계산하느냐에 따라 두 가지 방식으로 나뉩니다.

![팬아웃 온 Write vs 팬아웃 온 Read 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part14_대규모_시스템_설계_사례/14-2_사례_연구_URL_단축기_뉴스피드_설계/assets/diagram2_fanout_write_vs_read.svg)

- <u>**팬아웃 온 라이트(Fan-out on Write, push 모델)**</u>: 게시물이 작성되는 순간, 그 사람을 팔로우하는 모든 사용자의 피드함(캐시)에 미리 복사해둡니다. 읽을 때는 이미 채워진 자신의 피드함을 그대로 반환하면 되므로 읽기가 O(1)로 빠릅니다. 대신 쓰기 비용이 팔로워 수에 비례해 커집니다.
- <u>**팬아웃 온 리드(Fan-out on Read, pull 모델)**</u>: 게시물은 작성자의 타임라인에만 저장해두고, 사용자가 피드를 요청하는 순간 그 사람이 팔로우하는 모든 대상의 게시물을 모아 정렬해 보여줍니다. 쓰기는 게시물당 1건으로 저렴하지만, 읽을 때마다 팔로우 대상 수만큼 조회하고 병합해야 해 읽기 지연이 커집니다.

> **Q.** 그러면 읽기가 훨씬 많으니 무조건 팬아웃 온 라이트가 유리한 것 아닌가요?
> **A.** 대부분의 경우는 그렇습니다. 앞서 계산한 대로 읽기 QPS(2,315)가 쓰기 QPS(58)보다 훨씬 많으므로, 쓰기 시점에 미리 계산해두고 읽기를 가볍게 만드는 것이 유리합니다. 문제는 <u>**팔로워가 수백만~수천만 명에 달하는 유명 계정**</u>입니다. 이런 계정이 게시물을 하나 올릴 때마다 수백만 건의 팬아웃 쓰기가 한꺼번에 발생하면, 그 자체가 새로운 병목(핫스팟)이 됩니다.

### 3.4 하이브리드 전략

실무에서 널리 쓰이는 절충안은 두 방식을 팔로워 수 기준으로 나누어 적용하는 <u>**하이브리드 팬아웃**</u>입니다.

| 계정 유형 | 적용 방식 | 이유 |
|---|---|---|
| 일반 사용자 (팔로워 수 적음) | 팬아웃 온 라이트 | 팔로워가 적어 팬아웃 쓰기 비용이 작고, 읽기를 빠르게 만드는 이점이 큼 |
| 유명 계정 (팔로워 수 매우 많음) | 팬아웃 온 리드 | 게시물마다 수백만 건씩 쓰는 대신, 팔로워가 피드를 열 때 그 계정의 최신 글만 별도로 조회해 병합 |

> **원칙** 팬아웃 전략을 고를 때는 "읽기와 쓰기 중 무엇이 많은가"라는 14-1의 첫 번째 질문만으로는 부족합니다. <u>**같은 서비스 안에서도 사용자 유형에 따라 읽기·쓰기 비율이 극단적으로 달라질 수 있다는 점**</u>까지 고려해야 하며, 이때는 하나의 전략을 고정하기보다 기준(팔로워 수 임계치)을 두고 전략을 나누는 것이 현실적인 해법입니다.

## 4. 두 사례가 서로 다른 이유 — 정리

> **개요** 같은 "읽기 많고 쓰기 적은" 트래픽 패턴을 가진 두 서비스가 왜 설계 초점부터 달라지는지 비교합니다.

| 비교 항목 | URL 단축기 | 뉴스피드 |
|---|---|---|
| 읽기 대상 | 단축 코드 1개 → 원본 URL 1개 (관계 없음) | 팔로우한 다수의 사용자가 쓴 다수의 게시물 (관계 그래프) |
| 병목의 위치 | 단일 조회를 얼마나 빠르게 캐싱하는가 | 관계 그래프를 언제(쓸 때 vs 읽을 때) 계산하는가 |
| 핵심 설계 결정 | 단축 코드 생성 방식 + 캐시 히트율 | 팬아웃 시점 + 유명 계정 예외 처리 |
| Part 6-1 CAP 관점 | 단축 코드-URL 매핑은 한 번 정해지면 불변이라 일관성 요구가 낮음 | 최신 게시물 반영이 몇 초 늦어도 무방한 AP 성향 |

> **핵심**
> - 읽기:쓰기 비율이 같아도, 읽는 대상이 **독립된 키-값**인지 **관계로 얽힌 데이터**인지에 따라 설계의 초점이 완전히 달라집니다. URL 단축기는 캐싱 하나로 문제의 대부분이 해결되지만, 뉴스피드는 그 관계(팔로우 그래프)를 언제 계산할지부터 정해야 합니다.
> - URL 단축기의 핵심은 <u>**충돌 없는 단축 코드 생성(Base62 인코딩)과 읽기 경로의 캐시 어사이드 패턴**</u>이며, 뉴스피드의 핵심은 <u>**팬아웃 온 라이트와 팬아웃 온 리드 중 트래픽 패턴에 맞는 선택, 그리고 유명 계정을 위한 하이브리드 전략**</u>입니다.

## 요약

> **핵심**
> - URL 단축기는 단순 키-값 조회 문제로, **Base62 인코딩 기반 단축 코드 생성**과 **캐시 어사이드 패턴**으로 대부분의 읽기 트래픽을 방어합니다.
> - 뉴스피드는 관계 기반 집계 문제로, **팬아웃 온 라이트(쓰기 비용 ↑, 읽기 O(1))**와 **팬아웃 온 리드(쓰기 O(1), 읽기 비용 ↑)** 사이의 트레이드오프를 팔로워 규모에 따라 하이브리드로 선택합니다.
> - <u>**같은 "많이 읽고 가끔 쓰는" 패턴도 데이터 간 관계 구조가 다르면 전혀 다른 설계로 이어진다**</u>는 것이 이번 사례 연구의 핵심 결론입니다.
