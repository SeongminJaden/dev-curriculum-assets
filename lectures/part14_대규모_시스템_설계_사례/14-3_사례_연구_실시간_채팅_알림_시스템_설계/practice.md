# 14-3 실습 — 사례 연구: 실시간 채팅·알림 시스템 설계

> **참고** 제출물이 아니라, 오늘 배운 통신 방식 비교와 채팅 시스템 설계를 스스로 적용해보는 실습입니다.

## 실습 1. 통신 방식별 부하 직접 계산하기

- [ ] 동시 접속자 수를 100만 명으로 가정하고, 폴링 간격이 3초일 때 초당 발생하는 요청 수를 계산해보세요.
- [ ] 같은 100만 명이 웹소켓으로 연결되어 있다면, 새 메시지가 없는 평상시에 발생하는 요청 수는 얼마일지 생각해보세요.
- [ ] 위 두 계산 결과를 비교해서, 어느 지점부터 폴링이 현실적으로 불가능해지는지 본인의 기준을 세워보세요.

## 실습 2. 채팅 시스템 용량 추정 다시 해보기

- [ ] DAU와 동시 접속 비율을 강의와 다른 값으로 바꿔서 필요한 웹소켓 서버 대수를 다시 계산해보세요.
- [ ] 사용자당 하루 평균 메시지 수를 두 배로 늘리면 평균 메시지 QPS가 어떻게 바뀌는지 계산해보세요.

## 실습 3. 메시지 상태 전이 표 만들기

- [ ] 본인이 사용하는 메신저 앱 하나를 골라, 메시지를 보냈을 때 화면에 표시되는 상태(예: 시계 아이콘, 체크 표시)를 관찰해보세요.
- [ ] 그 상태들이 이 강의에서 다룬 "전송됨 → 전달됨 → 읽음" 중 어디에 해당하는지 표로 정리해보세요.
- [ ] 그룹 채팅에서 "읽음" 표시가 1:1 채팅과 어떻게 다르게 보여야 할지 생각해보세요. (예: "3명이 읽음")

## 생각해볼 질문

- [ ] 오프라인 메시지 저장소에 메시지가 몇 달째 쌓여만 있고 아무도 재접속하지 않는다면, 이 데이터는 어떻게 관리해야 할까요?
- [ ] 이 강의에서 다룬 메시지 큐 기반 알림 팬아웃 구조를, Part 3-2에서 배운 이벤트 기반 아키텍처의 장단점(결합도, 최종적 일관성)과 연결해 설명해보세요.
