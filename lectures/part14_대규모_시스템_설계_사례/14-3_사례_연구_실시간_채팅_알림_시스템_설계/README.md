# Part 14. 대규모 시스템 설계 사례

## 14-3. 사례 연구 — 실시간 채팅·알림 시스템 설계

> **학습 목표**
> - 폴링·롱폴링·웹소켓 세 가지 통신 방식이 지연 시간과 서버 자원 사이에서 어떤 트레이드오프를 갖는지 비교할 수 있다.
> - 메시지 큐를 이용해 하나의 이벤트를 여러 알림 채널(웹소켓 push, 모바일 푸시, 이메일)로 팬아웃하는 구조를 설계할 수 있다.
> - 읽음 확인과 오프라인 메시지 저장을 포함한 채팅 시스템의 고수준 설계를 요구사항부터 도출할 수 있다.

> **개요** "새 메시지가 왔는지 서버에 계속 물어보는 것"과 "서버가 준비되면 알아서 알려주는 것" 사이의 차이를 폴링·롱폴링·웹소켓 비교로 살펴본 뒤, 이 통신 방식 위에서 실시간 채팅과 알림 시스템을 요구사항부터 상세 설계까지 따라갑니다.

---

## 1. 폴링 대신 웹소켓을 쓰는 것은 무엇을 맞바꾸는 선택일까요?

> **Q.** 그냥 클라이언트가 몇 초마다 서버에 "새 메시지 있어?"라고 물어보면 되지 않을까요? 왜 굳이 웹소켓 같은 별도 기술이 필요할까요?
> **A.** 물어보는 방식(폴링)도 동작은 합니다. 문제는 <u>**대부분의 질문에 대한 답이 "없음"이라는 것**</u>입니다. 사용자가 실제로 메시지를 받는 순간은 하루 중 극히 일부인데, 폴링은 메시지가 있든 없든 정해진 간격마다 요청을 보냅니다. 접속자가 늘어날수록 이 "의미 없는 질문"의 총량이 폭발적으로 늘어나며, 이는 Part 6-2에서 다룬 C10K 문제, 즉 동시 접속자가 늘어날수록 서버가 각 연결을 유지하는 비용이 감당하기 어려워지는 문제와 정확히 같은 성격의 부담입니다. 웹소켓은 "물어보는" 방식 자체를 "서버가 준비되면 알려주는" 방식으로 바꾸어 이 낭비를 없애지만, 그 대가로 서버가 모든 접속자와의 연결을 계속 열어두어야 하는 새로운 부담을 집니다.

이 강의는 세 가지를 다룹니다.

- **통신 방식 비교** — 폴링, 롱폴링, 웹소켓이 지연과 자원 사이에서 어떻게 다른 선택을 하는가
- **채팅 시스템 설계** — 요구사항 파악부터 메시지 큐 기반 알림 팬아웃까지
- **읽음 확인과 오프라인 메시지** — 상대가 오프라인이거나 아직 메시지를 안 읽었을 때의 처리

## 2. 통신 방식 비교 — 폴링, 롱폴링, 웹소켓

> **개요** 클라이언트가 서버의 새 데이터를 확인하는 세 가지 방식을 지연 시간과 서버 자원 소모라는 두 축으로 비교합니다.

![폴링·롱폴링·웹소켓 통신 패턴 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part14_대규모_시스템_설계_사례/14-3_사례_연구_실시간_채팅_알림_시스템_설계/assets/diagram1_polling_vs_websocket.svg)

- <u>**폴링(Polling)**</u>: 클라이언트가 일정한 간격(예: 5초)마다 서버에 "새로운 것이 있는가"를 요청합니다. 구현이 가장 단순하지만, 새 메시지가 방금 도착했더라도 다음 폴링 시점까지 최대 간격만큼 지연되고, 대부분의 요청이 빈 응답으로 끝나 서버 자원을 낭비합니다.
- <u>**롱폴링(Long Polling)**</u>: 클라이언트가 요청을 보내면 서버는 새로운 데이터가 생기거나 타임아웃(예: 30초)에 도달할 때까지 응답을 미룹니다. 데이터가 생기면 즉시 응답하고, 클라이언트는 응답을 받는 즉시 새 요청을 다시 보냅니다. 폴링보다 요청 횟수와 지연이 줄지만, 서버가 각 연결을 오래 붙잡고 있어야 합니다.
- <u>**웹소켓(WebSocket)**</u>: 클라이언트와 서버가 처음 한 번 핸드셰이크로 연결을 맺으면, 그 이후에는 이 연결을 계속 유지한 채 양방향으로 데이터를 주고받습니다. 서버는 새 메시지가 생기는 즉시 그 연결을 통해 클라이언트에 push할 수 있어 지연이 가장 적지만, 접속자 수만큼의 연결을 서버가 항상 유지해야 합니다.

> **해석** 세 방식을 오른쪽으로 갈수록(폴링 → 롱폴링 → 웹소켓) 지연 시간은 짧아지지만, 서버가 연결 상태를 유지하는 비용은 커집니다. "60초당 요청 12회"에서 "요청 없이 연결 1개 유지"로 옮겨가는 과정은, 부담의 위치를 클라이언트-서버 간 반복 요청에서 서버의 연결 유지 비용으로 옮기는 것입니다.

> **참고** 여기서 말하는 "서버가 연결을 유지하는 비용"은 Part 6-2에서 비교한 멀티스레딩과 이벤트 루프 논의와 그대로 이어집니다. 채팅 서버가 접속자 수만큼 스레드를 하나씩 배정한다면 접속자가 만 명만 되어도 스택 메모리와 컨텍스트 스위칭 비용이 감당하기 어려워지지만(C10K 문제), 이벤트 루프와 논블로킹 I/O를 사용하면 스레드 하나(또는 소수)로 수만 개의 idle한 웹소켓 연결을 동시에 감시할 수 있습니다. 웹소켓 서버 대부분이 Node.js, Netty 같은 이벤트 루프 기반 프레임워크로 구현되는 이유가 여기에 있습니다.

### 실생활 예시 — 통신 방식별 60초 부하 비교

| 방식 | 60초당 요청 횟수 | 최악의 지연 | 서버가 짊어지는 비용 |
|---|---|---|---|
| 폴링(5초 간격) | 약 12회 | 최대 5초 | 대부분 빈 응답이지만 요청 자체는 매번 처리해야 함 |
| 롱폴링(최대 30초 대기) | 약 2회 | 거의 없음(즉시 응답) | 연결을 오래 붙잡아 두어야 함(동시 접속자만큼) |
| 웹소켓 | 연결 1회 이후 없음 | 밀리초 단위 | 접속자 전원의 연결 상태를 계속 유지해야 함 |

## 3. 실시간 채팅 시스템 설계

> **개요** 폴링 대신 웹소켓을 선택했다는 전제 위에서, 채팅 시스템을 14-1의 절차대로 요구사항부터 짚어봅니다.

### 3.1 요구사항 파악

- **기능 요구사항**: 사용자는 1:1 또는 그룹으로 메시지를 주고받을 수 있어야 하고, 상대가 메시지를 읽었는지 확인할 수 있어야 하며, 오프라인 상태였던 사용자는 재접속 시 그동안 받지 못한 메시지를 모두 받아야 합니다.
- **비기능 요구사항**: 메시지 전달 지연은 최대한 짧아야 하고(웹소켓 선택의 근거), 서버 재시작이나 일시적 장애가 있어도 메시지 자체가 유실되어서는 안 됩니다.

### 3.2 용량 추정

> **해석** DAU 500만 명 중 동시 접속 비율 10%, 사용자당 하루 평균 메시지 40건을 가정했을 때의 계산입니다.

| 항목 | 가정 | 계산 | 결과 |
|---|---|---|---|
| 일일 활성 사용자(DAU) | — | — | 5,000,000명 |
| 동시 접속자 수 | 동시 접속 비율 10% | 5,000,000 × 0.1 | 500,000명 |
| 웹소켓 서버 1대당 유지 가능 연결 | 이벤트 루프 기반 가정 | — | 약 50,000개 |
| 필요한 서버 대수 | 동시 접속자 ÷ 서버당 연결 | 500,000 ÷ 50,000 | 10대 |
| 하루 총 메시지 수 | 사용자당 40건/일 | 5,000,000 × 40 | 200,000,000건/일 |
| 평균 메시지 QPS | 하루 86,400초에 분산 | 200,000,000 ÷ 86,400 | 약 2,315 QPS |

> **팁** "동시 접속자 50만 명"이라는 숫자 자체가 이미 폴링을 배제하는 근거가 됩니다. 만약 이 50만 명이 5초 간격으로 폴링한다면 그것만으로 초당 10만 건의 요청이 발생하는데, 이 요청 대부분이 빈 응답이라는 점에서 이 강의 도입부에서 언급한 "의미 없는 질문의 폭증"이 숫자로 확인됩니다.

### 3.3 고수준 설계 — 메시지 큐를 이용한 알림 팬아웃

메시지 하나가 발생했을 때, 이를 수신자의 웹소켓 연결로 즉시 전달하는 것 외에도 모바일 푸시, 이메일, 오프라인 저장 등 여러 채널로 동시에 처리해야 합니다. 이 모든 처리를 발신 경로 하나에서 순서대로 처리하면(동기 호출) 가장 느린 채널이 전체를 지연시키므로, Part 3-2에서 다룬 <u>**이벤트 기반 아키텍처**</u>를 그대로 적용해 메시지 큐로 각 채널을 분리합니다.

![메시지 큐를 이용한 알림 팬아웃 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part14_대규모_시스템_설계_사례/14-3_사례_연구_실시간_채팅_알림_시스템_설계/assets/diagram2_notification_fanout.svg)

> **해석** 메시지 전송이라는 이벤트가 발생하면 메시지 브로커에 한 번만 발행되고, 웹소켓 서버(온라인 사용자에게 즉시 전달)·푸시 알림 서비스(앱이 꺼져 있어도 전달)·이메일 알림 서비스(긴급하지 않은 알림)·오프라인 메시지 저장소가 각자 이 이벤트를 구독해 독립적으로 처리합니다. 어느 한 채널(예: 이메일 발송)이 느려지거나 잠시 장애가 나도, 웹소켓을 통한 실시간 전달에는 영향을 주지 않습니다.

> **Q.** 왜 온라인 사용자에게 전달하는 것과 오프라인 메시지를 저장하는 것을 같은 큐에서 동시에 처리할까요?
> **A.** 이벤트가 발행되는 시점에는 수신자가 온라인인지 오프라인인지 미리 알 수 없기 때문입니다. 웹소켓 서버가 해당 사용자의 연결을 가지고 있다면 즉시 push하고, 연결이 없다면(오프라인) 오프라인 메시지 저장소에 기록해두는 방식으로 두 경로를 병행합니다. 이렇게 하면 수신자가 온라인이든 오프라인이든 메시지 발신자 입장에서는 같은 한 번의 발행으로 처리가 끝나며, 유실 없이 나중에 전달할 방법까지 함께 확보됩니다.

### 3.4 읽음 확인과 오프라인 메시지 저장

<u>**읽음 확인(Read Receipt)**</u>은 메시지가 상대에게 전달(delivered)되었는지, 실제로 읽혔는지(read)를 구분해 발신자에게 알려주는 기능입니다. 일반적으로 메시지는 아래와 같은 상태를 순서대로 거칩니다.

| 상태 | 의미 | 갱신 시점 |
|---|---|---|
| 전송됨(sent) | 발신자의 클라이언트가 메시지를 서버로 보냄 | 발신 즉시 |
| 전달됨(delivered) | 서버가 수신자의 기기까지 메시지를 push함 | 수신자가 온라인이고 push에 성공한 순간 |
| 읽음(read) | 수신자가 실제로 메시지를 확인함 | 수신자의 클라이언트가 읽음 이벤트를 서버로 보낸 순간 |

<u>**오프라인 메시지 저장(Offline Message Store)**</u>은 수신자가 오프라인일 때 "전달됨" 상태로 넘어가지 못한 메시지를 별도의 저장소(DB)에 보관해두는 장치입니다. 수신자가 재접속하면 웹소켓 서버가 이 저장소를 조회해, 그동안 쌓인 미읽음 메시지를 순서대로 전달합니다.

> **원칙** 읽음 확인 기능을 설계할 때는 <u>**메시지의 상태를 발신자와 수신자가 각자 다른 시점에 갱신한다는 점**</u>을 명확히 해야 합니다. "전달됨"은 서버와 수신자 기기 사이의 통신 결과이고, "읽음"은 수신자의 실제 행동(화면 확인)에 대한 결과입니다. 이 둘을 하나의 이벤트로 뭉뚱그리면, 수신자가 알림만 받고 아직 읽지 않았는데도 "읽음"으로 잘못 표시되는 문제가 생깁니다.

> **참고** 오프라인 메시지 저장은 Part 6-1에서 다룬 CAP 이론의 관점에서도 설명할 수 있습니다. 메시지 서비스는 수신자가 일시적으로 연결되지 않은 상태(분단과 유사한 상황)에서도 메시지를 잃어버리지 않아야 하므로, "지금 당장 전달"보다 "언젠가는 반드시 전달"을 우선하는 설계이며, 이는 강한 일관성보다 가용성과 최종적 전달 보장을 우선하는 선택에 가깝습니다.

## 4. 실생활 예시 — 그룹 채팅방에서 한 사람이 메시지를 보내는 순간

> **해석** 아래 표는 그룹 채팅방(멤버 5명, 그중 2명은 오프라인)에서 메시지 하나가 발행된 뒤 벌어지는 일을 순서대로 정리한 것입니다.

| 시점 | 벌어지는 일 |
|---|---|
| 발신 | 발신자의 클라이언트가 메시지를 웹소켓으로 서버에 전송, 상태는 "전송됨" |
| 이벤트 발행 | 서버가 메시지 브로커에 "새 메시지" 이벤트를 발행 |
| 온라인 멤버 3명 | 각자의 웹소켓 서버가 이벤트를 받아 즉시 push, 상태가 "전달됨"으로 갱신 |
| 오프라인 멤버 2명 | 오프라인 메시지 저장소에 메시지가 기록됨, "전달됨" 갱신은 보류 |
| 온라인 멤버가 메시지를 확인 | 클라이언트가 읽음 이벤트를 서버로 전송, 해당 멤버만 "읽음"으로 갱신 |
| 오프라인 멤버 재접속 | 웹소켓 서버가 저장소에서 미읽음 메시지를 가져와 전달, 이후 "전달됨"으로 갱신 |

## 요약

> **핵심**
> - 폴링은 구현이 단순하지만 대부분의 요청이 빈 응답으로 낭비되고, 롱폴링은 요청 횟수를 줄이지만 서버가 연결을 오래 붙잡아야 하며, 웹소켓은 <u>**지연을 가장 줄이는 대신 접속자 수만큼의 연결 유지 비용을 서버가 계속 짊어지는**</u> 선택입니다.
> - 채팅·알림 시스템은 Part 3-2의 이벤트 기반 아키텍처를 그대로 적용해, 메시지 발행을 메시지 큐로 분리하고 웹소켓 push·모바일 푸시·이메일·오프라인 저장을 각자 독립적으로 처리합니다.
> - 읽음 확인은 <u>**전송됨 → 전달됨 → 읽음**</u>의 서로 다른 시점을 구분해 갱신해야 하며, 오프라인 메시지 저장은 수신자가 연결되지 않은 상황에서도 메시지 유실 없이 나중에 전달을 보장하는 장치입니다.
