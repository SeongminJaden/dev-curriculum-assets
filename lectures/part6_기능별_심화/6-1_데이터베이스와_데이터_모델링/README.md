# Part 6. 기능별 심화

## 6-1. 데이터베이스와 데이터 모델링 — 관계형 모델을 깨면서까지 NoSQL을 쓰는 이유는 무엇인가

> **학습 목표**
> - 관계형 모델의 정규화가 무엇을 방지하기 위한 절차인지, ACID 네 속성이 각각 무엇을 보장하는지 설명할 수 있다.
> - CAP 이론의 세 속성(Consistency, Availability, Partition Tolerance)을 정의하고, 왜 셋을 동시에 만족할 수 없는지 논리적으로 설명할 수 있다.
> - 서비스의 데이터 특성에 따라 RDBMS와 NoSQL 중 무엇을 선택해야 하는지 근거를 들어 판단할 수 있다.

> **개요** 관계형 데이터베이스가 정규화와 ACID로 무엇을 지키는지 먼저 살펴본 뒤, 여러 대의 서버로 데이터를 분산시키는 순간 그 보장이 왜 흔들리는지, CAP 이론이 이 한계를 어떻게 공식화했는지, 그리고 NoSQL이 어떤 트레이드오프를 선택해 이 문제를 우회하는지를 다룹니다.

---

## 1. 왜 데이터를 "표"로 정리할까요? — 관계형 모델과 정규화

> **개요** 에드거 커드(E. F. Codd)가 1970년 제안한 관계형 모델과, 그 위에서 데이터 중복을 제거하는 정규화 절차가 왜 필요한지 다룹니다.

<u>**관계형 모델(Relational Model)**</u>은 IBM 연구원 에드거 커드가 1970년 논문 "A Relational Model of Data for Large Shared Data Banks"에서 제안한 개념으로, 모든 데이터를 행(row)과 열(column)로 이루어진 표(테이블)로 표현하고, 테이블 간의 관계는 값(주로 기본키·외래키)으로 연결한다는 아이디어입니다. 이 아이디어 위에서 SQL이라는 질의 언어가 정의되었고, 오늘날의 MySQL, PostgreSQL, Oracle 같은 RDBMS(Relational Database Management System)가 모두 이 모델을 구현합니다.

> **Q.** 데이터를 그냥 하나의 큰 표에 다 몰아넣으면 안 될까요? 왜 여러 테이블로 쪼갤까요?
> **A.** 하나의 표에 모든 정보를 몰아넣으면 같은 값이 여러 행에 중복해서 저장됩니다. 예를 들어 한 사용자가 주문을 10번 했다면, 그 사용자의 이름과 주소가 10개 행에 똑같이 반복됩니다. 이 상태에서 사용자가 주소를 바꾸면 10개 행을 전부 찾아 고쳐야 하고, 하나라도 놓치면 같은 사용자의 주소가 행마다 다르게 보이는 모순이 생깁니다. 이런 문제를 <u>**갱신 이상(Update Anomaly)**</u>, <u>**삽입 이상(Insertion Anomaly)**</u>, <u>**삭제 이상(Deletion Anomaly)**</u>이라 부르며, 이를 막기 위한 절차가 <u>**정규화(Normalization)**</u>입니다.

정규화는 단계별로 지켜야 할 규칙(정규형)을 두고 있습니다.

- <u>**제1정규형(1NF)**</u>: 각 열에는 더 이상 나눌 수 없는 하나의 값만 저장합니다(하나의 칸에 여러 취미를 쉼표로 몰아넣지 않습니다).
- <u>**제2정규형(2NF)**</u>: 1NF를 만족하면서, 기본키의 일부에만 종속된 열(부분 함수 종속)을 별도 테이블로 분리합니다.
- <u>**제3정규형(3NF)**</u>: 2NF를 만족하면서, 기본키가 아닌 열이 다른 기본키가 아닌 열에 종속되는 관계(이행 함수 종속)를 제거합니다.

실무 RDBMS 설계는 대부분 3NF 수준까지 정규화한 뒤, 조회 성능이 중요한 일부 테이블만 의도적으로 역정규화(비정규화)합니다.

![관계형 모델 vs NoSQL 문서 모델 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part6_기능별_심화/6-1_데이터베이스와_데이터_모델링/assets/diagram1_rdbms_vs_nosql.svg)

> **해석** 왼쪽 RDBMS 구조는 사용자·식당·주문·주문항목을 각각 별도 테이블로 분리하고 외래키(FK)로 연결합니다. 사용자 주소가 바뀌어도 `Users` 테이블 한 행만 고치면 되므로 갱신 이상이 생기지 않습니다. 오른쪽 NoSQL 문서 구조는 반대로, 주문 하나를 읽을 때 필요한 정보(사용자 이름, 식당 이름, 상품 목록)를 처음부터 문서 하나에 중복해서 넣어둡니다.

## 2. ACID — 트랜잭션이 지켜야 할 네 가지 약속

> **개요** RDBMS가 여러 작업을 하나의 단위로 묶어 처리할 때 지키는 ACID 네 속성을 계좌 이체 시나리오로 살펴봅니다.

<u>**트랜잭션(Transaction)**</u>이란 "모두 성공하거나, 모두 실패해야 하는 하나의 작업 단위"입니다. RDBMS는 이 트랜잭션이 아래 네 가지를 반드시 지키도록 보장하며, 이를 <u>**ACID**</u>라고 부릅니다.

- <u>**원자성(Atomicity)**</u>: 트랜잭션 안의 여러 작업은 전부 반영되거나 전부 취소됩니다. 중간까지만 반영되는 상태는 존재하지 않습니다.
- <u>**일관성(Consistency)**</u>: 트랜잭션 전후로 데이터베이스는 정의된 규칙(제약조건, 잔액은 0 이상이어야 한다 등)을 항상 만족합니다.
- <u>**격리성(Isolation)**</u>: 동시에 실행되는 여러 트랜잭션은 서로의 중간 상태를 보지 못하며, 마치 순서대로 실행된 것처럼 결과가 나옵니다.
- <u>**지속성(Durability)**</u>: 한 번 커밋(commit)된 트랜잭션의 결과는 서버가 그 직후 정전이 나더라도 사라지지 않습니다.

### 실생활 예시 — 계좌 이체

| ACID 속성 | 계좌 이체(A 계좌 → B 계좌 10만 원)에서의 의미 |
|---|---|
| 원자성 | A 계좌에서 10만 원을 빼는 작업과 B 계좌에 10만 원을 더하는 작업 중 하나만 성공하는 일은 없어야 한다 |
| 일관성 | 이체 후에도 두 계좌 잔액의 합은 이체 전과 같아야 한다 |
| 격리성 | 같은 순간 다른 이체가 A 계좌를 건드리더라도, 두 이체는 서로 뒤섞이지 않고 순서대로 처리된 것과 같은 결과가 나와야 한다 |
| 지속성 | 이체가 완료되었다고 화면에 뜬 순간, 그 즉시 서버가 꺼져도 이체 기록은 남아 있어야 한다 |

> **Q.** ACID는 왜 이렇게 엄격하게 지켜질까요?
> **A.** 금융, 재고, 예약처럼 "숫자가 틀리면 실제 손해로 이어지는" 도메인에서는 일시적인 편의보다 정확성이 우선이기 때문입니다. RDBMS는 락(lock)과 로그 기반 복구 메커니즘으로 이 네 속성을 구현하는데, 이 메커니즘 자체가 비용(락 대기, 로그 기록)입니다. 즉 ACID는 공짜가 아니라, 정확성을 위해 성능과 확장성 일부를 의도적으로 지불하는 설계입니다.

## 3. 분산 환경에서 생기는 문제 — CAP 이론

> **개요** 데이터를 여러 대의 서버에 나누어 저장하는 순간 ACID가 왜 그대로 유지되기 어려운지, 이를 공식화한 CAP 이론을 다룹니다.

서비스 사용자가 늘어나면 데이터를 서버 한 대에만 둘 수 없습니다. 여러 대의 서버에 데이터를 복제해 분산시켜야 하는데, 이때 새로운 문제가 생깁니다. <u>**CAP 이론**</u>은 컴퓨터과학자 에릭 브루어(Eric Brewer)가 2000년 강연에서 처음 제시한 추측(conjecture)으로, 2002년 세스 길버트(Seth Gilbert)와 낸시 린치(Nancy Lynch)가 수학적으로 증명하면서 이론으로 정립되었습니다.

CAP 이론은 분산 데이터 시스템이 아래 세 속성을 동시에 완전히 만족할 수 없다고 말합니다.

- <u>**일관성(Consistency)**</u>: 모든 노드가 같은 시점에 항상 같은 데이터를 반환합니다(정확히는 선형화 가능성, Linearizability에 가깝습니다).
- <u>**가용성(Availability)**</u>: 장애가 없는 모든 노드는 모든 요청에 대해 (에러 없이) 응답을 반환해야 합니다.
- <u>**분단 내성(Partition Tolerance)**</u>: 노드 간 네트워크 통신 일부가 끊기더라도 시스템 전체는 계속 동작해야 합니다.

![CAP 이론 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part6_기능별_심화/6-1_데이터베이스와_데이터_모델링/assets/diagram2_cap_theorem.svg)

> **Q.** 왜 이 세 가지를 동시에 만족할 수 없을까요?
> **A.** 노드가 여러 대인 이상 네트워크 분단(P)은 확률의 문제일 뿐, 언젠가 반드시 발생합니다(케이블 단절, 스위치 장애, 순간적인 패킷 유실 등). 분단이 실제로 일어난 그 순간을 생각해봅시다. 노드 1과 노드 2 사이의 통신이 끊긴 상태에서 노드 1에 쓰기 요청이 들어왔다고 합시다. 이때 시스템은 두 가지 선택지만 있습니다. ① 노드 2와 동기화가 끝날 때까지 응답을 미루거나 거부한다 — 이는 가용성(A)을 포기하는 것입니다. ② 노드 2와 동기화되지 않은 채로 즉시 응답한다 — 이는 노드 2가 같은 요청에 다른 값을 반환할 수 있으므로 일관성(C)을 포기하는 것입니다. 분단이 존재하는 한 이 둘 중 하나를 반드시 골라야 하므로, C·A·P 셋을 동시에 완전히 만족하는 분산 시스템은 존재할 수 없습니다. 참고로 CA(일관성+가용성만 만족)는 애초에 분단이 없는 단일 노드에서나 성립하는 조합이라, 여러 대로 분산된 시스템에는 해당하지 않습니다.

> **참고** CAP 이론은 "분단이 발생했을 때"라는 조건부 선택을 말하는 것이지, 평상시에도 C나 A 중 하나를 완전히 포기하고 산다는 뜻은 아닙니다. 대부분의 실제 시스템은 평소에는 C와 A를 모두 높은 수준으로 제공하다가, 분단이라는 예외 상황에서만 둘 중 하나에 우선순위를 둡니다.

## 4. NoSQL은 무엇을 포기하고 무엇을 얻는가

> **개요** RDBMS가 CP·강한 일관성 쪽에 무게를 둔 것과 달리, 다수의 NoSQL이 AP·가용성 쪽에 무게를 두는 이유와 그 결과로 얻는 BASE 특성을 다룹니다.

NoSQL은 하나의 기술이 아니라 관계형 모델을 벗어난 여러 데이터 모델을 통칭하는 이름입니다.

- <u>**문서형(Document)**</u>: 데이터를 JSON과 비슷한 문서 하나에 중첩해 저장합니다. (예: MongoDB)
- <u>**키-값형(Key-Value)**</u>: 키 하나에 값 하나를 대응시켜 매우 빠르게 읽고 씁니다. (예: Redis, DynamoDB)
- <u>**컬럼 패밀리형(Column-family)**</u>: 방대한 행을 여러 서버에 분산 저장하는 데 최적화되어 있습니다. (예: Cassandra)
- <u>**그래프형(Graph)**</u>: 노드와 관계 자체를 1급 데이터로 다뤄, 관계 탐색에 강합니다. (예: Neo4j)

많은 NoSQL, 특히 분산 환경을 겨냥한 제품은 ACID 대신 <u>**BASE**</u>(Basically Available, Soft state, Eventually consistent — 기본적으로 가용하고, 상태는 임시적일 수 있으며, 결국에는 일관성을 회복한다) 모델을 따릅니다. 이는 CAP에서 A(가용성)를 우선하고, C(일관성)는 "즉시"가 아니라 "결국에는(eventually)" 맞춰지는 것으로 완화한 선택입니다.

### 실생활 예시 — 배달 앱 주문 데이터를 RDBMS와 NoSQL로 각각 모델링하면

| 항목 | RDBMS로 모델링 | NoSQL(문서형)로 모델링 |
|---|---|---|
| 결제·잔액 정보 | 정규화된 `Payments` 테이블 + 트랜잭션으로 처리. 이중 결제·잔액 불일치가 절대 있으면 안 됨 | 굳이 쓰지 않음 — 강한 일관성이 필요한 영역이라 RDBMS 유지 |
| 주문 상세 화면 조회 | 주문·사용자·식당·상품 테이블을 조인(JOIN)해서 조회 | 화면에 필요한 정보를 미리 하나의 문서에 중첩해두어 조인 없이 즉시 조회 |
| 배달 기사 실시간 위치 | 매초 UPDATE가 발생하면 락 경합이 심해짐 | 키-값형에 최신 좌표만 덮어써 매우 빠르게 갱신 (잠깐 낡은 값이 보여도 무방) |
| 트래픽 폭주 시 우선순위 | 정합성을 지키기 위해 일부 요청을 지연·거부할 수 있음(CP 성향) | 서버가 살아있는 한 응답을 계속하는 것을 우선(AP 성향) |

> **팁** 실무에서는 "RDBMS냐 NoSQL이냐"를 서비스 전체 단위가 아니라 **데이터 종류 단위**로 결정하는 경우가 많습니다. 같은 배달 앱 안에서도 결제 데이터는 RDBMS에, 실시간 위치나 리뷰 캐시는 NoSQL에 두는 폴리글랏 퍼시스턴스(Polyglot Persistence)가 일반적입니다.

> **원칙** 데이터 저장소를 고를 때는 "이 데이터가 틀리면 즉시 손해로 이어지는가(→ ACID·RDBMS 우선), 아니면 잠깐 낡아도 서비스가 멈추지 않는 것이 더 중요한가(→ BASE·NoSQL 우선)"를 먼저 물어야 합니다. 유행이나 성능 수치만 보고 결정하면, 나중에 정합성이 반드시 필요한 데이터에서 문제가 드러납니다.

## 요약

> **핵심**
> - 정규화는 데이터 중복을 제거해 갱신·삽입·삭제 이상을 막기 위한 절차이며, ACID(원자성·일관성·격리성·지속성)는 RDBMS가 트랜잭션의 정확성을 보장하는 네 가지 약속입니다.
> - CAP 이론은 분산 시스템이 **일관성(C)·가용성(A)·분단 내성(P)** 세 속성을 동시에 완전히 만족할 수 없다고 말하며, 네트워크 분단은 피할 수 없으므로 실제 선택은 **C와 A 중 하나**입니다.
> - NoSQL은 CAP에서 가용성(A)을 우선하고 일관성을 "결국에는 맞는다(BASE)"로 완화하는 대가로, RDBMS보다 쉽게 수평 확장하고 유연한 스키마를 가집니다. 어떤 저장소를 쓸지는 유행이 아니라 **그 데이터가 정합성을 얼마나 엄격하게 요구하는가**로 결정해야 합니다.
