# Part 6. 기능별 심화

## 6-3. API 설계 패턴 — REST·GraphQL·gRPC는 각각 어떤 통신 문제를 풀기 위해 등장했는가

> **학습 목표**
> - REST가 등장한 역사적 배경과, REST를 REST답게 만드는 제약 조건들을 설명할 수 있다.
> - GraphQL이 REST의 오버페칭·언더페칭 문제를 어떤 방식으로 해결하는지 설명할 수 있다.
> - gRPC가 내부 마이크로서비스 통신에서 어떤 문제를 겨냥해 등장했는지 설명하고, 상황에 맞는 통신 방식을 선택할 수 있다.

> **개요** API가 왜 "설계"의 대상인지 먼저 짚은 뒤, REST가 자원 중심 설계로 무엇을 표준화했는지, GraphQL이 REST의 한계를 어떻게 극복했는지, gRPC가 내부 서비스 간 통신에서 어떤 문제를 풀기 위해 등장했는지 차례로 살펴봅니다.

---

## 1. API란 무엇이고 왜 "설계"가 필요한가

> **개요** 서로 다른 프로그램이 통신하려면 계약(contract)이 필요하다는 점과, 그 계약을 정하는 방식이 시대마다 왜 달라졌는지를 다룹니다.

<u>**API(Application Programming Interface)**</u>는 서로 다른 소프트웨어가 통신할 때 지키기로 약속한 규칙입니다. 클라이언트와 서버가 각자 내부 구현을 몰라도, 이 규칙만 지키면 데이터를 주고받을 수 있습니다.

> **Q.** 규칙이 하나만 있으면 될 텐데, 왜 REST·GraphQL·gRPC처럼 여러 방식이 계속 등장했을까요?
> **A.** API가 연결하는 대상의 성격이 시대에 따라 달라졌기 때문입니다. 2000년대 초에는 "서로 다른 회사의 서버끼리, 느슨하게, 문서만 보고" 통신하는 것이 문제였고, 2010년대 중반에는 "하나의 서버가 여러 종류의 화면(웹·모바일)에 각기 다른 모양의 데이터를 내려줘야 하는 것"이 문제였으며, 최근에는 "한 회사 내부의 수십~수백 개 마이크로서비스가 서로 빠르고 정확하게 통신하는 것"이 문제입니다. REST·GraphQL·gRPC는 이 세 가지 문제에 각각 대응하며 등장했습니다.

## 2. REST — 자원을 URL로, 상태 변화를 HTTP 메서드로

> **개요** 로이 필딩(Roy Fielding)이 2000년 박사논문에서 정리한 REST 제약 조건이 어떤 문제(제각각인 API 설계)를 풀었는지, 그리고 REST가 태생적으로 안고 있는 한계(오버페칭·언더페칭)를 다룹니다.

<u>**REST(Representational State Transfer)**</u>는 로이 필딩이 2000년 박사논문 "Architectural Styles and the Design of Network-based Software Architectures"에서 웹의 성공 요인을 분석하며 정리한 아키텍처 스타일입니다. 이 논문이 등장하기 전에는 API마다 설계 방식이 제각각이었고(SOAP처럼 XML로 원격 프로시저를 호출하는 방식이 대표적), 서버마다 새로운 규약을 익혀야 했습니다.

REST는 아래 제약 조건을 지킬 것을 제안합니다.

- <u>**클라이언트-서버 분리**</u>: 클라이언트와 서버는 서로의 내부 구현에 의존하지 않습니다.
- <u>**무상태성(Stateless)**</u>: 서버는 요청 사이에 클라이언트의 상태를 기억하지 않으며, 요청 하나에 처리에 필요한 모든 정보가 담겨야 합니다.
- <u>**캐시 가능(Cacheable)**</u>: 응답은 캐시 가능 여부를 명시할 수 있어야 합니다.
- <u>**일관된 인터페이스(Uniform Interface)**</u>: 자원은 URL로 식별하고, 자원에 대한 행위는 HTTP 메서드(GET·POST·PUT·DELETE)로 표현합니다.
- <u>**계층화 시스템(Layered System)**</u>: 클라이언트는 프록시·게이트웨이 같은 중간 계층이 있는지 몰라도 동작해야 합니다.

> **Q.** REST가 "자원을 URL로 표현한다"는 원칙이 왜 그렇게 중요할까요?
> **A.** 이 원칙 덕분에 API가 HTTP라는 기존 웹 인프라(캐시 서버, 프록시, 브라우저)를 그대로 활용할 수 있기 때문입니다. `GET /orders/5`는 같은 자원을 가리키는 요청임이 URL만 보고도 명확하고, 그 결과를 중간의 캐시 서버가 그대로 재사용할 수 있습니다. SOAP처럼 모든 요청이 같은 엔드포인트로 몰리고 동작이 메시지 본문 안에만 있다면, 중간 계층은 요청의 의미를 알 수 없어 이런 최적화를 할 수 없습니다.

REST는 자원 단위로 API를 설계하기 때문에, 화면 하나가 여러 자원의 데이터를 조합해야 할 때 문제가 생깁니다. 필요한 것보다 많은 필드를 받는 것을 <u>**오버페칭(Overfetching)**</u>, 필요한 필드가 부족해 여러 번 다시 요청해야 하는 것을 <u>**언더페칭(Underfetching)**</u>이라 부릅니다.

## 3. GraphQL — 클라이언트가 필요한 데이터의 모양을 직접 정의한다

> **개요** 페이스북이 모바일 앱에서 겪은 오버페칭·언더페칭 문제를 GraphQL이 어떻게 풀었는지, 그 대가로 무엇을 잃는지 다룹니다.

<u>**GraphQL**</u>은 페이스북이 2012년 내부적으로 개발해 2015년 공개한 쿼리 언어이자 런타임입니다. 등장 배경은 구체적입니다. 페이스북 뉴스피드 화면은 게시글·작성자·댓글·좋아요 수 등 여러 자원의 데이터를 한 번에 필요로 했는데, REST 방식으로는 이를 위해 여러 엔드포인트를 호출하거나(언더페칭 해소를 위한 다중 호출), 화면마다 필요한 조합을 미리 만든 전용 엔드포인트를 계속 늘려야 했습니다(오버페칭 회피를 위한 API 폭증). 모바일 네트워크처럼 왕복 지연이 큰 환경에서는 이 다중 호출 자체가 성능 문제였습니다.

GraphQL은 이를 뒤집었습니다. 서버는 자신이 가진 데이터 전체를 하나의 <u>**스키마(Schema)**</u>로 선언하고, 단일 엔드포인트만 열어둡니다. 클라이언트는 이 엔드포인트에 "내가 정확히 어떤 필드가 필요한지"를 쿼리로 적어 보내고, 서버는 그 모양 그대로 응답을 조립해 돌려줍니다.

![REST vs GraphQL vs gRPC 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part6_기능별_심화/6-3_API_설계_패턴/assets/diagram1_rest_graphql_grpc.svg)

> **Q.** 그러면 GraphQL이 REST를 완전히 대체하지 못한 이유는 무엇일까요?
> **A.** REST의 강점이었던 "URL 단위 캐싱"을 그대로 쓰기 어렵기 때문입니다. GraphQL은 요청이 항상 같은 엔드포인트로 가고 본문(쿼리)이 요청마다 달라서, HTTP 캐시 서버가 URL만 보고 캐시할 수 없습니다. 또한 서버는 다양한 조합의 쿼리를 모두 처리할 수 있어야 하므로, 리졸버(resolver) 설계가 잘못되면 하나의 GraphQL 쿼리 뒤에서 데이터베이스 조회가 N번 일어나는 N+1 문제가 REST보다 오히려 눈에 띄지 않게 숨어들 수 있습니다. 그래서 공개 웹 API처럼 캐시 효율이 중요한 영역에서는 지금도 REST가 널리 쓰입니다.

## 4. gRPC — 내부 서비스 간 고성능·계약 기반 통신

> **개요** 서비스 수가 급증한 마이크로서비스 환경에서 gRPC가 어떤 문제(직렬화 비용, 느슨한 계약)를 겨냥해 등장했는지 다룹니다.

<u>**gRPC**</u>는 구글이 사내에서 쓰던 원격 프로시저 호출(RPC) 시스템 Stubby의 후속으로 2015년 오픈소스로 공개했습니다. 겨냥한 문제는 REST·GraphQL과 다릅니다. 마이크로서비스 아키텍처에서는 하나의 사용자 요청을 처리하는 동안에도 내부적으로 수십 개의 서비스가 서로를 호출합니다. 이 통신이 텍스트 기반 JSON이라면 ① 직렬화·역직렬화 비용이 누적되고, ② 어떤 필드가 어떤 타입인지가 문서에만 적혀 있어 서비스 A와 B의 담당자가 합의 없이 필드를 바꾸면 런타임에야 오류가 드러납니다.

gRPC는 <u>**Protocol Buffers(Protobuf)**</u>라는 바이너리 직렬화 포맷과 `.proto` 파일로 이 문제를 풉니다. `.proto` 파일에 함수 시그니처와 메시지 타입을 명시하면, 이로부터 각 언어별 클라이언트·서버 코드가 자동 생성되어 계약이 코드 수준에서 강제됩니다. 또한 HTTP/2 위에서 동작해 하나의 연결로 여러 요청·응답을 동시에 주고받는 양방향 스트리밍을 지원합니다.

![같은 화면 하나를 그리기 위한 세 가지 통신 방식 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part6_기능별_심화/6-3_API_설계_패턴/assets/diagram2_request_flow_compare.svg)

> **해석** 위 다이어그램은 "주문 상세 화면 하나를 그리는" 동일한 목표를, REST는 자원별 엔드포인트 3번 호출로, GraphQL은 필요한 필드를 지정한 단일 쿼리로, gRPC는 스키마로 계약이 고정된 단일 바이너리 호출로 처리하는 차이를 보여줍니다.

> **Q.** gRPC가 이렇게 빠르고 계약도 명확한데, 왜 REST나 GraphQL을 완전히 대체하지 못할까요?
> **A.** 브라우저가 gRPC를 직접 호출하기 어렵기 때문입니다. gRPC는 HTTP/2의 트레일러(trailer) 등 브라우저 표준 API가 지원하지 않는 기능에 의존해, 웹 클라이언트에서 쓰려면 별도의 프록시(gRPC-Web)가 필요합니다. 또한 사람이 응답을 눈으로 바로 읽을 수 없는 바이너리 포맷이라 디버깅 편의성도 떨어집니다. 그래서 gRPC는 "내부 서비스 간 통신"에 집중되어 쓰이고, 외부에 공개하는 API는 여전히 REST나 GraphQL로 감싸는 경우가 많습니다.

## 5. 실생활 예시 — 배달 앱이 세 방식을 함께 쓰는 이유

> **해석** 하나의 배달 앱 서비스 안에서도 통신 상대와 목적에 따라 REST·GraphQL·gRPC가 함께 쓰일 수 있습니다.

| 통신 구간 | 선택한 방식 | 이유 |
|---|---|---|
| 외부 제휴사가 주문 데이터를 조회하는 공개 API | REST | 문서화·캐싱·범용성이 중요하고, 제휴사마다 별도 클라이언트 코드를 만들지 않아도 됨 |
| 앱 화면(주문 상세: 사용자+주문+상품 정보 조합) | GraphQL | 화면마다 필요한 필드 조합이 달라, 매번 새 REST 엔드포인트를 만들지 않고 클라이언트가 필요한 필드만 조합해서 요청 |
| 내부 주문 서비스 ↔ 재고 서비스 ↔ 결제 서비스 통신 | gRPC | 초당 호출 횟수가 많고, 팀 간 계약을 스키마로 강제해 실수를 줄이며, 직렬화 비용을 최소화해야 함 |

> **팁** "API를 REST로 만들지, GraphQL로 만들지, gRPC로 만들지"는 유행이 아니라 **통신 상대가 누구인가(브라우저인가 내부 서비스인가)와 데이터 요구가 얼마나 다양한가**로 결정해야 합니다. 세 방식은 경쟁 관계라기보다 서로 다른 통신 문제를 겨냥한 도구이며, 실제로는 한 서비스 안에서 함께 쓰이는 경우가 더 흔합니다.

> **원칙** 새로운 통신 방식을 도입하기 전에는 항상 "지금 이 통신에서 병목이 정확히 무엇인가(캐싱 부재인가, 오버페칭인가, 직렬화 비용인가)"를 먼저 확인해야 합니다. 병목을 확인하지 않고 최신 기술을 도입하면, REST로도 충분한 공개 API에 불필요하게 GraphQL 서버의 복잡도를 더하는 식의 과잉 설계로 이어집니다.

## 요약

> **핵심**
> - REST는 자원을 URL로, 자원에 대한 행위를 HTTP 메서드로 표현하는 규약(Roy Fielding, 2000)으로, 웹 인프라의 캐시·범용성을 그대로 활용하지만 화면 요구가 다양해지면 오버페칭·언더페칭 문제가 생깁니다.
> - GraphQL(Facebook, 2015 공개)은 단일 엔드포인트에서 클라이언트가 필요한 필드를 직접 지정하게 해 오버페칭·언더페칭을 줄이지만, 그 대가로 URL 기반 캐싱과 서버 측 복잡도 관리를 잃습니다.
> - gRPC(Google, 2015 공개)는 Protocol Buffers와 HTTP/2로 내부 마이크로서비스 간 직렬화 비용과 계약 불명확성을 해결하지만, 브라우저 직접 호출과 가독성에서는 불리해 주로 서비스 간 통신에 쓰입니다.
> - 세 방식은 등장 순서상의 대체 관계가 아니라, **통신 상대와 병목의 종류가 다른 문제를 각각 겨냥한 것**이며 실무에서는 함께 쓰이는 것이 일반적입니다.
