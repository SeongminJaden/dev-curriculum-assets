# Part 6. 기능별 심화

## 6-2. 동시성과 비동기 프로그래밍 — 스레드를 늘리는 것과 비동기로 처리하는 것은 무엇이 다른가

> **학습 목표**
> - 동시성(Concurrency)과 병렬성(Parallelism)의 차이를 구분하고, 서버가 왜 "동시에 많은 요청"을 다뤄야 하는지 설명할 수 있다.
> - 멀티스레딩이 컨텍스트 스위칭을 통해 동시 요청을 처리하는 방식과, 그 과정에서 발생하는 비용을 설명할 수 있다.
> - 이벤트 루프·논블로킹 I/O가 멀티스레딩과 근본적으로 다른 지점을 설명하고, 어떤 상황에 어느 방식이 유리한지 판단할 수 있다.

> **개요** 동시에 몰려드는 요청을 처리하는 두 가지 근본 전략, 즉 스레드를 늘리는 멀티스레딩과 스레드 하나로 순회하는 이벤트 루프 방식을 비교합니다. 스레드를 늘리는 방식이 왜 한계에 부딪히는지 컨텍스트 스위칭 비용으로 설명하고, 이벤트 루프가 어떻게 그 비용 없이 대량의 동시 연결을 처리하는지 다룹니다.

---

## 1. 동시성이란 무엇인가 — 동시성과 병렬성은 다릅니다

> **개요** "동시에 처리한다"는 말이 실제로는 두 가지 다른 상황(동시성과 병렬성)을 가리킬 수 있다는 점과, 서버가 왜 이 문제를 피할 수 없는지를 다룹니다.

Go 언어 설계자 중 한 명인 롭 파이크(Rob Pike)는 2012년 강연에서 "<u>**동시성은 병렬성이 아니다(Concurrency is not Parallelism)**</u>"라는 문장으로 이 둘을 구분했습니다.

- <u>**병렬성(Parallelism)**</u>: 여러 작업을 물리적으로 동시에(같은 순간에) 실행하는 것입니다. CPU 코어가 여러 개 있어야 가능합니다.
- <u>**동시성(Concurrency)**</u>: 여러 작업의 실행 흐름을 겹쳐서 다루는 것입니다. 한 순간에는 하나만 실제로 실행되더라도, 짧은 시간 단위로 번갈아 처리하면 마치 여러 작업이 동시에 진행되는 것처럼 보입니다.

> **Q.** 서버는 왜 이 문제에서 자유로울 수 없을까요?
> **A.** 웹 서버, 채팅 서버, API 서버는 모두 "한 명의 사용자만" 상대하지 않습니다. 수백~수만 명의 사용자가 비슷한 시각에 요청을 보내고, 서버는 그 요청들을 어떤 순서로든 겹쳐서 처리해야 합니다. 이때 서버가 가진 CPU 코어 수는 유한하므로(보통 사용자 수보다 훨씬 적으므로), 실제로는 병렬성보다 동시성으로 이 문제를 풀 수밖에 없는 경우가 대부분입니다. 이 강의에서 비교하는 멀티스레딩과 이벤트 루프는 둘 다 "동시성을 구현하는 방식"이며, 그중 멀티스레딩만 코어가 여러 개일 때 부분적으로 병렬성도 함께 얻습니다.

> **참고** 요청의 성격도 중요합니다. 요청 처리 시간의 대부분이 CPU 연산(암호화, 이미지 처리)에 쓰이는 작업을 <u>**CPU 바운드(CPU-bound)**</u>, 대부분이 디스크·네트워크·DB 응답을 "기다리는" 시간인 작업을 <u>**I/O 바운드(I/O-bound)**</u>라 부릅니다. 일반적인 웹 백엔드 요청은 DB 조회를 기다리는 시간이 대부분을 차지하는 I/O 바운드에 가깝습니다.

## 2. 멀티스레딩 — 요청마다 실행 흐름을 하나씩 배정한다

> **개요** 프로세스와 스레드의 관계, OS 스케줄러가 CPU 시간을 스레드에 나누어 주는 방식, 그리고 이 과정에서 발생하는 컨텍스트 스위칭 비용을 다룹니다.

<u>**프로세스(Process)**</u>는 실행 중인 프로그램에 할당된 독립된 메모리 공간(코드·힙·전역변수)이고, <u>**스레드(Thread)**</u>는 그 프로세스 안에서 실제로 명령어를 실행하는 흐름의 단위입니다. 한 프로세스 안의 여러 스레드는 힙과 코드는 공유하지만, 자신만의 스택과 레지스터 값(프로그램 카운터 등 실행 상태)은 따로 가집니다.

<u>**멀티스레딩(Multithreading)**</u> 방식의 서버는 요청이 들어올 때마다(또는 미리 만들어둔 스레드 풀에서) 스레드 하나를 배정해 그 요청 전체를 처리하게 합니다. CPU 코어 수는 한정되어 있으므로, OS 스케줄러는 아주 짧은 시간 단위(타임 슬라이스)로 실행 중인 스레드를 바꿔가며 여러 스레드에 CPU 시간을 나누어 줍니다. 이렇게 실행 중인 스레드를 다른 스레드로 바꾸는 과정을 <u>**컨텍스트 스위칭(Context Switching)**</u>이라 합니다.

컨텍스트 스위칭은 공짜가 아닙니다. OS는 현재 스레드의 레지스터 값을 커널 스택에 저장하고, 다음 실행할 스레드를 스케줄러 큐에서 고른 뒤, 그 스레드의 저장된 레지스터 값을 복원해야 합니다. 여기까지가 <u>**직접 비용**</u>입니다. 더 큰 문제는 <u>**간접 비용**</u>입니다. 새로 실행되는 스레드가 사용하는 데이터는 이전 스레드가 채워둔 CPU 캐시나 TLB(Translation Lookaside Buffer, 가상주소-물리주소 변환 캐시)에 없을 가능성이 높고, 이 경우 캐시를 다시 채우는 동안 CPU는 상대적으로 느려집니다.

![컨텍스트 스위칭의 4단계와 상대적 비용](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part6_기능별_심화/6-2_동시성과_비동기_프로그래밍/assets/diagram2_context_switch_cost.svg)

> **Q.** 그러면 스레드를 최대한 많이 만들면 더 많은 요청을 동시에 처리할 수 있지 않을까요?
> **A.** 어느 지점까지는 그렇지만, 스레드 수가 늘어날수록 두 가지 비용이 함께 늘어납니다. 첫째, 스레드마다 스택 메모리(보통 수백 KB~수 MB)가 필요하므로 스레드가 많아지면 메모리 자체가 부족해집니다. 둘째, 스레드 수가 코어 수를 훨씬 넘어서면 스케줄러가 그만큼 자주 컨텍스트 스위칭을 해야 하고, 방금 설명한 간접 비용(캐시 재적재)이 누적되어 오히려 스레드 하나당 처리 속도가 떨어집니다. 특히 요청 대부분이 I/O 바운드라면, 스레드는 응답을 기다리는 동안(Blocked 상태) CPU를 쓰지도 않으면서 스택 메모리와 스케줄링 비용만 차지하게 됩니다. 이 현상은 2000년대 초 동시 접속 1만 명을 처리하려는 서버가 공통으로 겪었던 문제라 해서 <u>**C10K 문제(C10K Problem)**</u>라 불렸습니다.

## 3. 비동기 프로그래밍 — 이벤트 루프와 논블로킹 I/O

> **개요** 이벤트 루프가 스레드를 늘리지 않고도 대량의 I/O 바운드 요청을 처리할 수 있는 이유와, 이를 가능하게 하는 논블로킹 I/O의 원리를 다룹니다.

<u>**이벤트 루프(Event Loop)**</u>는 C10K 문제에 대한 다른 접근입니다. 요청마다 스레드를 배정하는 대신, 단 하나(또는 소수)의 스레드가 "처리할 작업이 담긴 큐"를 끊임없이 순회하면서 작업을 하나씩 꺼내 실행합니다.

이 방식이 성립하려면 <u>**논블로킹 I/O(Non-blocking I/O)**</u>가 필요합니다. 전통적인(블로킹) I/O 호출은 디스크나 네트워크 응답이 올 때까지 그 스레드를 멈춰 세웁니다. 반면 논블로킹 I/O는 "이 작업 좀 부탁해"라고 커널(또는 별도의 이벤트 알림 메커니즘, 예: 리눅스의 epoll, macOS의 kqueue, 윈도우의 IOCP)에 위임만 해두고, 스레드는 결과를 기다리지 않은 채 즉시 다음 작업으로 넘어갑니다. 위임한 작업이 끝나면 그 결과를 처리할 콜백(callback)이 다시 큐에 등록되고, 이벤트 루프가 자기 차례가 되었을 때 그 콜백을 꺼내 실행합니다.

![멀티스레딩 vs 이벤트 루프 처리 흐름 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part6_기능별_심화/6-2_동시성과_비동기_프로그래밍/assets/diagram1_thread_vs_eventloop.svg)

> **해석** 왼쪽 멀티스레딩 구조는 요청 하나당 스레드 하나를 점유하며, I/O 응답을 기다리는 동안에도(Blocked) 그 스레드의 스택 메모리는 계속 차지된 상태입니다. 오른쪽 이벤트 루프 구조는 스레드가 단 하나이며, I/O 요청을 커널에 맡겨두고 곧바로 다음 콜백을 처리하러 넘어갑니다. 컨텍스트 스위칭 자체가 거의 발생하지 않으므로, 그 비용이 통째로 사라집니다.

> **Q.** 그렇다면 이벤트 루프가 멀티스레딩보다 항상 더 나은 선택일까요?
> **A.** 아닙니다. 이벤트 루프의 강점은 "대기 시간이 긴 I/O 바운드 작업을 아주 많이 동시에 처리하는 것"에서 나옵니다. 만약 콜백 하나가 이미지 리사이징이나 암호화처럼 CPU를 오래 붙잡는 연산(CPU 바운드)을 수행한다면, 그 콜백이 끝날 때까지 스레드 전체(=이벤트 루프 전체)가 멈춥니다. 다른 모든 대기 중인 요청이 그 하나의 연산이 끝날 때까지 기다려야 하므로, CPU 바운드 작업에는 오히려 멀티스레딩(또는 멀티프로세싱)이 유리합니다. Node.js가 무거운 연산을 워커 스레드(worker thread)나 별도 프로세스로 분리하도록 권장하는 이유가 여기에 있습니다.

## 4. 실생활 예시 — 채팅 서버는 왜 동시 접속자를 스레드 대신 이벤트 루프로 처리할까요

> **해석** 아래 표는 동시 접속자 수만 명 규모의 채팅 서버를 멀티스레딩과 이벤트 루프 각각으로 구현했을 때의 차이를 보여줍니다. 채팅 접속은 대부분의 시간 동안 "메시지가 오기를 기다리는" 전형적인 I/O 바운드 작업입니다.

| 항목 | 멀티스레딩(연결마다 스레드) | 이벤트 루프(단일 스레드 + 논블로킹) |
|---|---|---|
| 접속자 1만 명 유지 비용 | 스레드 1만 개의 스택 메모리 + 잦은 컨텍스트 스위칭 | 스레드 1개(또는 소수)가 1만 개의 소켓을 논블로킹으로 감시 |
| 대부분의 시간 | 스레드들이 "메시지 대기" 상태(Blocked)로 CPU 없이 메모리만 점유 | 대기 중인 연결은 커널의 이벤트 알림 메커니즘에 위임되어 스레드를 차지하지 않음 |
| 메시지 도착 시 | 해당 스레드가 스케줄러에 의해 깨어남(컨텍스트 스위칭 발생) | 이벤트 루프가 콜백 큐에서 해당 이벤트를 꺼내 순서대로 처리 |
| 확장 한계 | 스레드 수가 늘수록 스위칭 비용이 커져 성능 저하 | 콜백 하나가 CPU를 오래 잡으면 전체 루프가 지연 |

> **팁** 실무에서는 "이 서버가 다루는 요청이 I/O 바운드인가 CPU 바운드인가"를 먼저 판단하는 것이 설계의 출발점입니다. Node.js, Nginx, Redis처럼 이벤트 루프 기반 시스템이 널리 쓰이는 이유는 이들이 다루는 작업(HTTP 요청 중계, 키-값 조회)이 대부분 I/O 바운드이기 때문입니다.

> **원칙** 동시성 문제를 해결할 때는 "스레드를 늘릴 것인가, 비동기로 처리할 것인가"를 정하기 전에 **각 요청이 CPU를 실제로 쓰는 시간과 무언가를 기다리는 시간 중 어느 쪽이 더 긴가**를 먼저 측정해야 합니다. 이 판단 없이 방식부터 고르면, I/O 바운드 서버에 스레드를 무한정 늘리거나 CPU 바운드 서버를 이벤트 루프 하나로 처리하는 반대의 실수를 하게 됩니다.

## 요약

> **핵심**
> - 동시성(여러 작업의 흐름을 겹쳐 다룸)과 병렬성(여러 작업을 물리적으로 동시에 실행)은 다른 개념이며, 서버가 마주치는 문제는 대부분 동시성 문제입니다.
> - 멀티스레딩은 요청마다 스레드를 배정해 OS 스케줄러가 컨텍스트 스위칭으로 CPU를 나누어 쓰게 하며, 이 과정에는 레지스터 저장/복원(직접 비용)과 캐시·TLB 재적재(간접 비용)가 함께 듭니다.
> - 이벤트 루프는 단일 스레드가 논블로킹 I/O로 위임한 작업들의 완료 콜백을 큐에서 꺼내 순서대로 처리해, 컨텍스트 스위칭 비용 없이 대량의 I/O 바운드 요청을 동시에 다룹니다. 다만 콜백 하나가 CPU를 오래 잡으면 루프 전체가 멈추므로, CPU 바운드 작업에는 여전히 멀티스레딩(또는 멀티프로세싱)이 유리합니다.
