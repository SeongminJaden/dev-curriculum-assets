# Part 19. 이산수학과 비트 연산

## 19-1. 대수 법칙과 멱등성 — 왜 "재시도해도 안전한가"는 수학 문제인가

> **학습 목표**
> - 멱등성의 수학적 정의 $f(f(x)) = f(x)$를 설명하고, 어떤 연산이 멱등한지 판별할 수 있다.
> - 결합법칙·교환법칙·분배법칙이 각각 어떤 코드 최적화(병렬 리듀스, 순서 무관 병합, 연산 재배치)를 정당화하는지 근거를 들어 설명할 수 있다.
> - HTTP 메서드별 멱등성 차이를 대수 법칙의 사례로 이해하고, 재시도 안전성을 설계에 반영할 수 있다.

> **개요** 네트워크는 실패합니다. 요청을 보냈는데 응답이 오지 않았을 때, 그 요청이 서버에 도달하지 못한 것인지 도달했지만 응답만 유실된 것인지 클라이언트는 알 수 없습니다. 이때 "그냥 한 번 더 보내도 되는가"라는 질문에 답하는 것은 재시도 로직이 아니라 그 연산이 가진 **대수적 성질**입니다. 이 강의는 멱등성·결합법칙·교환법칙·분배법칙이라는 네 가지 법칙이 실제 코드에서 어떤 최적화와 안전성을 열어주는지 살펴봅니다.

## 1. 멱등성이란 무엇인가 — 두 번 해도 한 번 한 것과 같다

> **Q.** "여러 번 실행해도 결과가 같다"는 말은 너무 당연해 보입니다. 이게 왜 별도의 이름을 가질 만큼 중요한 성질인가요?
> **A.** 당연해 보이는 이유는 우리가 평소에 멱등한 연산과 그렇지 않은 연산을 뭉뚱그려 생각하기 때문입니다. "잔액을 100원으로 설정한다"는 몇 번을 실행해도 잔액은 100원입니다. 하지만 "잔액에서 100원을 뺀다"는 두 번 실행하면 200원이 빠집니다. 두 연산은 코드에서 겨우 한 글자 차이(`=` vs `-=`)지만, 네트워크 장애 상황에서의 안전성은 정반대입니다. 멱등성은 이 결정적 차이에 이름을 붙여, "재시도해도 되는가"라는 질문을 사람의 직관이 아닌 성질의 검증으로 바꿔놓습니다.

<u>**멱등성(Idempotency)**</u>은 어떤 연산을 한 번 적용한 결과와 여러 번 적용한 결과가 같은 성질입니다. 함수로 표현하면 다음과 같습니다.

$$f(f(x)) = f(x) \quad \Longrightarrow \quad f^n(x) = f(x) \ \ (n \geq 1)$$

집합 연산에서도 같은 성질이 나타납니다.

$$A \cup A = A, \qquad A \cap A = A$$

이 용어는 19세기 미국 수학자 벤저민 퍼스(Benjamin Peirce)가 1870년 저작 *Linear Associative Algebra*에서 도입했습니다. 라틴어 *idem*(같은) + *potent*(거듭제곱)의 합성으로, "거듭제곱해도 자기 자신인 원소"라는 뜻입니다.

> **참고** 멱등성과 <u>**순수성(purity)**</u>은 다른 개념입니다. 순수 함수는 부작용이 없고 같은 입력에 항상 같은 출력을 냅니다. 멱등한 연산은 부작용이 있어도 됩니다 — 다만 그 부작용이 두 번째 실행부터 상태를 더 바꾸지 않아야 합니다. `DELETE /users/1`은 파일 시스템을 바꾸므로 순수하지 않지만, 두 번 호출해도 "사용자 1이 없는 상태"라는 결과는 같으므로 멱등합니다.

## 2. 멱등한 연산과 아닌 연산 — 판별해봅시다

> **개요** 실무 코드에서 마주치는 연산들을 멱등한 것과 아닌 것으로 나누어, 판별 기준을 손에 익힙니다.

![멱등한 연산과 비멱등 연산의 재시도 결과 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part19_이산수학과_비트_연산/19-1_대수_법칙과_멱등성/assets/diagram1_idempotency.svg)

> **해석** 위 그림은 같은 요청이 네트워크 문제로 두 번 도달했을 때 두 연산이 어떻게 갈라지는지를 보여줍니다. 왼쪽의 "잔액을 100으로 설정"은 첫 번째 실행 후 상태가 100이 되고, 두 번째 실행도 100을 만들어 최종 상태가 동일합니다 — 상태가 이미 <u>**고정점(fixed point)**</u>에 도달했기 때문입니다. 오른쪽의 "잔액에서 100 차감"은 실행할 때마다 상태가 계속 이동하므로 고정점이 없고, 중복 실행이 곧 데이터 손상입니다. 멱등성의 본질은 "연산의 결과가 그 연산의 고정점인가"입니다.

| 연산 | 멱등 여부 | 이유 |
|---|---|---|
| `set(key, value)` | 멱등 | 몇 번을 써도 값은 `value` |
| `counter += 1` | 비멱등 | 실행마다 값이 이동 |
| `list.append(x)` | 비멱등 | 원소가 계속 쌓임 |
| `set.add(x)` | 멱등 | 집합은 중복을 담지 않음 ($A \cup \{x\} \cup \{x\} = A \cup \{x\}$) |
| `abs(x)` | 멱등 | $\lvert\,\lvert x \rvert\,\rvert = \lvert x \rvert$ |
| `max(a, x)` | 멱등 | $\max(\max(a,x), x) = \max(a,x)$ |
| `sort(list)` | 멱등 | 이미 정렬된 것을 다시 정렬해도 같음 |
| `DROP TABLE t` | 비멱등 | 두 번째 실행은 에러 |
| `DROP TABLE IF EXISTS t` | 멱등 | 없으면 아무것도 안 함 |

> **팁** `IF EXISTS`, `CREATE OR REPLACE`, `INSERT ... ON CONFLICT DO NOTHING` 같은 SQL 구문은 전부 비멱등 연산을 멱등하게 만드는 장치입니다. 데이터베이스 마이그레이션 스크립트를 이런 형태로 작성하면, 중간에 실패한 마이그레이션을 처음부터 다시 돌려도 안전합니다.

## 3. HTTP 메서드의 멱등성 — 수학적 성질이 프로토콜에 새겨진 사례

> **Q.** GET·PUT·DELETE는 멱등이고 POST는 아니라고들 하는데, 이건 규칙으로 정한 건가요, 아니면 성질에서 따라 나오는 건가요?
> **A.** 각 메서드가 하기로 약속된 일의 의미에서 따라 나옵니다. PUT은 "이 URI의 리소스를 이 표현으로 **대체하라**"이므로 `set`과 같은 대입 연산이고, 대입은 본질적으로 멱등합니다. DELETE는 "이 리소스가 없는 상태로 만들라"이므로 두 번째 호출도 같은 상태를 남깁니다(응답 코드는 404로 달라질 수 있지만, 멱등성은 **서버 상태**에 대한 성질이지 응답에 대한 성질이 아닙니다). 반면 POST는 "이 데이터를 하위 리소스로 **추가하라**"이므로 `append`와 같고, 실행할 때마다 새 리소스가 생깁니다. 즉 규칙이 성질을 만든 것이 아니라, 성질을 관찰해 규격에 명시한 것입니다.

| 메서드 | 안전(safe) | 멱등 | 대응하는 연산 |
|---|---|---|---|
| GET | O | O | 읽기 — 상태를 바꾸지 않음 |
| HEAD | O | O | 읽기(본문 없음) |
| PUT | X | O | 대입 `x = v` |
| DELETE | X | O | "없는 상태로" 설정 |
| POST | X | **X** | 추가 `append` |
| PATCH | X | 경우에 따라 다름 | 필드 설정이면 멱등, 증가 연산이면 비멱등 |

> **참고** 이 구분은 HTTP 규격(RFC 9110, 2022년 IETF 발행 — 이전의 RFC 7231을 대체)의 "Method Properties" 절에 명시되어 있습니다. **안전(safe)** 은 서버 상태를 전혀 바꾸지 않는다는 더 강한 성질이며, 안전한 메서드는 자동으로 멱등합니다. 역은 성립하지 않습니다 — DELETE는 멱등하지만 안전하지 않습니다.

> **팁** POST를 멱등하게 만드는 실무 표준 기법은 <u>**멱등성 키(idempotency key)**</u>입니다. 클라이언트가 요청마다 고유 키(UUID 등)를 헤더에 담아 보내면, 서버는 그 키를 저장해두고 같은 키로 다시 오는 요청에는 저장해둔 첫 응답을 재생합니다. Stripe API의 `Idempotency-Key` 헤더가 대표적인 구현이며, 이 방식으로 결제 중복 발생을 원천 차단합니다. REST 리소스 설계 전반은 Part 6의 `6-3 API 설계 패턴`을 참고하세요.

## 4. 결합법칙 — 병렬 리듀스와 분산 집계를 여는 열쇠

> **개요** 결합법칙이 성립하는지 여부가 곧 "이 집계를 여러 대의 머신에 쪼개 돌릴 수 있는가"를 결정합니다.

<u>**결합법칙(Associativity)**</u>은 연산의 **묶는 순서**를 바꿔도 결과가 같다는 성질입니다.

$$(a \oplus b) \oplus c = a \oplus (b \oplus c)$$

![결합법칙이 병렬 리듀스를 가능하게 하는 원리](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part19_이산수학과_비트_연산/19-1_대수_법칙과_멱등성/assets/diagram2_associative_reduce.svg)

> **해석** 위 그림의 왼쪽은 결합법칙이 없는 세계입니다. `((((a+b)+c)+d)+e)+f`처럼 반드시 왼쪽부터 순서대로 접어야 하므로, 원소가 $n$개면 단계 수도 $n-1$개이고 어떤 두 단계도 동시에 실행할 수 없습니다. 오른쪽은 결합법칙이 성립할 때입니다. `(a+b)`, `(c+d)`, `(e+f)`를 서로 다른 코어나 서로 다른 머신이 **동시에** 계산한 뒤 그 결과를 다시 합칠 수 있고, 어떻게 묶든 답이 같음이 수학적으로 보장됩니다. 단계 수는 $n-1$에서 $\lceil \log_2 n \rceil$로 줄어듭니다. 병렬 리듀스가 빠른 것은 하드웨어 덕분이 아니라 연산이 결합법칙을 만족하기 때문입니다.

| 연산 | 결합법칙 | 병렬 리듀스 가능? |
|---|---|---|
| `+`, `*` (정수) | O | 가능 |
| `min`, `max` | O | 가능 |
| 문자열 이어붙이기 | O | 가능(순서는 유지해야 함) |
| 집합 합집합 `∪` | O | 가능 |
| `-` (뺄셈) | X | 불가 — $(5-3)-1 \neq 5-(3-1)$ |
| 부동소수점 `+` | **근사적으로만** | 주의 — 반올림 오차로 결과가 달라짐 |

> **참고** 부동소수점 덧셈은 결합법칙을 **엄밀히는 만족하지 않습니다**. $(10^{20} + (-10^{20})) + 1 = 1$이지만 $10^{20} + ((-10^{20}) + 1) = 0$이 될 수 있습니다. 이 때문에 병렬 합산은 실행할 때마다 미세하게 다른 값을 낼 수 있으며, 수치 재현성이 중요한 분야에서는 이것이 실제 버그로 취급됩니다. 부동소수점 표현의 근본 원인은 Part 19-4에서 다루는 2의 보수와는 별개로, IEEE 754 규격에서 비롯됩니다.

> **팁** MapReduce의 <u>**컴바이너(combiner)**</u>, Spark의 `reduceByKey`, SQL의 `SUM`/`COUNT`/`MAX` 같은 집계 함수가 여러 노드에 분산되어 실행될 수 있는 근거가 바로 결합법칙입니다. 반대로 `MEDIAN`(중앙값)은 결합법칙이 성립하지 않아 부분 결과를 단순히 합칠 수 없고, 그래서 분산 환경에서 정확한 중앙값 계산이 유독 비싼 연산입니다.

## 5. 교환법칙 — 순서를 몰라도 되는 자유

> **개요** 결합법칙이 "묶는 순서"의 자유라면, 교환법칙은 "도착 순서"의 자유입니다. 분산 시스템에서 이 차이는 결정적입니다.

<u>**교환법칙(Commutativity)**</u>은 피연산자의 **순서**를 바꿔도 결과가 같다는 성질입니다.

$$a \oplus b = b \oplus a$$

> **Q.** 결합법칙만 있으면 병렬 처리는 되는데, 교환법칙까지 필요한 상황은 언제인가요?
> **A.** 데이터가 **어떤 순서로 도착할지 통제할 수 없을 때**입니다. 결합법칙은 순서는 그대로 두고 묶음만 바꿉니다 — 문자열 이어붙이기는 결합법칙은 만족하지만 `"ab" + "cd" ≠ "cd" + "ab"`이므로 조각이 순서대로 도착해야 합니다. 반면 서로 다른 지역의 서버가 네트워크 지연 때문에 제각각의 순서로 갱신을 주고받는 상황에서는, 어떤 순서로 병합해도 모든 노드가 같은 최종 상태에 수렴해야 합니다. 이때 필요한 것이 교환법칙입니다.

이 성질을 정면으로 활용한 자료구조가 <u>**CRDT(Conflict-free Replicated Data Type, 충돌 없는 복제 자료형)**</u>입니다. 2011년 마렉 샤피로(Marc Shapiro)와 누노 프레기사(Nuno Preguiça) 등이 발표한 논문에서 정식화되었으며, 병합 연산이 **결합·교환·멱등**의 세 성질을 모두 만족하도록 설계합니다.

```python
# G-Set(Grow-only Set): 가장 단순한 CRDT
def merge(replica_a, replica_b):
    return replica_a | replica_b   # 합집합

# 세 성질이 모두 성립하므로:
#  결합: (A|B)|C == A|(B|C)   -> 어떤 묶음으로 병합해도 동일
#  교환: A|B == B|A           -> 어떤 순서로 도착해도 동일
#  멱등: A|A == A             -> 같은 갱신이 중복 전달돼도 안전
```

> **해석** 이 세 성질을 모두 갖춘 병합 연산을 쓰면, 네트워크가 메시지를 **재정렬하고, 중복 전달하고, 지연시켜도** 모든 복제본이 결국 같은 상태로 수렴합니다. 이것이 <u>**결과적 일관성(eventual consistency)**</u>을 코디네이션(합의 프로토콜) 없이 달성하는 방법이며, Redis의 CRDT 기반 지리 분산 복제나 여러 협업 편집기의 동기화 엔진이 이 원리 위에 서 있습니다. 세 법칙이 함께 성립하는 구조를 수학에서는 <u>**반격자(semilattice)**</u>라고 부릅니다.

## 6. 분배법칙 — 컴파일러와 쿼리 옵티마이저가 하는 일

> **개요** 분배법칙은 두 연산 사이의 관계를 규정하며, 연산 재배치를 통한 성능 최적화의 근거가 됩니다.

<u>**분배법칙(Distributivity)**</u>은 한 연산이 다른 연산 위로 퍼져나갈 수 있다는 성질입니다.

$$a \times (b + c) = (a \times b) + (a \times c)$$

집합에서는 두 방향 모두 성립합니다.

$$A \cap (B \cup C) = (A \cap B) \cup (A \cap C), \qquad A \cup (B \cap C) = (A \cup B) \cap (A \cup C)$$

> **해석** 분배법칙은 **같은 답을 내는 두 가지 계산 경로**를 보장하므로, 그중 더 싼 쪽을 고를 자유를 줍니다. 왼쪽 $a \times (b+c)$는 덧셈 1회와 곱셈 1회, 오른쪽은 곱셈 2회와 덧셈 1회입니다. 곱셈이 비싼 하드웨어에서는 왼쪽이 유리하므로 컴파일러가 오른쪽 형태를 왼쪽으로 되돌리며(이를 <u>**인수분해(factoring)**</u> 또는 공통 부분식 제거라 합니다), 반대로 벡터화가 가능한 상황에서는 펼치는 쪽이 유리할 수도 있습니다.

> **팁** 데이터베이스 쿼리 옵티마이저의 <u>**조건 밀어내리기(predicate pushdown)**</u>도 분배법칙의 응용입니다. `필터(A UNION B)`를 `필터(A) UNION 필터(B)`로 바꾸면 각 테이블에서 미리 행을 걸러내 병합할 데이터 양 자체를 줄일 수 있습니다. PostgreSQL이나 Spark SQL의 실행 계획에서 필터가 스캔 노드 쪽으로 내려가 있는 것을 볼 수 있는데, 이 변형이 안전한 이유는 필터(선택)와 합집합 사이에 분배법칙이 성립하기 때문입니다.

## 요약

> **핵심**
> - **멱등성** $f(f(x)) = f(x)$는 "재시도해도 안전한가"를 판별하는 성질입니다. 대입·`max`·합집합·`DROP IF EXISTS`는 멱등하고, 증가·`append`·`DROP TABLE`은 아닙니다. HTTP에서 GET/PUT/DELETE가 멱등하고 POST가 아닌 것은 규칙이 아니라 각 메서드가 뜻하는 연산의 성질에서 따라 나옵니다.
> - **결합법칙**은 묶는 순서의 자유를 주어 병렬 리듀스와 분산 집계를 가능하게 하고($n-1$ 단계 → $\log n$ 단계), **교환법칙**은 도착 순서의 자유를 주어 CRDT의 순서 무관 병합을 가능하게 합니다. 결합·교환·멱등이 모두 성립하면 코디네이션 없이 결과적 일관성에 도달합니다.
> - **분배법칙**은 같은 답을 내는 여러 계산 경로를 보장해, 컴파일러의 인수분해와 쿼리 옵티마이저의 조건 밀어내리기 같은 재배치 최적화를 정당화합니다.
> - 대수 법칙은 추상적인 장식이 아니라 **어떤 최적화가 안전한지 판별하는 검증 도구**입니다. 최적화가 가능한지 고민될 때는 "이 연산이 어떤 법칙을 만족하는가"를 먼저 물어보세요.
