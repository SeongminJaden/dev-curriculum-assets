# Part 19. 이산수학과 비트 연산

## 19-4. 비트 연산과 2의 보수 — 왜 뺄셈 회로가 따로 없어도 되는가

> **학습 목표**
> - AND·OR·XOR·NOT·시프트 연산자의 동작을 비트 단위로 설명하고, 산술 시프트와 논리 시프트의 차이를 구분할 수 있다.
> - XOR의 대수적 성질을 이용해 스왑·중복 찾기 같은 문제를 추가 메모리 없이 해결할 수 있다.
> - 2의 보수로 음수를 표현하는 방식을 설명하고, 왜 이 표현이 덧셈기 하나로 뺄셈까지 처리하게 해주는지 근거를 들어 설명할 수 있다.

> **개요** `x & (x - 1)`이 왜 가장 낮은 1비트를 지우는지, `-x`가 왜 비트를 뒤집고 1을 더한 값인지, `for (int i = 0; i < n; i++)`에서 `n`이 21억을 넘으면 왜 음수가 되는지. 이 강의는 프로그래밍 언어의 비트 연산자와 정수 표현 방식을 다룹니다. 이 연산들을 실제로 수행하는 논리 게이트와 가산기 회로는 Part 18에서 다루므로, 여기서는 **코드에서 무엇이 일어나는가**에 집중합니다.

## 1. 비트 연산자 다섯 가지

> **Q.** 요즘 언어에는 `set`이나 `bool` 배열이 있는데, 굳이 비트 연산을 알아야 할 이유가 있나요?
> **A.** 세 가지 이유가 있습니다. 첫째, 표준 라이브러리와 시스템 코드가 비트 연산으로 쓰여 있어 읽을 수 있어야 합니다 — 파일 열기 플래그 `O_RDONLY | O_CREAT`, 권한 마스크 `0o755`, 색상 값 `(r << 16) | (g << 8) | b`가 전부 비트 연산입니다. 둘째, 부분집합을 정수 하나로 표현하면 `set`보다 수십 배 빠르고 메모리를 훨씬 적게 씁니다. 셋째, 정수 오버플로와 음수 표현을 이해하지 못하면 원인을 알 수 없는 버그를 만납니다.

| 연산자 | 이름 | 규칙 | 예시(4비트) |
|---|---|---|---|
| `&` | AND | 둘 다 1일 때만 1 | `1100 & 1010 = 1000` |
| `\|` | OR | 하나라도 1이면 1 | `1100 \| 1010 = 1110` |
| `^` | XOR | 서로 다를 때만 1 | `1100 ^ 1010 = 0110` |
| `~` | NOT | 모든 비트 반전 | `~1100 = 0011` (부호 포함 시 주의) |
| `<<` | 왼쪽 시프트 | 왼쪽으로 밀기 = $\times 2^k$ | `0011 << 2 = 1100` |
| `>>` | 오른쪽 시프트 | 오른쪽으로 밀기 = $\div 2^k$ | `1100 >> 2 = 0011` |

![비트 연산자의 동작과 시프트](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part19_이산수학과_비트_연산/19-4_비트_연산과_2의_보수/assets/diagram1_bit_operators.svg)

> **해석** 위 그림의 왼쪽은 AND·OR·XOR의 진리표를 비트열 위에서 보여줍니다. 각 자릿수가 **서로 독립적으로** 계산된다는 점이 핵심입니다 — 덧셈과 달리 자리올림이 없으므로, 64비트 정수에 대한 AND 한 번은 64개의 독립 판단을 CPU 한 사이클에 처리합니다. 이것이 비트 연산이 빠른 근본적인 이유입니다. 오른쪽은 시프트를 보여줍니다. 왼쪽 시프트는 각 비트를 한 칸씩 왼쪽으로 밀고 빈 자리를 0으로 채우므로 2를 곱한 것과 같고, 오른쪽 시프트는 그 반대로 2로 나눈 몫이 됩니다.

> **참고** 시프트가 곱셈·나눗셈과 같은 이유는 10진법에서 자릿수를 미는 것이 10배가 되는 것과 정확히 같은 원리입니다. 2진법에서 한 자리 밀면 $2^1$배가 됩니다. 다만 **컴파일러가 이미 이 최적화를 해줍니다** — `x * 8`을 `x << 3`으로 직접 쓰는 것은 현대 컴파일러에서 성능 이득이 없고 가독성만 해칩니다. 실제로 사용하는 곳은 곱셈이 아니라 비트 위치를 만들 때(`1 << i`)입니다.

## 2. 산술 시프트 vs 논리 시프트 — 음수에서 갈라진다

> **개요** 오른쪽 시프트는 두 종류가 있고, 그 차이는 음수를 다룰 때만 드러납니다.

<u>**논리 시프트(logical shift)**</u>는 빈 자리를 항상 0으로 채웁니다. <u>**산술 시프트(arithmetic shift)**</u>는 빈 자리를 **원래의 최상위 비트(부호 비트)**로 채웁니다.

| 값(8비트) | 논리 `>>> 1` | 산술 `>> 1` |
|---|---|---|
| `00001000` (=8) | `00000100` (=4) | `00000100` (=4) |
| `11111000` (=−8) | `01111100` (=124) | `11111100` (=−4) |

> **해석** 양수에서는 두 시프트가 같습니다. 음수에서만 갈라지는데, 산술 시프트는 부호 비트를 복제해 채우므로 "−8을 2로 나누면 −4"라는 산술적 의미를 보존합니다. 논리 시프트는 부호를 무시하고 비트를 순수한 비트 패턴으로 다루므로 −8이 갑자기 124라는 큰 양수가 됩니다. 이 부호 비트 복제를 <u>**부호 확장(sign extension)**</u>이라 하며, 좁은 타입을 넓은 타입으로 변환할 때(`int8` → `int32`)도 같은 원리가 쓰입니다.

| 언어 | 산술 시프트 | 논리 시프트 |
|---|---|---|
| Java | `>>` | `>>>` |
| JavaScript | `>>` | `>>>` |
| C/C++ | 부호 있는 타입에서 `>>` (구현 정의였으나 C++20부터 산술로 규정) | 부호 없는 타입에서 `>>` |
| Python | `>>` (정수가 무한 정밀도라 항상 산술적 의미) | 없음 |

> **팁** JavaScript에서 `x >>> 0`은 값을 부호 없는 32비트 정수로 강제 변환하는 관용구로 자주 쓰입니다. 반대로 배열 인덱스 계산에서 실수로 `>>>`를 쓰면 음수 인덱스가 거대한 양수로 바뀌어 엉뚱한 곳을 참조하게 되므로 주의해야 합니다.

## 3. XOR의 성질 — 되돌릴 수 있는 연산

> **Q.** AND와 OR는 직관적인데 XOR는 어디에 쓰는지 잘 모르겠습니다. 왜 이 연산만 특별 대접을 받나요?
> **A.** XOR만이 **정보를 잃지 않기** 때문입니다. `a & b`의 결과가 0이라고 해서 `a`가 무엇이었는지 알 수 없고, `a | b`가 1이어도 마찬가지입니다. 하지만 `a ^ b`는 `b`를 알면 언제나 `a`를 복원할 수 있습니다. 즉 XOR는 각 인자에 대해 전단사이며, 이 되돌릴 수 있음이 스왑·암호화·오류 검출·중복 찾기의 공통 기반입니다.

XOR의 핵심 성질은 다음 네 가지입니다.

$$a \oplus a = 0, \qquad a \oplus 0 = a, \qquad a \oplus b = b \oplus a, \qquad (a \oplus b) \oplus c = a \oplus (b \oplus c)$$

> **해석** 첫 두 성질이 XOR를 특별하게 만듭니다. $a \oplus a = 0$은 "같은 값을 두 번 XOR하면 사라진다"는 뜻이고, $a \oplus 0 = a$는 0이 항등원이라는 뜻입니다. 뒤의 두 성질(교환·결합)까지 합치면 **XOR 열에서는 순서가 무관하고 짝을 이루는 값이 저절로 소거됩니다**. 19-1에서 다룬 교환·결합법칙이 여기서 다시 등장하는 셈입니다.

```python
# 1) 임시 변수 없는 스왑
a, b = 3, 5
a ^= b      # a = 3^5
b ^= a      # b = 5^(3^5) = 3   (5^5=0 소거)
a ^= b      # a = (3^5)^3 = 5

# 2) 하나만 홀수 번 나오는 값 찾기 — O(n) 시간, O(1) 공간
def find_unique(nums):
    result = 0
    for x in nums:
        result ^= x     # 짝을 이룬 값들은 서로 소거되고 홀수 개인 값만 남습니다
    return result

# 3) 가장 낮은 1비트만 남기기 (2의 보수 성질 이용)
lowest_bit = x & -x
```

> **참고** XOR 스왑은 유명하지만 **실무에서 쓰지 마세요**. `a`와 `b`가 같은 메모리 주소를 가리키면 값이 0으로 파괴되고, 현대 컴파일러는 일반적인 스왑을 이미 레지스터 교환으로 최적화하므로 성능 이득도 없습니다. 반면 `find_unique` 패턴은 실용적이며, RAID 5의 패리티 디스크가 XOR로 데이터를 복구하는 원리도 같습니다 — 여러 디스크의 데이터를 XOR해 저장해두면, 디스크 하나가 고장 나도 나머지를 다시 XOR해 잃은 값을 복원할 수 있습니다.

## 4. 비트마스킹 — 부분집합을 정수 하나로

> **개요** 19-2에서 본 "각 원소마다 포함/미포함"이라는 부분집합의 구조를 그대로 비트에 대응시킵니다.

원소 $n$개의 집합에서 $i$번 원소를 포함하면 $i$번째 비트를 1로 둡니다. 그러면 부분집합 하나가 $0 \le \text{mask} < 2^n$인 정수 하나가 되고, 집합 연산이 비트 연산으로 바뀝니다.

| 집합 연산 | 비트 연산 |
|---|---|
| 원소 $i$ 추가 | `mask \| (1 << i)` |
| 원소 $i$ 제거 | `mask & ~(1 << i)` |
| 원소 $i$ 토글 | `mask ^ (1 << i)` |
| 원소 $i$ 포함 여부 | `(mask >> i) & 1` |
| 합집합 | `a \| b` |
| 교집합 | `a & b` |
| 차집합 | `a & ~b` |
| 공집합 / 전체집합 | `0` / `(1 << n) - 1` |
| 원소 개수 | `bin(mask).count('1')` (C++ `popcount`) |

```python
# 모든 부분집합 순회 — 2^n가지를 빠짐없이 한 번씩
for mask in range(1 << n):
    subset = [items[i] for i in range(n) if (mask >> i) & 1]
    ...

# 가장 낮은 1비트 지우기 — 켜진 비트 개수만큼만 반복
while mask:
    mask &= mask - 1          # Kernighan 기법
```

> **해석** `mask & (mask - 1)`이 가장 낮은 1비트를 지우는 이유는 이렇습니다. `mask - 1`은 가장 낮은 1비트를 0으로 바꾸고 그 아래의 모든 0을 1로 바꿉니다(예: `1011000 - 1 = 1010111`). 원래 값과 AND하면 그 자리와 아래가 전부 사라지고 위쪽만 남습니다. 따라서 이 반복은 전체 비트 수가 아니라 **켜진 비트 수만큼만** 돕니다. 이 기법은 브라이언 커니핸(Brian Kernighan)의 이름으로 알려져 있습니다.

> **팁** 실무에서 비트마스크의 가장 흔한 용도는 <u>**플래그(flag)**</u>입니다. 리눅스의 파일 권한(`rwx` 3비트씩), `open()`의 `O_RDONLY | O_CREAT | O_APPEND`, HTTP 클라이언트의 옵션 플래그가 모두 이 패턴입니다. 값을 2의 거듭제곱으로 정의하면 여러 옵션을 정수 하나에 담고 `flags & O_CREAT`로 한 번에 검사할 수 있습니다.

## 5. 2의 보수 — 음수를 표현하는 최선의 방법

> **Q.** 음수를 표현하는 가장 단순한 방법은 최상위 비트를 부호로 쓰는 것 아닌가요? 왜 굳이 비트를 뒤집고 1을 더하는 복잡한 방식을 쓰나요?
> **A.** 단순한 방식(부호-크기 표현)에는 두 가지 문제가 있습니다. 첫째, `+0`과 `−0`이 따로 존재해 비교가 번거로워집니다. 둘째, 그리고 결정적으로, **덧셈이 그냥 되지 않습니다** — $5 + (-3)$을 계산하려면 부호를 먼저 보고 어느 쪽이 큰지 판단해 뺄셈으로 분기해야 합니다. 2의 보수는 이 두 문제를 동시에 해결합니다. 0의 표현이 하나뿐이고, 부호를 전혀 신경 쓰지 않고 그냥 더하기만 하면 정답이 나옵니다.

<u>**2의 보수(Two's Complement)**</u>에서 음수 $-x$는 다음과 같이 표현됩니다.

$$-x = \lnot x + 1 = 2^n - x \pmod{2^n}$$

![2의 보수 표현과 덧셈으로 처리되는 뺄셈](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part19_이산수학과_비트_연산/19-4_비트_연산과_2의_보수/assets/diagram2_twos_complement.svg)

> **해석** 위 그림의 왼쪽 원은 4비트 정수의 "시계"입니다. 2의 보수의 본질은 **모듈러 산술**입니다 — 4비트 값은 $2^4 = 16$으로 나눈 나머지 세계에 살고 있으므로, $-3$과 $13$은 같은 값입니다($-3 \equiv 13 \pmod{16}$). 시계에서 3시간 뒤로 가는 것과 13시간 앞으로 가는 것이 같은 자리에 도착하는 것과 정확히 같은 이치입니다. 오른쪽은 그 결과를 보여줍니다. $5 - 3$을 계산할 때 뺄셈 회로를 쓰는 대신 $5 + 13 = 18$을 계산하고 4비트를 넘어가는 자리올림을 그냥 버리면 $18 - 16 = 2$가 남습니다. 즉 **넘침을 버리는 것이 곧 나머지 연산**이므로, 하드웨어는 덧셈기 하나만 있으면 됩니다. 실제 덧셈기 회로의 구성은 Part 18을 참고하세요.

$n$비트 2의 보수의 표현 범위는 다음과 같습니다.

$$-2^{n-1} \ \le \ x \ \le \ 2^{n-1} - 1$$

| 타입 | 비트 | 최솟값 | 최댓값 |
|---|---|---|---|
| `int8` | 8 | −128 | 127 |
| `int16` | 16 | −32,768 | 32,767 |
| `int32` | 32 | −2,147,483,648 | 2,147,483,647 |
| `int64` | 64 | 약 $-9.2 \times 10^{18}$ | 약 $9.2 \times 10^{18}$ |

> **참고** 범위가 음수 쪽으로 하나 더 넓은 비대칭이 생기는 이유는 0이 양수 쪽 표현을 하나 차지하기 때문입니다. 이 비대칭은 실제 버그를 만듭니다 — `int32`에서 `abs(-2147483648)`은 표현 가능한 양수 범위를 벗어나 그대로 음수가 반환됩니다. 마찬가지로 `Math.abs(Integer.MIN_VALUE)`가 음수를 반환하는 것은 Java의 문서화된 동작입니다.

## 6. 오버플로 — 소리 없이 값이 뒤집히는 순간

> **개요** 표현 범위를 넘으면 어떤 일이 일어나는지, 그리고 그것이 실제로 어떤 사고로 이어졌는지 봅니다.

<u>**정수 오버플로(Integer Overflow)**</u>는 연산 결과가 표현 범위를 넘어 자리올림이 버려지면서 값이 반대편으로 순환하는 현상입니다.

```java
int max = 2147483647;      // int32 최댓값
System.out.println(max + 1);   // -2147483648  — 최솟값으로 순환

// 이진 탐색의 고전적 오버플로 버그
int mid = (low + high) / 2;        // low+high가 int 범위를 넘으면 음수가 됩니다
int mid = low + (high - low) / 2;  // 안전한 형태
```

> **해석** 위의 이진 탐색 버그는 조슈아 블로크(Joshua Bloch)가 2006년 구글 리서치 블로그 글에서 공개적으로 지적하면서 널리 알려졌습니다. Java 표준 라이브러리의 `Arrays.binarySearch`에도 이 버그가 9년간 존재했으며, 배열 크기가 $2^{30}$을 넘는 상황에서만 드러나기 때문에 오랫동안 발견되지 않았습니다. 오버플로 버그의 전형적 특징이 바로 이것입니다 — 평소에는 완벽히 동작하다가 특정 규모에서만 조용히 잘못된 답을 냅니다.

> **참고** 언어마다 오버플로 처리가 다릅니다. **C/C++** 에서 부호 있는 정수의 오버플로는 <u>**미정의 동작(undefined behavior)**</u>이라 컴파일러가 "오버플로는 일어나지 않는다"고 가정하고 최적화해버릴 수 있습니다. **Java**는 순환(wrap-around)으로 규정합니다. **Rust**는 디버그 빌드에서 패닉을 일으키고 릴리스 빌드에서는 순환하며, `checked_add`·`wrapping_add`처럼 의도를 명시하는 메서드를 제공합니다. **Python**의 정수는 임의 정밀도라 오버플로가 아예 없지만, 그 대가로 연산이 느리고 메모리를 더 씁니다.

> **팁** 실무에서 오버플로를 피하는 방법은 세 가지입니다. (1) 값의 최대 크기를 미리 계산해 충분히 넓은 타입을 고릅니다 — 금액·타임스탬프·누적 카운터는 기본적으로 64비트를 쓰세요. (2) `low + (high-low)/2`처럼 중간 값이 커지지 않도록 식을 변형합니다. (3) 언어가 제공하는 검사 연산(`Math.addExact`, `checked_add`)이나 임의 정밀도 타입(`BigInteger`, `decimal`)을 씁니다. 2038년 문제(32비트 유닉스 타임스탬프가 넘치는 시점)가 바로 (1)을 하지 않아 생긴 대규모 사례입니다.

## 요약

> **핵심**
> - AND·OR·XOR는 각 비트를 **독립적으로** 계산하므로 자리올림이 없고, 64비트 연산 하나가 64개의 판단을 한 사이클에 처리합니다. 시프트는 $2^k$ 곱셈·나눗셈과 같으며, 실제 용도는 곱셈보다 비트 위치 생성(`1 << i`)입니다.
> - **산술 시프트**는 부호 비트를 복제해 채우고(부호 확장), **논리 시프트**는 0으로 채웁니다. 차이는 음수에서만 드러나며, Java·JavaScript는 `>>`와 `>>>`로 구분합니다.
> - **XOR**는 $a \oplus a = 0$, $a \oplus 0 = a$에 교환·결합법칙까지 갖춰 **정보를 잃지 않고 되돌릴 수 있는** 유일한 비트 연산이며, 중복 소거·패리티 복구(RAID 5)의 기반입니다.
> - **비트마스킹**은 부분집합을 정수 하나로 표현합니다. `1 << i`로 원소를 지정하고 `|`·`&`·`^`로 집합 연산을 하며, `mask &= mask - 1`은 켜진 비트 수만큼만 도는 순회 기법입니다.
> - **2의 보수**($-x = \lnot x + 1$)는 본질적으로 $2^n$에 대한 모듈러 산술이라, 자리올림을 버리는 것이 곧 나머지 연산이 됩니다. 덕분에 부호를 신경 쓰지 않고 덧셈기 하나로 뺄셈까지 처리할 수 있고, 0의 표현도 하나뿐입니다.
> - **오버플로**는 평소엔 멀쩡하다가 특정 규모에서만 조용히 잘못된 값을 냅니다. `low + (high-low)/2` 같은 식 변형과 넉넉한 타입 선택이 기본 방어책입니다.
