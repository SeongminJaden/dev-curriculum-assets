# Part 12. 모바일 개발

## 12-1. 네이티브 vs 크로스플랫폼 — 하나의 코드베이스는 무엇을 희생해서 만들어지는가

> **학습 목표**
> - 네이티브 개발(iOS/Android)이 UI를 그리는 방식과, 크로스플랫폼 프레임워크가 이를 대체하는 두 가지 접근(브리지, 자체 렌더링 엔진)을 구조적으로 구분할 수 있다.
> - React Native의 JS-네이티브 브리지와 Flutter의 자체 렌더링 엔진(Skia)이 각각 어떤 방식으로 동작하는지 설명할 수 있다.
> - "하나의 코드베이스"를 선택했을 때 실제로 희생되는 것이 무엇인지, 트레이드오프 관점에서 판단할 수 있다.

> **개요** 같은 앱을 iOS와 Android 두 번 만드는 대신, 코드 하나로 두 플랫폼을 동시에 지원하려는 시도가 크로스플랫폼 개발입니다. 이 강의에서는 네이티브가 화면을 그리는 원리부터 시작해, React Native와 Flutter가 이 원리를 각각 어떻게 우회하는지, 그리고 그 대가로 무엇을 내주는지를 살펴봅니다.

---

## 1. 왜 "하나의 코드베이스"가 필요할까요?

> **Q.** iOS는 Swift(또는 Objective-C)로, Android는 Kotlin(또는 Java)으로 따로 만들면 되는데, 왜 굳이 하나의 코드로 두 플랫폼을 지원하려 할까요?
> **A.** 같은 기능을 두 언어, 두 팀, 두 코드베이스로 각각 구현하고 유지보수하는 비용이 크기 때문입니다. 버튼 하나의 동작을 바꾸려 해도 iOS 팀과 Android 팀이 각각 코드를 수정하고, 각각 테스트하고, 각각 배포해야 합니다. 스타트업처럼 인력이 제한적인 조직에서는 이 이중 비용이 곧 출시 속도의 문제로 이어집니다. 크로스플랫폼 프레임워크는 이 비용을 "코드는 하나, 결과물은 둘"로 줄이려는 시도입니다.

이 질문에 답하기 전에, 먼저 네이티브 앱이 화면을 어떻게 그리는지부터 봐야 합니다. 크로스플랫폼 프레임워크가 "우회"하는 대상이 바로 이 과정이기 때문입니다.

---

## 2. 네이티브는 화면을 어떻게 그리는가

> **개요** iOS와 Android는 각각 자신의 언어로 작성된 코드를 플랫폼 고유의 UI 툴킷을 통해 화면에 그립니다. 이 과정 자체가 크로스플랫폼과 비교되는 기준점이 됩니다.

- <u>**iOS**</u>: Swift(또는 Objective-C)로 작성한 코드는 ARM 네이티브 머신 코드로 컴파일됩니다. 화면은 UIKit(명령형, `UIView` 트리)이나 SwiftUI(선언형)를 통해 구성되며, 최종적으로 Core Animation이 각 뷰를 레이어로 합성해 GPU로 전달합니다.
- <u>**Android**</u>: Kotlin(또는 Java)으로 작성한 코드는 JVM 바이트코드로 컴파일된 뒤, ART(Android Runtime)가 이를 기기에서 실행 가능한 네이티브 코드로 AOT/JIT 변환합니다. 화면은 View 시스템이나 Jetpack Compose를 통해 구성되고, 마찬가지로 시스템 컴포지터가 최종 합성을 담당합니다.

두 경우 모두 <u>**OS가 직접 제공하는 UI 컴포넌트(버튼, 리스트, 텍스트 필드 등)를 그대로 사용**</u>한다는 공통점이 있습니다. 즉 화면에 보이는 버튼은 실제로 iOS의 `UIButton`이거나 Android의 네이티브 위젯입니다. 이 덕분에 폰트 렌더링, 스크롤 관성, 접근성(스크린 리더) 동작이 OS 버전이 올라갈 때마다 자동으로 최신 상태를 따라갑니다.

> **참고** 네이티브 개발의 대가는 코드베이스가 플랫폼 수만큼 늘어난다는 점입니다. 같은 로직(네트워크 통신, 데이터 검증 등)도 Swift와 Kotlin으로 각각 구현해야 합니다.

## 3. 크로스플랫폼의 두 갈래 — 브리지 방식과 자체 렌더링 엔진 방식

> **개요** 크로스플랫폼 프레임워크는 "네이티브 UI 컴포넌트를 그대로 쓰되 로직만 공유"하는 방식과 "네이티브 UI 컴포넌트를 아예 쓰지 않고 직접 그리는" 방식으로 나뉩니다. React Native가 전자, Flutter가 후자의 대표 사례입니다.

![네이티브·React Native·Flutter 렌더링 아키텍처 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part12_모바일_개발/12-1_네이티브_vs_크로스플랫폼/assets/diagram1_rendering_architecture.svg)

### 3.1 React Native — JS-네이티브 브리지

React Native는 비즈니스 로직과 UI 선언을 JavaScript(정확히는 JSX)로 작성합니다. 이 코드는 별도의 JS 엔진(Hermes 또는 JavaScriptCore)에서 실행되며, 실제 화면에 그려지는 것은 <u>**진짜 네이티브 컴포넌트**</u>입니다. `<View>`, `<Text>` 같은 컴포넌트는 최종적으로 iOS의 `UIView`, Android의 네이티브 View로 변환됩니다.

문제는 이 둘을 어떻게 연결하느냐입니다. 초기 아키텍처에서는 JS 스레드와 네이티브 스레드 사이에 <u>**브리지(Bridge)**</u>가 있어, 두 쪽이 주고받는 모든 메시지(레이아웃 값, 이벤트, 함수 호출 인자)를 JSON으로 직렬화해 비동기 큐에 쌓아 전달했습니다. 이 방식은 스레드 간 결합을 느슨하게 만든다는 장점이 있지만, 짧은 시간에 메시지가 몰리면(예: 스크롤이나 애니메이션처럼 프레임마다 값이 바뀌는 상황) 직렬화·역직렬화 비용과 큐잉 지연이 체감되는 끊김으로 이어질 수 있습니다.

> **참고** 최신 React Native(New Architecture)는 JSI(JavaScript Interface)를 통해 JS와 네이티브가 JSON 직렬화 없이 서로의 객체를 직접 참조하고, 필요하면 동기적으로 호출할 수 있도록 개선되었습니다. Fabric(새 렌더러)과 TurboModules(지연 로딩되는 네이티브 모듈)가 이 구조를 뒷받침합니다. 다만 이는 브리지의 병목을 줄이는 개선이지, "레이아웃과 이벤트를 JS와 네이티브가 나눠 처리한다"는 근본 구조 자체를 없애지는 않습니다.

### 3.2 Flutter — 자체 렌더링 엔진

Flutter는 접근 자체가 다릅니다. Dart로 작성한 코드는 개발 중에는 JIT으로, 배포용 빌드에서는 <u>**AOT로 컴파일되어 ARM/x86 네이티브 머신 코드**</u>가 됩니다. 여기까지는 네이티브와 비슷하지만, 결정적 차이는 화면을 그리는 방법입니다.

Flutter는 OS가 제공하는 버튼·리스트 같은 네이티브 위젯을 <u>**전혀 사용하지 않습니다**</u>. 대신 OS로부터 그림을 그릴 수 있는 캔버스(하나의 네이티브 `SurfaceView`/`UIView`)만 받아, 그 위에 모든 버튼·텍스트·아이콘을 Flutter 엔진이 직접 픽셀 단위로 그립니다. 이 렌더링은 2D 그래픽 라이브러리인 Skia(최근에는 Impeller로 전환 중)가 담당하며, iOS든 Android든 동일한 방식으로 동작합니다. Material Design(Android 스타일)과 Cupertino(iOS 스타일) 위젯 세트를 Flutter가 직접 구현해 제공하는 이유가 여기 있습니다 — 진짜 네이티브 위젯이 아니라 그것을 흉내 낸 그림이기 때문입니다.

## 4. 그래서 무엇을 희생하는가

> **개요** "하나의 코드베이스"라는 이득의 대가로 React Native와 Flutter가 각각 무엇을 내주는지 정리합니다.

![코드 공유와 맞바꾸는 것들](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part12_모바일_개발/12-1_네이티브_vs_크로스플랫폼/assets/diagram2_tradeoff.svg)

- <u>**플랫폼 고유의 룩앤필 자동 반영**</u>: 네이티브는 OS 업데이트로 버튼 애니메이션이나 접근성 기능이 바뀌면 자동으로 따라갑니다. React Native는 실제 네이티브 컴포넌트를 쓰므로 이 이득을 상당 부분 유지하지만, Flutter는 위젯을 직접 그리기 때문에 OS가 디자인 언어를 바꾸면 Flutter 팀이 위젯 세트를 다시 그려 업데이트해줄 때까지 기다려야 합니다.
- <u>**접근성 트리와의 자연스러운 통합**</u>: 스크린 리더 같은 보조 기술은 OS의 네이티브 UI 트리를 기준으로 동작합니다. React Native는 실제 네이티브 뷰를 쓰므로 비교적 자연스럽게 연결되지만, Flutter는 캔버스 위에 그린 그림이므로 별도의 시맨틱 트리(semantics tree)를 엔진이 직접 구성해 OS 접근성 API에 맞춰 노출해야 합니다.
- <u>**프레임마다 값이 바뀌는 상호작용의 예측 가능성**</u>: React Native는 JS 스레드와 네이티브 스레드가 분리되어 있어, 복잡한 애니메이션이나 제스처를 다룰 때 두 스레드 간 통신 비용을 신경 써야 합니다(New Architecture로 완화되는 중). Flutter는 UI를 자신의 엔진이 처음부터 끝까지 직접 그리므로 이런 스레드 간 조율 문제 자체가 없지만, 대신 위젯 트리 전체를 자신의 엔진으로 렌더링해야 하는 부담을 집니다.
- <u>**앱 용량**</u>: Flutter는 Skia 렌더링 엔진 자체를 앱에 포함시켜 배포하므로 순정 네이티브 앱보다 기본 용량이 늘어납니다. React Native도 JS 엔진(Hermes)을 함께 배포해야 하므로 마찬가지로 순정 네이티브보다 용량이 늘어납니다.
- <u>**"완전한 코드 공유"라는 전제 자체**</u>: 카메라, 블루투스, 결제 SDK처럼 플랫폼 고유 기능을 다루려면 결국 네이티브 모듈(React Native의 Native Module, Flutter의 Platform Channel)을 Swift/Kotlin으로 따로 작성해야 하는 경우가 많습니다. 즉 "코드베이스 하나"는 UI와 공통 로직 수준에서는 사실이지만, 플랫폼 경계에 걸친 기능에서는 여전히 네이티브 지식이 필요합니다.

### 실생활 예시 — 두 회사가 크로스플랫폼을 만든 이유

> **해석** React Native는 Meta(당시 Facebook)가, Flutter는 Google이 각각 만들고 자사 제품 일부에 실제로 적용하고 있는 프레임워크입니다. 두 회사 모두 "네이티브 대비 무엇을 희생하는가"를 각자의 방식으로 최소화하려 했다는 공통점이 있습니다.

| 프레임워크 | 만든 회사 | 접근 방식 | 무엇을 유지하려 했는가 |
|---|---|---|---|
| React Native | Meta | JS-네이티브 브리지, 실제 네이티브 컴포넌트 사용 | 플랫폼 고유 룩앤필과 접근성 트리 |
| Flutter | Google | 자체 렌더링 엔진(Skia)으로 전체를 직접 그림 | 두 플랫폼에서 완전히 동일한 렌더링 결과와 프레임 성능의 예측 가능성 |

> **팁** 어떤 방식이 "더 낫다"가 아니라 무엇을 더 중요하게 여기느냐의 문제입니다. 플랫폼 고유의 느낌이 중요한 앱(주로 iOS/Android 사용자 경험 규범을 따르는 앱)은 React Native 쪽이, 브랜드 고유의 일관된 디자인을 모든 플랫폼에서 동일하게 유지하고 싶은 앱은 Flutter 쪽이 상대적으로 유리한 경우가 많습니다.

## 요약

> **핵심**
> - 네이티브 앱은 OS가 제공하는 UI 컴포넌트를 직접 사용해 플랫폼 고유의 룩앤필과 접근성을 자동으로 따라가지만, iOS·Android 코드베이스를 각각 유지해야 합니다.
> - React Native는 <u>**JS-네이티브 브리지**</u>를 통해 로직은 JavaScript로, 화면은 실제 네이티브 컴포넌트로 그리는 절충안입니다. 대가는 두 스레드 간 통신 비용입니다.
> - Flutter는 <u>**자체 렌더링 엔진(Skia)**</u>으로 네이티브 위젯을 아예 쓰지 않고 모든 화면을 직접 그립니다. 대가는 OS의 디자인 변화를 자동으로 따라가지 못하고, 접근성 트리를 직접 구성해야 한다는 점입니다.
> - "하나의 코드베이스"는 UI·공통 로직 수준의 절약이며, 플랫폼 고유 기능(카메라, 결제 등)에는 여전히 네이티브 코드가 필요한 경우가 많습니다.
