# Part 12. 모바일 개발

## 12-3. 모바일 성능과 배터리 최적화 — "빠른 것"보다 "적게 쓰는 것"이 우선인 이유

> **학습 목표**
> - 모바일 기기가 서버·데스크톱과 달리 배터리라는 유한한 자원 제약을 갖는다는 점이 설계에 어떤 영향을 주는지 설명할 수 있다.
> - 셀룰러 라디오의 전력 상태 전이와 "테일 타임(tail time)" 개념을 통해 네트워크 요청 배칭이 왜 배터리를 아끼는지 설명할 수 있다.
> - iOS·Android가 백그라운드 작업을 제한하는 구체적 메커니즘(Background Task, Doze 모드, WorkManager)을 구분해 설명할 수 있다.

> **개요** 서버는 전원 콘센트에 꽂혀 있어 "빠르게 처리하는 것"이 곧 좋은 설계입니다. 하지만 모바일 기기는 수천 mAh 남짓한 배터리로 하루를 버텨야 합니다. 이 강의에서는 왜 모바일 환경에서는 처리 속도보다 "에너지를 얼마나 적게 쓰는가"가 우선하는 설계 기준이 되는지, 네트워크 라디오와 OS의 백그라운드 제한 메커니즘을 통해 살펴봅니다.

---

## 1. 왜 배터리가 성능보다 먼저 고려되어야 할까요?

> **Q.** 서버 애플리케이션을 최적화할 때는 "얼마나 빠른가"가 핵심 지표인데, 모바일 앱에서는 왜 "얼마나 적게 쓰는가"가 우선일까요?
> **A.** 서버는 전력이 사실상 무제한으로 공급되고, 발열도 액티브 냉각(팬, 수랭)으로 처리할 수 있습니다. 반면 모바일 기기는 <u>**유한한 배터리 용량**</u>과 <u>**패시브 냉각(별도의 팬 없이 기기 자체로 열을 방출)**</u>이라는 물리적 제약을 동시에 짊어집니다. 처리 속도를 높이려고 CPU 클럭을 계속 끌어올리면 발열이 쌓여 오히려 시스템이 스스로 속도를 낮추는 <u>**서멀 스로틀링(thermal throttling)**</u>이 걸립니다. 게다가 배터리 소모는 사용자가 가장 직접적으로 체감하는 품질 지표이기도 합니다. 화면 반응은 0.1초 느려져도 잘 못 느끼지만, 배터리가 하루 종일 빨리 닳는 앱은 사용자가 곧바로 알아차리고 삭제로 이어집니다.

이 질문에 답하려면 먼저 배터리를 가장 많이 소모하는 두 요소, <u>**네트워크 라디오와 CPU**</u>가 어떻게 전력을 쓰는지부터 봐야 합니다.

## 2. 네트워크 요청이 배터리를 소모하는 방식 — 라디오의 상태 전이

> **개요** 셀룰러 모뎀(라디오)은 데이터를 주고받지 않을 때 자동으로 저전력 상태로 내려가지만, 상태를 전환하는 데도 비용이 들고 전송이 끝난 뒤에도 일정 시간 고전력 상태를 유지합니다. 이 특성 때문에 요청 방식 자체가 배터리 소모량을 좌우합니다.

![라디오 전력 상태 전이](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part12_모바일_개발/12-3_모바일_성능과_배터리_최적화/assets/diagram1_radio_state.svg)

셀룰러 라디오는 대략 다음과 같은 상태를 오갑니다.

1. <u>**Idle(유휴)**</u>: 데이터를 주고받지 않아 전력을 거의 쓰지 않는 상태입니다.
2. <u>**Promotion(승격)**</u>: 앱이 네트워크 요청을 시작하면 라디오가 고전력 상태로 전환됩니다. 이 전환 자체에 시간과 전력이 듭니다.
3. <u>**Full Power(전송)**</u>: 실제 데이터를 주고받는 동안 가장 높은 전력을 사용합니다.
4. <u>**Tail Time(테일 타임)**</u>: 데이터 전송이 끝난 뒤에도 라디오는 곧바로 Idle로 내려가지 않고, 혹시 이어질 다음 요청에 대비해 <u>**일정 시간 고전력 상태를 유지**</u>합니다. 매번 Idle로 내려갔다가 다시 Promotion을 거치는 것보다, 잠시 대기하는 편이 평균적으로 더 효율적이기 때문에 설계된 특성입니다.

> **참고** 문제는 이 테일 타임이 요청 하나하나에 딸려 온다는 점입니다. 작은 요청을 10번 흩어서 보내면 테일 타임도 10번 반복해서 소모되지만, 같은 데이터를 하나로 모아 한 번에 보내면 테일 타임도 한 번만 소모됩니다. <u>**네트워크 요청을 배칭(batching)해야 하는 이유가 바로 여기 있습니다.**</u> 배칭은 단순히 요청 횟수를 줄이는 것이 아니라, 라디오가 고전력 상태를 유지하는 총 시간 자체를 줄이는 조치입니다.

![배칭 전후 라디오 전력 소모 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part12_모바일_개발/12-3_모바일_성능과_배터리_최적화/assets/diagram2_batching_comparison.svg)

### 실생활 예시 — 배터리를 유독 많이 먹는 앱의 공통점

> **해석** 사용자들이 "배터리를 많이 먹는다"고 느끼는 앱은 대개 아래 표의 패턴 중 하나 이상을 가지고 있습니다.

| 배터리 소모가 큰 패턴 | 원인 | 개선 방향 |
|---|---|---|
| 실시간 위치 추적(내비게이션, 배달 기사용 앱) | GPS를 고정밀도로 계속 켜두고, 위치를 서버에 자주 전송 | 이동 중에는 주기를 좁히고, 정지 상태에서는 저전력 위치 모드로 전환 |
| 짧은 주기로 서버 상태를 반복 확인(폴링) | 몇 초~몇 십 초마다 작은 요청을 반복해 라디오 테일 타임이 계속 갱신됨 | 폴링 대신 푸시 알림(FCM/APNs)으로 전환하거나 요청을 배칭 |
| 백그라운드에서 계속 화면을 그리는 애니메이션 | 화면에 보이지 않는데도 GPU가 계속 프레임을 렌더링 | 백그라운드 진입 시 렌더링·타이머 중지 |

## 3. CPU와 백그라운드 작업 — OS가 실행을 강하게 제한하는 이유

> **개요** OS는 화면에 보이지 않는 앱이 CPU와 네트워크를 무제한으로 쓰지 못하도록 여러 겹의 제한 장치를 두고 있습니다.

- <u>**iOS의 Background Task**</u>: 앱이 백그라운드로 전환되면 기본적으로 짧은 시간(수십 초 단위) 안에 진행 중이던 작업을 마무리해야 합니다. 이 시간을 넘기면 시스템이 해당 작업을 강제로 종료합니다. 오디오 재생, 위치 추적, 백그라운드 페치처럼 계속 실행이 필요한 경우에만 별도로 선언된 <u>**백그라운드 모드(Background Modes)**</u>를 통해 예외적으로 더 긴 실행이 허용됩니다.
- <u>**Android의 Doze 모드와 App Standby**</u>: 기기 화면이 꺼진 채 일정 시간 방치되면 Doze 모드가 작동해, 앱의 백그라운드 네트워크 접근과 작업 실행을 주기적인 "유지보수 윈도우"에만 몰아서 허용합니다. 최근에 사용하지 않은 앱은 App Standby Buckets에 따라 실행 빈도가 더 강하게 제한됩니다.
- <u>**WorkManager로 작업을 OS에 위임**</u>: Android는 개별 앱이 각자 타이머로 백그라운드 작업을 직접 실행하는 대신, `WorkManager` 같은 시스템 스케줄러에 "언젠가 조건이 맞을 때 실행해 달라"고 위임하도록 권장합니다. 여러 앱의 작업 요청을 시스템이 한꺼번에 모아 처리하면, 기기 전체의 라디오·CPU 깨우기 횟수 자체를 줄일 수 있습니다.

> **Q.** CPU를 최대 클럭으로 밀어붙여 작업을 최대한 빨리 끝내고 곧바로 저전력 상태로 돌아가는 전략(race-to-idle)은 배터리에 나쁜 전략 아닌가요?
> **A.** 항상 그렇지는 않습니다. 배터리 소모량은 <u>**순간 전력 사용량과 그 상태가 지속되는 시간의 곱(에너지 = 전력 × 시간)**</u>으로 결정됩니다. 짧고 굵게 처리한 뒤 즉시 유휴 상태로 돌아가는 것이, 낮은 클럭으로 오래 붙잡고 있는 것보다 총 에너지 소모가 더 적을 수 있습니다. 문제가 되는 것은 "빠르게 처리하는 것" 자체가 아니라, <u>**불필요하게 자주 깨어나거나, 화면에 보이지도 않는데 계속 CPU·라디오를 붙잡고 있는 패턴**</u>입니다. 즉 최적화의 목표는 처리 속도가 아니라 총 사용 에너지를 최소화하는 것이며, 배칭과 백그라운드 제한은 모두 "깨어 있는 총 시간"을 줄이는 방향의 조치입니다.

## 4. 정리 — 네 가지 실천 원칙

> **원칙**
> - <u>**네트워크 요청은 모아서 보낸다.**</u> 라디오의 테일 타임이 요청 횟수만큼 반복되므로, 여러 개의 작은 요청보다 하나로 배칭된 요청이 총 에너지 소모가 적습니다.
> - <u>**폴링 대신 푸시를 쓴다.**</u> 서버 상태 변화를 짧은 주기로 반복 확인하는 대신, 변화가 있을 때만 서버가 알려주는 푸시 알림을 사용하면 불필요한 라디오 깨우기 자체가 사라집니다.
> - <u>**백그라운드 작업은 OS 스케줄러에 위임한다.**</u> 앱이 직접 타이머를 돌리는 대신 WorkManager나 Background Task API에 조건부 실행을 맡기면, 시스템이 여러 앱의 요청을 묶어 깨우는 횟수 자체를 줄여줍니다.
> - <u>**화면에 보이지 않는 리소스는 즉시 반환한다.**</u> GPS, 카메라, 애니메이션 타이머처럼 켜둔 채로 방치하면 계속 전력을 소모하는 리소스는 백그라운드 전환 시점에 명시적으로 꺼야 합니다.

## 요약

> **핵심**
> - 모바일 기기는 유한한 배터리와 패시브 냉각이라는 물리적 제약 때문에, "빠른 처리"보다 <u>**"총 에너지 소모를 줄이는 것"**</u>이 우선하는 설계 기준이 됩니다.
> - 셀룰러 라디오는 데이터 전송이 끝난 뒤에도 일정 시간 고전력 상태(테일 타임)를 유지하므로, <u>**요청을 배칭하면 테일 타임이 반복되는 횟수 자체를 줄일 수 있습니다.**</u>
> - iOS의 Background Task 시간 제한, Android의 Doze 모드·App Standby·WorkManager는 모두 화면에 보이지 않는 앱이 배터리를 무제한으로 소모하지 못하도록 OS가 거는 장치입니다.
> - 에너지 소모는 전력과 시간의 곱으로 결정되므로, 문제의 본질은 처리 속도가 아니라 <u>**불필요하게 자주 깨어나는 패턴을 줄이는 것**</u>입니다.
