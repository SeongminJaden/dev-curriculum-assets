# Part 12. 모바일 개발

## 12-2. 모바일 앱 생명주기와 상태 관리 — 백그라운드 전환은 왜 특별한가

> **학습 목표**
> - iOS와 Android가 각각 정의하는 앱 생명주기 상태와, 상태가 전환될 때 호출되는 콜백을 설명할 수 있다.
> - 앱이 백그라운드로 전환된 뒤 OS가 프로세스를 종료할 수 있다는 사실이 왜 "특별한 처리"를 요구하는지 설명할 수 있다.
> - 상태를 저장하고 복원하는 실제 메커니즘(iOS 상태 복원, Android `onSaveInstanceState`·`ViewModel`)을 구분해 적용할 수 있다.

> **개요** 웹 브라우저 탭은 닫아도 서버가 그대로 남아 있고, 새로고침하면 다시 데이터를 받아오면 그만입니다. 하지만 모바일 앱은 사용자가 눈치채지 못하는 사이 프로세스 자체가 통째로 사라질 수 있습니다. 이 강의에서는 iOS와 Android의 생명주기 상태를 각각 살펴보고, 왜 "백그라운드로 보내는 것"이 단순히 화면을 숨기는 것 이상의 의미를 갖는지 다룹니다.

---

## 1. 웹과 무엇이 다를까요?

> **Q.** 웹페이지도 탭을 다른 곳으로 전환하면 화면이 안 보이는데, 모바일 앱의 백그라운드 전환은 왜 다르게 취급해야 할까요?
> **A.** 웹은 탭이 안 보여도 브라우저 프로세스와 페이지 상태(대부분)가 그대로 메모리에 남아 있고, 설령 새로고침되어도 서버에서 필요한 데이터를 다시 받아오면 사용자는 큰 차이를 느끼지 못합니다. 반면 모바일 OS는 <u>**메모리가 부족해지면 사용자 눈에 보이지 않는 앱의 프로세스를 예고 없이 강제 종료**</u>합니다. 사용자는 "앱을 그대로 두고 왔다"고 생각하지만, 실제로는 완전히 새 프로세스로 다시 시작되는 경우가 흔합니다. 이 간극을 메우지 못하면 입력 중이던 텍스트, 스크롤 위치, 진행 중이던 화면이 통째로 사라집니다.

이 문제를 이해하려면 먼저 iOS와 Android가 각각 정의하는 생명주기 상태부터 봐야 합니다.

---

## 2. iOS 앱 생명주기

> **개요** iOS는 앱을 5가지 상태(Not Running, Inactive, Active, Background, Suspended)로 구분하고, 상태가 바뀔 때마다 `UIApplicationDelegate`(또는 SwiftUI의 `scenePhase`)로 알립니다.

- <u>**Not Running**</u>: 앱이 실행되지 않았거나, 실행 중이었지만 시스템에 의해 종료된 상태입니다.
- <u>**Inactive**</u>: 앱이 포그라운드에 있지만 이벤트를 받지 못하는 짧은 전이 상태입니다. 예를 들어 전화가 오거나 알림 센터를 내렸을 때 잠깐 이 상태를 거칩니다.
- <u>**Active**</u>: 포그라운드에서 정상적으로 사용자 입력을 받는, 우리가 흔히 "앱을 쓰고 있다"고 생각하는 상태입니다.
- <u>**Background**</u>: 화면에는 보이지 않지만 코드가 실행 중인 상태입니다. iOS는 기본적으로 이 상태에서 실행을 계속할 수 있는 시간을 짧게 제한합니다.
- <u>**Suspended**</u>: 코드는 더 이상 실행되지 않지만, 프로세스와 메모리는 시스템에 남아 있어 다시 포그라운드로 돌아올 때 빠르게 재개할 수 있는 상태입니다.

주요 콜백은 `application(_:didFinishLaunchingWithOptions:)`(최초 실행), `applicationDidBecomeActive(_:)`(Active 진입), `applicationWillResignActive(_:)`(Active 이탈 시작), `applicationDidEnterBackground(_:)`(Background 진입), `applicationWillEnterForeground(_:)`(Background 이탈)입니다. SwiftUI는 이를 `.active`, `.inactive`, `.background` 세 가지 `scenePhase` 값으로 단순화해 제공합니다.

> **참고** `applicationWillTerminate(_:)`는 앱이 포그라운드나 백그라운드 실행 중에 명시적으로 종료될 때만 호출이 보장됩니다. <u>**Suspended 상태에서 시스템이 메모리 확보를 위해 프로세스를 직접 정리하는 경우에는 이 콜백이 호출되지 않습니다.**</u> 그래서 중요한 데이터를 이 콜백에만 의존해 저장하면 안 되고, `applicationDidEnterBackground(_:)` 시점에 미리 저장해두어야 합니다.

## 3. Android 액티비티 생명주기

> **개요** Android는 `Activity`의 생명주기를 `onCreate → onStart → onResume → onPause → onStop → onDestroy` 콜백 순서로 정의하며, 화면 회전 같은 구성 변경(configuration change)과 프로세스 종료를 구분해서 다룹니다.

![iOS/Android 생명주기 콜백 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part12_모바일_개발/12-2_모바일_앱_생명주기와_상태_관리/assets/diagram2_callback_comparison.svg)

- `onCreate()`: 액티비티가 처음 생성될 때 한 번 호출되며, 화면 구성 요소를 초기화합니다.
- `onStart()` → `onResume()`: 화면이 보이기 시작하고(Started), 사용자 입력을 받을 수 있는 상태(Resumed)가 됩니다.
- `onPause()`: 다른 화면이 일부라도 겹쳐 보이면 호출됩니다. 이 시점에는 짧고 가벼운 저장 작업만 해야 합니다.
- `onStop()`: 화면이 완전히 가려지면 호출됩니다. 이 전후로 `onSaveInstanceState(Bundle)`가 호출되어 UI 상태를 임시 저장할 기회를 줍니다.
- `onDestroy()`: 액티비티가 완전히 제거될 때 호출되지만, <u>**시스템이 메모리 부족으로 프로세스를 강제 종료하는 경우에는 호출되지 않을 수 있습니다.**</u>

> **참고** 화면 회전 같은 구성 변경이 일어나면 기본적으로 액티비티가 `onDestroy()` 후 `onCreate()`로 다시 생성됩니다. 이때 `onSaveInstanceState(Bundle)`에 담긴 값은 `onCreate(savedInstanceState)`나 `onRestoreInstanceState`로 복원할 수 있습니다. 다만 이 `Bundle`은 프로세스 간 전달(Binder IPC)을 거치므로 <u>**용량이 크면 안 됩니다.**</u> 이미지 원본이나 대용량 리스트를 통째로 넣는 것이 아니라, 스크롤 위치나 입력 중이던 텍스트처럼 가벼운 값만 담아야 합니다.

### 3.1 화면 회전과 프로세스 종료는 다른 문제입니다

> **Q.** 화면 회전 때문에 생기는 상태 손실과, 백그라운드에서 프로세스가 종료되어 생기는 상태 손실은 같은 문제 아닌가요?
> **A.** 겉보기 증상(화면이 초기 상태로 돌아간 것처럼 보임)은 비슷하지만 원인과 해법은 다릅니다. Jetpack의 `ViewModel`은 구성 변경에서 액티비티가 재생성되어도 살아남도록 설계되어, 화면 회전으로 인한 데이터 재로딩을 막아줍니다. 하지만 `ViewModel`은 <u>**프로세스 자체가 종료되면 함께 사라집니다.**</u> 프로세스 종료 이후까지 살아남으려면 `ViewModel`과 연동되는 `SavedStateHandle`(내부적으로 `onSaveInstanceState`와 같은 `Bundle` 메커니즘을 사용)이나, Room·DataStore 같은 영구 저장소에 데이터를 직접 저장해야 합니다.

## 4. 왜 백그라운드 전환에 특별한 처리가 필요한가

> **개요** 백그라운드 전환이 단순한 "화면 숨김"이 아니라 네 가지 실질적 위험을 동반한다는 점을 정리합니다.

![앱 생명주기 상태 전이](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part12_모바일_개발/12-2_모바일_앱_생명주기와_상태_관리/assets/diagram1_lifecycle_states.svg)

1. <u>**프로세스가 예고 없이 종료될 수 있다**</u>: iOS의 Suspended, Android의 Stopped 상태는 프로세스를 살려두려는 시도이지만, 시스템 메모리가 부족해지면 우선순위가 낮은(화면에 보이지 않는) 프로세스부터 정리됩니다. 저장하지 않은 데이터는 그대로 사라집니다.
2. <u>**백그라운드 실행 시간이 제한된다**</u>: 배터리 소모를 막기 위해 OS는 백그라운드에서 코드가 실행될 수 있는 시간을 짧게 제한합니다. 위치 추적, 오디오 재생처럼 계속 실행이 필요한 작업은 별도로 선언된 백그라운드 모드나 스케줄러(iOS의 Background Task, Android의 `WorkManager`)를 통해서만 예외적으로 허용됩니다.
3. <u>**민감한 정보가 화면 전환기에 노출될 수 있다**</u>: 앱을 백그라운드로 보내는 순간 OS는 마지막 화면을 스크린샷으로 캡처해 앱 전환 화면에 보여줍니다. 은행 앱이나 인증 화면이 백그라운드 진입 시 화면을 흐리게 처리하는 것은 이 캡처를 막기 위한 조치입니다.
4. <u>**점유하고 있던 리소스를 반환해야 한다**</u>: 카메라, GPS, 네트워크 소켓처럼 배터리를 많이 쓰는 리소스를 화면이 보이지 않는데도 계속 붙잡고 있으면 불필요한 배터리 소모로 이어집니다. 백그라운드 진입 시점은 이런 리소스를 명시적으로 해제해야 하는 신호이기도 합니다.

### 실생활 예시 — 지도 앱이 몇 분 뒤 돌아왔을 때

> **해석** 아래 표는 지도 앱을 켜서 경로를 검색한 뒤 다른 앱을 몇 분간 쓰다가 돌아왔을 때, 상태 저장 처리가 되어 있는 경우와 아닌 경우를 비교합니다.

| 상황 | 상태 저장을 하지 않은 경우 | 상태 저장을 제대로 한 경우 |
|---|---|---|
| 검색창에 입력하던 목적지 | 사라지고 빈 검색창으로 초기화 | 입력하던 텍스트 그대로 복원 |
| 이미 계산된 경로 | 처음부터 다시 검색해야 함 | 계산된 경로와 지도 위치 그대로 복원 |
| 진행 중이던 내비게이션 | 내비게이션이 끊기고 처음 화면으로 이동 | 마지막 안내 지점부터 자연스럽게 재개 |

> **팁** 이 차이는 대부분 "프로세스가 실제로 종료되었는가"에서 갈립니다. 잠깐 다른 앱을 켰다 오는 정도로는 대개 프로세스가 살아 있어 문제가 드러나지 않습니다. 하지만 메모리를 많이 쓰는 다른 앱(카메라, 게임)을 거쳐 오면 iOS/Android가 지도 앱 프로세스를 정리했을 가능성이 커지고, 이때 상태 저장 로직의 유무가 사용자 경험을 완전히 갈라놓습니다.

## 요약

> **핵심**
> - iOS는 Not Running·Inactive·Active·Background·Suspended 5가지 상태를, Android는 `onCreate`~`onDestroy` 콜백 순서로 생명주기를 관리합니다.
> - <u>**백그라운드로 전환된 앱의 프로세스는 OS 판단에 따라 예고 없이 종료될 수 있으며**</u>, `applicationWillTerminate`나 `onDestroy`처럼 "종료 직전"에 호출될 것 같은 콜백조차 호출이 보장되지 않습니다.
> - 그래서 상태 저장은 종료 시점이 아니라 <u>**백그라운드로 전환되는 시점(iOS의 `applicationDidEnterBackground`, Android의 `onSaveInstanceState`/`onStop`)에 선제적으로 이루어져야 합니다.**</u>
> - 화면 회전으로 인한 상태 손실(구성 변경)과 프로세스 종료로 인한 상태 손실은 원인이 다르며, Android는 이를 `ViewModel`(구성 변경 대응)과 `SavedStateHandle`·영구 저장소(프로세스 종료 대응)로 구분해 해결합니다.
