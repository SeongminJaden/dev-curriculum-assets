# Part 15. 자료구조 심화

## 15-3. 힙과 우선순위 큐 — 트리인데 왜 배열에 담을까

> **학습 목표**
> - 완전 이진 트리의 성질을 이용해 힙을 포인터 없이 배열로 표현하고, 부모·자식 인덱스를 수식으로 계산할 수 있다.
> - sift-up과 sift-down의 동작을 추적하며 삽입·삭제가 $O(\log n)$인 이유를 설명할 수 있다.
> - heapify가 $n \log n$이 아니라 $O(n)$인 이유를 레벨별 비용 합으로 증명하고, 우선순위 큐가 필요한 문제를 식별할 수 있다.

> **개요** "가장 작은 값을 계속 꺼내야 한다"는 요구는 놀랄 만큼 자주 등장합니다. 매번 전체를 훑으면 $O(n)$, 매번 정렬하면 $O(n \log n)$입니다. 힙은 이 문제를 $O(\log n)$으로 낮추는데, 그 비결은 **완전한 정렬을 포기하고 최솟값 하나만 정확히 아는 것**입니다. 이 강의는 그 절충이 어떻게 배열 하나로 구현되는지 봅니다.

---

## 1. 우선순위 큐 — 힙보다 먼저 이해해야 할 개념

> **Q.** 힙과 우선순위 큐는 같은 말인가요?
> **A.** 아닙니다. <u>**우선순위 큐(Priority Queue)**</u>는 "가장 우선순위가 높은 원소를 꺼낸다"는 **인터페이스(추상 자료형)**이고, <u>**힙(Heap)**</u>은 그 인터페이스를 효율적으로 구현하는 **자료구조**입니다. 우선순위 큐를 정렬된 배열이나 균형 트리로 구현할 수도 있습니다. 다만 힙이 가장 널리 쓰이는 이유는 삽입과 삭제가 모두 $O(\log n)$이면서 추가 포인터 메모리가 전혀 필요 없기 때문입니다.

| 구현 방식 | 삽입 | 최솟값 삭제 | 최솟값 조회 | 추가 메모리 |
|---|---|---|---|---|
| 정렬 안 된 배열 | $O(1)$ | $O(n)$ | $O(n)$ | 없음 |
| 정렬된 배열 | $O(n)$ | $O(1)$ | $O(1)$ | 없음 |
| 균형 이진 탐색 트리 | $O(\log n)$ | $O(\log n)$ | $O(\log n)$ | 노드당 포인터 2~3개 |
| **힙** | $O(\log n)$ | $O(\log n)$ | $O(1)$ | **없음(배열 그대로)** |

> **해석** 정렬 안 된 배열은 넣기만 싸고 꺼내기가 비싸며, 정렬된 배열은 정반대입니다. 힙은 두 연산을 모두 $O(\log n)$으로 균형 있게 맞춘 절충안입니다. 균형 트리도 같은 복잡도를 내지만 노드마다 포인터를 저장해야 하고 캐시 지역성이 나쁩니다. 힙은 15-1에서 본 배열의 연속 배치를 그대로 쓰므로 상수 계수에서 크게 앞섭니다.

## 2. 힙 성질 — 정렬보다 약한, 그러나 충분한 조건

<u>**최소 힙(Min-Heap)**</u>은 다음 <u>**힙 성질(heap property)**</u>을 만족하는 완전 이진 트리입니다.

$$\forall v \neq \text{root} : \text{key}(\text{parent}(v)) \leq \text{key}(v)$$

> **참고** 여기서 조건이 걸린 것은 **부모와 자식 사이뿐**입니다. 형제끼리는 아무 순서 관계가 없고, 왼쪽 서브트리와 오른쪽 서브트리 사이에도 관계가 없습니다. BST의 불변식이 트리 전체에 전역 순서를 강제하는 것과 대조적입니다. 그래서 힙의 중위 순회는 정렬 상태가 아니며, 힙에서 "$k$번째로 작은 값"이나 "특정 값의 존재 여부"를 묻는 건 $O(n)$입니다. 힙이 값싸게 알려주는 건 오직 **최솟값 하나**입니다. 이 "덜 알기 때문에 더 싸다"는 절충이 힙 설계의 핵심입니다.

<u>**완전 이진 트리(Complete Binary Tree)**</u>는 마지막 레벨을 제외한 모든 레벨이 꽉 차 있고, 마지막 레벨은 왼쪽부터 빈틈없이 채워진 트리입니다. 이 조건은 두 가지 결과를 보증합니다. 첫째, 노드가 $n$개일 때 높이가 항상 정확히 $\lfloor \log_2 n \rfloor$입니다. 15-2의 BST와 달리 **편향될 수 없습니다**. 둘째, 빈틈이 없으므로 레벨 순서로 번호를 매기면 배열의 연속 인덱스와 정확히 일치합니다.

## 3. 배열 표현 — 포인터를 없애는 인덱스 산술

![힙의 완전 이진 트리와 배열 표현의 대응](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part15_자료구조_심화/15-3_힙과_우선순위_큐/assets/diagram1_heap_array.svg)

0-기반 인덱스에서 노드 $i$의 관계는 다음과 같습니다.

$$\text{parent}(i) = \left\lfloor \frac{i-1}{2} \right\rfloor, \quad \text{left}(i) = 2i+1, \quad \text{right}(i) = 2i+2$$

> **해석** 위 그림에서 트리의 각 노드 아래에 적힌 번호가 그대로 배열 인덱스입니다. 인덱스 1의 자식은 3과 4, 인덱스 4의 부모는 $\lfloor 3/2 \rfloor = 1$로 정확히 맞아떨어집니다. 이 대응이 성립하는 이유는 완전 이진 트리에 빈칸이 없기 때문입니다. 만약 중간에 빈 노드가 있으면 번호가 밀려 식이 깨집니다. 15-1에서 연결 리스트의 대가로 지적한 포인터 오버헤드와 캐시 미스를, 힙은 이 산술식 하나로 완전히 제거합니다. 노드 100만 개짜리 힙도 포인터를 단 하나도 저장하지 않습니다.

> **팁** 곱하기 2와 나누기 2는 비트 시프트(`i << 1`, `i >> 1`)로 처리되므로 사실상 비용이 없습니다. 일부 구현이 1-기반 인덱싱을 쓰는 이유도 이것입니다. 1-기반에서는 $\text{left}(i) = 2i$, $\text{parent}(i) = i/2$로 식이 더 깔끔해집니다.

## 4. sift-up과 sift-down — 힙을 고치는 두 가지 손짓

> **개요** 힙의 모든 연산은 결국 두 가지 복구 동작으로 환원됩니다. 하나는 너무 작은 값을 위로 올리는 것, 다른 하나는 너무 큰 값을 아래로 내리는 것입니다.

**삽입**은 새 값을 배열 맨 뒤(= 트리의 마지막 자리)에 붙이는 것으로 시작합니다. 완전 이진 트리 조건은 이것으로 이미 만족됩니다. 하지만 힙 성질은 깨졌을 수 있으므로, 새 값이 부모보다 작으면 부모와 자리를 바꾸며 위로 올라갑니다. 이것이 <u>**sift-up(상향 이동)**</u>입니다.

**최솟값 삭제**는 조금 더 영리합니다. 루트를 그냥 빼면 트리에 구멍이 생기므로, **배열의 마지막 원소를 루트로 옮기고** 배열 크기를 하나 줄입니다. 완전성은 유지됐지만 이제 루트에 큰 값이 앉아 있으니, 두 자식 중 더 작은 쪽과 비교해 자리를 바꾸며 내려갑니다. 이것이 <u>**sift-down(하향 이동)**</u>입니다.

```python
def sift_up(heap, i):
    while i > 0:
        p = (i - 1) // 2
        if heap[p] <= heap[i]: break      # 부모가 더 작으면 완료
        heap[p], heap[i] = heap[i], heap[p]
        i = p

def sift_down(heap, i, n):
    while True:
        smallest, l, r = i, 2*i + 1, 2*i + 2
        if l < n and heap[l] < heap[smallest]: smallest = l
        if r < n and heap[r] < heap[smallest]: smallest = r
        if smallest == i: break            # 자식이 모두 크면 완료
        heap[i], heap[smallest] = heap[smallest], heap[i]
        i = smallest
```

> **해석** 두 함수 모두 한 번의 반복마다 정확히 한 레벨씩 이동하며, 각 레벨에서 하는 일은 비교 1~2회와 교환 1회로 상수입니다. 따라서 최악의 경우 비용은 트리의 높이와 같고, 완전 이진 트리의 높이는 $\lfloor \log_2 n \rfloor$이므로 삽입과 삭제 모두 $O(\log n)$입니다. 여기서 **완전 이진 트리라는 조건이 복잡도 보장의 근거**임을 알 수 있습니다. 15-2의 BST는 편향되면 $O(n)$으로 무너졌지만, 힙은 구조상 편향될 수가 없습니다.

## 5. heapify가 $O(n)$인 이유

> **Q.** 정렬 안 된 배열 $n$개를 힙으로 만들려면, 원소를 하나씩 삽입하면 $O(n \log n)$입니다. 그런데 왜 heapify는 $O(n)$이라고 할까요?
> **A.** 삽입 방식과 반대 방향으로 진행하기 때문입니다. <u>**heapify**</u>는 배열의 **뒤쪽 절반(잎 노드들)은 이미 각자 크기 1의 유효한 힙**이라는 사실에서 출발해, 마지막 내부 노드부터 루트까지 거꾸로 올라오며 sift-down만 수행합니다. 핵심은 **대부분의 노드가 아래쪽에 몰려 있고, 아래쪽 노드일수록 내려갈 거리가 짧다**는 점입니다.

높이 $h$인 노드는 최대 $h$번 내려가고, 그런 노드는 최대 $\lceil n / 2^{h+1} \rceil$개 있습니다. 총 비용은 다음과 같습니다.

$$T(n) = \sum_{h=0}^{\lfloor \log_2 n \rfloor} \left\lceil \frac{n}{2^{h+1}} \right\rceil \cdot O(h) \;\leq\; O\left(n \sum_{h=0}^{\infty} \frac{h}{2^{h+1}}\right) = O(n \cdot 1) = O(n)$$

> **해석** 결론을 만드는 것은 $\sum_{h=0}^{\infty} h/2^h = 2$라는 수렴하는 급수입니다. 직관적으로 풀면 이렇습니다. 전체 노드의 절반은 잎이라 이동 거리가 0이고, 4분의 1은 최대 1칸, 8분의 1은 최대 2칸 내려갑니다. 노드 수는 절반씩 줄어드는데 이동 거리는 1씩만 늘어나므로, 곱한 총합이 $n$의 상수배로 수렴합니다. 반대로 삽입을 $n$번 반복하는 방식은 **루트에서 시작해 아래로 넓어지는** 방향이라, 노드가 많은 아래쪽에서 이동 거리도 길어져 $O(n \log n)$이 됩니다. 같은 결과를 만드는 두 절차의 비용이 방향 하나로 갈리는 것입니다.

> **참고** 이 heapify를 정렬에 활용한 것이 1964년 J. W. J. 윌리엄스(J. W. J. Williams)가 발표한 <u>**힙 정렬(Heapsort)**</u>입니다. 배열을 $O(n)$에 힙으로 만든 뒤, 루트(최댓값)를 맨 뒤로 보내는 작업을 $n$번 반복해 $O(n \log n)$에 정렬합니다. 추가 메모리 없이 제자리에서 최악에도 $O(n \log n)$을 보장하는 몇 안 되는 정렬입니다.

## 6. 어디에 쓰이는가

![우선순위 큐의 실무 활용](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part15_자료구조_심화/15-3_힙과_우선순위_큐/assets/diagram2_pq_usage.svg)

> **해석** 위 사례들의 공통 구조는 "매 단계에서 아직 처리하지 않은 것 중 가장 우선순위가 높은 하나를 꺼낸다"입니다. 다익스트라 최단 경로 알고리즘은 아직 확정되지 않은 정점 중 거리가 가장 짧은 것을 꺼내는데, 이 부분을 배열 선형 탐색으로 하면 $O(V^2)$, 힙으로 하면 $O((V+E)\log V)$가 됩니다. OS 스케줄러는 실행 대기 중인 프로세스 중 우선순위가 가장 높은 것을 고릅니다. 이벤트 시뮬레이션은 앞으로 일어날 이벤트 중 시각이 가장 이른 것을 꺼냅니다. 상위 $k$개 추출은 크기 $k$짜리 힙을 유지하며 전체를 한 번 훑어, 전체 정렬($O(n \log n)$) 없이 $O(n \log k)$에 끝냅니다.

> **참고** Python의 `heapq`는 최소 힙만 제공하며, 리스트를 그대로 힙으로 다룹니다(별도 타입이 없습니다). 최대 힙이 필요하면 값의 부호를 뒤집어 넣는 것이 관용적인 방법입니다. Java의 `PriorityQueue`는 기본이 최소 힙이고 `Comparator`로 순서를 바꿉니다. 두 구현 모두 내부는 배열 기반 이진 힙입니다.

> **팁** 힙은 임의 원소의 우선순위 변경(decrease-key)을 값싸게 지원하지 않습니다. 해당 원소가 배열 어디에 있는지 찾는 데만 $O(n)$이 걸리기 때문입니다. 실무에서는 보통 갱신된 값을 새로 밀어 넣고, 꺼낼 때 이미 처리된 항목이면 버리는 <u>**lazy deletion**</u> 기법을 씁니다. 다익스트라 구현에서 자주 보이는 패턴입니다.

## 요약

> **핵심**
> - 우선순위 큐는 인터페이스이고 힙은 그 구현체입니다. 힙이 표준으로 자리 잡은 이유는 삽입·삭제 $O(\log n)$을 **추가 포인터 메모리 없이** 배열 하나로 달성하기 때문입니다.
> - 힙 성질은 부모-자식 사이에만 걸리는 약한 조건이라 전체 정렬 정보를 주지 않습니다. 대신 최솟값 하나를 $O(1)$에 알려주며, 이 "덜 알기에 더 싸다"가 힙의 설계 철학입니다.
> - 완전 이진 트리라는 조건이 배열 표현($\text{left}(i)=2i+1$, $\text{parent}(i)=\lfloor (i-1)/2 \rfloor$)과 높이 $\lfloor \log_2 n \rfloor$ 보장을 동시에 만들어냅니다. 힙은 BST와 달리 **구조상 편향될 수 없습니다**.
> - heapify가 $O(n)$인 것은 아래에서 위로 진행하기 때문입니다. 노드 수가 많은 아래쪽일수록 이동 거리가 짧아, 총합이 $\sum h/2^h = 2$로 수렴합니다. 방향만 뒤집으면 같은 결과가 $O(n \log n)$이 됩니다.
