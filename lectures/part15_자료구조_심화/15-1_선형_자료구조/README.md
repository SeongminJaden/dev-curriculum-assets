# Part 15. 자료구조 심화

## 15-1. 선형 자료구조 — 배열과 연결 리스트는 메모리에서 무엇이 다른가

> **학습 목표**
> - 배열과 연결 리스트의 메모리 배치 차이를 설명하고, 그 차이가 각 연산 비용으로 이어지는 과정을 도출할 수 있다.
> - 동적 배열의 증폭(amortized) 분석을 통해 `append`가 왜 평균 $O(1)$인지 계산할 수 있다.
> - 스택·큐·덱을 배열 기반과 연결 리스트 기반으로 각각 구현할 때의 트레이드오프를 근거로 구조를 선택할 수 있다.

> **개요** 1-3에서 배열은 접근 $O(1)$·삽입 $O(n)$, 연결 리스트는 그 반대라는 표를 봤습니다. 이번 강의는 그 표를 반복하지 않고, **왜 그런 숫자가 나오는지**를 메모리 수준에서 봅니다. 주소 계산식, 포인터 한 칸의 비용, 재할당이 일어나는 순간, 그리고 CPU 캐시가 이 모든 것에 어떻게 개입하는지가 이 강의의 주제입니다.

---

## 1. 배열은 왜 인덱스 접근이 공짜에 가까울까요?

> **Q.** "배열은 인덱스로 바로 접근한다"는 말을 자주 듣는데, 컴퓨터는 도대체 어떻게 100만 번째 칸의 위치를 즉시 알아낼까요?
> **A.** 계산하기 때문입니다. 배열은 같은 크기의 원소를 메모리에 **빈틈없이 연속**으로 배치합니다. 그래서 배열의 시작 주소와 원소 하나의 크기만 알면, $i$번째 원소의 주소는 곱셈과 덧셈 한 번으로 구할 수 있습니다. 탐색이 아니라 산술 연산이므로 $i$가 0이든 100만이든 비용이 같습니다.

<u>**배열(Array)**</u>의 원소 주소는 다음 식으로 결정됩니다.

$$\text{addr}(A[i]) = \text{base} + i \times \text{sizeof(element)}$$

> **해석** 이 식에 $i$가 곱셈 항으로만 등장한다는 점이 핵심입니다. 어떤 $i$를 넣어도 연산 횟수는 곱셈 1회 + 덧셈 1회로 고정되므로 $O(1)$입니다. 동시에 이 식은 배열의 제약도 설명합니다. 원소 크기가 모두 같아야 하고(그래야 곱셈이 성립), 메모리가 연속이어야 합니다(그래야 base 기준 계산이 성립). 배열 중간에 값을 끼워 넣으면 이 연속성이 깨지므로, 뒤 원소를 전부 한 칸씩 밀어 연속성을 복구해야 합니다. 그 밀기 작업이 $O(n)$의 정체입니다.

> **참고** 연속 배치에는 복잡도 표에 드러나지 않는 보너스가 있습니다. CPU는 메모리를 한 바이트씩이 아니라 <u>**캐시 라인(cache line)**</u> 단위(보통 64바이트)로 읽어옵니다. 배열을 순회하면 한 번 읽을 때 뒤따르는 원소들이 함께 캐시에 올라오므로 이후 접근이 거의 공짜가 됩니다. 이것을 <u>**참조 지역성(locality of reference)**</u>이라 하며, 같은 $O(n)$ 순회라도 배열이 연결 리스트보다 실측에서 몇 배 빠른 이유입니다.

## 2. 동적 배열 — 크기가 고정인 배열이 어떻게 늘어날까요?

> **개요** Python의 `list`, Java의 `ArrayList`, C++의 `std::vector`는 모두 크기가 자유롭게 늘어납니다. 하지만 그 안에 있는 것은 여전히 고정 크기 배열입니다. 이 모순이 어떻게 해소되는지 봅니다.

<u>**동적 배열(Dynamic Array)**</u>은 실제 원소 수(`size`)보다 큰 용량(`capacity`)의 배열을 미리 잡아두고, 용량이 차면 **더 큰 배열을 새로 할당해 전부 복사**한 뒤 기존 배열을 버립니다. 이 복사는 명백히 $O(n)$입니다. 그런데도 `append`를 $O(1)$이라 부르는 이유는 재할당이 **매번** 일어나지 않기 때문입니다.

용량을 2배씩 늘린다고 하면, $n$번 추가하는 동안 발생하는 총 복사 횟수는 다음과 같습니다.

$$1 + 2 + 4 + \cdots + \frac{n}{2} = \sum_{k=0}^{\log_2 n - 1} 2^k = n - 1 < n$$

> **해석** $n$번의 추가를 하는 동안 복사는 총 $n$회 미만입니다. 따라서 추가 1회당 평균 복사 비용은 $\frac{n}{n} = O(1)$입니다. 이렇게 "개별 연산은 가끔 비싸지만 긴 구간의 평균은 싸다"를 증명하는 기법을 <u>**분할 상환 분석(amortized analysis)**</u>이라 하며, 1985년 로버트 타잔(Robert Tarjan)이 논문 *Amortized Computational Complexity*에서 체계화했습니다. 중요한 건 **2배씩** 늘려야 이 결론이 나온다는 점입니다. 만약 매번 1칸씩만 늘리면 총 복사가 $1+2+\cdots+n = O(n^2)$이 되어 추가 1회당 $O(n)$으로 무너집니다.

> **참고** 실제 성장 배수는 구현마다 다릅니다. Java `ArrayList`는 1.5배, C++ `std::vector`는 대개 2배, CPython의 `list`는 약 1.125배에 상수를 더하는 완만한 곡선을 씁니다. 배수가 크면 복사는 줄지만 메모리 낭비가 늘어나는 트레이드오프입니다.

> **팁** 넣을 원소 개수를 미리 안다면 용량을 먼저 확보하세요(`ArrayList<>(10000)`, `vector::reserve`, C#의 `List.Capacity`). 재할당과 복사를 통째로 없앨 수 있습니다.

## 3. 연결 리스트 — 포인터로 산 것과 잃은 것

![배열과 연결 리스트의 메모리 배치 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part15_자료구조_심화/15-1_선형_자료구조/assets/diagram1_memory_layout.svg)

> **해석** 위 그림에서 배열은 하나의 연속된 메모리 블록을 차지하는 반면, 연결 리스트의 노드들은 힙 여기저기에 흩어져 있고 오직 포인터로만 이어져 있습니다. 이 배치 차이가 모든 것을 결정합니다. 연결 리스트는 원소를 끼워 넣을 때 **아무것도 밀지 않아도** 되므로(포인터 두 개만 고쳐 쓰면 됩니다) 삽입이 $O(1)$입니다. 대신 $i$번째 노드의 주소를 계산할 방법이 없어 첫 노드부터 링크를 $i$번 따라가야 하므로 접근이 $O(n)$입니다. 게다가 각 노드가 흩어져 있어 순회할 때마다 캐시 미스가 발생합니다.

<u>**단일 연결 리스트(Singly Linked List)**</u>는 노드마다 다음 노드 포인터 하나를 갖습니다. 뒤로만 갈 수 있으므로 어떤 노드를 삭제하려면 그 **앞 노드**를 알아야 합니다. <u>**이중 연결 리스트(Doubly Linked List)**</u>는 이전 노드 포인터를 추가로 두어, 노드 하나만 손에 쥐고도 $O(1)$에 삭제할 수 있게 합니다. <u>**원형 연결 리스트(Circular Linked List)**</u>는 마지막 노드가 첫 노드를 가리켜 끝이 없으며, 라운드 로빈 스케줄러나 순환 버퍼처럼 "끝나면 처음으로" 성질이 자연스러운 문제에 씁니다.

| 구분 | 노드당 포인터 | 임의 노드 삭제 | 역방향 순회 | 노드 오버헤드 |
|---|---|---|---|---|
| 단일 연결 리스트 | 1개 | $O(n)$ (앞 노드 탐색 필요) | 불가 | 작음 |
| 이중 연결 리스트 | 2개 | $O(1)$ | 가능 | 큼(포인터 1개분 추가) |
| 원형 연결 리스트 | 1 또는 2개 | 구현 따름 | 구현 따름 | 단일/이중과 동일 |

> **참고** 포인터의 비용은 무시할 수 없습니다. 64비트 환경에서 4바이트 정수 하나를 이중 연결 리스트에 담으면 포인터 2개(16바이트) + 정렬 패딩까지 붙어, 실제 데이터의 **5배 이상** 메모리를 씁니다. 같은 정수를 배열에 담으면 4바이트입니다. "삽입이 $O(1)$"이라는 이점만 보고 연결 리스트를 고르면 메모리와 캐시에서 크게 손해 볼 수 있습니다.

> **팁** 실무에서 순수 연결 리스트를 직접 쓸 일은 생각보다 드뭅니다. 다만 LRU 캐시(이중 연결 리스트 + 해시 맵), OS의 프로세스 대기열, 자바 `LinkedHashMap`의 순서 유지처럼 **"임의 위치의 원소를 $O(1)$에 떼어내 다른 곳으로 옮겨야 하는"** 요구가 있을 때는 대체 불가능한 선택지입니다.

## 4. 스택과 큐 — 제약이 만들어내는 힘

> **Q.** 배열이 있으면 아무 위치나 다룰 수 있는데, 굳이 "한쪽 끝에서만 넣고 뺀다"고 스스로를 제한하는 스택이 왜 필요할까요?
> **A.** 제약이 곧 보증이기 때문입니다. 스택으로 선언한 자료는 중간이 임의로 건드려지지 않는다는 것이 타입 수준에서 보장되므로, 코드를 읽는 사람도 컴파일러도 그 사실에 의존할 수 있습니다. 또 연산이 양 끝으로 한정되면 구현이 항상 $O(1)$로 떨어지고 자료구조가 단순해집니다.

<u>**스택(Stack)**</u>은 <u>**LIFO(Last In First Out)**</u> 규칙을 따릅니다. `push`와 `pop`이 모두 같은 끝에서 일어나므로 동적 배열의 맨 뒤를 쓰면 두 연산 모두 분할 상환 $O(1)$입니다. 함수 호출 스택, 수식의 괄호 검사, 깊이 우선 탐색의 되추적, 편집기의 실행 취소가 모두 스택입니다.

<u>**큐(Queue)**</u>는 <u>**FIFO(First In First Out)**</u> 규칙입니다. 여기서 초보자가 자주 밟는 지뢰가 있습니다. 배열의 0번 인덱스에서 `dequeue`하면 뒤 원소를 전부 앞으로 당겨야 해 $O(n)$이 됩니다.

```python
# 잘못된 큐: 매 dequeue마다 전체 원소를 앞으로 당겨 O(n)
q = []
q.append(x)     # O(1)
q.pop(0)        # O(n)  ← 함정

# 올바른 큐: 양 끝 O(1)이 보장되는 덱 사용
from collections import deque
q = deque()
q.append(x)     # O(1)
q.popleft()     # O(1)
```

이 문제의 정석 해법이 <u>**원형 버퍼(Circular Buffer)**</u>입니다. 고정 크기 배열에 `front`와 `rear` 두 인덱스를 두고, 끝에 닿으면 나머지 연산으로 처음으로 되감습니다.

$$\text{rear}_{\text{new}} = (\text{rear} + 1) \bmod \text{capacity}$$

> **해석** 이 한 줄의 나머지 연산 덕분에 원소를 단 하나도 옮기지 않고 배열을 무한히 재활용할 수 있습니다. 대신 "가득 참"과 "비었음"이 `front == rear`로 똑같이 보이는 문제가 생겨, 보통 한 칸을 비워두거나 원소 개수를 따로 세어 구분합니다.

<u>**덱(Deque, Double-Ended Queue)**</u>은 양쪽 끝 모두에서 삽입·삭제를 허용해 스택과 큐를 모두 대체합니다. Python의 `collections.deque`는 고정 크기 블록들의 이중 연결 리스트로 구현되어, 연결 리스트의 양 끝 $O(1)$과 배열의 캐시 지역성을 절충합니다.

## 5. 그래서 무엇을 언제 쓰는가

![선형 자료구조 선택 기준](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part15_자료구조_심화/15-1_선형_자료구조/assets/diagram2_structure_choice.svg)

> **해석** 위 결정 흐름은 "무엇이 빠른가"가 아니라 **"어떤 연산을 가장 자주 하는가"**에서 출발합니다. 인덱스 접근과 순회가 지배적이면 거의 항상 동적 배열이 정답이며, 이것이 대부분의 언어에서 기본 리스트 타입이 동적 배열인 이유입니다. 양 끝에서만 다룬다면 덱, 중간 임의 위치를 자주 떼었다 붙였다 해야 하고 그 위치를 이미 손에 쥐고 있다면 이중 연결 리스트로 갑니다.

| 요구 사항 | 권장 구조 | 근거 |
|---|---|---|
| 인덱스 접근·전체 순회가 지배적 | 동적 배열 | 주소 계산 $O(1)$, 캐시 지역성 |
| 뒤에서만 넣고 뺌 | 동적 배열(스택) | 분할 상환 $O(1)$, 추가 메모리 없음 |
| 앞뒤 양 끝에서 넣고 뺌 | 덱 / 원형 버퍼 | 양 끝 $O(1)$, 원소 이동 없음 |
| 이미 아는 위치를 자주 떼어 옮김 | 이중 연결 리스트 | 포인터 재배선만으로 $O(1)$ |
| 원소 개수 상한이 정해짐 | 고정 배열 / 원형 버퍼 | 재할당 없음, 예측 가능한 지연 |

> **팁** 확신이 안 서면 동적 배열로 시작하세요. 현대 하드웨어에서 캐시 지역성의 이득은 복잡도 표가 보여주는 것보다 훨씬 크며, 원소 수천 개 수준에서는 $O(n)$ 배열 삽입이 $O(1)$ 연결 리스트 삽입보다 빠른 경우도 흔합니다. 구조를 바꾸는 건 병목이 측정으로 확인된 뒤여도 늦지 않습니다.

## 요약

> **핵심**
> - 배열의 $O(1)$ 접근은 탐색이 아니라 `base + i × size`라는 **주소 계산**에서 나오며, 이 식이 성립하려면 원소 크기가 같고 메모리가 연속이어야 한다는 제약이 따라옵니다.
> - 동적 배열의 `append`가 $O(1)$인 것은 용량을 **배수로** 늘리기 때문이며, 총 복사 횟수가 $n$ 미만이라는 분할 상환 분석으로 증명됩니다. 1칸씩 늘리면 이 성질이 무너집니다.
> - 연결 리스트는 $O(1)$ 삽입·삭제를 얻는 대신 노드당 포인터 오버헤드와 캐시 미스를 지불합니다. 이 대가가 값어치 있는 경우는 **위치를 이미 아는 원소를 자주 옮겨야 할 때**입니다.
> - 스택·큐·덱은 새로운 저장 방식이 아니라 배열이나 연결 리스트 위에 **연산을 제한해 얹은 인터페이스**이며, 그 제약이 모든 연산의 $O(1)$ 보장과 코드의 의도 명확성을 만들어냅니다.
