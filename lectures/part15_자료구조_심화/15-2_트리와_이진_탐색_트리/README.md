# Part 15. 자료구조 심화

## 15-2. 트리와 이진 탐색 트리 — $O(\log n)$은 어떻게 무너지고 어떻게 지켜지는가

> **학습 목표**
> - 트리의 기본 용어(루트·차수·높이·깊이)를 정의하고, 이진 트리의 노드 수와 높이의 관계를 수식으로 유도할 수 있다.
> - 전위·중위·후위·레벨 순회를 직접 수행하고, 각 순회가 어떤 문제에 적합한지 구분할 수 있다.
> - 이진 탐색 트리의 삽입·탐색·삭제를 구현하고, 편향 트리가 발생하는 조건과 균형 트리가 이를 어떻게 막는지 설명할 수 있다.

> **개요** 1-3에서 균형 이진 탐색 트리는 "최악에도 $O(\log n)$"이라고 소개했습니다. 그런데 균형이라는 단어가 왜 굳이 붙어 있을까요? 이번 강의는 평범한 이진 탐색 트리가 정렬된 데이터를 만나는 순간 어떻게 연결 리스트로 퇴화하는지를 보고, AVL과 레드-블랙 트리가 그 붕괴를 막기 위해 무엇을 지불하는지 살펴봅니다.

---

## 1. 트리 용어 — 그림 없이도 통하는 공용어

> **Q.** 트리를 설명할 때 높이, 깊이, 레벨, 차수 같은 말이 뒤섞여 나옵니다. 이들이 정확히 어떻게 다를까요?
> **A.** 기준점이 다릅니다. **깊이(depth)**는 루트에서 그 노드까지 내려온 간선 수라 위에서 잰 값이고, **높이(height)**는 그 노드에서 가장 먼 잎까지의 간선 수라 아래에서 잰 값입니다. 그래서 루트의 깊이는 항상 0이고, 잎의 높이는 항상 0입니다. **트리의 높이**는 관례상 루트의 높이를 뜻합니다. **차수(degree)**는 방향이 아니라 개수의 문제로, 한 노드가 가진 자식의 수입니다.

<u>**트리(Tree)**</u>는 사이클이 없는 연결 그래프 중에서 한 노드를 <u>**루트(root)**</u>로 지정한 계층 구조입니다. 노드가 $n$개인 트리의 간선은 항상 정확히 $n-1$개입니다. 루트를 제외한 모든 노드가 자신을 부모와 잇는 간선을 정확히 하나씩 갖기 때문입니다.

| 용어 | 정의 | 예 |
|---|---|---|
| 루트(root) | 부모가 없는 유일한 노드 | 파일 시스템의 `/` |
| 잎(leaf) | 자식이 없는 노드 | 빈 디렉터리, 파일 |
| 차수(degree) | 한 노드의 자식 수 | 이진 트리는 최대 2 |
| 깊이(depth) | 루트에서 해당 노드까지의 간선 수 | 루트는 0 |
| 높이(height) | 해당 노드에서 가장 먼 잎까지의 간선 수 | 잎은 0 |
| 서브트리(subtree) | 어떤 노드와 그 모든 자손이 이루는 트리 | 재귀 구현의 기반 |

> **참고** 트리의 정의 자체가 재귀적입니다. "트리는 루트 하나와, 그 자식들을 루트로 하는 서브트리들의 집합"입니다. 그래서 트리를 다루는 코드는 대부분 재귀로 짜면 놀랄 만큼 짧아집니다. 반대로 재귀 없이 짜려면 스택을 손으로 관리해야 합니다.

<u>**이진 트리(Binary Tree)**</u>는 모든 노드의 차수가 2 이하인 트리입니다. 높이가 $h$인 이진 트리가 가질 수 있는 최대 노드 수는 각 레벨의 노드 수를 더한 값입니다.

$$n_{\max} = 2^0 + 2^1 + \cdots + 2^h = 2^{h+1} - 1$$

> **해석** 이 식을 $h$에 대해 풀면 $h = \log_2(n+1) - 1$이 됩니다. 즉 **노드를 빈틈없이 채웠을 때만** 높이가 $\log n$에 도달합니다. 이것이 이 강의 전체의 핵심입니다. 트리 기반 자료구조의 $O(\log n)$은 공짜로 주어지는 성질이 아니라, **높이를 $\log n$ 근처로 유지했을 때만** 성립하는 조건부 보장입니다. 반대편 극단으로, 모든 노드가 자식을 하나씩만 가지면 높이는 $n-1$이 되어 사실상 연결 리스트가 됩니다.

## 2. 순회 — 같은 트리를 네 가지 순서로 읽기

> **개요** 선형 구조는 읽는 순서가 하나뿐이지만, 트리는 갈래가 있어 방문 순서를 정해야 합니다. 그 선택이 곧 알고리즘의 목적을 드러냅니다.

<u>**순회(Traversal)**</u>는 트리의 모든 노드를 빠짐없이 한 번씩 방문하는 절차입니다. 깊이 우선 계열 셋은 "현재 노드를 언제 처리하느냐"만 다릅니다.

```python
def preorder(node):    # 전위: 현재 → 왼쪽 → 오른쪽
    if not node: return
    visit(node); preorder(node.left); preorder(node.right)

def inorder(node):     # 중위: 왼쪽 → 현재 → 오른쪽
    if not node: return
    inorder(node.left); visit(node); inorder(node.right)

def postorder(node):   # 후위: 왼쪽 → 오른쪽 → 현재
    if not node: return
    postorder(node.left); postorder(node.right); visit(node)
```

![이진 트리의 네 가지 순회 순서](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part15_자료구조_심화/15-2_트리와_이진_탐색_트리/assets/diagram1_traversal.svg)

> **해석** 위 그림의 같은 트리를 네 방식으로 읽으면 완전히 다른 수열이 나옵니다. 각 순서에는 존재 이유가 있습니다. **전위**는 부모를 자식보다 먼저 처리하므로 트리를 복제하거나 직렬화할 때 씁니다(부모가 있어야 자식을 매달 수 있으니까요). **중위**는 이진 탐색 트리에서 **정렬된 순서**를 그대로 뱉어내며, 이것이 이진 탐색 트리의 가장 강력한 성질 중 하나입니다. **후위**는 자식을 모두 처리한 뒤 부모를 처리하므로 트리를 메모리에서 해제하거나 디렉터리 용량을 합산할 때 씁니다. **레벨 순회**는 큐를 써서 위에서 아래로 층층이 읽으며, 최단 경로 성격의 문제에 쓰입니다.

| 순회 | 방문 순서 | 자료구조 | 대표 용도 |
|---|---|---|---|
| 전위(preorder) | 현재 → 좌 → 우 | 재귀/스택 | 트리 복제, 직렬화, 수식의 전위 표기 |
| 중위(inorder) | 좌 → 현재 → 우 | 재귀/스택 | BST의 정렬 출력, 범위 질의 |
| 후위(postorder) | 좌 → 우 → 현재 | 재귀/스택 | 메모리 해제, 디렉터리 크기 합산 |
| 레벨(level-order) | 위에서 아래로 층별 | **큐** | 트리의 너비 계산, 층별 처리 |

> **팁** 순회 코드는 모두 $O(n)$이고 모든 노드를 정확히 한 번 방문합니다. 다만 재귀 깊이는 **트리의 높이**만큼 쌓이므로, 편향된 트리에서 재귀 순회는 스택 오버플로를 일으킬 수 있습니다. 깊이가 수만 단계에 이를 수 있는 트리라면 명시적 스택으로 바꾸는 편이 안전합니다.

## 3. 이진 탐색 트리 — 불변식 하나가 만드는 $O(h)$

<u>**이진 탐색 트리(Binary Search Tree, BST)**</u>를 정의하는 것은 단 하나의 <u>**불변식(invariant)**</u>입니다.

> 모든 노드 $v$에 대해, 왼쪽 서브트리의 **모든** 키 < $v$의 키 < 오른쪽 서브트리의 **모든** 키

> **참고** "왼쪽 자식보다 크다"가 아니라 **왼쪽 서브트리 전체**보다 크다는 점이 중요합니다. 직계 자식만 비교하는 조건으로 착각하면, 손자 노드가 조건을 어기는 트리를 BST로 잘못 판정하게 됩니다. BST 검증 문제가 면접 단골인 이유가 바로 이 함정입니다.

이 불변식 덕분에 **탐색**은 매 노드에서 크기 비교 한 번으로 절반의 서브트리를 통째로 버립니다. 삽입도 같은 경로를 따라 내려가다 빈 자리에 새 노드를 매답니다. 두 연산 모두 루트에서 잎까지의 경로 길이, 즉 $O(h)$입니다.

**삭제**만 세 경우로 나뉩니다.

| 경우 | 처리 방법 |
|---|---|
| 자식이 없음(잎) | 그냥 제거 |
| 자식이 하나 | 그 자식을 자기 자리로 끌어올림 |
| 자식이 둘 | **중위 후속자**(오른쪽 서브트리의 최솟값) 또는 중위 선행자(왼쪽 서브트리의 최댓값)를 찾아 값을 덮어쓰고, 그 노드를 대신 삭제 |

> **해석** 자식이 둘인 경우가 까다로운 이유는 빈자리 하나에 두 서브트리를 매달 수 없기 때문입니다. 해법은 "불변식을 깨지 않는 유일한 후보"를 찾는 것입니다. 삭제할 노드보다 큰 값 중 가장 작은 값(= 오른쪽 서브트리의 최솟값)을 올리면, 그 값은 왼쪽 서브트리 전체보다는 크고 오른쪽 서브트리의 나머지 전체보다는 작으므로 불변식이 그대로 유지됩니다. 그리고 그 최솟값 노드는 정의상 왼쪽 자식이 없으므로, 이제 훨씬 쉬운 경우 1이나 2로 환원됩니다.

## 4. 편향 트리 — $O(\log n)$이 무너지는 순간

> **Q.** 이진 탐색 트리는 $O(\log n)$이라고 배웠는데, 왜 실무에서는 그냥 BST를 쓰지 말라고 할까요?
> **A.** BST의 복잡도는 $O(\log n)$이 아니라 $O(h)$이며, $h$가 $\log n$이라는 보장이 어디에도 없기 때문입니다. 정렬된 데이터 1, 2, 3, ..., $n$을 순서대로 삽입해보면 모든 값이 직전 노드의 오른쪽 자식이 되어, 높이 $n-1$의 오른쪽으로만 뻗은 <u>**편향 트리(skewed tree)**</u>가 만들어집니다. 그러면 모든 연산이 $O(n)$으로 무너집니다. 문제는 정렬된 입력이 실무에서 예외가 아니라 **기본값**에 가깝다는 점입니다. 자동 증가 ID, 타임스탬프, 파일명 정렬 결과가 모두 그렇습니다.

![편향 트리와 균형 트리의 높이 차이](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part15_자료구조_심화/15-2_트리와_이진_탐색_트리/assets/diagram2_balance.svg)

> **해석** 같은 7개의 키를 담았는데도 왼쪽 편향 트리는 높이 6, 오른쪽 균형 트리는 높이 2입니다. 노드 수가 100만 개로 늘면 격차는 100만 대 20이 됩니다. 두 트리는 담긴 데이터도 같고 BST 불변식도 똑같이 지키지만, **삽입 순서**만 다릅니다. 자료구조의 성능이 데이터 자체가 아니라 데이터가 도착한 순서에 좌우된다는 것은 명백한 설계 결함이며, 그래서 삽입할 때마다 스스로 형태를 고치는 자가 균형 트리가 필요해집니다.

## 5. 균형 트리 — 무엇을 지불하고 무엇을 사는가

<u>**자가 균형 이진 탐색 트리(self-balancing BST)**</u>는 삽입·삭제 후 <u>**회전(rotation)**</u>이라는 국소적 구조 변경으로 높이를 $O(\log n)$ 안에 묶습니다. 회전은 부모-자식 관계를 뒤집으면서도 중위 순회 순서를 그대로 보존하므로, BST 불변식을 깨지 않고 높이만 줄일 수 있습니다.

<u>**AVL 트리**</u>는 1962년 소련의 게오르기 아델손-벨스키(Georgy Adelson-Velsky)와 예브게니 란디스(Evgenii Landis)가 발표한 최초의 자가 균형 트리로, 이름은 두 사람의 이니셜에서 왔습니다. 모든 노드에서 다음 조건을 강제합니다.

$$|\text{height}(\text{left}) - \text{height}(\text{right})| \leq 1$$

<u>**레드-블랙 트리(Red-Black Tree)**</u>는 1978년 레오 기바스(Leo Guibas)와 로버트 세지윅(Robert Sedgewick)이 제시한 구조로, 각 노드에 색을 부여하고 "루트에서 모든 잎까지의 경로에 검은 노드 수가 같다", "빨간 노드는 빨간 자식을 가질 수 없다" 같은 규칙으로 균형을 유지합니다. 이 규칙들의 결과로 가장 긴 경로가 가장 짧은 경로의 2배를 넘지 못하게 됩니다.

| 비교 항목 | AVL 트리 | 레드-블랙 트리 |
|---|---|---|
| 균형 기준 | 높이 차 $\leq 1$ | 검은 높이 동일 + 색 규칙 |
| 높이 상한 | 약 $1.44 \log_2 n$ | 약 $2 \log_2 n$ |
| 탐색 속도 | 더 빠름(더 낮은 트리) | 조금 느림 |
| 삽입·삭제 회전 | 더 잦음 | 더 적음(삽입 최대 2회) |
| 유리한 워크로드 | 읽기 위주 | 쓰기가 섞인 범용 |

> **참고** 실무 표준 라이브러리는 대부분 레드-블랙 트리를 택했습니다. Java의 `TreeMap`·`TreeSet`, C++의 `std::map`·`std::set`, 리눅스 커널의 프로세스 스케줄러(CFS)가 모두 레드-블랙 트리입니다. 균형이 조금 느슨한 대신 삽입·삭제 시 재조정 비용이 낮아, 읽기와 쓰기가 섞인 일반적인 상황에서 총 비용이 유리하기 때문입니다. Java `HashMap`이 충돌이 심한 버킷을 트리로 바꿀 때 쓰는 것도 레드-블랙 트리입니다.

> **팁** 해시 테이블이 평균 $O(1)$인데 왜 균형 트리를 쓸까요? 트리는 **순서**를 유지하기 때문입니다. "점수가 80점 이상 90점 이하인 사용자", "최근 1시간 내 로그", "다음으로 큰 키" 같은 **범위 질의와 정렬 순회**는 해시 테이블이 원리적으로 할 수 없고 트리는 자연스럽게 해내는 일입니다.

## 요약

> **핵심**
> - 이진 트리의 높이는 노드를 빈틈없이 채웠을 때만 $\log_2(n+1) - 1$에 도달합니다. 트리 자료구조의 $O(\log n)$은 **높이가 관리될 때만** 성립하는 조건부 보장입니다.
> - 순회 네 가지는 "현재 노드를 언제 처리하느냐"의 차이이며, 각각 직렬화(전위)·정렬 출력(중위)·해제와 집계(후위)·층별 처리(레벨)라는 뚜렷한 용도를 갖습니다.
> - BST를 정의하는 것은 "왼쪽 **서브트리 전체** < 노드 < 오른쪽 **서브트리 전체**"라는 불변식 하나이며, 자식 둘인 노드의 삭제는 중위 후속자로 값을 대체해 이 불변식을 지킵니다.
> - 정렬된 순서로 삽입하면 BST는 높이 $n-1$의 편향 트리로 퇴화해 모든 연산이 $O(n)$이 됩니다. AVL과 레드-블랙 트리는 회전으로 높이를 $O(\log n)$에 묶어 이를 막으며, 실무 표준 라이브러리는 재조정 비용이 낮은 레드-블랙 트리를 주로 택합니다.
