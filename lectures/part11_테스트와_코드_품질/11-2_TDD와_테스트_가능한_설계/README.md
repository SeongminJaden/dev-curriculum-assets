# Part 11. 테스트와 코드 품질

## 11-2. TDD와 테스트 가능한 설계 — 테스트를 먼저 쓰는 것이 왜 더 나은 설계로 이어지는가

> **학습 목표**
> - Red-Green-Refactor 사이클의 각 단계가 하는 역할을 순서대로 설명할 수 있다.
> - TDD가 "버그를 줄이는 기법"이 아니라 "설계를 유도하는 기법"이라고 불리는 이유를 설명할 수 있다.
> - 의존성 주입(Dependency Injection)이 테스트 가능성을 어떻게 높이는지 코드 수준에서 설명할 수 있다.

> **개요** 테스트 주도 개발(Test-Driven Development, TDD)은 코드를 먼저 짜고 나중에 테스트를 붙이는 순서를 뒤집습니다. 이 순서 전환이 왜 단순한 습관 차이가 아니라, 결과적으로 더 테스트하기 쉬운(=더 나은) 설계로 이어지는지를 살펴봅니다.

---

## 1. TDD란 무엇인가 — Red-Green-Refactor

> **개요** TDD의 창시자로 알려진 켄트 벡(Kent Beck)이 정리한 세 단계 사이클을 살펴봅니다.

테스트 주도 개발은 켄트 벡(Kent Beck)이 정리하고 확산시킨 개발 방법론으로, 프로덕션 코드를 작성하기 전에 그 코드가 통과해야 할 테스트를 먼저 작성합니다. 이 과정은 아래 세 단계를 짧은 주기로 반복합니다.

![Red-Green-Refactor 사이클 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part11_테스트와_코드_품질/11-2_TDD와_테스트_가능한_설계/assets/diagram1_cycle.svg)

1. <u>**Red**</u> — 아직 존재하지 않는 기능에 대한 테스트를 먼저 작성합니다. 구현이 없으므로 이 테스트는 반드시 실패합니다.
2. <u>**Green**</u> — 그 테스트를 통과시키는 데 필요한 최소한의 코드만 작성합니다. 완성도나 아름다움은 이 단계의 목표가 아닙니다.
3. <u>**Refactor**</u> — 테스트가 통과하는 상태를 유지한 채로 코드의 구조를 개선합니다. 중복을 제거하고 이름을 명확히 하는 등, 동작은 그대로 두고 설계만 다듬습니다.

> **참고** "Red-Green-Refactor"라는 이름은 대부분의 테스트 도구가 실패한 테스트를 빨간색으로, 통과한 테스트를 초록색으로 표시하는 관행에서 왔습니다.

> **Q.** 테스트를 나중에 작성해도 결국 같은 테스트가 만들어지는 것 아닌가요? 순서가 왜 중요할까요?
> **A.** 순서가 결과물의 성격을 바꾸기 때문입니다. 코드를 먼저 작성하면, 테스트는 "이미 짜여진 구현"에 맞춰 끼워 넣어야 합니다. 이 경우 구현이 테스트하기 어려운 구조여도 이미 코드가 완성되어 있으니 그대로 두고 테스트 쪽에서 억지로 우회하게 됩니다. 반대로 테스트를 먼저 쓰면, 개발자는 "이 기능을 어떻게 호출하면 검증하기 쉬울까"를 코드를 작성하기 전에 먼저 고민하게 됩니다. 그 결과 만들어지는 인터페이스 자체가 처음부터 테스트하기 쉬운 모양을 갖추게 됩니다.

## 2. TDD는 왜 "설계 기법"이라고 불리는가

> **개요** TDD의 진짜 가치가 버그 감소가 아니라 설계 유도에 있다는 관점을 다룹니다.

<u>**TDD를 "테스트를 많이 만드는 습관"으로만 이해하면 핵심을 놓칩니다.**</u> TDD의 실질적인 효과는 테스트를 먼저 쓰기 위해 개발자가 강제로 거치게 되는 질문에서 나옵니다. 테스트를 먼저 작성하려면 "이 함수를 어떻게 호출할 것인가", "이 클래스가 무엇에 의존해야 하는가"를 구현보다 먼저 결정해야 합니다. 이 강제된 질문이 자연스럽게 아래와 같은 설계 습관으로 이어집니다.

- 함수나 클래스가 <u>**한 번에 하나의 책임만 지도록**</u> 작아집니다. 책임이 여러 개면 테스트 하나를 작성하기 위해 준비해야 할 조건이 너무 많아지기 때문입니다.
- 외부 의존성(데이터베이스, 네트워크, 현재 시각 등)이 <u>**코드 내부에 직접 박혀 있지 않고 바깥에서 주입되는 구조**</u>로 바뀝니다. 테스트에서는 실제 데이터베이스 없이 검증하고 싶기 때문입니다.
- 함수의 입력과 출력이 <u>**명확하게 드러납니다.**</u> 테스트가 "무엇을 넣으면 무엇이 나오는가"를 검증해야 하므로, 숨겨진 전역 상태에 의존하는 코드는 테스트를 작성하기 어렵다는 이유로 자연스럽게 걸러집니다.

> **핵심** 그래서 TDD를 두고 흔히 "테스트 주도 개발이 아니라 테스트 주도 설계(Test-Driven Design)"라고도 부릅니다. <u>**테스트를 먼저 쓰기 어렵다는 사실 자체가, 그 코드의 설계에 문제가 있다는 신호로 작동합니다.**</u>

## 3. 테스트 가능성을 낮추는 원인 — 의존성이 코드에 박혀 있을 때

> **개요** 코드 내부에서 의존성을 직접 생성하는 방식이 왜 테스트를 어렵게 만드는지, 구체적인 예시로 살펴봅니다.

테스트를 짜기 어려운 코드에는 공통된 특징이 있습니다. 함수나 클래스가 자신에게 필요한 협력 대상(데이터베이스 연결, 외부 API 클라이언트, 현재 시각 등)을 <u>**스스로 생성해서 사용**</u>한다는 점입니다.

### 3.1 의존성이 코드 내부에 박혀 있는 경우 (의사코드)

```
class OrderService:
    def place_order(self, item, quantity):
        db = MySQLConnection("prod-db-host")   # 내부에서 직접 생성
        price = ExternalPriceApi().fetch(item)  # 내부에서 직접 생성
        db.save(item, quantity, price)
        return price * quantity
```

> **해석** 이 코드를 테스트하려면 실제 MySQL 서버와 실제 외부 가격 API가 살아 있어야 합니다. 네트워크가 끊기거나 외부 API가 잠깐 응답하지 않으면 테스트도 함께 실패합니다. 게다가 "가격이 음수로 내려왔을 때 어떻게 되는가" 같은 예외 상황을 재현하려면, 실제 외부 API가 그런 값을 돌려주는 순간까지 기다려야 합니다.

### 3.2 의존성을 바깥에서 주입받는 경우 (의사코드)

```
class OrderService:
    def __init__(self, db, price_api):
        self.db = db
        self.price_api = price_api

    def place_order(self, item, quantity):
        price = self.price_api.fetch(item)
        self.db.save(item, quantity, price)
        return price * quantity
```

> **해석** `db`와 `price_api`를 생성자에서 외부로부터 전달받도록 바꾸면, 테스트에서는 실제 데이터베이스나 API 대신 가짜 객체(mock)를 넣어줄 수 있습니다. "가격 API가 음수를 반환하면 어떻게 되는가"를 확인하고 싶다면, 실제 API를 기다릴 필요 없이 가짜 객체가 그 순간 음수를 반환하도록 설정하기만 하면 됩니다.

> **참고** 이렇게 <u>**필요한 협력 대상을 스스로 만들지 않고 외부에서 전달받는 설계 방식**</u>을 의존성 주입(Dependency Injection, DI)이라고 부릅니다. DI 자체는 원래 "코드를 유연하게 만드는" 목적으로 쓰이지만, 부수적으로 테스트 가능성을 크게 높여주는 효과가 있습니다.

## 4. 실생활 예시 — 같은 버그를 재현하는 두 가지 비용

> **개요** 의존성 주입 여부가 실제로 버그를 재현하고 고치는 비용에 어떤 차이를 만드는지 봅니다.

| 상황 | 의존성이 코드에 박혀 있을 때 | 의존성을 주입받을 때 |
|---|---|---|
| "결제 API가 타임아웃을 반환하면 주문이 취소되어야 한다"는 요구사항 검증 | 실제 결제 API가 타임아웃을 일으키는 상황을 재현해야 함 (거의 불가능) | 가짜 결제 API 객체가 타임아웃 예외를 던지도록 설정만 하면 즉시 검증 가능 |
| 테스트 실행 속도 | 매번 실제 네트워크 호출이 발생해 느리고, 외부 서비스 장애 시 테스트도 함께 실패 | 네트워크 호출이 없어 밀리초 단위로 끝나고, 외부 서비스 상태와 무관 |
| 새 개발자가 코드를 이해하는 속도 | 생성자 안까지 들어가야 실제 의존 대상을 알 수 있음 | 생성자 매개변수만 봐도 이 클래스가 무엇에 의존하는지 바로 드러남 |

> **핵심** 의존성 주입은 <u>**테스트를 쉽게 만드는 부수 효과를 넘어서, 코드가 무엇에 의존하는지를 인터페이스만으로 드러내는 설계 효과**</u>도 함께 가져옵니다.

## 5. TDD를 둘러싼 오해

> **개요** TDD에 대해 흔히 생기는 두 가지 오해를 짚습니다.

> **참고** TDD를 처음 접할 때 흔히 하는 오해입니다.
> - "TDD를 하면 버그가 0이 된다" — 아닙니다. TDD는 설계를 검증 가능한 형태로 유도할 뿐, 요구사항 자체를 잘못 이해했다면 테스트도 잘못된 기준으로 작성됩니다.
> - "TDD는 모든 코드에 항상 적용해야 한다" — 아닙니다. 요구사항이 자주 바뀌는 프로토타입 단계나, UI의 미세한 배치처럼 자동 테스트로 검증하기 비효율적인 영역에서는 TDD의 이점이 상대적으로 작습니다. TDD는 <u>**로직이 명확히 정의될 수 있는 영역**</u>에서 가장 큰 효과를 냅니다.

---

## 요약

> **핵심**
> - TDD는 Red(실패하는 테스트 작성) → Green(최소 구현으로 통과) → Refactor(구조 개선)의 사이클을 반복하는 개발 방법론입니다.
> - TDD의 핵심 가치는 버그 감소 자체가 아니라, <u>**테스트를 먼저 쓰기 위해 강제로 거치는 질문이 결과적으로 책임이 작고 의존성이 명확한 설계로 이어진다**</u>는 데 있습니다.
> - 의존성 주입은 협력 대상을 코드 내부에서 직접 생성하지 않고 바깥에서 전달받는 방식으로, <u>**외부 상태 없이도 다양한 상황을 가짜 객체로 즉시 재현할 수 있게 해 테스트 가능성을 크게 높입니다.**</u>
