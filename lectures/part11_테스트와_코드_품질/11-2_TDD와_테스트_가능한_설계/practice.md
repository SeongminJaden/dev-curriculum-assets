# 11-2 실습 — TDD와 테스트 가능한 설계

> **참고** 제출물이 아니라, Red-Green-Refactor 사이클과 의존성 주입을 직접 손으로 익혀보는 실습입니다.

## 실습 1. Red-Green-Refactor 한 바퀴 돌려보기

- [ ] 간단한 함수 하나를 정해주세요. (예: 문자열을 뒤집는 함수, 두 날짜 사이의 일수를 계산하는 함수)
- [ ] 그 함수가 아직 없다고 가정하고, 통과해야 할 테스트를 먼저 글이나 의사코드로 적어보세요. (Red 단계)
- [ ] 그 테스트를 통과시키는 데 필요한 최소한의 구현을 적어보세요. (Green 단계)
- [ ] 통과 상태를 유지하면서 구현을 더 깔끔하게 고쳐보세요. (Refactor 단계)
- [ ] 세 단계를 거치는 동안 처음 생각했던 함수 시그니처(입력·출력)가 바뀌었는지 비교해보세요.

## 실습 2. 의존성 주입으로 바꿔보기

- [ ] 아래와 같이 내부에서 의존성을 직접 생성하는 의사코드를 하나 만들어보세요. (강의의 `OrderService` 예시를 참고해도 됩니다)
- [ ] 그 코드를 테스트하려면 어떤 준비가 필요한지 적어보세요. (예: 실제 서버 실행, 실제 API 키 발급)
- [ ] 의존성을 생성자에서 전달받는 구조로 바꿔보세요.
- [ ] 바뀐 구조에서는 같은 상황(예: 외부 API 오류)을 어떻게 가짜 객체로 재현할 수 있을지 적어보세요.

## 생각해볼 질문

- [ ] "테스트를 먼저 쓰기 어렵다"는 사실이 왜 설계 문제의 신호로 해석될 수 있는지, 이번 강의 내용을 근거로 설명해보세요.
- [ ] 여러분이 만약 요구사항이 매일 바뀌는 프로토타입을 만든다면, TDD를 얼마나 엄격하게 적용할지 이유와 함께 적어보세요.
