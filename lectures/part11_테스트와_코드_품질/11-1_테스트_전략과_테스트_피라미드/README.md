# Part 11. 테스트와 코드 품질

## 11-1. 테스트 전략과 테스트 피라미드 — 단위·통합·E2E는 왜 비율이 달라야 하는가

> **학습 목표**
> - 단위·통합·E2E 테스트 각각의 정의와 검증 범위를 구분해서 설명할 수 있다.
> - 테스트 피라미드가 삼각형 모양을 갖는 이유를 속도·비용·신뢰도 트레이드오프로 설명할 수 있다.
> - 프로젝트 상황에 맞게 테스트 비율을 어떻게 설계해야 하는지 판단할 수 있다.

> **개요** 테스트는 "많이 짤수록 좋다"가 아니라 "어떤 레벨에 얼마나 배분하는가"가 품질과 개발 속도를 동시에 결정합니다. 이 강의에서는 테스트의 세 레벨을 정의하고, 왜 그 비율이 삼각형 모양이어야 하는지를 살펴봅니다.

---

## 1. 테스트는 왜 필요한가

> **개요** 코드를 검증하는 절차가 없을 때 리팩터링과 기능 추가가 왜 위험해지는지 살펴봅니다.

> **Q.** 코드가 지금 잘 동작하고 있다면, 굳이 테스트를 작성해야 할까요?
> **A.** "지금 잘 동작한다"는 사실은 "앞으로도 잘 동작한다"를 보장하지 않기 때문입니다. 코드는 배포된 이후에도 계속 변경됩니다. 새 기능을 추가하거나 성능을 개선하려고 기존 코드를 건드릴 때마다, 그 변경이 이미 동작하던 다른 부분을 망가뜨리지 않았는지 확인할 방법이 필요합니다. 테스트는 사람이 매번 수동으로 눈으로 확인하는 작업을 자동화된 절차로 바꿔줍니다.

<u>**테스트가 없는 코드베이스에서는 리팩터링 자체가 위험한 작업이 됩니다.**</u> 개발자가 "이 코드를 더 깔끔하게 고치고 싶다"고 생각해도, 그 변경이 안전한지 확인할 방법이 없으므로 결국 손을 대지 않고 방치하게 됩니다. 이런 코드가 쌓이면 코드베이스 전체가 점점 다루기 어려워집니다.

### 실생활 예시 — 할인 로직을 리팩터링하다가 생긴 버그

| 상황 | 테스트가 없었을 때 | 테스트가 있었을 때 |
|---|---|---|
| 할인 계산 함수를 더 읽기 쉽게 리팩터링 | 코드는 깔끔해졌지만, 정기 할인과 쿠폰 할인이 중복 적용되는 경계 조건을 놓침 | 리팩터링 직후 자동 테스트가 실패해서 즉시 문제 확인 |
| 문제 발견 시점 | 실제 결제가 발생한 뒤, 사용자가 "할인이 이상하게 적용됐다"고 문의할 때 | 코드를 커밋하기 전, 개발자의 로컬 환경 |
| 수정 비용 | 이미 발생한 결제 건들을 역으로 조사하고 환불 처리까지 필요 | 함수 한 줄 수정 후 재실행으로 종료 |

> **핵심** 테스트의 본질적인 역할은 "버그를 없애는 것"이 아니라, <u>**변경이 안전한지를 사람이 아니라 기계가 즉시 확인해 주는 것**</u>입니다. 이 덕분에 개발자는 겁내지 않고 코드를 계속 개선할 수 있습니다.

## 2. 테스트의 세 레벨

> **개요** 단위·통합·E2E 테스트가 각각 무엇을 검증 대상으로 삼는지, 정의를 정확히 구분합니다.

테스트는 "무엇을 하나의 단위로 보고 검증하는가"에 따라 크게 세 레벨로 나뉩니다.

- <u>**단위 테스트(Unit Test)**</u>: 함수나 클래스 하나처럼 더 이상 쪼갤 필요가 없는 최소 단위를 독립적으로 검증합니다. 데이터베이스, 네트워크, 파일 시스템 같은 외부 의존성은 실제로 호출하지 않고 대역(mock, stub)으로 대체합니다.
- <u>**통합 테스트(Integration Test)**</u>: 여러 모듈이 실제로 맞물려 동작할 때를 검증합니다. 예를 들어 서비스 코드가 실제 데이터베이스와 통신해서 원하는 결과를 저장·조회하는지 확인합니다.
- <u>**E2E 테스트(End-to-End Test)**</u>: 실제 사용자의 흐름을 처음부터 끝까지 그대로 재현합니다. 브라우저를 실제로 띄워 로그인부터 결제 완료까지 클릭 흐름을 자동으로 수행하는 식입니다.

> **참고** 세 레벨을 가르는 기준은 "테스트 코드의 길이"가 아니라 <u>**"검증 범위에 실제 의존성이 얼마나 포함되는가"**</u>입니다. 단위 테스트는 의존성을 최대한 걷어내고, E2E 테스트는 실제 환경에 최대한 가깝게 만듭니다.

### 실생활 예시 — 배달 앱의 "주문하기" 기능을 세 레벨로 검증하기

| 레벨 | 검증 대상 | 예시 |
|---|---|---|
| 단위 테스트 | 총 금액 계산 함수 하나 | 상품 3개 가격과 할인율을 넣었을 때 계산된 합계가 맞는지 확인 (DB 호출 없음) |
| 통합 테스트 | 주문 저장 로직과 실제 데이터베이스 | 주문 API를 호출했을 때 실제 테스트 DB에 주문 레코드가 정확히 저장되는지 확인 |
| E2E 테스트 | 사용자 화면부터 서버까지 전체 흐름 | 실제 브라우저에서 메뉴 선택 → 결제 → 주문 완료 화면까지 자동으로 클릭해서 확인 |

## 3. 테스트 피라미드 — 왜 비율이 달라야 하는가

> **개요** 세 레벨의 테스트가 왜 같은 개수로 존재하면 안 되는지, 속도·비용·신뢰도의 트레이드오프로 설명합니다.

테스트 피라미드(Test Pyramid)는 마이크 콘(Mike Cohn)이 저서 *Succeeding with Agile*(2009)에서 제시한 뒤, 마틴 파울러(Martin Fowler) 등을 통해 널리 알려진 개념입니다. 핵심 주장은 단순합니다. <u>**단위 테스트를 가장 많이, 통합 테스트를 그다음으로, E2E 테스트를 가장 적게 작성해야 한다**</u>는 것입니다.

![테스트 피라미드 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part11_테스트와_코드_품질/11-1_테스트_전략과_테스트_피라미드/assets/diagram1_pyramid.svg)

> **Q.** 왜 세 레벨을 똑같은 비율로 섞으면 안 될까요?
> **A.** 레벨이 올라갈수록 실제 의존성을 더 많이 포함하기 때문에, 신뢰도는 올라가지만 그 대가로 속도가 느려지고 비용이 커지며 결과가 불안정(flaky)해질 위험도 커지기 때문입니다. 이 트레이드오프를 무시하고 E2E 테스트 위주로 검증하면, 테스트 전체를 돌리는 데 시간이 너무 오래 걸려서 결국 테스트를 자주 돌리지 않게 됩니다. 반대로 단위 테스트만 있으면 모듈 각각은 검증되어도 실제로 맞물렸을 때 문제가 생기는 경우를 놓칩니다.

### 3.1 레벨별 트레이드오프

![테스트 레벨별 속도·신뢰도 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part11_테스트와_코드_품질/11-1_테스트_전략과_테스트_피라미드/assets/diagram2_tradeoff.svg)

- <u>**속도**</u>: 단위 테스트는 외부 의존성이 없어 밀리초 단위로 끝납니다. E2E 테스트는 실제 브라우저·서버·데이터베이스를 모두 띄워야 하므로 초에서 분 단위까지 걸립니다.
- <u>**비용**</u>: 단위 테스트는 작성과 유지보수가 상대적으로 쉽습니다. E2E 테스트는 테스트 환경 구축, 테스트 데이터 준비, 화면 구조 변경에 따른 유지보수 부담이 훨씬 큽니다.
- <u>**신뢰도(확신)**</u>: 반대로 실제 사용자 흐름을 그대로 재현하는 E2E 테스트가 "실제로 서비스가 동작한다"는 확신을 가장 강하게 줍니다. 단위 테스트는 개별 부품이 옳다는 확신만 줄 뿐, 부품들이 맞물렸을 때도 옳다는 보장은 하지 못합니다.

> **참고** 그래서 피라미드는 "느리고 비싸지만 확신을 주는 테스트는 적게, 빠르고 저렴한 테스트는 많이" 배치하는 전략입니다. 대다수의 버그를 저렴한 단위 테스트로 걸러내고, E2E 테스트는 "전체 흐름이 실제로 이어지는가"라는 최종 확인 용도로만 소수 유지합니다.

### 3.2 피라미드가 뒤집히면 생기는 문제 — 아이스크림콘 안티패턴

E2E 테스트 비중이 지나치게 커서 피라미드가 뒤집힌 모양을 <u>**아이스크림콘 안티패턴(Ice Cream Cone Anti-pattern)**</u>이라고 부릅니다.

> **참고** 아이스크림콘 형태의 테스트 스위트에서 흔히 나타나는 증상입니다.
> - 테스트 전체를 실행하는 데 시간이 오래 걸려서, 개발자가 코드를 수정할 때마다 테스트를 돌리지 않고 미룹니다.
> - 화면 UI가 조금만 바뀌어도 관련 없는 E2E 테스트가 무더기로 실패해서, 실패 원인을 찾는 데 드는 시간이 늘어납니다.
> - 네트워크 지연이나 타이밍 문제로 실패와 성공이 매번 바뀌는 "불안정한(flaky)" 테스트가 늘어나, 팀이 테스트 실패를 신뢰하지 않게 됩니다.

> **핵심** <u>**테스트 피라미드의 목적은 테스트를 없애는 것이 아니라, 같은 신뢰도를 더 빠르고 저렴하게 얻을 수 있는 레벨로 검증을 최대한 옮기는 것입니다.**</u> 통합 테스트나 E2E 테스트로만 확인할 수 있는 것(실제 DB 연동, 실제 사용자 흐름)은 그 레벨에 남기고, 나머지는 단위 테스트로 내려보냅니다.

## 4. 프로젝트 상황에 따른 비율 조정

> **개요** 피라미드의 정확한 비율이 모든 프로젝트에 고정된 숫자로 존재하지는 않는다는 점을 다룹니다.

테스트 피라미드는 "단위 테스트 70%, 통합 테스트 20%, E2E 테스트 10%"처럼 고정된 비율표가 아니라, <u>**아래로 갈수록 넓어지는 삼각형 모양이라는 상대적 원칙**</u>입니다. 실제 비율은 프로젝트의 성격에 따라 달라집니다.

- 순수 계산 로직이 많은 라이브러리라면 단위 테스트 비중이 압도적으로 커야 합니다.
- 여러 외부 시스템과의 연동이 핵심인 백엔드 서비스라면 통합 테스트 비중이 상대적으로 커질 수 있습니다.
- 결제처럼 사용자 흐름 전체가 한 번이라도 끊기면 안 되는 기능은, 비중은 작더라도 핵심 시나리오에 대한 E2E 테스트를 반드시 포함해야 합니다.

> **참고** 중요한 것은 정확한 비율 숫자를 외우는 것이 아니라, <u>**"이 검증을 더 아래 레벨(더 빠르고 저렴한 레벨)에서 할 수는 없는가?"**</u>를 매번 스스로 묻는 습관입니다.

## 요약

> **핵심**
> - 단위·통합·E2E 테스트는 검증 범위에 포함되는 실제 의존성의 양으로 구분되며, <u>**의존성이 많아질수록 신뢰도는 올라가지만 속도와 비용 면에서는 불리해집니다.**</u>
> - 테스트 피라미드는 이 트레이드오프 때문에 <u>**빠르고 저렴한 단위 테스트를 가장 많이, 느리고 비싼 E2E 테스트를 가장 적게 배치**</u>해야 한다는 원칙입니다.
> - 피라미드가 뒤집히면(아이스크림콘 안티패턴) 테스트 전체가 느려지고 불안정해져서, 오히려 테스트가 개발 속도를 떨어뜨리는 역효과가 납니다.
