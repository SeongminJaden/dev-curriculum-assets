# 11-3 실습 — 코드 품질과 설계 원칙(SOLID)

> **참고** 제출물이 아니라, SOLID 다섯 원칙을 직접 코드 수준에서 적용해 보는 실습입니다.

## 실습 1. 위반 코드 찾아 고치기

- [ ] 최근에 작성했던 코드(과제, 프로젝트 등) 중 하나의 클래스나 함수가 여러 가지 일을 한꺼번에 하고 있지 않은지 살펴보세요.
- [ ] 그 클래스가 "몇 가지 이유로 바뀔 수 있는지" 하나씩 나열해보세요. (SRP 점검)
- [ ] 이유가 두 개 이상이라면, 각 이유별로 클래스를 분리해보세요.

## 실습 2. 새 요구사항으로 원칙 점검하기

- [ ] 강의의 `PaymentProcessor` 예시(또는 여러분이 만든 코드)에 새로운 결제 수단을 하나 추가한다고 가정해보세요.
- [ ] 위반 코드 버전에서는 몇 개의 파일·함수를 수정해야 하는지 적어보세요.
- [ ] 준수 코드 버전(공통 인터페이스 + 개별 클래스)에서는 몇 개의 파일을 새로 추가하기만 하면 되는지 적어보세요.
- [ ] 두 버전의 차이가 왜 생기는지, OCP 정의를 근거로 한 문단으로 설명해보세요.

## 실습 3. 다섯 원칙 자가 진단

- [ ] 여러분이 최근에 작성한 클래스 하나를 골라, 아래 다섯 질문에 각각 예/아니오로 답해보세요.
  - [ ] SRP — 이 클래스가 바뀔 수 있는 이유는 하나뿐인가요?
  - [ ] OCP — 새 기능을 추가할 때 이 클래스의 기존 코드를 수정하지 않아도 되나요?
  - [ ] LSP — 이 클래스를 상속한 하위 클래스가 있다면, 상위 클래스 대신 넣어도 동작이 그대로인가요?
  - [ ] ISP — 이 클래스가 구현한 인터페이스의 메서드를 실제로 전부 사용하고 있나요?
  - [ ] DIP — 이 클래스는 구체적인 구현이 아니라 추상(인터페이스)에 의존하고 있나요?
- [ ] "아니오"가 나온 항목이 있다면, 어떻게 고치면 좋을지 한 문장씩 적어보세요.

## 생각해볼 질문

- [ ] SOLID 다섯 원칙이 공통으로 겨냥하는 하나의 목표가 무엇인지, 이번 강의 내용을 근거로 설명해보세요.
- [ ] 11-2에서 배운 의존성 주입과, 이번 강의의 의존성 역전 원칙(DIP)이 어떤 관계인지 설명해보세요.
