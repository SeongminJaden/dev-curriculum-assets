# Part 9. 보안

## 9-1. 웹 애플리케이션 보안 기초 — OWASP Top 10은 왜 매년 같은 얼굴로 돌아오는가

> **학습 목표**
> - OWASP과 OWASP Top 10 문서의 제작 방식을 이해하고, 이것이 왜 업계 표준 참고자료로 통용되는지 설명할 수 있다.
> - SQL 인젝션, XSS, 인증·접근 제어 취약점이 공통으로 딛고 서 있는 원리(신뢰 경계 붕괴)를 설명할 수 있다.
> - 프레임워크와 언어가 계속 바뀌는데도 같은 유형의 취약점이 왜 해마다 반복해서 등장하는지 구조적으로 설명할 수 있다.

> **개요** 웹 애플리케이션 취약점은 매년 새로운 이름의 사고로 보도되지만, 그 뿌리를 파보면 대부분 같은 실수로 수렴합니다. OWASP Top 10을 축으로 대표 취약점들의 발생 원리를 살펴보고, "왜 반복되는가"라는 구조적인 질문에 답합니다.

---

## 1. OWASP Top 10이란 무엇이고 왜 신뢰받는가

> **개요** OWASP이라는 조직의 성격과, Top 10 문서가 어떤 방식으로 만들어지기에 특정 회사의 의견이 아니라 업계 표준으로 인정받는지를 다룹니다.

<u>**OWASP(Open Worldwide Application Security Project)**</u>는 2001년 설립된 비영리 재단으로, 특정 벤더나 제품과 이해관계 없이 웹 애플리케이션 보안 지식을 공개적으로 공유하는 것을 목표로 합니다. <u>**OWASP Top 10**</u>은 이 재단이 3~4년 주기로 발표하는 문서로, 수백 개 조직이 제출한 실제 취약점 진단 데이터와 수천 명의 보안 전문가 설문을 근거로 "발생 빈도와 피해 규모가 가장 큰 10개 취약점 카테고리"를 선정합니다.

> **참고** Top 10은 개별 버그 하나하나의 목록이 아니라 <u>**카테고리**</u>의 목록입니다. 예를 들어 "SQL 인젝션"은 더 큰 범주인 "Injection(인젝션)" 카테고리에 속한 대표 사례 중 하나입니다. 이 구조 때문에 문서가 몇 년마다 개정되어도 카테고리의 근본 원리는 거의 바뀌지 않습니다.

> **Q.** 정부나 특정 대기업이 만든 표준도 아닌데, 왜 OWASP Top 10이 사실상 업계 표준으로 통용될까요?
> **A.** 이해관계 없는 커뮤니티가 실제 침해사고 데이터를 근거로 만든 합의이기 때문입니다. 특정 벤더가 만든 기준이라면 "자사 제품이 막아주는 취약점만 강조한다"는 의심을 받기 쉽지만, OWASP은 비영리이고 데이터 출처와 선정 방법론을 공개합니다. 그 결과 PCI-DSS 같은 결제 보안 규정이나 다수 기업의 내부 보안 점검 기준이 OWASP Top 10을 직접 인용합니다.

## 2. 모든 취약점의 공통 뿌리 — 신뢰 경계의 붕괴

> **개요** 서로 달라 보이는 취약점들이 사실은 같은 원리, 즉 "신뢰할 수 없는 데이터를 신뢰해버리는 지점"에서 발생한다는 것을 짚습니다.

애플리케이션은 항상 <u>**신뢰 경계(Trust Boundary)**</u>를 사이에 두고 동작합니다. 경계 바깥(사용자 입력, 외부 API 응답, 업로드된 파일)에서 들어온 데이터는 원칙적으로 신뢰할 수 없고, 경계 안쪽(서버 내부 로직, 데이터베이스 쿼리 구조, HTML 문서 구조)은 개발자가 의도한 대로만 움직여야 합니다.

![신뢰 경계와 SQL 인젝션이 발생하는 요청 흐름](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part9_보안/9-1_웹_애플리케이션_보안_기초/assets/diagram1_sql_injection_flow.svg)

> **해석** 위 다이어그램의 위쪽 경로(정상 요청)는 사용자 입력이 "값"으로만 취급되어 쿼리 구조 바깥에 머무릅니다. 아래쪽 경로(공격 요청)는 입력값이 이스케이프 없이 쿼리 문자열에 그대로 이어붙여지면서, 원래는 "값"이어야 할 데이터가 "쿼리 구조를 바꾸는 코드"로 승격됩니다. 이 승격이 일어나는 지점이 바로 신뢰 경계가 무너지는 지점입니다.

거의 모든 인젝션 계열 취약점(SQL 인젝션, XSS, 커맨드 인젝션)은 이 한 문장으로 요약됩니다. <u>**데이터와 코드(또는 명령)를 같은 문자열 안에서 구분하지 못하면, 공격자가 데이터인 척 코드를 끼워 넣을 수 있습니다.**</u>

## 3. SQL 인젝션 — 데이터를 쿼리 구조로 착각시키기

> **개요** SQL 인젝션이 어떤 원리로 발생하며, 왜 준비된 문(Prepared Statement)이 근본적인 해결책인지를 다룹니다.

<u>**SQL 인젝션(SQL Injection)**</u>은 사용자 입력이 SQL 쿼리 문자열에 이스케이프 없이 그대로 삽입될 때, 입력값에 포함된 따옴표나 연산자가 쿼리의 문법 구조 자체를 바꿔버리는 취약점입니다. 예를 들어 로그인 폼에서 아이디 입력값을 검증 없이 문자열로 이어 붙이면, 입력값 안에 포함된 작은따옴표(`'`)가 원래 문자열 값의 경계를 끝내버리고 그 뒤에 이어지는 텍스트가 SQL 문법의 일부(조건식)로 해석될 수 있습니다. 이 원리 때문에 "비밀번호를 몰라도 조건절 자체를 항상 참으로 만드는" 형태의 우회가 교과서적으로 언급됩니다.

### 실생활 예시 — 로그인 폼이 신뢰 경계를 검증하지 않을 때

> **해석** 아래 표는 "이 아이디로 저장된 비밀번호와 입력값이 같은가"를 검사하는 쿼리에 이스케이프 처리가 없을 때 벌어지는 일을 개념적으로 정리한 것입니다. 실제 침투 절차가 아니라, 왜 이런 구조가 위험한지 이해하기 위한 설명입니다.

| 상황 | 서버가 조립하는 쿼리(개념) | 결과 |
|---|---|---|
| 정상 로그인 | 아이디와 비밀번호를 값으로 대입해 일치 여부를 확인 | 값이 일치할 때만 로그인 성공 |
| 이스케이프 없는 입력 결합 | 입력값 안의 특수문자가 쿼리 문법의 일부로 해석됨 | 조건절 구조 자체가 바뀌어, 값 비교라는 원래 의도가 깨짐 |
| 준비된 문(Prepared Statement) 사용 | 쿼리 구조를 먼저 고정하고, 입력값은 오직 데이터 자리에만 바인딩 | 입력값에 어떤 특수문자가 있어도 쿼리 구조는 바뀌지 않음 |

> **원칙** SQL 인젝션의 근본적인 방어는 "특수문자를 필터링하는 것"이 아니라 <u>**준비된 문(Prepared Statement, Parameterized Query)을 사용해 쿼리 구조와 데이터를 애초에 분리하는 것**</u>입니다. 필터링은 우회 방법이 계속 발견되는 대응 방식이지만, 구조 분리는 입력값이 무엇이든 쿼리 문법에 관여할 수 없게 만듭니다. 대부분의 ORM(Object-Relational Mapping)이 내부적으로 이 방식을 강제하는 이유가 여기에 있습니다.

## 4. XSS — 데이터를 실행 가능한 스크립트로 착각시키기

> **개요** 크로스 사이트 스크립팅이 발생하는 원리와 저장형·반사형·DOM 기반의 차이, 그리고 출력 이스케이프가 왜 근본 대책인지를 다룹니다.

<u>**크로스 사이트 스크립팅(Cross-Site Scripting, XSS)**</u>은 사용자 입력이 이스케이프 없이 HTML 문서에 그대로 삽입되어, 브라우저가 그 입력을 "표시할 텍스트"가 아니라 "실행할 스크립트 태그"로 해석하는 취약점입니다. 원리는 SQL 인젝션과 동일합니다. 데이터와 코드를 구분하는 경계가 무너진 것입니다. 다만 그 경계가 데이터베이스 쿼리 문법이 아니라 <u>**HTML/JavaScript 문법**</u>이라는 점만 다릅니다.

- <u>**저장형(Stored) XSS**</u>: 악성 스크립트가 게시글·댓글처럼 서버에 저장되어, 그 페이지를 여는 모든 사용자에게 실행됩니다. 피해 범위가 가장 넓습니다.
- <u>**반사형(Reflected) XSS**</u>: URL 파라미터 등 요청 값이 그대로 응답 페이지에 반영되어 실행됩니다. 피해자가 조작된 링크를 직접 클릭해야 발동합니다.
- <u>**DOM 기반(DOM-based) XSS**</u>: 서버를 거치지 않고, 브라우저 내 자바스크립트가 페이지 URL이나 사용자 입력을 직접 DOM에 삽입할 때 발생합니다.

> **Q.** SQL 인젝션은 준비된 문으로 막을 수 있는데, XSS는 왜 여전히 Top 10에 남아 있을까요?
> **A.** 출력해야 할 위치가 훨씬 다양하기 때문입니다. SQL은 "쿼리 안"이라는 한 곳만 지키면 되지만, 웹 페이지는 HTML 본문, HTML 속성, JavaScript 문자열, CSS, URL 등 데이터가 삽입되는 위치마다 이스케이프 규칙이 다릅니다. 어느 한 위치라도 이스케이프를 빠뜨리면 취약점이 생기므로, 실무에서는 위치별 이스케이프를 자동화하는 템플릿 엔진과, 스크립트 실행 자체를 제한하는 <u>**콘텐츠 보안 정책(Content Security Policy, CSP)**</u>을 함께 사용해 이중으로 방어합니다.

## 5. 인증·접근 제어 취약점 — "누구인지"와 "무엇을 할 수 있는지"를 혼동하기

> **개요** 인증(Authentication)과 인가(Authorization)를 구분하지 못할 때 발생하는 취약점, 특히 접근 제어 취약점이 2021년 OWASP Top 10에서 왜 1위에 올랐는지를 다룹니다.

<u>**인증(Authentication)**</u>은 "이 사용자가 누구인가"를 확인하는 절차이고, <u>**인가(Authorization)**</u>는 "이 사용자가 이 자원에 접근할 권한이 있는가"를 확인하는 절차입니다. 많은 사고는 인증에는 성공했지만 인가 검증이 빠져서 발생합니다. 대표적으로 <u>**안전하지 않은 직접 객체 참조(Insecure Direct Object Reference, IDOR)**</u>는 로그인한 사용자가 자신의 게시글 URL에 있는 ID 번호를 다른 값으로 바꾸기만 해도, 서버가 "이 게시글이 정말 이 사용자 것인가"를 재검증하지 않아 타인의 데이터에 접근하게 되는 문제입니다.

> **참고** 2021년 개정판 OWASP Top 10에서 <u>**Broken Access Control(취약한 접근 제어)**</u>이 처음으로 1위에 올랐습니다. 이는 개발자들이 "로그인 여부"는 꼼꼼히 검사하면서도, 로그인한 사용자가 "그 특정 자원"에 접근할 권한이 있는지는 매 요청마다 서버에서 재확인하지 않는 실수가 실제로 가장 흔했다는 것을 보여줍니다.

## 6. 왜 같은 취약점이 매년 반복해서 등장하는가

> **개요** 이 강의의 핵심 질문에 대한 답으로, 언어와 프레임워크가 계속 발전하는데도 같은 유형의 취약점이 반복되는 구조적인 이유를 정리합니다.

![OWASP 취약점이 반복되는 구조적 순환](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part9_보안/9-1_웹_애플리케이션_보안_기초/assets/diagram2_recurring_cycle.svg)

> **Q.** 새로운 언어와 프레임워크가 계속 나오고, 자동으로 이스케이프를 처리해주는 도구도 많아졌는데, 왜 SQL 인젝션과 XSS는 10년 넘게 Top 10에서 사라지지 않을까요?
> **A.** 도구가 발전하는 속도보다 "신뢰 경계를 어디에 그어야 하는가"라는 사고방식이 개발자 개개인에게 전수되는 속도가 느리기 때문입니다. 새 프레임워크는 기본값(default)으로는 안전하게 설계되어 있어도, 개발자가 "성능을 위해", "레거시 코드와 맞추기 위해" 그 기본값을 우회하는 순간 같은 취약점이 그대로 재현됩니다. 또한 매년 새로운 개발자가 유입되고, 마감 압박 속에서 "이 입력은 내부 관리자만 쓰니까 괜찮겠지"라는 가정이 반복적으로 생깁니다. 즉 기술이 아니라 <u>**신뢰를 검증 없이 확장하는 습관**</u>이 반복되는 것이므로, 프레임워크를 바꾸는 것만으로는 근본 원인이 사라지지 않습니다.

> **팁** 코드 리뷰에서 "이 값은 어디서 왔는가"를 먼저 묻는 습관이 취약점 예방에 가장 효과적입니다. 값의 출처가 사용자 입력이든, 외부 API 응답이든, 신뢰 경계 바깥이라면 그 값이 쿼리 구조·HTML 구조·셸 명령 구조에 영향을 줄 수 있는지부터 점검해야 합니다.

## 요약

> **핵심**
> - OWASP Top 10은 특정 취약점이 아니라 실제 침해사고 데이터에 근거한 **취약점 카테고리** 목록이며, 이해관계 없는 커뮤니티가 만들었기 때문에 업계 표준으로 통용됩니다.
> - SQL 인젝션, XSS 등 인젝션 계열 취약점의 공통 원리는 **신뢰 경계 붕괴** — 신뢰할 수 없는 입력이 데이터가 아니라 코드/명령 구조의 일부로 해석되는 것입니다.
> - 인증·접근 제어 취약점은 "누구인지 확인하는 것"과 "무엇을 할 수 있는지 매 요청마다 재검증하는 것"을 혼동할 때 발생합니다.
> - 같은 취약점이 매년 반복되는 이유는 도구의 한계가 아니라, **신뢰 경계를 검증 없이 확장하는 개발 습관이 새로운 기술로 자동 전수되지 않기 때문**입니다.
