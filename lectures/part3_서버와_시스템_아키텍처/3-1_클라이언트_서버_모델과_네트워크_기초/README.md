# Part 3. 서버와 시스템 아키텍처

## 3-1. 클라이언트-서버 모델과 네트워크 기초 — HTTP 요청은 어떤 계층을 거치는가

> **학습 목표**
> - 클라이언트-서버 모델의 정의와 P2P(Peer-to-Peer) 모델과의 구조적 차이를 설명할 수 있다.
> - OSI 7계층과 TCP/IP 4계층의 대응 관계, 그리고 각 계층에서 데이터가 캡슐화되는 원리를 설명할 수 있다.
> - TCP 3-way handshake의 각 단계와 그것이 반드시 세 번이어야 하는 이유를 설명할 수 있다.

> **개요** 배달 앱에서 "주문하기" 버튼을 누르는 순간 만들어지는 HTTP 요청 한 건이, 실제로는 여러 개의 논리적 계층을 순서대로 통과하며 각 계층마다 정보를 덧붙인 뒤에야 물리적인 전기 신호가 되어 서버까지 도달합니다. 이 강의에서는 그 계층 구조와, 연결이 시작되는 순간 벌어지는 TCP 3-way handshake를 다룹니다.

## 1. 클라이언트-서버 모델이란

> **개요** 왜 대부분의 웹·앱 서비스가 "요청하는 쪽"과 "응답하는 쪽"이 분리된 구조를 갖는지, P2P 모델과 비교해 살펴봅니다.

> **Q.** 사용자의 스마트폰끼리 직접 연결해서 데이터를 주고받으면 안 될까요? 왜 굳이 중간에 서버를 둘까요?
> **A.** 안 되는 것은 아니지만, 그렇게 설계된 시스템을 P2P(Peer-to-Peer)라고 부릅니다. 클라이언트-서버 모델은 Kurose와 Ross의 『컴퓨터 네트워킹』에서 정의하듯, 항상 켜져 있고(always-on) 고정된 주소를 가진 <u>**서버**</u>가 다수의 <u>**클라이언트**</u>로부터 오는 요청을 처리하는 비대칭 구조입니다. 배달 앱의 경우 사용자의 스마트폰(클라이언트)은 언제 꺼질지 알 수 없고 IP 주소도 계속 바뀌지만, 배달 앱 회사의 서버는 항상 같은 주소로 접속 가능해야 재고·주문·결제 상태를 일관되게 관리할 수 있습니다. 이 "누가 항상 그 자리에 있는가"라는 비대칭성이 클라이언트-서버 구조를 선택하는 근본 이유입니다.

| 구분 | 클라이언트-서버 | P2P |
|---|---|---|
| 역할 | 서버(고정, 응답 전담)와 클라이언트(가변, 요청 전담)가 분리됨 | 모든 노드가 클라이언트이자 서버 역할을 동시에 수행 |
| 가용성 요구 | 서버가 항상 켜져 있어야 함 | 특정 노드가 꺼져도 다른 노드가 대체 가능 |
| 확장 방식 | 서버 자원(하드웨어)을 늘려야 함 | 참여 노드가 늘수록 오히려 자원이 늘어남 |
| 대표 사례 | 웹 서비스, 배달 앱, 은행 시스템 | 토렌트, 초기 블록체인 네트워크 |

> **참고** 실무에서 완전한 P2P 시스템은 드뭅니다. 대부분의 상용 서비스는 사용자 데이터의 일관성·보안·과금을 중앙에서 통제해야 하므로 클라이언트-서버 구조를 기본으로 채택합니다. 이후 Part 3~7에서 다루는 서버 확장, 로드 밸런싱, 인증 개념은 모두 이 비대칭 구조를 전제로 합니다.

## 2. OSI 7계층과 TCP/IP 4계층 — 요청은 계층을 타고 내려간다

> **개요** 국제표준 참조 모델인 OSI 7계층과, 실제 인터넷에서 쓰이는 TCP/IP 4계층이 서로 어떻게 대응하는지, 그리고 각 계층을 통과할 때마다 데이터에 무엇이 덧붙는지(캡슐화) 살펴봅니다.

OSI(Open Systems Interconnection) 모델은 국제표준화기구(ISO)가 1984년 발표한 참조 모델로, 네트워크 통신을 7개의 독립된 계층으로 나눕니다. Andrew Tanenbaum의 『Computer Networks』는 이 계층화의 목적을 "각 계층이 하위 계층의 구현 방식을 몰라도 정해진 인터페이스만 지키면 동작하게 하는 것"이라고 설명합니다. 반면 실제 인터넷 프로토콜 스택은 이보다 단순한 TCP/IP 4계층(또는 5계층) 모델을 따릅니다.

![OSI 7계층과 TCP/IP 4계층 대응 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part3_서버와_시스템_아키텍처/3-1_클라이언트_서버_모델과_네트워크_기초/assets/diagram1_layers.svg)

> **해석** 왼쪽 OSI 7계층 중 응용·표현·세션 계층은 TCP/IP에서 하나의 <u>**응용 계층(Application Layer)**</u>으로 통합됩니다. 프로그래머가 실제로 코드를 작성하는 계층이 바로 여기입니다. 데이터 링크·물리 계층 역시 TCP/IP에서는 하나의 <u>**링크 계층(Link Layer)**</u>으로 묶입니다. 오른쪽 박스 안의 용어(메시지, 세그먼트, 패킷, 프레임)는 각 계층에서 데이터 단위를 부르는 이름(PDU, Protocol Data Unit)입니다.

각 계층의 역할은 다음과 같습니다.

1. **응용 계층(Application)** — HTTP, DNS, FTP 등 사용자가 실제로 사용하는 프로토콜이 동작합니다. 배달 앱의 "주문하기"는 여기서 HTTP 요청(`POST /orders`)으로 만들어집니다.
2. **전송 계층(Transport)** — TCP 또는 UDP가 동작하며, 데이터를 세그먼트 단위로 나누고 포트 번호로 어떤 프로그램에 전달할지 결정합니다.
3. **인터넷 계층(Internet)** — IP가 동작하며, 세그먼트에 출발지·목적지 IP 주소를 붙여 패킷으로 만들고 라우팅 경로를 결정합니다.
4. **링크 계층(Link)** — 이더넷, Wi-Fi 프로토콜이 동작하며, 패킷에 MAC 주소를 붙여 프레임으로 만들고 실제 물리 매체(전선, 전파)로 전송합니다.

> **원칙** <u>**캡슐화(Encapsulation)**</u>는 상위 계층에서 내려온 데이터를 그대로 감싸고, 자신의 계층에서 필요한 헤더만 앞에 덧붙이는 원칙입니다. 응용 계층의 HTTP 메시지는 전송 계층에서 TCP 헤더가 붙어 세그먼트가 되고, 인터넷 계층에서 IP 헤더가 붙어 패킷이 되며, 링크 계층에서 이더넷 헤더가 붙어 프레임이 됩니다. 서버에 도착하면 반대 순서로 헤더를 하나씩 벗겨내는 <u>**역캡슐화(Decapsulation)**</u>가 일어납니다. 이 원칙 덕분에 링크 계층이 이더넷에서 Wi-Fi로 바뀌어도 상위 계층(HTTP, TCP)의 코드는 전혀 수정할 필요가 없습니다.

> **Q.** 왜 계층을 하나로 합치지 않고 굳이 여러 개로 쪼갤까요?
> **A.** 관심사를 분리하기 위함입니다. 웹 개발자는 HTTP 요청·응답 형식만 신경 쓰면 되고, 그 아래에서 패킷이 어떤 경로로 라우팅되는지, 신호가 구리선을 타는지 광섬유를 타는지는 몰라도 됩니다. Part 1-1에서 본 SDLC의 단계 분리와 같은 이유입니다. 계층 하나만 교체해도 나머지 계층은 그대로 재사용할 수 있습니다.

## 3. 전송 계층: TCP와 3-way Handshake

> **개요** 전송 계층의 두 프로토콜(TCP, UDP)이 어떤 문제를 다르게 풀고 있는지, 그리고 TCP가 연결을 맺을 때 정확히 세 번의 메시지를 주고받는 이유를 다룹니다.

TCP(Transmission Control Protocol)와 UDP(User Datagram Protocol)는 같은 전송 계층에 있지만 상반된 설계를 택했습니다.

| 구분 | TCP | UDP |
|---|---|---|
| 연결 방식 | 연결 지향(connection-oriented) — 핸드셰이크 필요 | 비연결형(connectionless) — 바로 전송 |
| 신뢰성 | 순서 보장, 손실 시 재전송 보장 | 보장 없음 (손실되면 그냥 사라짐) |
| 속도 | 신뢰성 확인 절차만큼 지연 발생 | 지연이 적음 |
| 대표 사용처 | HTTP, 파일 전송, 결제 API | 실시간 스트리밍, 온라인 게임, DNS 조회 |

배달 앱의 주문·결제 API가 TCP를 쓰는 이유는 명확합니다. 주문 데이터 하나라도 유실되면 안 되기 때문입니다. 반대로 실시간 배달 기사 위치 갱신처럼 "약간 유실되어도 다음 갱신으로 덮어쓰이면 되는" 데이터는 UDP 기반 프로토콜을 쓰기도 합니다.

TCP가 신뢰성을 확보하는 첫 단계가 바로 <u>**3-way Handshake**</u>입니다. 데이터를 주고받기 전에 클라이언트와 서버가 서로 "나는 연결을 시작할 준비가 되었다"는 사실을 확인하는 절차입니다.

![TCP 3-way Handshake 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part3_서버와_시스템_아키텍처/3-1_클라이언트_서버_모델과_네트워크_기초/assets/diagram2_handshake.svg)

> **해석** 클라이언트는 자신의 초기 시퀀스 번호 `x`를 담아 SYN을 보냅니다. 서버는 이를 받았다는 확인(`ack=x+1`)과 자신의 초기 시퀀스 번호 `y`를 함께 담아 SYN-ACK으로 응답합니다. 마지막으로 클라이언트는 서버의 시퀀스 번호를 확인했다는 뜻으로 `ack=y+1`을 담은 ACK을 보냅니다. 이 시점부터 연결은 ESTABLISHED 상태가 되어 실제 데이터(HTTP 요청 본문)를 주고받기 시작합니다.

> **Q.** 왜 2번이 아니라 정확히 3번을 주고받아야 할까요?
> **A.** 서버가 보낸 SYN-ACK이 클라이언트에게 실제로 도착했는지, 서버 입장에서는 확인할 방법이 없기 때문입니다. 만약 핸드셰이크를 2단계(SYN → SYN-ACK)에서 끝낸다면, 서버는 클라이언트가 SYN-ACK을 받지 못했는데도 연결이 성립했다고 착각하고 자원(메모리, 소켓)을 미리 할당해 버립니다. 세 번째 ACK은 "나는 당신의 응답까지 확실히 받았다"는 사실을 서버에게 알려주는 마지막 확인이며, 이 확인이 도착한 뒤에야 서버는 안심하고 연결을 위한 자원을 확정합니다.

> **참고** 연결을 끊을 때는 반대로 4-way handshake(FIN, ACK, FIN, ACK)가 일어납니다. 종료는 연결보다 한 단계가 더 필요한데, TCP는 양방향 통신이므로 클라이언트가 보낼 데이터가 끝났다는 것과 서버가 보낼 데이터가 끝났다는 것을 각각 별도로 확인해야 하기 때문입니다.

## 4. 대역폭-지연 곱 — 네트워크 성능을 수치로 이해하기

> **개요** 네트워크가 "얼마나 빠른가"를 이야기할 때 대역폭 하나만으로는 부족한 이유를, 대역폭-지연 곱(Bandwidth-Delay Product)이라는 공식으로 설명합니다.

네트워크 성능을 나타내는 대표적인 두 지표는 <u>**대역폭(Bandwidth, $B$)**</u>과 <u>**왕복 지연시간(Round-Trip Time, $RTT$)**</u>입니다. 대역폭은 초당 전송 가능한 최대 데이터량(bps)이고, RTT는 패킷이 상대방까지 갔다가 돌아오는 데 걸리는 시간입니다. 이 둘을 곱한 값이 대역폭-지연 곱입니다.

$$BDP = B \times RTT$$

$BDP$는 "상대방의 확인 응답을 기다리지 않고, 이미 네트워크 회선 위에 떠 있을 수 있는 데이터의 최대량"을 의미합니다. Kurose & Ross는 이를 "파이프의 부피"에 비유합니다. 대역폭이 파이프의 굵기라면, RTT는 파이프의 길이입니다. TCP는 이 값을 감안해 <u>**혼잡 윈도우(Congestion Window)**</u> 크기를 조절합니다. 만약 클라이언트가 서버의 ACK을 기다리기 전까지 $BDP$보다 적은 양의 데이터만 보낸다면, 파이프의 일부만 채운 채 나머지는 비워두는 셈이 되어 대역폭을 다 쓰지 못합니다.

> **팁** 지리적으로 먼 리전(예: 서울 사용자가 미국 서버에 접속)일수록 $RTT$가 커져 $BDP$도 커집니다. 이런 경우 TCP 윈도우 크기를 충분히 키우지 않으면 회선의 실제 대역폭을 다 활용하지 못하는 문제가 생깁니다. AWS나 GCP 같은 클라우드 사업자가 사용자와 가까운 리전에 서버를 두라고 권장하는 이유, 그리고 CDN(Content Delivery Network)이 존재하는 이유가 바로 이 $RTT$를 줄이기 위함입니다.

## 5. 실생활 예시 — 배달 앱 주문 버튼이 서버에 닿기까지

> **해석** 아래 표는 "주문하기" 버튼 클릭 한 번이 계층을 통과하는 과정을 정리한 것입니다.

| 계층 | 이 순간 벌어지는 일 |
|---|---|
| 응용 계층 | 앱이 주문 정보를 JSON으로 만들어 `POST /orders` HTTP 요청을 구성 |
| 전송 계층 | (연결이 없다면 먼저 3-way handshake) TCP가 요청을 세그먼트로 나누고 포트 번호(443)를 붙임 |
| 인터넷 계층 | IP가 스마트폰의 사설 IP와 서버의 공인 IP를 담아 패킷으로 만듦 |
| 링크 계층 | Wi-Fi 또는 LTE 프로토콜이 MAC 주소를 붙여 프레임으로 만들고 기지국·공유기로 전송 |
| (서버 도착) | 서버는 프레임 → 패킷 → 세그먼트 → 메시지 순으로 역캡슐화하며 최종적으로 HTTP 요청을 복원 |

> **핵심**
> - 클라이언트-서버 모델은 <u>**항상 켜져 있는 서버**</u>와 <u>**가변적인 다수의 클라이언트**</u>가 비대칭적으로 통신하는 구조이며, P2P와 달리 데이터 일관성과 접근 통제를 중앙에서 관리하기 유리합니다.
> - HTTP 요청은 응용 → 전송 → 인터넷 → 링크 계층을 순서대로 내려가며 각 계층마다 헤더가 추가되는 <u>**캡슐화**</u>를 거칩니다. 계층을 분리하는 이유는 하위 계층의 구현이 바뀌어도 상위 계층 코드를 그대로 재사용하기 위함입니다.
> - TCP는 데이터를 주고받기 전 <u>**3-way handshake(SYN → SYN-ACK → ACK)**</u>로 연결을 확립하며, 세 번째 ACK은 서버가 자원을 안전하게 할당하기 위한 마지막 확인입니다.
> - 네트워크 성능은 대역폭만으로 판단할 수 없으며, $BDP = B \times RTT$가 "확인 응답 없이 전송 가능한 데이터량"을 결정합니다.
