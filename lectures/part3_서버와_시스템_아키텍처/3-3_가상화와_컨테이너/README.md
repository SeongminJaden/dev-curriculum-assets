# Part 3. 서버와 시스템 아키텍처

## 3-3. 가상화와 컨테이너 — Docker는 가상 머신과 무엇이 다른가

> **학습 목표**
> - 하이퍼바이저 기반 가상화(Type 1/Type 2)의 구조와, 각 가상 머신이 독립된 커널을 갖는 원리를 설명할 수 있다.
> - 컨테이너 엔진이 호스트 커널을 공유하면서도 프로세스를 격리하는 원리(네임스페이스, cgroups)를 설명할 수 있다.
> - VM과 컨테이너를 자원 오버헤드, 시작 시간, 격리 수준 관점에서 비교하고 각각이 적합한 상황을 판단할 수 있다.

> **개요** 서버 한 대 위에서 여러 애플리케이션을 서로 침범하지 않게 격리해 실행하려는 목적은 가상 머신과 컨테이너가 동일합니다. 다만 "무엇을 통째로 복제하고, 무엇을 공유할 것인가"에 대한 답이 다르며, 이 차이가 곧 무게 차이로 이어집니다.

---

## 1. 가상화란 무엇인가 — 하이퍼바이저

> **개요** 물리적 하드웨어 한 대를 여러 개의 독립된 컴퓨터처럼 보이게 만드는 하이퍼바이저의 원리를 다룹니다.

<u>**가상화(Virtualization)**</u>는 하나의 물리적 하드웨어 위에 여러 개의 독립된 가상 컴퓨터를 만들어내는 기술입니다. 이를 가능하게 하는 소프트웨어 계층이 <u>**하이퍼바이저(Hypervisor)**</u>, 다른 말로 VMM(Virtual Machine Monitor)입니다. 하이퍼바이저는 실제 CPU, 메모리, 디스크를 가상의 CPU·메모리·디스크로 흉내 내어 각 가상 머신(VM)에 제공합니다. 각 VM은 이 가상 하드웨어 위에 자신만의 완전한 운영체제(Guest OS)를 처음부터 새로 설치하고 부팅합니다.

하이퍼바이저는 두 가지 유형으로 나뉩니다.

| 유형 | 설명 | 대표 사례 |
|---|---|---|
| Type 1 (Bare-metal) | 하드웨어 위에 하이퍼바이저가 직접 설치되어 동작 | VMware ESXi, Xen, KVM |
| Type 2 (Hosted) | 일반 운영체제 위에 애플리케이션처럼 하이퍼바이저를 설치 | VirtualBox, VMware Workstation |

> **Q.** VM마다 운영체제를 통째로 새로 설치하는 게 왜 무거울까요?
> **A.** 커널, 드라이버, 시스템 서비스까지 포함한 운영체제 전체가 VM 개수만큼 중복되기 때문입니다. VM 3개를 띄우면 리눅스 커널도 3벌, 부팅 절차도 3번, 커널이 관리하는 메모리도 VM마다 별도로 확보해야 합니다. 하드웨어 수준까지 완전히 격리된다는 강력한 장점이 있지만, 그 대가로 부팅에 수십 초가 걸리고 이미지 크기도 수 기가바이트에 이릅니다.

## 2. 컨테이너란 무엇인가 — 커널 공유

> **개요** 컨테이너 엔진이 운영체제 전체를 복제하지 않고도 격리를 구현하는 원리를 다룹니다.

<u>**컨테이너(Container)**</u>는 VM과 달리 자체 커널을 갖지 않습니다. 대신 호스트 운영체제의 커널을 여러 컨테이너가 공유하면서, 커널이 제공하는 격리 기능만으로 "마치 독립된 시스템처럼 보이는" 환경을 만듭니다. 이런 방식을 <u>**OS 수준 가상화(OS-level virtualization)**</u>라고 부르며, Docker가 대표적인 컨테이너 엔진입니다. 이 격리는 리눅스 커널이 제공하는 두 가지 기능으로 구현됩니다.

- <u>**네임스페이스(Namespace)**</u> — 프로세스가 "자신 말고는 아무것도 없다"고 착각하게 만드는 격리 기법입니다. PID 네임스페이스는 컨테이너 안의 프로세스가 자신을 PID 1로 보게 하고, NET 네임스페이스는 독립된 네트워크 인터페이스를 갖게 하며, MNT 네임스페이스는 독립된 파일시스템 마운트 지점을 보게 합니다. 실제로는 같은 커널 위에서 동작하지만, 커널이 "이 프로세스에게는 이 범위만 보여준다"고 필터링하는 것입니다.
- <u>**cgroups(Control Groups)**</u> — 프로세스 그룹이 사용할 수 있는 CPU 시간, 메모리, 디스크 I/O의 상한선을 강제하는 기능입니다. 컨테이너 하나가 폭주해 호스트 전체 자원을 독점하는 것을 막습니다.

> **원칙** 컨테이너의 격리는 "새로운 운영체제를 만드는 것"이 아니라 <u>**같은 커널을 보는 시야를 프로세스마다 다르게 제한하는 것**</u>입니다. 이 원칙 덕분에 컨테이너는 운영체제를 부팅할 필요 없이 애플리케이션 프로세스 하나만 시작하면 되고, 그만큼 VM보다 가볍고 빠릅니다.

## 3. VM과 컨테이너 구조 비교

> **개요** 지금까지 설명한 원리를 층 구조 그림으로 비교하고, 핵심 수치를 표로 정리합니다.

![하이퍼바이저와 컨테이너 엔진 구조 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part3_서버와_시스템_아키텍처/3-3_가상화와_컨테이너/assets/diagram1_hypervisor_vs_container.svg)

> **해석** 왼쪽 VM 구조에서는 하이퍼바이저 위에 Guest OS가 애플리케이션 개수만큼 따로 존재합니다. 오른쪽 컨테이너 구조에서는 Host OS가 단 하나이며, 그 위의 컨테이너 엔진이 애플리케이션들을 직접 격리해 실행합니다. 커널을 포함한 운영체제 계층이 통째로 반복되는지 여부가 두 구조의 근본적인 차이입니다.

| 비교 항목 | 가상 머신 (VM) | 컨테이너 |
|---|---|---|
| 커널 | VM마다 독립된 커널 | 호스트 커널을 공유 |
| 시작 시간 | 수십 초 (OS 부팅 필요) | 1초 내외 (프로세스 시작 수준) |
| 이미지 크기 | 보통 기가바이트 단위 | 보통 수십~수백 메가바이트 |
| 격리 수준 | 하드웨어 수준의 강한 격리 | 커널 리소스를 공유하는 약한 격리 |
| 밀도 (한 서버에 띄울 수 있는 개수) | 상대적으로 적음 | 상대적으로 많음 |

## 4. Docker의 구체적 구조

> **개요** 컨테이너 엔진의 대표 사례인 Docker가 이미지·컨테이너·레지스트리를 어떻게 연결하는지 다룹니다.

![Docker 아키텍처 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part3_서버와_시스템_아키텍처/3-3_가상화와_컨테이너/assets/diagram2_docker_architecture.svg)

> **해석** Docker Client는 `docker build`, `docker run` 같은 명령을 Docker Daemon에 전달합니다. Daemon은 `Dockerfile`을 읽어 <u>**이미지(Image)**</u>를 만들고, 필요하면 Docker Hub 같은 레지스트리에서 이미지를 내려받습니다(`pull`). 이미지는 애플리케이션 실행에 필요한 파일과 설정을 겹겹이 쌓은 읽기 전용 계층(Layer)들의 모음이며, 이 이미지를 실행한 결과가 <u>**컨테이너(Container)**</u>입니다. 같은 이미지로 컨테이너를 여러 개 띄워도 이미지 계층은 공유되고, 컨테이너마다 변경 사항만 별도의 쓰기 계층에 저장됩니다.

> **팁** 이미지가 계층(Layer) 구조로 되어 있다는 사실은 실무에서 빌드 속도에 직접 영향을 줍니다. `Dockerfile`에서 자주 바뀌지 않는 명령(예: 의존성 설치)을 앞쪽에, 자주 바뀌는 명령(예: 소스 코드 복사)을 뒤쪽에 배치하면 Docker가 캐시된 계층을 재사용해 빌드 시간을 크게 줄일 수 있습니다.

## 5. 왜 이 트레이드오프가 성립하는가

> **개요** 컨테이너가 가벼운 대신 잃는 것, 그리고 실무에서 이 트레이드오프를 어떻게 절충하는지 다룹니다.

컨테이너가 가벼운 이유는 커널을 공유하기 때문이고, 바로 그 이유 때문에 격리가 VM보다 약합니다. 컨테이너 하나에서 커널의 취약점을 이용한 공격이 성공하면, 이론적으로 같은 호스트의 커널을 공유하는 다른 컨테이너에도 영향을 줄 수 있습니다. VM은 각자 완전히 다른 커널을 쓰므로 이런 경로가 원천적으로 차단됩니다.

Kubernetes 같은 컨테이너 오케스트레이션 도구가 컨테이너를 표준 배포 단위로 채택한 이유는 이 가벼움에 있습니다. 짧은 시작 시간 덕분에 트래픽 급증에 맞춰 수십~수백 개의 컨테이너를 초 단위로 늘렸다 줄일 수 있고(오토스케일링), 이미지 크기가 작아 같은 서버에 더 많은 애플리케이션을 밀도 있게 배치할 수 있습니다. 이는 Part 3-2에서 다룬 마이크로서비스 아키텍처가 실제로 운영 가능해진 배경이기도 합니다. 서비스 수십 개를 VM 단위로 각각 띄운다면 자원 낭비가 감당하기 어려운 수준이 되지만, 컨테이너 단위라면 현실적입니다.

> **참고** 격리와 가벼움을 동시에 얻으려는 절충안도 존재합니다. gVisor나 Kata Containers 같은 기술은 컨테이너마다 경량화된 커널 계층이나 샌드박스를 추가로 두어, VM에 준하는 격리 수준을 컨테이너에 가까운 가벼움으로 제공하려 시도합니다. 다만 이런 절충안은 이 강의의 범위를 넘는 심화 주제입니다.

## 6. 실생활 예시 — 서버 한 대에서 개발·스테이징·운영 환경 분리하기

> **해석** 아래 표는 같은 물리 서버 한 대에서 서로 다른 환경 세 개를 격리해 운영해야 할 때, VM과 컨테이너를 각각 선택했을 때의 차이를 보여줍니다.

| 상황 | VM으로 분리 | 컨테이너로 분리 |
|---|---|---|
| 환경 3개(개발/스테이징/운영) 준비 | Guest OS 3벌을 설치·부팅, 디스크 수십 GB 소모 | 이미지 레이어를 공유하며 컨테이너 3개만 실행, 디스크 수백 MB 수준 |
| 신규 기능 테스트를 위해 환경 재시작 | OS 부팅까지 포함해 수십 초 소요 | 프로세스 재시작 수준으로 수 초 내 완료 |
| 커널 버전을 환경마다 다르게 유지해야 하는 경우 | 가능 (VM마다 독립된 커널) | 불가능 (호스트 커널을 공유하므로 VM 또는 별도 호스트 필요) |

## 요약

> **핵심**
> - 하이퍼바이저는 하드웨어를 가상화해 VM마다 <u>**독립된 커널을 통째로 실행**</u>시키며, 그만큼 강한 격리와 무거운 자원 소모를 함께 얻습니다.
> - 컨테이너는 <u>**호스트 커널을 공유**</u>하면서 네임스페이스로 프로세스의 시야를 제한하고 cgroups로 자원 사용량을 제한해 격리를 흉내 냅니다.
> - 이 차이 때문에 컨테이너는 VM보다 시작이 빠르고 이미지가 작지만, 커널을 공유하는 만큼 격리 수준은 상대적으로 낮습니다.
> - Kubernetes를 비롯한 현업의 오케스트레이션 도구가 컨테이너를 표준 단위로 삼는 이유는 이 <u>**가벼움이 곧 빠른 오토스케일링과 높은 서버 밀도**</u>로 이어지기 때문입니다.
