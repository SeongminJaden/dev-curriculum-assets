# Part 3. 서버와 시스템 아키텍처

## 3-4. 클라우드 컴퓨팅 모델 — IaaS·PaaS·SaaS의 경계는 어떻게 나뉘는가

> **학습 목표**
> - NIST가 정의한 클라우드 컴퓨팅의 5가지 필수 특성을 설명할 수 있다.
> - IaaS, PaaS, SaaS 각 모델에서 고객과 제공자의 책임이 어느 계층에서 나뉘는지 구체적으로 설명할 수 있다.
> - 온프레미스 대비 클라우드가 트래픽 급증에 유리한 이유를 탄력성(elasticity) 개념으로 설명할 수 있다.

> **개요** 서버를 직접 사서 데이터센터에 설치하던 시절과 달리, 오늘날 대부분의 서비스는 클라우드 위에서 동작합니다. 이 강의에서는 클라우드가 무엇을 대신 관리해주는지, 그리고 IaaS·PaaS·SaaS의 경계가 "책임을 어디까지 넘기는가"로 어떻게 갈리는지를 다룹니다.

---

## 1. 클라우드 컴퓨팅의 정의

> **개요** 미국 국립표준기술연구소(NIST)가 정의한 클라우드 컴퓨팅의 공식 정의와 5가지 특성을 살펴봅니다.

미국 국립표준기술연구소(NIST)는 클라우드 컴퓨팅을 "최소한의 관리 노력이나 서비스 제공자와의 상호작용만으로, 설정 가능한 컴퓨팅 자원(네트워크, 서버, 스토리지, 애플리케이션 등)의 공유 풀에 언제 어디서나 편리하게 온디맨드로 접근할 수 있게 하는 모델"로 정의합니다(NIST SP 800-145). 이 정의는 다음 5가지 특성으로 구체화됩니다.

- <u>**온디맨드 셀프서비스(On-demand self-service)**</u> — 사람과의 상담 없이 콘솔이나 API 호출만으로 서버를 즉시 확보할 수 있습니다.
- <u>**광범위한 네트워크 접근(Broad network access)**</u> — 표준 인터넷 프로토콜로 어디서든 접근 가능합니다.
- <u>**자원 풀링(Resource pooling)**</u> — 여러 고객이 물리적 자원을 가상화된 형태로 나눠 씁니다.
- <u>**신속한 탄력성(Rapid elasticity)**</u> — 필요에 따라 자원을 빠르게 늘리거나 줄일 수 있습니다.
- <u>**측정 가능한 서비스(Measured service)**</u> — 사용한 만큼만 자동으로 계량되어 과금됩니다.

> **Q.** 회사가 직접 서버를 사서 운영하는 온프레미스(On-premise)와 클라우드는 근본적으로 무엇이 다를까요?
> **A.** 온프레미스는 서버 구매부터 설치, 전력·냉각, 네트워크 배선, 장애 시 부품 교체까지 모든 계층을 회사가 직접 책임집니다. 클라우드는 이 중 상당 부분을 클라우드 사업자(AWS, GCP, Azure 등)에게 위임하고, 회사는 위임하지 않은 나머지 계층만 관리합니다. 어디까지 위임하느냐가 곧 IaaS·PaaS·SaaS를 가르는 기준입니다.

## 2. 책임 분리 모델 — 누가 무엇을 관리하는가

> **개요** 애플리케이션부터 네트워킹까지 9개 계층을 놓고, 온프레미스·IaaS·PaaS·SaaS 각각에서 고객과 제공자의 관리 경계가 어디서 그어지는지 살펴봅니다.

![클라우드 서비스 모델별 책임 분리 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part3_서버와_시스템_아키텍처/3-4_클라우드_컴퓨팅_모델/assets/diagram1_responsibility_model.svg)

> **해석** 위 그림의 각 열은 왼쪽부터 온프레미스, IaaS, PaaS, SaaS를 나타냅니다. 파란 칸은 <u>**고객이 직접 관리**</u>하는 계층, 회색 칸은 <u>**클라우드 제공자가 관리**</u>하는 계층입니다. 오른쪽으로 갈수록 회색 칸이 늘어나는 것을 확인할 수 있습니다. 이는 고객이 신경 써야 할 범위가 점점 줄어든다는 뜻이며, 동시에 고객이 직접 통제할 수 있는 범위도 함께 줄어든다는 뜻이기도 합니다.

---

## 3. IaaS — Infrastructure as a Service

> **개요** 인프라만 빌리고 그 위의 모든 것을 고객이 직접 구성하는 모델을 다룹니다.

<u>**IaaS**</u>는 가상화된 서버, 스토리지, 네트워크 같은 인프라 자원만 서비스 형태로 제공합니다. 대표 사례는 AWS EC2, GCP Compute Engine, Azure Virtual Machines입니다. 제공자는 물리 서버, 가상화 계층, 데이터센터의 전력·냉각·네트워크 배선까지 책임지고, 고객은 그 위에 자신이 원하는 운영체제를 설치하고, 미들웨어와 런타임을 구성하고, 애플리케이션을 배포합니다. 즉 온프레미스에서 하던 일 중 "물리적인 하드웨어를 직접 사고 유지보수하는 부분"만 빠진 형태입니다.

- 고객이 관리: OS, 미들웨어, 런타임, 데이터, 애플리케이션
- 제공자가 관리: 가상화, 서버(컴퓨트), 스토리지, 네트워킹

---

## 4. PaaS — Platform as a Service

> **개요** 애플리케이션을 실행할 플랫폼까지 제공자가 관리해주는 모델을 다룹니다.

<u>**PaaS**</u>는 IaaS의 인프라 위에 운영체제, 런타임, 미들웨어까지 미리 구성해 제공합니다. 대표 사례는 Heroku, AWS Elastic Beanstalk, Google App Engine, Google Cloud Run입니다. 고객은 서버에 접속해 패키지를 설치하거나 OS를 업데이트할 필요 없이, 애플리케이션 코드를 올리기만 하면 됩니다. 트래픽에 따른 스케일링, 로드 밸런싱, 런타임 패치까지 제공자가 자동으로 처리하는 경우가 많습니다.

- 고객이 관리: 데이터, 애플리케이션 코드
- 제공자가 관리: 런타임, 미들웨어, OS, 가상화, 서버, 스토리지, 네트워킹

> **참고** PaaS는 관리 부담을 크게 줄여주는 대신 유연성을 제한합니다. 특정 런타임 버전이나 미들웨어 설정이 플랫폼에서 지원하지 않으면, 고객이 임의로 커스터마이징하기 어렵습니다. "관리를 덜 하는 대가로 자유도를 덜 갖는다"는 절충이 여기서도 나타납니다.

---

## 5. SaaS — Software as a Service

> **개요** 애플리케이션 자체를 완성된 형태로 제공받는 모델을 다룹니다.

<u>**SaaS**</u>는 완성된 애플리케이션을 웹 브라우저나 클라이언트 앱을 통해 그대로 사용하는 모델입니다. Gmail, Notion, Salesforce, Slack이 대표 사례입니다. 고객은 인프라, 런타임, 심지어 애플리케이션 코드에 대해서도 전혀 신경 쓸 필요가 없고, 자신의 데이터와 설정(계정, 권한, 워크스페이스 구성)만 관리합니다.

- 고객이 관리: 데이터, 사용자 설정
- 제공자가 관리: 애플리케이션, 런타임, 미들웨어, OS, 가상화, 서버, 스토리지, 네트워킹 전체

| 구분 | IaaS | PaaS | SaaS |
|---|---|---|---|
| 고객이 직접 다루는 범위 | OS부터 애플리케이션까지 | 애플리케이션 코드와 데이터만 | 데이터·설정만 |
| 유연성 | 매우 높음 | 중간 | 낮음 |
| 관리 부담 | 높음 | 낮음 | 매우 낮음 |
| 대표 사례 | AWS EC2, GCP Compute Engine | Heroku, Elastic Beanstalk, Cloud Run | Gmail, Notion, Salesforce |

## 6. 왜 이 구분이 실무에서 중요한가

> **개요** 책임 분리 모델이 단순한 분류를 넘어, 보안과 확장성 설계에 실제로 어떤 영향을 주는지 다룹니다.

이 구분이 실무에서 중요한 첫 번째 이유는 <u>**보안 책임(Shared Responsibility Model)**</u> 때문입니다. 클라우드 사업자들은 "제공자가 관리하는 계층의 보안은 제공자 책임, 고객이 관리하는 계층의 보안은 고객 책임"이라는 원칙을 공통적으로 채택합니다. IaaS를 쓰면서 OS 패치를 게을리하면 그 취약점으로 인한 침해는 고객 책임입니다. 반대로 SaaS를 쓰면 애플리케이션 자체의 취약점은 제공자 책임이지만, 계정 비밀번호 관리나 접근 권한 설정은 여전히 고객 책임입니다.

두 번째 이유는 <u>**탄력성(Elasticity)**</u>입니다. 온프레미스는 트래픽이 몰릴 것을 대비해 미리 최대치를 예측해 서버를 구매해야 하고, 예측이 틀리면 자원이 남아 낭비되거나 부족해 장애가 납니다. 클라우드는 API 호출 한 번으로 서버 대수를 늘리거나 줄일 수 있어, 실제 수요 곡선에 훨씬 가깝게 자원을 맞출 수 있습니다.

> **원칙** 탄력성은 "필요할 때만 자원을 켜고, 필요 없어지면 곧바로 끈다"는 원칙이며, 이것이 가능한 이유는 Part 3-3에서 다룬 컨테이너의 빠른 시작 시간과, 자원을 잘게 쪼개 여러 고객에게 나눠주는 자원 풀링 덕분입니다. 오토스케일링 그룹이 CPU 사용률 같은 지표를 감시하다가 임계치를 넘으면 자동으로 인스턴스를 추가하는 방식이 대표적인 구현입니다.

## 7. 실생활 예시 — 쇼핑몰이 블랙프라이데이 트래픽에 대응하는 과정

![온프레미스와 클라우드의 탄력적 확장 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part3_서버와_시스템_아키텍처/3-4_클라우드_컴퓨팅_모델/assets/diagram2_elasticity.svg)

> **해석** 회색 점선은 실제 트래픽(수요) 곡선입니다. 온프레미스는 미리 정해둔 고정 용량(빨간 실선)을 넘는 순간부터 장애가 발생하고, 평소에는 남는 용량만큼 자원이 낭비됩니다. 클라우드의 오토스케일링(청록 실선)은 수요 곡선을 따라 용량을 함께 늘리고 줄여, 장애와 낭비를 동시에 줄입니다.

| 상황 | 온프레미스 서버 | 클라우드(IaaS + 오토스케일링) |
|---|---|---|
| 평소 트래픽 | 미리 구매한 고정 용량 그대로 유지 (여유분 낭비) | 낮은 사용량만큼만 인스턴스 유지, 비용 절감 |
| 세일 시작 30분 전 | 수동으로 서버를 추가하려 해도 조달·설치에 시간이 걸림 | 오토스케일링 정책에 따라 API 호출만으로 인스턴스 즉시 추가 |
| 트래픽이 예측을 초과 | 준비된 용량을 초과해 응답 지연·장애 발생 | 지표(CPU, 요청 큐 길이)를 감시하며 계속 자동 증설 |
| 세일 종료 후 | 추가 구매한 서버가 계속 유휴 상태로 남아 비용 낭비 | 트래픽 감소에 맞춰 자동으로 인스턴스 축소, 사용한 만큼만 과금 |

## 요약

> **핵심**
> - NIST 정의에 따르면 클라우드 컴퓨팅은 <u>**온디맨드 셀프서비스, 광범위한 네트워크 접근, 자원 풀링, 신속한 탄력성, 측정 가능한 서비스**</u>라는 5가지 특성을 갖습니다.
> - IaaS·PaaS·SaaS의 경계는 애플리케이션부터 네트워킹까지 이어지는 계층 중 <u>**고객이 직접 관리하는 범위가 어디까지인가**</u>로 정해지며, IaaS(OS부터)·PaaS(코드와 데이터만)·SaaS(데이터·설정만) 순으로 고객의 관리 범위가 줄어듭니다.
> - 관리 범위가 줄어드는 만큼 유연성도 줄어들고, 보안 책임도 <u>**제공자가 관리하는 계층은 제공자, 고객이 관리하는 계층은 고객**</u>이라는 공유 책임 모델을 따릅니다.
> - 클라우드가 트래픽 급증 대응에 유리한 이유는 <u>**탄력성**</u>, 즉 실제 수요 곡선에 맞춰 자원을 자동으로 늘리고 줄일 수 있기 때문이며, 이는 컨테이너의 빠른 시작 시간과 자원 풀링이 뒷받침합니다.
