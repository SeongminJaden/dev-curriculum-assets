# Part 3. 서버와 시스템 아키텍처

## 3-2. 시스템 아키텍처 패턴 — 모놀리식에서 마이크로서비스로 옮겨가는 기준

> **학습 목표**
> - 모놀리식, 마이크로서비스, 이벤트 기반 아키텍처의 구조적 차이를 설명할 수 있다.
> - 각 패턴이 개발 속도, 배포 단위, 장애 격리, 데이터 일관성 측면에서 어떤 트레이드오프를 갖는지 비교할 수 있다.
> - 조직 규모와 트래픽 특성에 따라 아키텍처를 언제, 왜 전환해야 하는지 판단 기준을 세울 수 있다.

> **개요** 하나의 코드베이스로 시작한 서비스가 사용자와 개발팀이 늘어나면서 왜 여러 개의 독립된 서비스로 쪼개지는지, 그리고 그 과정에서 무엇을 얻고 무엇을 잃는지를 모놀리식·마이크로서비스·이벤트 기반이라는 세 가지 패턴으로 비교합니다.

---

## 1. 모놀리식 아키텍처

> **개요** 대부분의 서비스가 첫 출발점으로 삼는 모놀리식 구조의 정의와, 그것이 초기 단계에서 합리적인 이유를 살펴봅니다.

<u>**모놀리식(Monolithic) 아키텍처**</u>는 주문, 결제, 재고, 회원 같은 서로 다른 기능들이 하나의 코드베이스, 하나의 프로세스, 하나의 배포 단위로 묶여 있는 구조입니다. 모든 모듈이 같은 메모리 공간에서 함수 호출로 서로를 부르고, 대개 하나의 데이터베이스를 공유합니다.

> **Q.** 모놀리식은 "옛날 방식"이라 나쁜 걸까요?
> **A.** 아닙니다. 서비스 초기, 즉 팀 규모가 작고 도메인 경계가 아직 불확실한 시기에는 모놀리식이 오히려 합리적인 선택입니다. 함수 호출 하나로 결제와 재고 차감을 같은 트랜잭션에 묶을 수 있어 데이터 일관성을 지키기 쉽고, 배포도 한 번에 끝나며, 개발자가 코드 전체를 한 저장소에서 검색할 수 있습니다. 아마존이나 넷플릭스도 초기에는 모놀리식으로 시작했습니다. 문제는 회사가 커진 뒤에도 이 구조를 유지할 때 발생합니다.

| 장점 | 단점 |
|---|---|
| 트랜잭션 일관성을 함수 호출 수준에서 보장하기 쉬움 | 코드베이스가 커질수록 빌드·테스트·배포 시간이 길어짐 |
| 배포가 단순함 (결과물 하나) | 한 모듈의 버그가 프로세스 전체를 다운시킬 수 있음(장애 전파) |
| 초기 개발 속도가 빠름 | 특정 기능(예: 결제)만 트래픽이 몰려도 전체를 스케일링해야 함 |
| 코드 탐색·리팩터링이 쉬움 | 팀이 커지면 같은 코드베이스에 여러 팀이 동시에 손을 대며 충돌이 잦아짐 |

## 2. 마이크로서비스 아키텍처

> **개요** 모놀리식의 한계를 팀·트래픽 규모가 커졌을 때 어떻게 해결하는지, 마이크로서비스의 정의와 설계 원칙을 다룹니다.

James Lewis와 Martin Fowler는 마이크로서비스를 "각각 독립적으로 배포 가능하고, 가벼운 메커니즘(대개 HTTP API)으로 통신하며, 비즈니스 역량을 중심으로 조직된 작은 서비스들의 집합"으로 정의합니다. 핵심은 단순히 "코드를 여러 서버에 나눠 담는 것"이 아니라, 각 서비스가 <u>**자신의 데이터베이스를 소유**</u>하고 <u>**독립적으로 배포**</u>될 수 있어야 한다는 점입니다.

이 경계를 어디에 그을지는 도메인 주도 설계(DDD)의 <u>**바운디드 컨텍스트(Bounded Context)**</u> 개념을 따르는 경우가 많습니다. "주문"과 "결제"는 서로 다른 팀이 서로 다른 언어(주문 팀은 "취소", 결제 팀은 "환불")로 부르는 별개의 개념 경계이므로, 이 경계를 그대로 서비스 경계로 삼습니다.

> **원칙** <u>**콘웨이의 법칙(Conway's Law)**</u>은 "시스템을 설계하는 조직은, 그 조직의 의사소통 구조를 그대로 닮은 설계를 만들어낸다"는 원칙입니다. 팀이 주문팀·결제팀·재고팀으로 나뉘어 있다면 시스템도 자연스럽게 그 경계를 따라 나뉘는 것이 유리합니다. 반대로 조직 구조와 어긋나게 서비스를 쪼개면, 배포 하나를 위해 여러 팀이 매번 조율해야 하는 비효율이 생깁니다. 마이크로서비스로의 전환이 기술 문제이면서 동시에 조직 설계 문제인 이유가 여기 있습니다.

| 장점 | 단점 |
|---|---|
| 서비스별 독립 배포 — 결제 팀이 재고 팀을 기다리지 않고 배포 가능 | 서비스 간 호출이 네트워크를 타므로 지연·장애 지점이 늘어남 |
| 트래픽이 몰리는 서비스만 선택적으로 스케일링 가능 | 여러 데이터베이스에 걸친 트랜잭션 일관성 보장이 어려움 |
| 장애가 한 서비스 안에 격리됨(전체 다운 방지) | 분산 시스템 특유의 디버깅·모니터링 복잡도 증가 |
| 팀마다 다른 언어·기술 스택 선택 가능 | 서비스 수만큼 배포 파이프라인, 로깅, 인증 체계를 관리해야 함 |

## 3. 이벤트 기반 아키텍처

> **개요** 마이크로서비스끼리 서로 직접 호출하는 방식(동기)과, 메시지 브로커를 거쳐 통신하는 방식(비동기)의 차이를 다룹니다.

마이크로서비스로 쪼갠다고 해서 통신 방식까지 자동으로 정해지는 것은 아닙니다. 서비스 A가 서비스 B의 REST API를 직접 호출하고 응답을 기다리는 방식을 <u>**동기(synchronous) 통신**</u>이라 하고, 서비스 A가 "주문이 생성됨"이라는 이벤트를 메시지 브로커(Kafka, RabbitMQ 등)에 발행(publish)하기만 하고 관심 있는 서비스들이 알아서 구독(subscribe)해 처리하는 방식을 <u>**이벤트 기반(event-driven) 아키텍처**</u>라 합니다.

![동기 호출과 이벤트 기반 통신 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part3_서버와_시스템_아키텍처/3-2_시스템_아키텍처_패턴/assets/diagram2_event_driven.svg)

> **해석** 위쪽 동기 호출 방식에서는 주문 서비스가 결제 서비스의 응답을, 결제 서비스가 다시 알림 서비스의 응답을 기다립니다. 체인 중 하나라도 느려지거나 장애가 나면 그 앞 단계 전체가 블로킹됩니다. 아래쪽 이벤트 기반 방식에서는 주문 서비스가 메시지 브로커에 이벤트만 발행하고 곧바로 자신의 작업을 끝냅니다. 결제 서비스와 알림 서비스는 각자의 속도로 이벤트를 가져가 처리하며, 서로의 존재나 상태를 몰라도 됩니다.

> **참고** 이벤트 기반 구조는 서비스 간 결합도를 크게 낮추지만 대가가 있습니다. "결제가 완료된 시점에 재고가 정확히 얼마나 남아 있었는가"를 즉시 알 수 없는 <u>**최종적 일관성(Eventual Consistency)**</u>을 받아들여야 하고, 이벤트가 여러 서비스를 거쳐 처리되는 과정을 추적하려면 분산 트레이싱 같은 별도 도구가 필요합니다.

## 4. 세 패턴의 구조 비교

> **개요** 지금까지 다룬 세 패턴을 배포 단위와 데이터 소유권 관점에서 그림으로 정리합니다.

![모놀리식과 마이크로서비스 구조 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part3_서버와_시스템_아키텍처/3-2_시스템_아키텍처_패턴/assets/diagram1_monolith_vs_msa.svg)

> **해석** 왼쪽 모놀리식은 여러 모듈이 하나의 큰 박스(하나의 배포 단위) 안에 있고, 하나의 데이터베이스를 공유합니다. 오른쪽 마이크로서비스는 API 게이트웨이 뒤에 서비스별로 독립된 박스와 데이터베이스가 존재합니다. 배포 단위의 개수가 곧 "무엇을 한 번에 배포해야 하는가"의 단위이기도 합니다.

| 비교 항목 | 모놀리식 | 마이크로서비스 | 이벤트 기반 |
|---|---|---|---|
| 배포 단위 | 전체 애플리케이션 1개 | 서비스 개수만큼 | 서비스 개수만큼 (+ 브로커) |
| 데이터 일관성 | 강한 일관성(단일 트랜잭션) | 서비스별로 분리, 조율 필요 | 최종적 일관성이 기본 |
| 장애 격리 | 낮음 (전체 다운 위험) | 높음 (서비스 단위로 격리) | 매우 높음 (발행자·구독자 분리) |
| 개발 초기 속도 | 빠름 | 느림 (인프라 구축 비용) | 느림 (브로커 운영 비용) |
| 운영 복잡도 | 낮음 | 높음 | 매우 높음 (추적·순서 보장 문제) |
| 스케일링 단위 | 전체 | 서비스 단위로 선택적 스케일링 | 서비스 단위 + 브로커 파티션 |

## 5. 언제 전환해야 하는가

> **개요** "마이크로서비스가 더 좋아 보이니 무조건 전환한다"가 아니라, 실제로 어떤 신호가 나타났을 때 전환을 고려해야 하는지 다룹니다.

전환을 고려할 만한 신호는 대체로 다음과 같습니다.

- 여러 팀이 같은 코드베이스에 배포를 걸어 놓고 서로의 배포를 기다리는 일이 반복될 때 (배포 병목)
- 특정 모듈(예: 결제)만 트래픽이 몰리는데, 전체 애플리케이션을 통째로 스케일링해야 할 때 (자원 낭비)
- 한 모듈의 버그가 무관한 다른 기능까지 다운시키는 사고가 반복될 때 (장애 격리 필요)
- 팀 조직이 이미 도메인별로 나뉘어 있어, 코드 경계와 조직 경계를 일치시키는 것이 자연스러울 때 (콘웨이의 법칙)

> **팁** 전환은 한 번에 전체를 다시 쓰는 방식보다, <u>**교살자 무화과 패턴(Strangler Fig Pattern)**</u>을 흔히 씁니다. 모놀리식 앞에 라우팅 계층을 두고, 기능을 하나씩 새로운 마이크로서비스로 옮기면서 라우팅 규칙만 바꿉니다. 옮겨진 기능은 모놀리식 쪽 코드를 지우고, 이 과정을 반복하면 결국 모놀리식이 무화과나무를 감싸며 자라는 교살자 식물처럼 서서히 사라집니다. 이렇게 하면 서비스 중단 없이 점진적으로 구조를 바꿀 수 있습니다.

## 6. 실생활 예시 — 쇼핑몰이 트래픽 급증에 대응하는 과정

> **해석** 아래 표는 같은 "블랙프라이데이 트래픽 급증" 상황에서 모놀리식과 마이크로서비스가 각각 어떻게 대응하는지 비교한 것입니다.

| 상황 | 모놀리식 쇼핑몰 서버 | 마이크로서비스 쇼핑몰 |
|---|---|---|
| 결제 요청이 평소의 20배로 급증 | 애플리케이션 전체(주문·재고·회원 포함)를 동일 비율로 증설해야 함 | 결제 서비스 인스턴스만 선택적으로 증설 |
| 재고 조회 로직에 버그 발생 | 프로세스 전체가 다운되어 결제까지 불가능해짐 | 재고 서비스만 장애, 결제·주문은 계속 동작 |
| 신규 기능(쿠폰) 배포 필요 | 전체 애플리케이션을 다시 빌드·배포해야 하므로 배포 중 서비스 전체 재시작 위험 | 쿠폰 서비스만 독립적으로 배포, 다른 서비스에 영향 없음 |

## 요약

> **핵심**
> - 모놀리식은 <u>**하나의 배포 단위와 하나의 데이터베이스**</u>를 공유하는 구조로, 초기 개발 속도와 트랜잭션 일관성에 유리하지만 코드베이스가 커질수록 배포 병목과 장애 전파 문제가 커집니다.
> - 마이크로서비스는 <u>**바운디드 컨텍스트 단위로 독립 배포·독립 데이터베이스**</u>를 갖는 구조이며, 콘웨이의 법칙에 따라 조직 구조와 서비스 경계를 일치시킬 때 효과적입니다.
> - 이벤트 기반 아키텍처는 서비스 간 <u>**동기 호출을 비동기 이벤트 발행/구독으로 대체**</u>해 결합도를 낮추지만, 그 대가로 최종적 일관성과 분산 추적 복잡도를 감수해야 합니다.
> - 전환 여부는 유행이 아니라 <u>**배포 병목, 선택적 스케일링 필요성, 장애 격리 필요성, 조직 구조**</u>라는 구체적 신호로 판단해야 하며, 전환은 교살자 무화과 패턴처럼 점진적으로 진행하는 것이 안전합니다.
