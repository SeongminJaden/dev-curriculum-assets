# Part 10. DevOps와 CI/CD

## 10-3. 모니터링과 관측성(Observability) — 로그·메트릭·트레이스는 각각 어떤 질문에 답하기 위한 도구인가

> **학습 목표**
> - 모니터링과 관측성이 서로 어떻게 다른 개념인지 설명할 수 있다.
> - 관측성의 세 기둥인 로그, 메트릭, 트레이스가 각각 어떤 질문에 답하기 위한 도구인지 구분해 설명할 수 있다.
> - 마이크로서비스처럼 분산된 시스템에서 분산 트레이싱이 왜 별도로 필요한지 설명할 수 있다.

> **개요** 10-1, 10-2강에서 코드를 자동으로 검증·배포하고 인프라를 코드로 재현하는 방법을 다뤘다면, 이번 강의는 그렇게 배포된 시스템이 실제로 운영되는 동안 "지금 무슨 일이 일어나고 있는가"를 알아내는 방법을 다룹니다. 로그·메트릭·트레이스라는 서로 다른 도구가 각각 어떤 질문에 답하는지 구분하는 것이 이 강의의 핵심입니다.

---

## 1. 모니터링과 관측성은 같은 말이 아니다

> **개요** 흔히 같은 의미로 섞어 쓰이는 모니터링과 관측성을 구분하고, 왜 이 구분이 실무에서 의미를 갖는지를 다룹니다.

<u>**모니터링(Monitoring)**</u>은 CPU 사용률, 응답 시간, 에러 발생 횟수처럼 미리 정해둔 지표를 지속적으로 감시하다가, 그 값이 정상 범위를 벗어나면 알림을 보내는 활동입니다. "이 지표가 임계값을 넘으면 알려달라"는 방식으로, 무엇을 감시할지를 <u>**사전에 정해두어야**</u> 합니다.

<u>**관측성(Observability)**</u>은 원래 제어 이론(Control Theory)에서 온 개념으로, "시스템 내부 상태를 외부에서 관측 가능한 출력만으로 얼마나 추론해낼 수 있는가"를 가리킵니다. 소프트웨어 시스템에 적용하면, 시스템이 왜 그런 상태가 되었는지를 <u>**사전에 어떤 질문을 할지 정해두지 않고도**</u> 사후에 수집된 데이터를 파고들어 알아낼 수 있는 능력을 뜻합니다.

> **Q.** 모니터링만으로는 부족하고 관측성까지 필요한 이유는 무엇일까요?
> **A.** 모니터링은 "미리 예상한 문제"를 감시하는 데는 강하지만, "한 번도 본 적 없는 새로운 형태의 장애"에는 취약하기 때문입니다. 예를 들어 "응답 시간이 500ms를 넘으면 알림"이라는 모니터링 규칙은 응답 시간이 실제로 느려졌을 때는 잘 작동하지만, "왜 느려졌는가"까지는 답해주지 않습니다. 관측성이 잘 갖춰진 시스템은 이 질문에 답하기 위해 필요한 세부 데이터(로그, 메트릭, 트레이스)를 평소에 충분히 수집해두어, 담당자가 사전에 준비하지 못한 질문이 갑자기 생기더라도 그 데이터를 뒤져 원인을 추론할 수 있게 해줍니다. 즉 모니터링이 "정해진 질문에 대한 답을 자동으로 알려주는 것"이라면, 관측성은 "어떤 질문이든 답할 수 있는 재료를 갖추는 것"에 가깝습니다.

## 2. 관측성의 세 기둥 — 로그, 메트릭, 트레이스

> **개요** 관측성을 구성하는 세 가지 데이터 유형이 각각 어떤 질문에 답하도록 설계되었는지를 다룹니다.

관측성을 실현하는 데 쓰이는 세 종류의 데이터를 흔히 <u>**관측성의 세 기둥(Three Pillars of Observability)**</u>이라 부릅니다. 세 기둥은 서로 대체 관계가 아니라, 서로 다른 질문에 답하도록 역할이 나뉘어 있습니다.

![관측성의 세 기둥이 답하는 질문](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part10_DevOps와_CI_CD/10-3_모니터링과_관측성/assets/diagram1_three_pillars.svg)

- <u>**로그(Logs)**</u> — "정확히 무슨 일이 있었는가?"에 답합니다. 특정 시점에 발생한 개별 이벤트(요청 수신, 오류 발생, 상태 변경)를 시간 순서대로 남긴 상세한 기록입니다. 하나의 사건을 가장 구체적으로 들여다볼 수 있지만, 로그 양이 많아지면 전체 경향을 파악하기는 어렵습니다.
- <u>**메트릭(Metrics)**</u> — "지금 시스템 상태는 어떠하고, 추세는 어떤가?"에 답합니다. CPU 사용률, 초당 요청 수, 에러율처럼 숫자로 집계된 값을 시간에 따라 기록한 시계열 데이터입니다. 개별 사건의 상세 내용은 알 수 없지만, 저장 비용이 로그보다 낮고 추세나 이상 징후를 한눈에 파악하기 좋습니다.
- <u>**트레이스(Traces)**</u> — "이 요청 하나는 어디를 거쳐갔고, 어디서 시간이 걸렸는가?"에 답합니다. 하나의 요청이 여러 구성 요소(서비스, 함수, 데이터베이스 호출)를 거치는 전체 경로와 각 구간에 걸린 시간을 기록합니다.

> **Q.** 셋 중 하나만 있으면 안 될까요?
> **A.** 안 됩니다. 각자 잘하는 질문과 못하는 질문이 뚜렷이 나뉘기 때문입니다. 메트릭만 있다면 "평균 응답 시간이 갑자기 늘었다"는 사실은 알 수 있어도, 그 요청이 정확히 어떤 값으로 실패했는지(로그가 답할 질문)나 시간이 정확히 어느 구간에서 소요됐는지(트레이스가 답할 질문)는 알 수 없습니다. 반대로 로그만 있다면 개별 사건은 상세히 볼 수 있어도, 수백만 건의 로그 중 "지금 전체적으로 상태가 나빠지고 있는가"라는 추세 질문에는 매번 로그를 집계해야 하므로 비효율적입니다. 실무에서 세 가지를 함께 수집하는 이유는, <u>**서로 답하지 못하는 질문의 공백을 메워주기 때문**</u>입니다.

## 3. 왜 분산 시스템에서는 트레이싱이 특히 중요해지는가

> **개요** 하나의 서버가 모든 로직을 처리하던 시절과 달리, 여러 서비스로 나뉜 분산 시스템에서 로그와 메트릭만으로는 왜 문제 원인을 찾기 어려워지는지, 그리고 분산 트레이싱이 이 공백을 어떻게 메우는지를 다룹니다.

하나의 서버(모놀리식 구조, Part 3-2)가 요청 전체를 처리하던 시절에는, 그 서버 하나의 로그만 순서대로 읽으면 요청이 어떻게 처리되었는지 대체로 재구성할 수 있었습니다.

하지만 마이크로서비스 구조에서는 사용자의 요청 하나가 주문 서비스, 재고 서비스, 결제 서비스, 배송 서비스처럼 여러 개의 독립된 서비스를 순서대로, 때로는 동시에 거쳐 갑니다. 각 서비스는 자신만의 로그를 남기며, 이 로그들은 서로 다른 서버에 각각 저장됩니다.

> **Q.** 이 상황에서 로그만으로 문제를 추적하기 어려운 이유는 무엇일까요?
> **A.** 로그가 서비스별로 흩어져 있어서, "느린 결제 요청 하나"가 정확히 어느 서비스들을 거쳤는지, 그리고 그중 어느 서비스에서 시간을 많이 썼는지를 알아내려면 담당자가 여러 서비스의 로그를 시간대를 맞춰가며 일일이 대조해야 하기 때문입니다. 서비스가 4~5개일 때도 쉽지 않은 이 작업은, 서비스 수가 수십 개로 늘어나면 사실상 사람이 손으로 하기 불가능한 수준이 됩니다. 게다가 여러 서비스가 동시에 많은 요청을 처리하고 있으므로, 로그만 봐서는 어떤 로그 줄이 "내가 찾는 그 요청"에 해당하는지조차 구분하기 어렵습니다.

이 문제를 해결하는 방법이 <u>**분산 트레이싱(Distributed Tracing)**</u>입니다. 요청이 시스템에 처음 들어오는 순간 고유한 <u>**트레이스 ID(Trace ID)**</u>를 부여하고, 이 요청이 거치는 모든 서비스 호출에 이 ID를 그대로 실어 전달합니다. 각 서비스 안에서 처리되는 구간은 <u>**스팬(Span)**</u>이라는 단위로 기록되며, 하나의 트레이스는 여러 개의 스팬이 모여 만들어집니다.

![분산 트레이스가 여러 서비스에 걸친 요청 경로를 재구성하는 방식](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part10_DevOps와_CI_CD/10-3_모니터링과_관측성/assets/diagram2_distributed_trace.svg)

> **해석** 위 다이어그램은 하나의 주문 요청이 트레이스 ID 하나를 공유한 채 4개의 서비스를 거치는 모습을 시간 순서대로 보여줍니다. 각 막대(스팬)의 길이는 해당 서비스에서 소요된 시간을 나타냅니다. 개별 서비스의 로그만 보면 "재고 서비스가 조금 느렸다"는 사실만 따로따로 보이지만, 트레이스 ID로 묶어 하나의 타임라인으로 재구성하면 <u>**전체 요청 시간의 대부분이 재고 서비스 구간에서 소요되었다**</u>는 것이 한눈에 드러납니다. 이는 로그를 아무리 많이 쌓아도 서비스별로 흩어져 있으면 얻기 어려운 정보입니다.

> **원칙** 분산 트레이싱이 실무 표준으로 자리 잡은 이유는, 서비스 수가 늘어날수록 "요청 하나의 전체 경로"라는 정보가 개별 서비스의 로그·메트릭만으로는 재구성 불가능해지기 때문입니다. 로그와 메트릭이 각 서비스 내부를 잘 보여주는 <u>**세로 방향(서비스별)**</u> 도구라면, 트레이스는 여러 서비스를 가로질러 요청 하나를 따라가는 <u>**가로 방향(요청별)**</u> 도구입니다. 분산 시스템에서는 이 가로 방향의 시야가 없으면 지연의 원인이 되는 서비스를 특정하는 데만 상당한 시간이 걸립니다.

### 실생활 예시 — 배달 앱 주문이 갑자기 느려진 경우

> **해석** 아래 표는 "주문 완료까지 평균 5초 걸리던 것이 갑자기 20초로 늘었다"는 장애가 발생했을 때, 로그만 있는 경우와 로그·메트릭·트레이스가 모두 갖춰진 경우를 비교한 것입니다.

| 상황 | 로그만 있는 경우 | 세 기둥이 모두 갖춰진 경우 |
|---|---|---|
| 문제 인지 | 사용자 불만이 접수되고 나서야 인지 | 메트릭 대시보드에서 응답 시간 급증을 즉시 확인 |
| 원인 후보 좁히기 | 주문·재고·결제·배송 서비스의 로그를 각각 열어 시간대를 눈으로 대조 | 트레이스에서 유독 긴 스팬을 가진 서비스를 즉시 특정 |
| 상세 원인 확인 | 어느 로그 줄이 해당 요청인지 특정하기 어려움 | 특정된 서비스·트레이스 ID로 관련 로그만 정확히 조회 |
| 소요 시간 | 여러 서비스 담당자가 모여 로그를 맞춰보는 데 상당한 시간 소요 | 대시보드 확인 후 최소한의 로그 조회만으로 원인 특정 |

> **팁** 이 비교에서 알 수 있듯, 로그는 "원인을 확정하는 마지막 단계"에서는 여전히 필요합니다. 다만 메트릭이 "문제가 생겼다는 사실"을, 트레이스가 "어디에서 생겼는가"를 먼저 좁혀주어야, 로그를 어디서부터 뒤져야 할지가 정해집니다.

## 요약

> **핵심**
> - <u>**모니터링**</u>은 미리 정한 지표를 감시해 알려진 문제를 발견하는 활동이고, <u>**관측성**</u>은 사전에 정하지 않은 질문에도 답할 수 있도록 데이터를 충분히 갖추는 능력입니다.
> - 관측성의 세 기둥 중 <u>**로그**</u>는 "정확히 무슨 일이 있었는가", <u>**메트릭**</u>은 "지금 상태와 추세가 어떠한가", <u>**트레이스**</u>는 "요청이 어디를 거쳐갔고 어디서 시간이 걸렸는가"에 답하며, 서로 대체할 수 없습니다.
> - 마이크로서비스처럼 요청이 여러 서비스를 거치는 분산 시스템에서는 로그와 메트릭만으로 전체 요청 경로를 재구성하기 어려워지며, **트레이스 ID로 여러 서비스의 스팬을 하나로 묶는 분산 트레이싱이 지연 원인을 특정하는 데 필수적**입니다.
