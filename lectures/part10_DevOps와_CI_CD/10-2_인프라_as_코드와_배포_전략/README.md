# Part 10. DevOps와 CI/CD

## 10-2. 인프라 as 코드(IaC)와 배포 전략 — 서버 설정을 코드로 관리하면 왜 "재현 가능한 인프라"가 되는가

> **학습 목표**
> - 수동으로 서버를 설정할 때 발생하는 "설정 드리프트" 문제를 설명할 수 있다.
> - 선언적(Declarative) IaC가 명령형(Imperative) 방식과 무엇이 다르고, 왜 재현성을 보장하는지 설명할 수 있다.
> - 블루/그린 배포와 카나리 배포가 각각 어떤 위험을 줄이기 위한 전략인지, 롤백이 왜 이 위에서 안전해지는지 설명할 수 있다.

> **개요** 10-1강에서 코드를 자동으로 검증하고 배포하는 파이프라인을 다뤘다면, 이번 강의는 그 코드가 실제로 올라갈 "서버"라는 대상 자체를 코드로 관리하는 방법과, 새 버전을 실제 사용자에게 안전하게 노출하는 배포 전략을 다룹니다.

---

## 1. 손으로 설정한 서버는 왜 서로 달라지는가 — 설정 드리프트

> **개요** 여러 대의 서버를 사람이 직접 설정할 때 발생하는 "설정 드리프트" 문제를 정의하고, 이것이 왜 장애로 이어지는지를 다룹니다.

서버 한 대를 준비하는 과정을 생각해봅니다. 운영체제를 설치하고, 필요한 소프트웨어를 설치하고, 환경 변수와 방화벽 규칙을 설정하고, 애플리케이션을 배치합니다. 이 작업을 관리자가 터미널에 명령어를 하나씩 입력해 수행한다면 어떤 문제가 생길까요?

> **Q.** 똑같은 절차서를 보고 설정했는데도 서버마다 미묘하게 달라지는 이유는 무엇일까요?
> **A.** 사람이 수행하는 작업은 완벽하게 재현되지 않기 때문입니다. 오늘 서버를 설정한 담당자와 6개월 뒤 새 서버를 추가하는 담당자가 다를 수 있고, 같은 사람이라도 그사이 절차서에 없는 긴급 패치를 하나둘 적용하다 보면 서버 간 상태가 조금씩 벌어집니다. 이렇게 <u>**의도한 설정과 실제 서버의 상태가 시간이 지나며 조금씩 어긋나는 현상**</u>을 <u>**설정 드리프트(Configuration Drift)**</u>라고 부릅니다. 결과적으로 겉보기엔 똑같아야 할 서버 10대가 실제로는 조금씩 다른 "제각각의 서버(Snowflake Server)"가 되어, "이 서버에서만 장애가 난다"는 재현 불가능한 문제를 만들어냅니다.

설정 드리프트가 위험한 이유는 단순히 서버가 다르다는 사실 자체가 아니라, <u>**그 차이를 아무도 정확히 알지 못한다**</u>는 데 있습니다. 문서화된 절차서와 실제 서버 상태가 어긋나 있으면, 장애가 발생했을 때 "정상 상태가 무엇인지"조차 기준을 세우기 어렵습니다.

## 2. 인프라를 코드로 정의한다 — Infrastructure as Code

> **개요** IaC의 정의와, 선언적 접근이 왜 인프라를 재현 가능하게 만드는지 원리를 다룹니다.

<u>**인프라 as 코드(Infrastructure as Code, IaC)**</u>는 서버, 네트워크, 데이터베이스 같은 인프라 자원의 설정을 사람이 손으로 조작하는 대신, 코드(설정 파일)로 정의하고 그 코드를 실행해 인프라를 만들거나 변경하는 방식입니다. Terraform, AWS CloudFormation과 같은 도구들이 이 방식을 지원합니다.

IaC 도구는 크게 두 가지 접근으로 나뉩니다.

- <u>**명령형(Imperative)**</u> 접근: "무엇을 어떤 순서로 실행할지" 절차를 하나하나 코드로 적습니다. 셸 스크립트로 서버를 설정하는 방식이 여기에 해당합니다.
- <u>**선언적(Declarative)**</u> 접근: "최종적으로 인프라가 어떤 상태여야 하는지"만 정의합니다. 실행 순서나 현재 상태와의 차이를 계산하는 일은 도구가 대신 처리합니다.

> **Q.** 선언적 접근이 재현 가능한 인프라를 만드는 데 왜 더 유리할까요?
> **A.** 명령형 스크립트는 "현재 상태가 무엇인지"를 가정하고 절차를 적습니다. 만약 실제 서버 상태가 스크립트가 가정한 것과 다르면(이미 누군가 손으로 값을 바꿔놓았다면), 같은 스크립트를 실행해도 결과가 매번 달라질 수 있습니다. 반면 선언적 접근은 "목표 상태"만 코드에 적어두고, 도구가 매번 실제 상태를 목표 상태와 비교해 그 차이만큼만 변경을 적용합니다. 즉 <u>**같은 코드를 몇 번을 실행하든 결과는 항상 코드에 적힌 목표 상태로 수렴**</u>합니다. 이 성질을 <u>**멱등성(Idempotency)**</u>이라고 부르며, 서버가 100대이든 1대이든 같은 코드를 적용하면 모두 같은 상태가 된다는 것을 보장하는 근거가 됩니다.

![선언적 IaC가 여러 환경의 일관성을 보장하는 원리](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part10_DevOps와_CI_CD/10-2_인프라_as_코드와_배포_전략/assets/diagram1_iac_flow.svg)

> **해석** 위 다이어그램은 하나의 IaC 코드가 개발·스테이징·운영 세 환경에 각각 적용되는 흐름을 보여줍니다. 사람이 세 환경을 따로 설정하면 서로 다른 결과물이 나올 위험이 있지만, 동일한 코드를 각 환경에 그대로 적용하면 세 환경 모두 코드에 적힌 것과 같은 구조를 갖게 됩니다. 서버가 손상되거나 삭제되더라도, 같은 코드를 다시 적용하는 것만으로 동일한 인프라를 다시 만들어낼 수 있습니다. 이것이 <u>**"재현 가능한 인프라"**</u>의 의미입니다. 또한 이 코드는 애플리케이션 코드와 마찬가지로 버전 관리 시스템(Part 1-4)에 저장되므로, "언제 누가 무엇을 바꿨는지"를 코드 변경 이력으로 그대로 추적할 수 있습니다.

## 3. 새 버전을 어떻게 안전하게 노출할 것인가 — 배포 전략

> **개요** 인프라가 코드로 재현 가능해졌다는 전제 위에서, 새 버전의 애플리케이션을 실제 트래픽에 안전하게 노출하는 대표적인 전략들을 다룹니다.

인프라를 코드로 관리할 수 있게 되면, "새 버전이 배포될 서버 한 벌을 통째로 새로 만드는" 것과 같은 작업이 훨씬 쉬워집니다. 이 능력을 바탕으로 아래와 같은 배포 전략이 가능해집니다.

- <u>**블루/그린 배포(Blue/Green Deployment)**</u>: 기존 버전(블루)이 트래픽을 처리하는 동안, 새 버전(그린)을 별도로 완전히 구성해둡니다. 새 버전 검증이 끝나면 트래픽을 블루에서 그린으로 한 번에 전환합니다.
- <u>**카나리 배포(Canary Deployment)**</u>: 새 버전에 트래픽의 일부(예: 5%)만 먼저 흘려보내고, 문제가 없는지 확인하며 점진적으로 비율을 늘려 100%까지 전환합니다.
- <u>**롤링 배포(Rolling Deployment)**</u>: 여러 대의 서버를 한꺼번에 교체하지 않고, 일부씩 순차적으로 새 버전으로 교체합니다.

![블루/그린 배포와 카나리 배포 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part10_DevOps와_CI_CD/10-2_인프라_as_코드와_배포_전략/assets/diagram2_deployment_strategies.svg)

> **Q.** 카나리(canary)라는 이름은 어디서 왔을까요?
> **A.** 과거 광부들이 유독 가스를 미리 감지하기 위해 카나리아 새를 탄광에 데리고 들어간 관행에서 온 이름입니다. 카나리아는 사람보다 먼저 가스에 반응했기 때문에 위험 신호 역할을 했습니다. 카나리 배포도 같은 발상입니다. <u>**전체 사용자 대신 일부 사용자만 먼저 새 버전에 노출시켜, 문제가 있다면 소수의 사용자만 영향을 받는 동안 미리 감지**</u>하려는 것입니다.

두 전략은 목적이 조금 다릅니다. 블루/그린은 "전환 자체를 순식간에 끝내 다운타임을 없애는 것"에 초점이 있고, 카나리는 "실제 트래픽으로 점진적으로 검증하며 위험을 최소화하는 것"에 초점이 있습니다. 실무에서는 두 방식을 함께 적용하기도 합니다.

## 4. 문제가 생기면 어떻게 되돌리는가 — 롤백

> **개요** 배포 전략과 IaC가 함께 갖춰졌을 때 롤백이 왜 안전하고 빠른 작업이 되는지를 다룹니다.

새 버전에 문제가 발견되면 이전 버전으로 되돌리는 <u>**롤백(Rollback)**</u>이 필요합니다.

> **Q.** 롤백은 왜 IaC와 자동화된 배포 전략이 없으면 위험한 작업이 될까요?
> **A.** 서버가 손으로 설정되어 있다면, "이전 버전으로 되돌린다"는 것이 정확히 무엇을 의미하는지부터 불분명합니다. 이전 버전의 애플리케이션 코드는 되돌릴 수 있어도, 그사이 손으로 바뀐 서버 설정까지 정확히 되돌리기는 어렵습니다. 반면 인프라와 애플리케이션이 모두 코드로 정의되어 있다면, 롤백은 "이전 버전의 코드를 다시 적용하는 것"과 동일한 작업이 됩니다. 블루/그린 배포에서는 이전 버전(블루)이 새 버전으로 트래픽이 전환된 뒤에도 잠시 그대로 남아 있는 경우가 많아, 문제가 발견되면 트래픽을 다시 블루로 전환하는 것만으로 즉시 되돌릴 수 있습니다. 카나리 배포에서는 애초에 트래픽의 일부만 새 버전에 노출되어 있으므로, 그 비율을 0%로 되돌리는 것만으로 영향 범위를 즉시 좁힐 수 있습니다.

> **원칙** 좋은 배포 전략의 공통점은 <u>**"문제가 생기지 않게 만드는 것"이 아니라 "문제가 생겼을 때 영향 범위를 좁히고 되돌리는 비용을 낮추는 것"**</u>에 있습니다. 아무리 테스트를 철저히 해도 실제 운영 트래픽에서만 드러나는 문제는 항상 존재하기 때문에, "얼마나 빨리, 얼마나 적은 사용자에게만 영향을 주며 되돌릴 수 있는가"가 배포 전략을 설계하는 핵심 기준이 됩니다.

### 실생활 예시 — 온라인 서점의 검색 기능 개편

> **해석** 아래 표는 검색 알고리즘을 새 버전으로 교체할 때, 서버가 수동 설정된 경우와 IaC + 카나리 배포를 함께 쓴 경우를 비교한 것입니다.

| 상황 | 수동 설정 서버 | IaC + 카나리 배포 |
|---|---|---|
| 새 버전 적용 범위 | 전체 서버에 한 번에 적용 (되돌리기 전 이미 모든 사용자에게 노출) | 트래픽의 5%에만 우선 적용 |
| 문제 발견 시 대응 | 이전 버전 코드를 다시 배포하고, 손으로 바뀐 설정까지 일일이 점검 | 카나리 비율을 0%로 되돌리는 설정 변경만으로 즉시 차단 |
| 영향받은 사용자 규모 | 전체 사용자 | 트래픽의 5%뿐 |
| 원인 분석 난이도 | 서버마다 설정이 달라 재현이 어려움 | 코드로 동일하게 재현된 환경이므로 원인 추적이 상대적으로 쉬움 |

## 요약

> **핵심**
> - 서버를 손으로 설정하면 시간이 지나며 <u>**설정 드리프트**</u>가 쌓여, 서버마다 재현 불가능한 차이가 생깁니다.
> - <u>**선언적 IaC**</u>는 "목표 상태"만 코드로 정의하고 도구가 실제 상태와의 차이를 계산해 적용하므로, 같은 코드를 몇 번 실행해도 항상 같은 결과(멱등성)를 보장해 재현 가능한 인프라를 만듭니다.
> - <u>**블루/그린 배포**</u>는 전환 자체의 다운타임을 없애고, <u>**카나리 배포**</u>는 일부 트래픽으로 점진적으로 위험을 검증합니다.
> - 롤백은 인프라와 배포가 모두 코드·설정으로 정의되어 있을 때만 "이전 상태를 다시 적용하는 예측 가능한 작업"이 되며, **좋은 배포 전략의 핵심은 문제를 완전히 막는 것이 아니라 영향 범위와 되돌리는 비용을 낮추는 것**입니다.
