# Part 10. DevOps와 CI/CD

## 10-1. CI/CD 파이프라인의 원리 — 빌드·테스트·배포를 자동화하는 것은 어떤 실수를 막기 위함인가

> **학습 목표**
> - 지속적 통합(CI)과 지속적 전달·배포(CD)의 개념을 구분해 설명할 수 있다.
> - CI/CD 파이프라인의 각 단계(빌드, 테스트, 배포)가 왜 자동화되어야 하는지 근거를 들어 설명할 수 있다.
> - "It works on my machine" 문제가 왜 발생하고, CI/CD가 이를 구조적으로 어떻게 제거하는지 설명할 수 있다.

> **개요** 코드를 작성하는 것과 그 코드를 사용자에게 안전하게 전달하는 것은 완전히 다른 문제입니다. 이 강의는 빌드·테스트·배포라는 반복 작업을 사람이 손으로 하지 않고 자동화된 파이프라인에 맡기는 이유를, 그 자동화가 막아주는 구체적인 실수의 종류를 통해 살펴봅니다.

---

## 1. "제 컴퓨터에서는 잘 되는데요" — 문제의 출발점

> **개요** CI/CD가 등장하기 전, 개발자가 코드를 작성한 뒤 실제 운영 서버까지 전달하는 과정에서 어떤 문제가 반복적으로 발생했는지를 살펴봅니다.

<u>**"It works on my machine"**</u>은 개발자 사이에서 오래된 농담이자 동시에 실제로 반복되는 장애의 원인입니다. 개발자의 노트북에서는 정상 동작하던 코드가, 다른 개발자의 컴퓨터나 실제 서버에서는 오류를 내는 상황을 가리킵니다.

이런 문제가 생기는 이유는 하나로 요약됩니다. <u>**개발 환경과 운영 환경이 완전히 같지 않기 때문**</u>입니다.

- 설치된 프로그래밍 언어나 라이브러리의 버전이 다를 수 있습니다.
- 운영체제나 시스템 설정(환경 변수, 타임존, 파일 경로 구분자)이 다를 수 있습니다.
- 개발자의 컴퓨터에는 있지만 서버에는 없는 설정 파일이나 캐시된 데이터가 존재할 수 있습니다.

> **Q.** 이런 환경 차이는 왜 미리 발견하지 못할까요?
> **A.** CI/CD가 없던 시절에는 "빌드해서 테스트하고 서버에 올리는" 전 과정을 사람이 손으로 수행했기 때문입니다. 개발자 한 명이 자신의 컴퓨터에서 코드를 실행해보고 "동작한다"고 판단하면, 그 판단이 곧 배포 승인 기준이 되었습니다. 하지만 이 판단은 오직 그 사람의 환경 안에서만 유효합니다. 실제 운영 환경에서 같은 코드가 어떻게 동작할지는 배포하고 나서야 알 수 있었습니다. 즉 <u>**검증이 사람의 기억과 손작업에 의존**</u>하는 구조였기 때문에, 검증 자체가 누락되거나 부정확해질 여지가 컸습니다.

수동 배포 시절에는 아래와 같은 실수가 반복적으로 발생했습니다.

- 테스트를 깜빡하고 배포 스크립트를 실행한다.
- 여러 명이 동시에 작업한 코드를 합칠 때(병합) 서로의 변경을 덮어써 버린다.
- 배포 순서를 착각해 아직 준비되지 않은 데이터베이스 변경 사항 없이 새 코드만 올린다.
- 체크리스트 항목 중 하나(캐시 초기화, 설정 파일 갱신 등)를 사람이 빠뜨린다.

CI/CD는 이 모든 단계를 "사람이 매번 정확히 기억해서 순서대로 수행하는 일"에서 "코드로 정의해두고 기계가 매번 동일하게 반복하는 일"로 바꾸는 접근입니다.

## 2. 지속적 통합(Continuous Integration) — 자주 합치고, 매번 검증한다

> **개요** CI의 정의와 핵심 원칙을 살펴보고, "자주 병합한다"는 규칙이 왜 병합 자체를 더 쉽게 만드는지를 다룹니다.

<u>**지속적 통합(Continuous Integration, CI)**</u>은 여러 개발자가 작성한 코드 변경 사항을 하루에도 여러 번, 공유 저장소의 메인 브랜치에 자주 병합하고, 병합될 때마다 자동으로 빌드와 테스트를 수행하는 개발 관행입니다. 소프트웨어 엔지니어 마틴 파울러(Martin Fowler)가 2000년대 초반 정리한 이 개념의 핵심은 "통합(병합)을 얼마나 자주 하는가"에 있습니다.

> **Q.** 왜 며칠에 한 번이 아니라 "하루에도 여러 번" 합치는 것이 더 안전할까요?
> **A.** 직관과 반대로 들리지만, 병합 간격이 길어질수록 병합 자체가 더 위험해지기 때문입니다. 두 사람이 같은 파일을 각자 일주일씩 수정한 뒤 합치면, 변경 범위가 넓어 어디서 충돌이 나는지 파악하기 어렵고 충돌 해결 과정에서 실수가 생기기 쉽습니다. 반대로 하루에 여러 번 작은 단위로 합치면, 한 번에 병합되는 변경량이 작아 충돌이 나더라도 원인을 빠르게 좁힐 수 있습니다. <u>**문제를 발견하는 시점과 실제 코드가 작성된 시점 사이의 간격을 줄이는 것**</u>이 CI의 핵심 원리이며, 이는 Part 1-1에서 다룬 "SDLC 단계를 짧게 반복할수록 수정 비용이 줄어든다"는 원리와 정확히 같은 논리입니다.

CI 파이프라인은 코드가 저장소에 병합될 때마다 최소한 다음 두 가지를 자동으로 수행합니다.

1. **빌드(Build)** — 소스 코드를 실행 가능한 형태로 컴파일하거나 패키징합니다. 이 단계에서 문법 오류나 의존성 누락이 즉시 드러납니다.
2. **테스트(Test)** — 미리 작성해둔 자동화 테스트(단위 테스트, 통합 테스트 등, Part 11에서 다룹니다)를 실행해 코드가 의도대로 동작하는지 검증합니다.

![CI/CD 파이프라인 단계](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part10_DevOps와_CI_CD/10-1_CI_CD_파이프라인의_원리/assets/diagram1_pipeline.svg)

> **참고** 빌드나 테스트 중 하나라도 실패하면 파이프라인은 그 지점에서 멈추고 실패를 알립니다. 이 실패는 "병합 직후, 매우 좁은 범위의 변경"에서 발생한 것이므로 원인을 찾기 쉽습니다. 만약 이 검증 없이 며칠 뒤 운영 서버에서 문제가 발견됐다면, 그사이 다른 코드가 계속 쌓여 원인 범위가 훨씬 넓어졌을 것입니다.

## 3. 지속적 전달과 지속적 배포 — CD의 두 가지 의미

> **개요** 같은 CD 약자로 불리지만 서로 다른 지속적 전달(Continuous Delivery)과 지속적 배포(Continuous Deployment)를 구분하고, 두 방식이 어떤 상황에 각각 적합한지를 다룹니다.

CI를 통과한 코드는 다음 단계로 넘어갑니다. 이때 "CD"는 두 가지 서로 다른 관행을 가리킬 수 있습니다.

- <u>**지속적 전달(Continuous Delivery)**</u>: 빌드와 테스트를 통과한 코드를 언제든 운영 환경에 배포할 수 있는 상태로 자동으로 준비해두되, 실제 운영 환경에 반영하는 마지막 버튼은 사람이 누릅니다.
- <u>**지속적 배포(Continuous Deployment)**</u>: 모든 검증 단계를 통과하면 사람의 개입 없이 자동으로 운영 환경까지 배포합니다.

> **Q.** 두 방식 중 어느 쪽이 더 나은 방식일까요?
> **A.** 우열의 문제가 아니라 <u>**얼마나 검증을 신뢰할 수 있는가**</u>의 문제입니다. 자동화된 테스트가 충분히 촘촘하게 갖춰져 있고, 문제가 생겼을 때 빠르게 되돌릴 수 있는 장치(Part 10-2에서 다룰 롤백 전략)까지 마련되어 있다면 지속적 배포까지 자동화해도 안전합니다. 반대로 자동 테스트로 잡아내지 못하는 위험(법적 검토가 필요한 변경, 결제처럼 민감한 기능)이 있다면, 사람이 마지막으로 확인하는 지속적 전달 방식이 더 적합합니다. 두 방식 모두 "빌드와 테스트까지는 반드시 자동화한다"는 원칙은 공유합니다.

## 4. 파이프라인이 막아주는 것 — 자동화 전후 비교

> **개요** 자동화된 파이프라인이 도입되기 전과 후, 같은 규모의 배포 작업이 어떻게 달라지는지를 시간과 실수 발생 지점 관점에서 비교합니다.

![수동 배포와 자동화 파이프라인의 소요 시간 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part10_DevOps와_CI_CD/10-1_CI_CD_파이프라인의_원리/assets/diagram2_time_comparison.svg)

> **해석** 위 다이어그램은 같은 배포 작업(빌드 → 테스트 → 배포)을 수동으로 수행할 때와 자동화된 파이프라인으로 수행할 때의 소요 시간을 개념적으로 비교한 것입니다. 수동 방식은 각 단계 사이에 사람이 다음 단계를 직접 실행하기까지의 대기 시간과 확인 시간이 끼어들며, 단계 수가 늘어날수록 이 대기 시간이 누적됩니다. 자동화된 파이프라인은 각 단계가 이전 단계 성공 즉시 연쇄적으로 실행되므로 이 대기 시간이 사라집니다. 실제 절감 폭은 조직과 프로젝트 규모에 따라 다르지만, "사람의 대기·확인 시간이 파이프라인 총 소요 시간의 상당 부분을 차지한다"는 구조 자체는 대부분의 수동 배포 프로세스에서 공통으로 관찰됩니다.

### 실생활 예시 — 쇼핑몰 서비스의 버그 수정 배포

> **해석** 아래 표는 "결제 오류를 수정하는 코드 한 줄"이 수동 배포 시절과 CI/CD 도입 이후 각각 어떻게 처리되는지 비교한 것입니다.

| 단계 | 수동 배포 시절 | CI/CD 파이프라인 도입 이후 |
|---|---|---|
| 코드 병합 | 개발자가 임의로 정한 시점에 병합, 충돌은 발견 시점에 수작업으로 해결 | 병합과 동시에 자동 빌드·테스트 시작 |
| 검증 | 개발자 본인 컴퓨터에서 수동으로 한 번 실행해보고 판단 | 정해진 테스트 스위트를 운영 환경과 동일한 조건에서 자동 실행 |
| 배포 실행 | 담당자가 서버에 접속해 배포 스크립트를 수동 실행, 체크리스트를 손으로 확인 | 검증을 통과하면 정의된 절차대로 자동 배포 |
| 문제 발견 시점 | 배포 후 사용자 불만이나 장애 알림으로 뒤늦게 발견 | 빌드·테스트 단계에서 대부분 사전 차단 |

> **팁** 이 비교에서 핵심은 "속도"만이 아닙니다. 자동화는 <u>**사람이 매번 정확히 같은 순서로, 빠짐없이 수행해야 하는 부담을 제거**</u>합니다. 사람은 실수하지 않으려 애쓰는 대신, 실수해도 기계가 같은 검증을 매번 동일하게 반복해주는 구조를 만드는 데 집중하게 됩니다.

## 요약

> **핵심**
> - "It works on my machine" 문제는 개발 환경과 운영 환경의 차이, 그리고 검증 과정이 사람의 손작업에 의존하는 구조에서 비롯됩니다.
> - <u>**지속적 통합(CI)**</u>은 코드를 자주 병합하고 매번 자동으로 빌드·테스트해, 병합 범위를 좁혀 문제 원인을 빠르게 좁힐 수 있게 합니다.
> - <u>**지속적 전달(Delivery)**</u>은 배포 가능한 상태까지, <u>**지속적 배포(Deployment)**</u>는 실제 운영 반영까지 자동화하며, 둘의 선택 기준은 자동 검증에 대한 신뢰도입니다.
> - CI/CD 파이프라인의 근본 목적은 **사람이 매번 정확히 기억하고 반복해야 했던 검증 절차를 기계에 위임해, 실수가 낄 자리를 구조적으로 줄이는 것**입니다.
