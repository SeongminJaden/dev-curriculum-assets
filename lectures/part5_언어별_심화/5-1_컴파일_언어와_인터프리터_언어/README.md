# Part 5. 프로그래밍 언어별 심화

## 5-1. 컴파일 언어와 인터프리터 언어 — Python은 왜 C보다 느릴까

> **학습 목표**
> - AOT(Ahead-Of-Time) 컴파일과 인터프리터 실행이 소스코드를 기계어로 바꾸는 서로 다른 경로임을 구조적으로 설명할 수 있다.
> - 바이트코드와 가상머신(VM)이 인터프리터 언어의 성능·이식성 트레이드오프에서 하는 역할을 설명할 수 있다.
> - JIT 컴파일이 두 방식의 장점을 어떻게 절충하는지 설명할 수 있다.

> **개요** 소스코드가 실행되는 데는 근본적으로 다른 두 경로가 있습니다. 실행 전에 전부 기계어로 번역해두는 AOT 컴파일과, 실행하면서 그때그때 해석하는 인터프리터 방식입니다. 이 강의는 Python이 C보다 느린 구조적 이유와 그 대가로 얻는 것, 그리고 두 세계를 절충하는 JIT 컴파일의 원리를 다룹니다.

---

## 1. 왜 같은 로직인데 언어마다 속도가 다를까요?

> **Q.** Python과 C로 똑같은 반복 계산을 짰는데, 왜 체감 속도가 이렇게 다를까요?
> **A.** 두 언어는 "소스코드를 누가, 언제 기계어로 바꾸는가"가 근본적으로 다르기 때문입니다. C는 실행되기 전에 컴파일러가 모든 코드를 미리 기계어로 바꿔둡니다. Python은 실행되는 바로 그 순간, 한 줄(정확히는 한 바이트코드 명령)씩 가상머신이 해석하면서 진행합니다. 이 "언제 번역하는가"의 차이가 실행 속도, 이식성, 개발 편의성을 모두 갈라놓습니다.

<u>**컴파일 언어(Compiled Language)**</u>는 실행 전에 소스코드 전체를 대상 플랫폼의 기계어로 미리 번역해두는 언어를, <u>**인터프리터 언어(Interpreted Language)**</u>는 소스코드(또는 그것을 변환한 중간 표현)를 실행 시점에 한 단위씩 해석하며 실행하는 언어를 말합니다. 이는 문법의 차이가 아니라 <u>**실행 모델(execution model)**</u>의 차이입니다.

![AOT 컴파일과 인터프리터 실행 경로 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part5_언어별_심화/5-1_컴파일_언어와_인터프리터_언어/assets/diagram1_compile_vs_interpret.svg)

> **참고** 이 구분은 언어 자체의 속성이 아니라 구현(implementation)의 속성입니다. 같은 Python 언어도 CPython(바이트코드 인터프리터), PyPy(JIT 포함), Cython(C로 트랜스파일 후 AOT 컴파일)처럼 서로 다른 실행 모델로 구현될 수 있습니다. "Python은 느리다"보다 정확한 표현은 "CPython의 기본 실행 모델은 인터프리터 방식이라 느리다"입니다.

## 2. AOT 컴파일 — C가 실행되는 경로

> **개요** 컴파일러가 소스코드를 기계어로 바꾸는 표준 단계와, 이 방식이 왜 실행 시점에 가장 빠른지를 다룹니다.

<u>**AOT(Ahead-Of-Time) 컴파일**</u>은 프로그램을 실행하기 전에 컴파일러가 아래 단계를 거쳐 소스코드를 대상 CPU의 기계어로 완전히 번역해두는 방식입니다.

1. **어휘 분석(Lexical Analysis)** — 소스코드 문자열을 토큰(예약어, 식별자, 연산자)으로 쪼갭니다.
2. **구문 분석(Parsing)** — 토큰을 문법 규칙에 따라 추상 구문 트리(AST)로 구성합니다.
3. **의미 분석(Semantic Analysis)** — 타입 검사, 변수 선언 확인 등 프로그램이 언어 규칙상 유효한지 확인합니다.
4. **최적화(Optimization)** — 불필요한 연산 제거, 반복문 인라이닝 등 실행 속도를 높이는 변환을 적용합니다.
5. **코드 생성(Code Generation)** — 대상 CPU의 명령어 집합(ISA)에 맞는 기계어를 출력합니다.

이 과정을 모두 마친 결과물이 실행파일(예: Linux의 ELF, Windows의 PE)이며, 운영체제는 이 파일을 그대로 메모리에 적재해 CPU가 바로 실행하게 합니다.

> **Q.** AOT 컴파일 언어가 실행 시점에 빠른 이유는 정확히 무엇일까요?
> **A.** 실행되는 순간에는 "번역"이라는 작업이 전혀 남아있지 않기 때문입니다. CPU는 이미 자신이 이해하는 명령어만 순서대로 가져와 실행합니다. 게다가 컴파일 시점에는 실행 시간 제약이 없으므로, 컴파일러가 시간을 들여 반복문 언롤링, 함수 인라이닝, 레지스터 할당 최적화 같은 고비용 최적화를 미리 적용해둘 수 있습니다.

> **참고** 대가도 있습니다. 실행파일은 컴파일할 때 지정한 CPU 아키텍처와 운영체제 조합(예: x86-64 Linux)에 종속되므로, 다른 플랫폼에서 실행하려면 그 플랫폼용으로 다시 컴파일해야 합니다. 또한 코드를 한 줄 고칠 때마다 컴파일 과정을 다시 거쳐야 하므로, 즉석에서 코드를 실험하는 대화형 개발(REPL)에는 근본적으로 불리합니다.

## 3. 인터프리터 언어 — Python이 실행되는 경로

> **개요** CPython이 소스코드를 바이트코드로 바꾸고, 가상머신이 그것을 한 명령씩 해석하는 과정을 다룹니다. 이 한 단계가 인터프리터 언어가 느린 구조적 원인입니다.

Python(CPython 기준)도 사실 소스코드를 한 글자씩 그때그때 읽어가며 실행하지는 않습니다. 실제 경로는 다음과 같습니다.

1. 소스코드를 파싱해 AST를 만듭니다(여기까지는 컴파일 언어와 동일합니다).
2. AST를 <u>**바이트코드(bytecode)**</u>라는 중간 표현으로 컴파일합니다. 이 바이트코드는 특정 CPU의 기계어가 아니라, CPython이라는 가상의 기계를 위한 명령어 집합입니다. (`__pycache__`의 `.pyc` 파일이 이 결과물입니다.)
3. <u>**가상머신(Virtual Machine, VM)**</u>이 이 바이트코드를 한 명령어씩 순서대로 가져와, 각 명령어에 대응하는 실제 동작(C로 구현된 함수)을 실행합니다. 이 반복 구조를 <u>**디스패치 루프(dispatch loop)**</u>라고 부릅니다.

![Python 인터프리터의 실행 파이프라인 개념도(diagram1 하단 행 참고)](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part5_언어별_심화/5-1_컴파일_언어와_인터프리터_언어/assets/diagram1_compile_vs_interpret.svg)

> **Q.** 그렇다면 Python이 느린 진짜 원인은 무엇일까요?
> **A.** "바이트코드로 바꾼다"는 단계 자체가 아니라, **매 명령어마다 VM이 개입해야 한다**는 점입니다. C에서는 `a + b`가 컴파일 시점에 CPU의 덧셈 명령어 하나로 확정됩니다. Python에서는 실행 중에 VM이 바이트코드 `BINARY_ADD`를 보고, `a`와 `b`가 각각 무슨 타입인지 런타임에 확인하고, 그에 맞는 덧셈 로직을 호출하는 과정을 매번 반복합니다. 이 "명령어 해석 + 타입 확인 + 함수 호출"이라는 오버헤드가 반복문 안에서 수백만 번 누적되면 체감 가능한 속도 차이로 나타납니다.

> **참고** Python의 모든 값(정수 포함)은 파이썬 객체(`PyObject`)로 힙에 저장되고, 변수는 그 객체를 가리키는 참조일 뿐입니다. C의 `int`가 스택에 고정폭 값으로 바로 놓이는 것과 달리, Python은 정수 하나를 더하려 해도 객체를 힙에서 참조 해제하고 타입을 확인하는 간접 단계를 거칩니다. 이 방식은 5-2강에서 다룰 동적 타입의 유연성을 가능하게 하는 대신, 속도를 대가로 치릅니다.

## 4. 그 대가로 무엇을 얻는가

> **개요** 인터프리터 언어가 속도를 희생하는 대신 실제로 얻는 것이 무엇인지, 핵심 질문에 대한 답을 정리합니다.

- **플랫폼 독립성** — 바이트코드는 특정 CPU가 아니라 VM을 대상으로 하므로, VM이 있는 곳이면 어디서든 같은 바이트코드가 그대로 실행됩니다. "한 번 작성하면 어디서나 실행"이라는 이식성은 여기서 나옵니다.
- **대화형 실행(REPL)** — 코드를 한 줄 입력하면 즉시 파싱·컴파일·실행까지 끝나므로, 컴파일-실행 주기가 사실상 사라져 실험과 디버깅이 빨라집니다.
- **런타임 유연성** — 실행 중에 코드 구조 자체를 바꾸는 `eval`, 동적 속성 추가, 몽키 패칭 같은 기능은 "실행 시점까지 번역을 미뤄둔다"는 인터프리터의 특성 없이는 자연스럽게 구현하기 어렵습니다.
- **개발 생산성** — 컴파일 단계와 엄격한 정적 검사가 없는 만큼, 짧은 스크립트나 프로토타입을 빠르게 작성할 수 있습니다.

> **핵심**
> Python이 C보다 느린 이유는 "나쁜 언어라서"가 아니라, **매 실행마다 VM이 명령어를 해석하는 오버헤드를 감수하는 대신, 플랫폼 독립성·대화형 실행·런타임 유연성을 얻는 설계 선택**이기 때문입니다.

## 5. 실생활 예시 — 같은 반복 연산, 다른 체감 속도

> **해석** 아래 표는 "숫자 목록을 순회하며 값을 누적하는" 동일한 로직을 Python의 순수 `for` 반복문과 C의 `for` 반복문으로 짰을 때, 내부적으로 벌어지는 일의 차이를 보여줍니다. 구체적인 배수는 하드웨어·버전에 따라 달라지므로 수치 대신 원리로 비교합니다.

| 구분 | Python (순수 반복문) | C (컴파일된 반복문) |
|---|---|---|
| 반복 한 번마다 벌어지는 일 | 바이트코드 명령 해석 → 각 값의 타입 확인 → 덧셈 함수 호출 → 결과를 새 객체로 힙에 생성 | 컴파일 시점에 확정된 덧셈 기계어 명령 한 개 실행 |
| 값의 표현 방식 | 정수도 힙에 있는 객체(참조 간접 접근) | 스택 위의 고정폭 값(직접 접근) |
| 체감 결과 | 반복 횟수가 커질수록 VM 오버헤드가 누적되어 느려짐 | 반복 횟수와 무관하게 일정한 기계어 실행 속도 유지 |
| 실무 대응 | 반복량이 많은 연산은 내부가 C로 구현된 NumPy 등으로 위임 | 별도 대응 불필요(이미 기계어) |

> **팁** 그래서 실무에서 Python으로 대량 반복 연산(이미지 처리, 행렬 계산)을 할 때는 순수 `for` 문을 직접 돌리기보다, 내부 구현이 C로 AOT 컴파일된 NumPy·Pandas 같은 라이브러리에 반복 작업을 통째로 위임합니다. "Python 코드"처럼 보여도, 실제 반복 연산은 C 수준에서 벌어지게 만드는 것이 정석입니다.

## 6. JIT 컴파일 — 두 세계의 절충

> **개요** 실행 중에 자주 쓰이는 코드만 골라 기계어로 컴파일하는 JIT(Just-In-Time) 컴파일의 동작 원리를 다룹니다.

인터프리터의 유연성과 AOT 컴파일의 속도를 동시에 취하려는 시도가 <u>**JIT(Just-In-Time) 컴파일**</u>입니다. V8(JavaScript 엔진), HotSpot(Java 가상머신), PyPy(Python 대체 구현)가 공통으로 쓰는 전략입니다.

![JIT 컴파일 파이프라인 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part5_언어별_심화/5-1_컴파일_언어와_인터프리터_언어/assets/diagram2_jit_compile.svg)

동작 원리는 다음과 같습니다.

1. 프로그램은 처음에 인터프리터(바이트코드 해석)로 실행을 시작합니다.
2. 실행 중인 함수·루프가 몇 번 호출되는지 **프로파일링**합니다.
3. 호출 횟수가 임계치를 넘으면 그 부분을 <u>**hot path**</u>(자주 실행되는 코드)로 표시합니다.
4. hot path만 골라 그 자리에서(Just-In-Time) 대상 CPU의 네이티브 기계어로 컴파일합니다.
5. 컴파일된 기계어를 캐시해두고, 이후 같은 코드가 실행될 때는 인터프리터를 건너뛰고 캐시된 기계어를 바로 씁니다.

> **Q.** 그러면 처음부터 전부 컴파일하면 되지, 왜 굳이 실행하다가 나중에 컴파일할까요?
> **A.** 프로그램 전체를 컴파일하는 데는 시간과 메모리 비용이 들고, 실제로는 코드의 일부(흔히 소수의 함수·루프)만 반복적으로 실행되며 전체 실행 시간을 좌우합니다. 한 번만 실행되고 마는 코드까지 공들여 컴파일하는 것은 낭비입니다. JIT는 "자주 실행되는 부분에만 컴파일 비용을 투자"함으로써 컴파일 비용 대비 이득을 극대화합니다.

> **참고** JIT에도 대가가 있습니다. 프로그램 시작 직후에는 아직 프로파일링이 끝나지 않아 인터프리터로 도는 구간(워밍업, warm-up)이 있고, 컴파일된 기계어를 저장할 메모리도 추가로 필요합니다. 그래서 아주 짧게 실행되고 끝나는 스크립트에는 JIT의 이득이 크지 않을 수 있습니다. CPython도 3.13부터 실험적인 JIT 컴파일러를 도입하기 시작했지만, 기본 실행 모델은 여전히 순수 바이트코드 인터프리터입니다.

> **핵심**
> - AOT 컴파일은 **실행 전에** 모든 코드를 기계어로 확정해 실행 시점 오버헤드를 없애고, 인터프리터는 **실행 중에** 매 명령어를 해석해 이식성과 유연성을 얻습니다.
> - JIT 컴파일은 **자주 실행되는 부분만 실행 중에 골라 기계어로 컴파일**함으로써, 인터프리터의 유연성을 유지하면서 AOT에 근접한 성능을 노리는 절충안입니다.
