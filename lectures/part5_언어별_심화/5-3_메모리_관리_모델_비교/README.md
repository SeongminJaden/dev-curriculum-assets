# Part 5. 프로그래밍 언어별 심화

## 5-3. 메모리 관리 모델 비교 — GC 없이 Rust는 어떻게 안전한가

> **학습 목표**
> - 수동 메모리 관리(C)에서 발생하는 대표적 오류(use-after-free, double free, 메모리 누수)의 원인을 설명할 수 있다.
> - 참조 카운팅과 추적 GC(traced GC)의 동작 원리와 트레이드오프를 설명할 수 있다.
> - Rust의 소유권·빌림 규칙을 정확한 규칙과 예시 코드로 설명하고, 이 규칙이 왜 GC 없이도 메모리 안전성을 보장하는지 설명할 수 있다.

> **개요** 힙에 할당된 메모리는 결국 누군가 해제해야 합니다. C는 이를 개발자에게, Python·Java는 런타임의 가비지 컬렉터(GC)에게 맡깁니다. Rust는 제3의 길을 택했습니다 — 소유권과 빌림이라는 규칙을 컴파일러가 정적으로 검사해, 런타임 개입 없이도 안전성을 보장합니다. 이 강의는 이 세 모델을 나란히 비교합니다.

---

## 1. 메모리 관리라는 근본 문제

> **Q.** 힙에 할당한 메모리를 "정확히 언제" 해제해야 할까요?
> **A.** 너무 일찍 해제하면 아직 그 메모리를 가리키는 참조가 남아있는 상태에서 그 참조를 사용하는 <u>**댕글링 포인터(dangling pointer)**</u> 문제가 생기고, 너무 늦게(또는 영영) 해제하지 않으면 쓰지 않는 메모리가 계속 쌓이는 <u>**메모리 누수(memory leak)**</u>가 됩니다. 정답은 "그 메모리를 아무도 더 이상 참조하지 않는 바로 그 순간"이지만, 이 순간을 누가 어떻게 판단하느냐에서 언어마다 다른 답을 내놓습니다.

## 2. 수동 관리 — C의 malloc/free

> **개요** 개발자가 해제 시점을 직접 코드로 결정하는 방식과, 이 자유가 만드는 대표적인 오류들을 다룹니다.

C는 `malloc()`으로 힙 메모리를 할당하고 `free()`로 명시적으로 해제합니다. 컴파일러도 런타임도 이 시점이 옳은지 검사해주지 않습니다.

```c
char *s = malloc(6);
strcpy(s, "hello");
free(s);
printf("%s\n", s);  // use-after-free: 이미 해제된 메모리에 접근
```

- **use-after-free** — 위 예시처럼 해제된 메모리에 접근하면, 그 메모리가 이미 다른 용도로 재사용되었을 수 있어 예측 불가능한 값을 읽거나 프로그램이 크래시합니다.
- **double free** — 같은 포인터에 `free()`를 두 번 호출하면 메모리 할당기(allocator)의 내부 상태가 손상되어 이후 할당·해제가 전부 불안정해집니다.
- **memory leak** — `free()` 호출을 잊으면 그 메모리는 프로그램이 끝날 때까지 회수되지 않습니다.

> **참고** 이 오류들이 무서운 이유는 컴파일도 되고, 대부분의 경우 실행도 되기 때문입니다. 오류가 항상 즉시 크래시로 나타나는 것도 아니라서, 실제 서비스에서 드물게만 발생하는 조건에서 뒤늦게 발견되는 보안 취약점(버퍼 오버플로, 임의 코드 실행)으로 이어지는 경우가 많습니다. C가 지금도 운영체제 커널·임베디드에 쓰이는 것은 이 자유가 주는 성능 때문이지만, 그 자유의 대가가 바로 이 오류들입니다.

## 3. 자동 관리 — 참조 카운팅과 추적 GC

> **개요** Python의 참조 카운팅과 Java의 추적 GC가 "해제 시점 판단"을 런타임에 자동화하는 원리와, 그 대가로 생기는 비용을 다룹니다.

<u>**참조 카운팅(Reference Counting)**</u>은 모든 객체에 "지금 몇 개의 참조가 나를 가리키고 있는가"를 세는 카운터를 붙여두고, 이 카운터가 0이 되는 순간 즉시 메모리를 회수하는 방식입니다. Python(CPython)의 기본 메모리 관리 방식이 여기 해당합니다.

```python
a = [1, 2, 3]   # 리스트 객체의 참조 카운트 = 1
b = a           # 참조 카운트 = 2
del a           # 참조 카운트 = 1
del b           # 참조 카운트 = 0 → 즉시 메모리 회수
```

> **Q.** 참조 카운팅만으로 충분하지 않은 경우는 언제일까요?
> **A.** 두 객체가 서로를 참조하는 <u>**순환 참조(reference cycle)**</u>가 생기면, 둘 다 참조 카운트가 0이 될 일이 없어 아무도 가리키지 않아도 회수되지 않습니다. 그래서 CPython은 참조 카운팅과 별도로, 도달 가능성을 주기적으로 검사해 순환 참조를 찾아내는 세대별(generational) 추적 GC를 함께 운영합니다.

<u>**추적 GC(Traced GC)**</u>는 Java(JVM)가 쓰는 방식으로, 프로그램(스택·전역 변수 등)에서 출발해 참조를 따라가며 "도달 가능한(reachable)" 객체를 표시(mark)하고, 표시되지 않은 객체를 회수(sweep)합니다. 참조 카운팅과 달리 순환 참조도 정확히 찾아냅니다.

> **참고** 자동 관리의 대가는 두 가지입니다. 첫째, GC가 도달 가능성을 검사하는 동안 프로그램 실행이 잠시 멈추는 <u>**stop-the-world 일시정지**</u>가 발생할 수 있습니다(최신 GC는 이를 최소화하려 병렬·동시 수집 기법을 씁니다). 둘째, 카운터나 마킹 정보를 관리하는 데 추가 메모리와 CPU 시간이 듭니다. 대신 개발자는 해제 시점을 전혀 신경 쓸 필요가 없어집니다.

## 4. Rust의 소유권과 빌림 — 컴파일 시점에 검증하기

> **개요** 이 강의의 핵심 질문 — GC 없이 Rust가 메모리 안전성을 보장하는 원리 — 에 정확한 규칙과 코드로 답합니다.

Rust는 런타임 GC도, 수동 `free()`도 요구하지 않습니다. 대신 컴파일러가 아래 규칙을 정적으로 검사해, 규칙을 위반하는 코드는 **아예 컴파일되지 않게** 막습니다.

### 소유권 규칙 (Ownership)

1. 모든 값은 정확히 하나의 <u>**소유자(owner)**</u>를 가진다.
2. 소유자가 스코프(중괄호 블록 등)를 벗어나면, 그 값은 자동으로 <u>**drop**</u>되어(소멸자가 호출되어) 메모리가 즉시 해제된다.
3. 값은 다른 변수에 대입되거나 함수에 전달될 때 <u>**이동(move)**</u>할 수 있으며, 이동 이후 원래 소유자는 무효화되어 더 이상 사용할 수 없다.

```rust
fn main() {
    let s1 = String::from("hello"); // s1이 이 문자열의 소유자
    let s2 = s1;                    // 소유권이 s2로 이동(move), s1은 이제 무효

    println!("{}", s1); // 컴파일 에러: value borrowed here after move
}
```

### 빌림 규칙 (Borrowing)

1. 특정 시점에, 하나의 값에 대해 **가변 참조(`&mut T`) 하나** 또는 **불변 참조(`&T`) 여러 개** 중 하나만 존재할 수 있다. 가변 참조와 불변 참조는 동시에 존재할 수 없다.
2. 모든 참조는 항상 유효한 대상을 가리켜야 한다 — 즉 원본 값보다 참조가 더 오래 살아남는 <u>**댕글링 참조(dangling reference)**</u>는 허용되지 않는다.

```rust
fn main() {
    let mut s = String::from("hello");
    let r1 = &s;       // 불변 참조
    let r2 = &mut s;   // 컴파일 에러: 불변 참조(r1)가 살아있는 동안 가변 참조를 만들 수 없다
    println!("{}, {}", r1, r2);
}
```

```rust
fn dangle() -> &String {
    let s = String::from("hello");
    &s  // 컴파일 에러: s는 함수가 끝나며 drop되므로, 이 참조는 댕글링 참조가 된다
}
```

![Rust 소유권 이동과 빌림 규칙 흐름 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part5_언어별_심화/5-3_메모리_관리_모델_비교/assets/diagram2_rust_ownership_flow.svg)

> **원칙** Rust의 <u>**빌림 검사기(Borrow Checker)**</u>는 컴파일 시점에 모든 참조의 <u>**라이프타임(lifetime, 유효 범위)**</u>을 정적으로 추론합니다. 위 두 규칙 중 하나라도 위반되면, 그 코드는 실행되기는커녕 컴파일 자체가 실패합니다. 즉 use-after-free, double free, 데이터 레이스 같은 오류는 "런타임에 발생하는 문제"에서 "컴파일 타임에 거부되는 문제"로 자리를 옮깁니다. 이 발상은 학계의 선형 타입(linear type) 이론과, 이를 실용적 언어에 적용한 Cyclone 같은 연구 언어의 계보를 잇습니다.

> **Q.** 그러면 Rust는 GC의 역할을 어떻게 대신하는 걸까요?
> **A.** GC가 "런타임에 도달 가능성을 검사해 해제 시점을 결정"한다면, Rust는 "컴파일 시점에 스코프 규칙으로 해제 시점을 이미 확정"해버립니다. 모든 값의 소유자가 스코프를 벗어나는 지점은 컴파일 시점에 정확히 알 수 있으므로, 컴파일러가 그 지점에 `drop` 호출을 자동으로 삽입합니다. GC가 하던 "판단"을, Rust는 컴파일러의 "정적 증명"으로 대체한 것입니다.

## 5. 실생활 예시 — 같은 use-after-free가 언어마다 겪는 운명

> **해석** 아래 표는 "이미 해제된 데이터를 실수로 다시 사용하려는" 동일한 상황이 세 모델에서 각각 어떻게 처리되는지 보여줍니다.

| 모델 | 같은 실수가 벌어졌을 때 | 문제가 드러나는 시점 |
|---|---|---|
| 수동 관리 (C) | `free(s)` 이후에도 `s`를 계속 사용 — 컴파일도, 대부분 실행도 된다 | 실행 중, 그것도 운이 나쁠 때만(조용히 잘못된 값을 반환할 수도 있음) |
| 자동 관리 (Python/Java) | 참조가 남아있는 한 GC가 회수하지 않으므로 이 상황 자체가 발생하지 않는다 | 대신 실수로 참조를 계속 들고 있으면 메모리 누수로 나타날 수 있음 |
| 소유권 (Rust) | 이동된 값이나 스코프를 벗어난 값을 참조하는 코드는 컴파일러가 애초에 거부한다 | 컴파일 시점 — 실행 파일 자체가 만들어지지 않는다 |

> **팁** 이 표가 보여주는 것은 "Rust가 무조건 더 안전하다"가 아니라, **같은 종류의 버그가 언어마다 다른 단계(실행 중 vs 컴파일 중)에서 처리된다**는 것입니다. 대신 Rust는 이 안전성을 얻는 대가로, 빌림 검사기의 규칙에 맞게 코드를 구조화해야 하는 학습 곡선을 요구합니다.

> **핵심**
> - 세 모델의 차이는 "안전한가"가 아니라 **"언제, 누가 해제 시점을 결정하는가"**입니다 — C는 개발자가 런타임에, GC 언어는 런타임 자체가, Rust는 컴파일러가 컴파일 시점에 결정합니다.
> - Rust는 **모든 값에 하나의 소유자만 허용하고(소유권), 참조의 가변성과 라이프타임을 제한(빌림)**하는 두 규칙을 컴파일 시점에 정적으로 검사함으로써, GC 없이도 use-after-free·data race 같은 오류를 원천 차단합니다.
> - 그 결과 런타임 오류였던 문제가 컴파일 타임 오류로 옮겨가며, 이는 GC의 stop-the-world 비용도, 수동 관리의 위험도 없는 대신 더 엄격한 컴파일 시점 규칙 준수를 요구합니다.
