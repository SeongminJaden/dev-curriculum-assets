# 5-3. 메모리 관리 모델 비교 — 실습

> **참고** 제출물이 아니라, 세 가지 메모리 관리 모델(수동 관리, 자동 관리, 소유권)의 차이를 직접 코드로 확인해보는 과정입니다. 아래 체크리스트를 순서대로 진행해보세요.

## 실습 1 — C에서 수동 관리 오류를 직접 재현하기

- [ ] `malloc()`으로 문자열 하나를 힙에 할당하고 값을 채운 뒤, `free()`로 해제해보세요.
- [ ] `free()` 호출 이후에도 같은 포인터로 값을 출력해보고, 어떤 일이 벌어지는지 관찰해보세요(정상 출력, 깨진 값, 크래시 중 무엇이든 기록하세요).
- [ ] 같은 포인터에 `free()`를 한 번 더 호출해보고(double free), 실행 결과를 기록해보세요.
- [ ] 두 실험에서 컴파일러가 이 실수를 막아주지 않은 이유를 본문에서 배운 "수동 관리" 개념으로 설명해보세요.

## 실습 2 — Python에서 참조 카운트 관찰하기

- [ ] `sys.getrefcount()`를 사용해, 리스트 객체 하나를 여러 변수에 대입할 때마다 참조 카운트가 어떻게 변하는지 확인해보세요.
- [ ] 서로를 참조하는 두 객체(순환 참조)를 만들어보고, 두 변수를 모두 `del`한 뒤에도 메모리가 바로 회수되지 않을 수 있는 이유를 설명해보세요.
- [ ] `gc` 모듈의 역할이 참조 카운팅만으로 부족한 부분을 어떻게 보완하는지 한 문단으로 정리해보세요.

## 실습 3 — Rust 소유권 규칙을 컴파일러로 직접 확인하기

- [ ] 본문의 `String` 이동(move) 예시 코드를 그대로 작성하고 컴파일해, 정확히 어떤 오류 메시지가 나오는지 확인해보세요.
- [ ] 오류가 난 코드를 `s1.clone()`을 사용해 수정하고, 왜 `clone()`이 이 오류를 해결하는지 설명해보세요.
- [ ] 본문의 가변 참조·불변 참조 동시 사용 예시 코드를 작성해 컴파일 오류를 재현하고, 오류 메시지에서 "빌림 규칙" 중 어떤 항목을 위반했는지 짚어보세요.
- [ ] 댕글링 참조를 반환하는 `dangle()` 함수 예시를 작성해 컴파일해보고, 컴파일러가 제안하는 해결 방법(예: 소유권을 그대로 반환하기)을 적용해 수정해보세요.

## 생각해볼 질문

- [ ] 세 모델(C, Python/Java, Rust) 중 실시간성이 중요한 게임 엔진이나 임베디드 시스템에서 GC 기반 언어를 기피하는 이유를 "stop-the-world 일시정지" 개념으로 설명해보세요.
- [ ] Rust의 빌림 검사기가 컴파일 시점에 거부하는 코드가, 만약 C였다면 실행 중 어떤 문제로 나타났을지 구체적인 시나리오로 설명해보세요.
