# Part 5. 프로그래밍 언어별 심화

## 5-2. 정적 타입과 동적 타입 — 타입을 언제 확정하는가

> **학습 목표**
> - 정적 타입 시스템과 동적 타입 시스템이 "언제 타입을 검사하는가"로 갈린다는 것을 구조적으로 설명할 수 있다.
> - 타입을 컴파일 시점에 고정하는 것이 어떤 종류의 오류를 방지하기 위한 장치인지 설명할 수 있다.
> - JavaScript의 동적 타입 때문에 생기는 실제 버그 패턴과, TypeScript가 이를 어떻게 컴파일 시점에 차단하는지 비교할 수 있다.

> **개요** Java·TypeScript처럼 컴파일 시점에 모든 값의 타입을 고정하는 언어와, Python·JavaScript처럼 실행 중에 값이 자기 타입을 들고 다니는 언어를 비교합니다. 정적 타이핑이 무엇을 방지하려는 장치인지, 그리고 그 대가로 동적 타이핑이 주는 유연성이 무엇인지를 다룹니다.

---

## 1. 타입 시스템은 왜 존재할까요?

> **Q.** 애초에 타입이라는 개념이 왜 필요할까요? 그냥 값만 계산하면 안 될까요?
> **A.** 타입은 "이 값에 어떤 연산을 허용할 것인가"를 정하는 규칙이기 때문입니다. 문자열 `"3"`에 정수 `1`을 더하는 연산은 언어마다 결과가 다르게 정의되어 있고, 어떤 언어는 이를 아예 프로그래머의 실수로 간주해 막아버립니다. <u>**타입 시스템(Type System)**</u>은 프로그램이 의미상 말이 되지 않는 연산(정수에 문자열을 곱한다, 존재하지 않는 속성에 접근한다)을 걸러내기 위한 형식적 규칙 체계입니다.

컴퓨터 과학자 로빈 밀너(Robin Milner)는 1978년 "잘 타입 지정된 프로그램은 잘못될 수 없다(Well-typed programs cannot go wrong)"는 유명한 명제를 남겼습니다. 이는 타입 검사를 통과한 프로그램이라면 적어도 "타입 오류로 인한 예기치 못한 동작"만큼은 실행 중에 절대 일어나지 않는다는, <u>**타입 건전성(Type Soundness)**</u>이라는 개념의 학문적 뿌리입니다.

---

## 2. 정적 타입과 동적 타입 — 검사 시점의 차이

> **개요** 정적 타입과 동적 타입이 "타입을 언제 확정하고 검사하는가"라는 단 하나의 기준으로 갈린다는 것을 살펴봅니다.

- <u>**정적 타입(Static Typing)**</u>: 변수와 표현식의 타입이 **컴파일 시점**에 결정되고 검사됩니다. Java, C, TypeScript, Rust가 여기 속합니다.
- <u>**동적 타입(Dynamic Typing)**</u>: 타입이 변수가 아니라 **값 자체**에 붙어 있고, 검사는 그 값에 연산이 실제로 적용되는 **실행 시점**에 이루어집니다. Python, JavaScript, Ruby가 여기 속합니다.

![정적 타입과 동적 타입의 검사 시점 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part5_언어별_심화/5-2_정적_타입과_동적_타입/assets/diagram1_type_check_timing.svg)

> **참고** 정적/동적 구분과 별개로 "강한 타입(strong typing)/약한 타입(weak typing)"이라는 축도 있습니다. 이는 암묵적 타입 변환을 얼마나 허용하는가의 문제로, 정적 타입 언어(C)도 약한 타입일 수 있고 동적 타입 언어(Python)도 강한 타입일 수 있습니다. 이 강의에서는 "언제 검사하는가"에 해당하는 정적/동적 축에 집중합니다.

## 3. 정적 타입이 방지하는 것 — 핵심 질문에 대한 답

> **개요** "타입을 컴파일 시점에 고정하는 것은 무엇을 방지하기 위함인가"라는 이 강의의 핵심 질문에 정면으로 답합니다.

컴파일러가 타입을 미리 고정하고 검사하는 근본적인 목적은, **실행되지 않은 코드 경로에 숨어 있는 타입 오류까지 실행 전에 찾아내는 것**입니다.

```java
// Java
int add(int a, int b) {
    return a + b;
}

add(3, "hello"); // 컴파일 자체가 실패한다 — 실행조차 되지 않는다
```

이 코드가 실제로 호출되는 경로는 프로그램 안에 단 한 번뿐일 수도 있고, 특정 조건에서만 실행되는 드문 분기일 수도 있습니다. 동적 타입 언어라면 이 분기가 실제로 실행될 때까지 오류가 드러나지 않습니다. 정적 타입 언어는 그 분기가 한 번도 실행되지 않아도, 컴파일이라는 단 한 번의 전수 검사로 모든 경로의 타입 정합성을 확인합니다.

> **원칙** 정적 타입 검사가 막으려는 것은 "오류가 일어나는 것" 자체가 아니라, **"오류가 실행 중, 그것도 하필 그 코드 경로가 실행되는 특정 순간에 발견되는 것"**입니다. 오류를 컴파일 시점으로 앞당길수록, 수정 비용은 줄어듭니다. 이는 1-1강에서 다룬 "오류 발견 시점을 앞당길수록 수정 비용이 준다"는 SDLC의 원칙이 타입 시스템 차원에서 반복되는 것입니다.

## 4. 동적 타입이 주는 유연성과 그 대가

> **개요** 동적 타입이 왜 지금도 널리 쓰이는지, 정적 타입과 맞바꾸는 것이 무엇인지 다룹니다.

동적 타입 언어는 컴파일 단계 자체가 없거나 가볍기 때문에 다음과 같은 유연성을 얻습니다.

- **덕 타이핑(Duck Typing)** — "오리처럼 걷고 오리처럼 울면 오리로 취급한다"는 관용구처럼, 객체의 실제 타입을 선언하지 않고도 필요한 메서드만 있으면 그대로 사용할 수 있습니다. 제네릭 프로그래밍을 별도 문법 없이 자연스럽게 구현합니다.
- **빠른 프로토타이핑** — 타입을 미리 설계하지 않아도 되므로, 데이터 구조가 자주 바뀌는 초기 개발 단계에서 코드를 빠르게 수정할 수 있습니다.
- **런타임 유연성** — 실행 중에 객체에 새 속성을 추가하거나, 함수를 다른 함수로 교체하는 등의 동적 기능을 자연스럽게 지원합니다.

> **Q.** 그렇다면 이 유연성의 대가는 정확히 무엇일까요?
> **A.** 타입 오류가 코드 어딘가에 숨어 있어도, 그 코드 경로가 실행되기 전까지는 아무도 알 수 없다는 것입니다. 테스트 커버리지가 100%가 아닌 이상, 프로덕션에 배포된 뒤에야 특정 사용자의 특정 입력값이 그 경로를 타면서 오류가 드러날 수 있습니다. 정적 타입이 "컴파일러가 대신 모든 경로를 검사"해주는 대신, 동적 타입은 "그 경로가 실행되는 순간"까지 검사를 미룹니다.

## 5. 실생활 예시 — JavaScript의 동적 타입이 만든 버그

> **해석** 아래 표는 JavaScript(동적 타입)에서 실제로 자주 나타나는 버그 패턴과, TypeScript(정적 타입 레이어)가 같은 상황을 컴파일 시점에 어떻게 차단하는지 보여줍니다.

| 상황 | JavaScript(동적 타입)에서 벌어지는 일 | TypeScript(정적 타입)가 막는 방법 |
|---|---|---|
| 함수에 잘못된 타입의 인자를 전달 | 오류 없이 실행되다가, 값이 쓰이는 시점에 `NaN`이나 `undefined`가 튀어나온다 | 매개변수 타입과 다르면 그 자리에서 컴파일 실패 |
| 존재하지 않는 객체 속성에 접근 | `undefined`를 조용히 반환하고, 그 `undefined`에 다시 접근할 때 비로소 `TypeError`가 발생 | 정의되지 않은 속성 접근을 컴파일 시점에 오류로 표시 |
| 함수 호출 시 인자 개수가 다름 | JavaScript는 인자 개수를 검사하지 않아 오타로 인자를 빠뜨려도 그대로 실행된다 | 함수 시그니처와 호출부의 인자 개수·타입을 대조해 불일치 시 컴파일 실패 |
| 문자열과 숫자의 암묵적 덧셈 | `"1" + 1`은 문자열 결합이 되어 `"11"`이 되지만, 개발자는 산술 덧셈 `2`를 기대했을 수 있다 | 결합 대상의 타입이 다르면 의도한 연산인지 타입 수준에서 드러난다 |

![JavaScript와 TypeScript에서 같은 실수가 흘러가는 길 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part5_언어별_심화/5-2_정적_타입과_동적_타입/assets/diagram2_js_vs_ts_bug_flow.svg)

> **팁** 실무에서 TypeScript가 널리 도입된 이유는 "타입을 쓰는 게 트렌디해서"가 아니라, 표에 나온 것과 같은 버그 패턴이 대규모 JavaScript 코드베이스에서 반복적으로 발생했고, 그 발견 시점을 프로덕션에서 컴파일 시점(개발자의 에디터 안)으로 앞당기려는 실용적 필요 때문이었습니다.

## 6. 점진적 타입 지정 — 둘 사이의 다리

> **개요** 완전한 정적 타입과 완전한 동적 타입 사이에서, 실무가 택한 절충안인 점진적 타이핑을 다룹니다.

TypeScript는 JavaScript 문법을 그대로 포함하면서, 타입 표기를 선택적으로 추가할 수 있는 <u>**점진적 타이핑(Gradual Typing)**</u>을 채택합니다. 타입을 명시한 부분은 컴파일 시점에 검사되고, 타입을 명시하지 않은 부분(`any` 등)은 기존 JavaScript처럼 동작합니다. 컴파일이 끝나면 타입 정보는 제거(<u>**타입 소거, Type Erasure**</u>)되어 순수 JavaScript로 변환됩니다 — 즉 타입 검사는 컴파일 시점에만 존재하고, 런타임에는 어떤 타입 정보도 남지 않습니다.

Python도 PEP 484(2014)를 통해 타입 힌트(`def add(a: int, b: int) -> int:`) 문법을 도입했지만, 이는 `mypy` 같은 외부 도구가 검사할 뿐 Python 인터프리터 자체는 이 힌트를 강제하지 않습니다. 즉 Python의 타입 힌트는 "문서화 + 외부 검사 도구용 힌트"에 가깝고, 언어 차원의 정적 타입 보장은 아닙니다.

> **참고** 두 경우 모두 "정적 타입의 안전성"과 "동적 타입의 기존 생태계·유연성"을 동시에 취하려는 절충이라는 점에서 목적이 같습니다. 다만 TypeScript는 컴파일러가 직접 오류를 컴파일 실패로 막는 반면, Python 타입 힌트는 강제성이 없는 부가 도구라는 차이가 있습니다.

> **핵심**
> - 정적 타입은 **컴파일 시점**에, 동적 타입은 **실행 시점**에 타입을 검사하며, 이 차이가 오류 발견 시점을 가릅니다.
> - 타입을 컴파일 시점에 고정하는 이유는 **실행되지 않은 코드 경로에 숨은 타입 오류까지 실행 전에 전부 찾아내기 위함**입니다.
> - 동적 타입은 그 대가로 검사를 실행 시점까지 미루는 대신, 덕 타이핑과 런타임 유연성, 빠른 프로토타이핑을 얻습니다.
> - TypeScript의 점진적 타이핑은 두 세계를 절충해, 기존 JavaScript 자산을 유지하면서 정적 검사의 이점을 부분적으로 도입하는 방식입니다.
