# Part 1. 개발 기초

## 1-1. 소프트웨어 개발의 전체 그림 — 요구사항부터 배포까지

> **학습 목표**
> - 소프트웨어 개발 생명주기(SDLC)의 각 단계와 그 존재 이유를 설명할 수 있다.
> - 하나의 요청이 사용자로부터 하드웨어까지 어떤 계층을 거치는지 구조적으로 그릴 수 있다.
> - 이후 파트(컴퓨터 구조론, 서버·아키텍처, 백엔드/프론트엔드)가 이 전체 그림의 어느 위치에 해당하는지 말할 수 있다.

> **개요** 코드 한 줄을 배우기 전에, 그 코드가 태어나서(기획) 사용자에게 도달하기까지(배포·운영) 전체 여정과, 실행되는 순간 거치는 계층 구조를 먼저 살펴봅니다. 이 두 장의 지도가 앞으로 배울 모든 파트의 좌표계가 됩니다.

---

## 1. 왜 "전체 그림"부터 봐야 할까요?

> **Q.** 언어 문법이나 프레임워크 사용법부터 배우지 않고, 왜 개발 생명주기와 계층 구조부터 볼까요?
> **A.** 개별 기술은 전체 그림 속 정확한 위치를 모르면 "언제 무엇을 써야 하는가"를 판단할 수 없기 때문입니다. 같은 코드라도 기획 단계의 산출물인지, 운영 중 발견된 버그 수정인지에 따라 접근 방식이 완전히 달라집니다. 마찬가지로 프론트엔드 코드 한 줄이 사용자 화면부터 하드웨어까지 이어지는 6개 계층 중 정확히 어디에 있는지 알아야, 성능 문제가 생겼을 때 "어느 계층을 의심해야 하는가"를 판단할 수 있습니다.

이 강의는 두 개의 지도를 그립니다.

- **지도 1 — 시간 축**: 소프트웨어가 기획부터 운영까지 거치는 단계 (SDLC)
- **지도 2 — 계층 축**: 하나의 요청이 실행되는 순간 거치는 물리적·논리적 계층

## 2. 소프트웨어 개발 생명주기 (SDLC)

> **개요** 소프트웨어 공학에서 정의하는 6단계 생명주기를 살펴보고, 각 단계가 왜 분리되어 있는지를 다룹니다.

소프트웨어 개발 생명주기(Software Development Life Cycle, SDLC)는 1970년 윈스턴 로이스(Winston Royce)가 대규모 소프트웨어 프로젝트의 관리 문제를 정리하면서 처음 체계화한 개념으로, 이후 폭포수(Waterfall) 모델과 애자일(Agile) 모델 등 다양한 실행 방식으로 발전했습니다. 방식은 달라도 아래 6단계 자체는 거의 모든 개발 방법론에 공통으로 존재합니다.

![소프트웨어 개발 생명주기 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part1_개발기초/1-1_소프트웨어_개발의_전체_그림/assets/diagram1_sdlc.svg)

1. **요구사항 정의** — 사용자와 이해관계자가 원하는 것이 무엇인지 문서로 확정합니다.
2. **설계** — 요구사항을 만족시킬 시스템의 구조(모듈, 데이터베이스, 인터페이스)를 결정합니다.
3. **구현** — 설계를 실제 코드로 작성합니다.
4. **빌드·테스트** — 코드를 실행 가능한 형태로 컴파일하고, 의도대로 동작하는지 검증합니다.
5. **배포** — 검증된 결과물을 실제 사용자가 쓰는 운영 환경에 반영합니다.
6. **운영·유지보수** — 실제 사용 중 발생하는 문제를 모니터링하고 개선하며, 그 결과가 다시 새로운 요구사항이 됩니다.

> **참고** 실제 현업에서는 이 단계를 한 번만 순서대로 거치지 않습니다. 애자일 방법론에서는 1~5단계를 1~4주 단위의 짧은 주기(스프린트)로 반복하며, 매 반복마다 6단계에서 얻은 피드백을 다음 반복의 1단계에 반영합니다. 다이어그램의 점선 화살표가 이 피드백 순환을 나타냅니다.

### 실생활 예시 — 배달 앱에 리뷰 기능을 추가하는 경우

> **해석** 아래 표는 "배달 앱에 별점 리뷰 기능을 추가해 달라"는 요청이 SDLC 6단계를 어떻게 통과하는지 보여줍니다.

| 단계 | 실제로 벌어지는 일 |
|---|---|
| 요구사항 정의 | 기획자가 "주문 완료 후 별점과 사진을 남길 수 있어야 한다"를 문서로 확정 |
| 설계 | 리뷰 데이터를 저장할 테이블 구조, 화면 흐름(주문 내역 → 리뷰 작성 버튼)을 설계 |
| 구현 | 백엔드 개발자가 리뷰 저장 API를, 프론트엔드 개발자가 별점 입력 화면을 작성 |
| 빌드·테스트 | 별점이 1~5 범위를 벗어나면 저장되지 않는지, 사진 업로드가 정상 동작하는지 검증 |
| 배포 | 검증이 끝난 코드를 실제 서비스 서버에 반영 |
| 운영·유지보수 | 리뷰에 욕설이 섞여 들어오는 문제를 발견해, 다음 반복의 요구사항(필터링 기능)으로 연결 |

> **핵심**
> - SDLC의 각 단계가 분리된 이유는 **하나의 실수가 다음 단계 전체를 무효화하는 비용을 줄이기 위함**입니다. 설계 단계에서 발견했으면 문서 한 줄 수정으로 끝날 문제가, 배포 후 발견되면 전체 시스템을 되돌려야 할 수도 있습니다.
> - 애자일이 반복 주기를 짧게 가져가는 이유도 같습니다. **문제를 발견하는 시점과 실제 발생 시점 사이의 간격을 줄일수록 수정 비용이 줄어듭니다.**

## 3. 코드는 어떻게 "실행되는 것"이 되는가 — 계층 구조로 보기

> **개요** 배포까지 끝난 코드가 실제 사용자의 요청 한 번을 처리할 때 통과하는 6개 계층을 살펴봅니다. 이 계층 구조가 이후 Part 2~9의 목차와 그대로 대응됩니다.

사용자가 배달 앱에서 "주문하기" 버튼을 누르는 순간부터, 그 요청이 실제로 처리되어 화면에 결과가 표시되기까지는 아래와 같은 계층을 순서대로 통과합니다.

![요청 처리 계층 구조 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part1_개발기초/1-1_소프트웨어_개발의_전체_그림/assets/diagram2_stack.svg)

- <u>**사용자 계층**</u>: 브라우저나 앱이 버튼 클릭을 감지합니다.
- <u>**프론트엔드 계층**</u>: 클릭 이벤트를 서버가 이해할 수 있는 HTTP 요청으로 변환합니다.
- <u>**네트워크 계층**</u>: 이 요청이 TCP/IP를 통해 물리적으로 떨어진 서버까지 전달됩니다.
- <u>**백엔드 서버 계층**</u>: 요청을 받아 "재고가 있는가", "결제가 가능한가" 같은 비즈니스 로직을 처리합니다.
- <u>**운영체제 계층**</u>: 이 처리 과정에서 필요한 CPU 시간과 메모리를 프로세스에 배분합니다.
- <u>**하드웨어 계층**</u>: 실제 트랜지스터의 전기 신호로 명령어가 실행됩니다.

> **Q.** 개발자가 프론트엔드나 백엔드 코드만 작성하는데, 왜 운영체제와 하드웨어 계층까지 알아야 할까요?
> **A.** 평소에는 몰라도 됩니다. 하지만 "똑같은 코드인데 특정 서버에서만 느리다", "메모리가 부족하다는 오류가 난다"처럼 상위 계층의 논리만으로는 설명되지 않는 문제가 반드시 나타납니다. 이런 문제는 하위 계층(운영체제의 메모리 관리, CPU의 캐시 구조)을 이해해야 원인을 좁힐 수 있습니다. 이 커리큘럼이 Part 2(컴퓨터 구조론)부터 시작해 위로 올라가는 이유가 여기 있습니다.

> **참고** 위 6개 계층은 이후 커리큘럼과 다음과 같이 대응됩니다.
> - 하드웨어 계층 → Part 2(컴퓨터 구조론), Part 4(x86 vs ARM, 통합 메모리 아키텍처)
> - 운영체제·네트워크 계층 → Part 3(서버와 시스템 아키텍처)
> - 백엔드 서버 계층 → Part 6(기능별 심화), Part 7(백엔드 개발)
> - 프론트엔드·사용자 계층 → Part 8(프론트엔드 개발)
> - 모든 계층을 관통하는 위협 → Part 9(보안)

## 요약

> **핵심**
> - SDLC는 요구사항 정의 → 설계 → 구현 → 빌드·테스트 → 배포 → 운영·유지보수의 6단계이며, **단계를 분리하는 이유는 오류 발견 시점을 앞당겨 수정 비용을 줄이기 위함**입니다.
> - 하나의 요청은 사용자 → 프론트엔드 → 네트워크 → 백엔드 서버 → 운영체제 → 하드웨어의 6개 계층을 통과하며, **이 커리큘럼의 Part 구성은 이 계층 구조를 그대로 따라갑니다.**
