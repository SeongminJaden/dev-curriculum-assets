# 1-2. 프로그래밍 패러다임 — 실습

> **참고** 이 실습은 제출물이 아니라, 강의에서 배운 세 가지 사고방식을 스스로 코드로 옮겨보는 과정입니다. 정답은 하나가 아니며, 각 단계를 마칠 때마다 아래 체크리스트에 표시해보세요.

## 실습 1 — 같은 문제를 세 패러다임으로 풀어보기

"학생 명단에서 평균 점수를 구하는" 문제를 익숙한 언어 하나로 세 번 풀어보세요.

- [ ] 절차적 스타일로 작성하세요. 반복문과 누적 변수(`total`, `count`)만 사용하고, 함수 하나 안에서 처음부터 끝까지 처리하세요.
- [ ] 객체지향 스타일로 다시 작성하세요. `ClassRoom` 같은 객체를 만들고, 학생 목록을 내부 상태로 갖게 한 뒤 `getAverageScore()` 메서드를 호출하는 방식으로 바꾸세요.
- [ ] 함수형 스타일로 다시 작성하세요. `map`과 `reduce`(또는 사용 언어가 제공하는 동등한 고차 함수)만 사용하고, 중간에 값을 바꾸는 변수를 만들지 마세요.
- [ ] 세 코드를 나란히 놓고 줄 수, 새 함수/클래스 추가 여부를 비교해보세요.

## 실습 2 — 변경에 대한 강건성 비교

- [ ] "평균 대신 중앙값을 구해야 한다"는 요구사항 변경이 생겼다고 가정하고, 세 코드 중 어떤 것을 고치기 가장 쉬웠는지 적어보세요.
- [ ] "낙제 기준 이하 학생은 평균 계산에서 제외한다"는 조건이 추가됐다고 가정하고, 같은 비교를 반복해보세요.
- [ ] 두 변경 실험 결과를 바탕으로, 어떤 상황에서 객체지향이 유리했고 어떤 상황에서 함수형이 유리했는지 한 문단으로 정리해보세요.

## 실습 3 — 실무 코드에서 패러다임 찾아보기

- [ ] 최근에 본 오픈소스 코드나 과제 코드 하나를 골라, 그 코드가 절차적/객체지향/함수형 중 어떤 사고방식에 가까운지 판단해보세요.
- [ ] 그렇게 판단한 근거(전역 상태를 공유하는지, 객체가 메시지를 주고받는지, 순수 함수와 불변 데이터를 쓰는지)를 두세 줄로 적어보세요.
