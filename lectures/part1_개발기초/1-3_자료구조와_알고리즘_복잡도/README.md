# Part 1. 개발 기초

## 1-3. 자료구조와 알고리즘 복잡도 — Big-O는 왜 "느리다/빠르다"보다 정확한 언어인가

> **학습 목표**
> - Big-O 표기법의 수학적 정의를 이해하고, 알고리즘의 시간복잡도를 직접 도출할 수 있다.
> - 대표적인 복잡도 클래스(O(1), O(log n), O(n), O(n log n), O(n²), O(2ⁿ))를 구분하고 각각의 대표 예시를 설명할 수 있다.
> - 배열·연결 리스트·해시 테이블·이진 탐색 트리의 연산별 복잡도 차이를 근거로, 상황에 맞는 자료구조를 선택할 수 있다.

> **개요** "이 코드가 느려요"라는 말은 하드웨어 성능, 데이터 양, 구현 방식 중 무엇이 원인인지 알려주지 않습니다. 이 강의는 데이터 양이 늘어날 때 연산 횟수가 얼마나 늘어나는지를 정확히 기술하는 언어인 Big-O 표기법을 정의하고, 자료구조 선택이 이 복잡도에 어떤 영향을 주는지 살펴봅니다.

---

## 1. 왜 "빠르다/느리다" 대신 Big-O로 말할까요?

> **Q.** "이 알고리즘이 저 알고리즘보다 빠르다"고 실측하면 되지 않을까요? 왜 굳이 별도의 표기법이 필요할까요?
> **A.** 실측 시간은 CPU 성능, 프로그래밍 언어, 그날의 서버 부하에 따라 매번 달라지기 때문입니다. 어제 내 노트북에서 0.3초 걸린 코드가 오늘 더 빠른 서버에서는 0.1초가 걸릴 수 있습니다. 이런 값은 "이 알고리즘이 데이터 양이 늘어날수록 얼마나 나빠지는가"라는 근본적인 질문에 답하지 못합니다. Big-O 표기법은 하드웨어나 언어와 무관하게, 입력 크기가 커질 때 연산 횟수가 늘어나는 **추세**만을 표현합니다. 그래서 서로 다른 컴퓨터, 서로 다른 언어로 구현된 두 알고리즘도 Big-O로는 공정하게 비교할 수 있습니다.

<u>**시간복잡도(Time Complexity)**</u>는 입력 크기 $n$이 커질 때 알고리즘이 수행하는 연산 횟수가 어떤 함수 형태로 증가하는지를 나타냅니다. 이 개념을 계산 이론에 도입한 사람은 도널드 커누스(Donald Knuth)로, 1976년 논문에서 점근적 표기법(asymptotic notation)을 컴퓨터 과학의 표준 도구로 체계화했습니다. 다만 표기법 자체의 수학적 뿌리는 더 오래되어, 1894년 독일 수학자 파울 바흐만(Paul Bachmann)이 정수론에서 처음 사용한 표기를 에드문트 란다우(Edmund Landau)가 정교화한 것입니다.

## 2. Big-O의 정의 — 정확히 무엇을 보장하는 표기인가

> **개요** Big-O 표기법의 형식적 정의를 수식으로 살펴보고, 각 기호가 실제로 무엇을 의미하는지 하나씩 풀어봅니다.

함수 $f(n)$이 알고리즘의 실제 연산 횟수라고 할 때, $f(n) = O(g(n))$이라는 표기는 다음을 의미합니다.

$$f(n) = O(g(n)) \iff \exists\, c > 0,\ \exists\, n_0 \in \mathbb{N} : \forall n \geq n_0,\ 0 \leq f(n) \leq c \cdot g(n)$$

> **해석** 이 수식을 하나씩 풀면 다음과 같습니다.
> - $f(n)$: 알고리즘이 입력 크기 $n$에 대해 실제로 수행하는 연산 횟수
> - $g(n)$: 그 증가 추세를 비교할 기준 함수(예: $n$, $n^2$, $\log n$)
> - $c > 0$: 적당히 큰 상수 하나를 곱하는 것은 허용한다는 뜻 — 즉 정확한 계수(2배, 3배 차이)는 무시하고 **증가하는 형태**만 본다는 뜻입니다.
> - $n_0$: 입력이 아주 작을 때(예: $n=1, 2$)의 예외적인 경우는 무시하고, 입력이 충분히 커진 이후의 경향만 본다는 뜻입니다.
> - 전체 의미: "$n$이 충분히 커지면, $f(n)$은 $g(n)$에 상수를 곱한 값보다 절대 커지지 않는다" — 즉 $g(n)$이 $f(n)$의 <u>**점근적 상한(asymptotic upper bound)**</u>이라는 뜻입니다.

> **참고** Big-O는 "최악의 경우(worst case)" 상한을 보장하는 표기입니다. 이와 짝을 이루는 표기로 최선의 경우를 나타내는 Big-Ω(오메가), 평균적인 정확한 증가율을 나타내는 Big-Θ(세타)가 있습니다. 실무에서 "이 알고리즘은 O(n)이다"라고 말할 때는 보통 최악의 경우를 기준으로 삼습니다. 최악의 경우를 보장할 수 있어야 시스템 설계 시 안전한 상한선을 잡을 수 있기 때문입니다.

## 3. 대표적인 복잡도 클래스

> **개요** 실무에서 가장 자주 마주치는 6개의 복잡도 클래스를 증가율이 작은 순서대로 살펴보고, 각각의 대표 알고리즘 예시를 확인합니다.

![입력 크기에 따른 연산 수 증가 비교 그래프](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part1_개발기초/1-3_자료구조와_알고리즘_복잡도/assets/diagram1_bigo_graph.svg)

| 복잡도 | 이름 | 대표 예시 | $n=100$만일 때 체감 |
|---|---|---|---|
| $O(1)$ | 상수 시간 | 배열 인덱스로 값 접근 | 즉시 |
| $O(\log n)$ | 로그 시간 | 정렬된 배열의 이진 탐색 | 약 20회 비교로 종료 |
| $O(n)$ | 선형 시간 | 정렬 안 된 배열의 순차 탐색 | 최대 100만 회 비교 |
| $O(n \log n)$ | 선형로그 시간 | 병합 정렬, 퀵 정렬(평균) | 약 2000만 회 연산 |
| $O(n^2)$ | 제곱 시간 | 이중 반복문(버블 정렬) | 약 1조 회 연산 |
| $O(2^n)$ | 지수 시간 | 부분집합 전수 탐색 | 현실적으로 계산 불가능 |

> **해석** 위 그래프는 정확한 축척이 아니라 개념도이지만, 곡선의 순서 자체는 수학적으로 엄밀합니다. $n$이 커질수록 $O(n^2)$과 $O(2^n)$은 다른 곡선과의 격차가 기하급수적으로 벌어집니다. 예를 들어 원소 30개를 대상으로 한 부분집합 전수 탐색($O(2^n)$)은 $2^{30}$, 즉 약 10억 번의 연산이 필요합니다. 반면 같은 크기의 데이터를 이진 탐색($O(\log n)$)으로 찾는다면 5번 이내의 비교로 끝납니다.

> **Q.** $O(n^2)$ 알고리즘을 $O(n \log n)$으로 바꾸는 것이 실무에서 왜 그렇게 중요하게 다뤄질까요?
> **A.** 데이터 양이 10배로 늘면 $O(n)$ 알고리즘의 연산량도 10배로 늘지만, $O(n^2)$ 알고리즘의 연산량은 100배로 늘어나기 때문입니다. 서비스 초기에는 데이터가 적어 두 알고리즘의 체감 속도 차이가 거의 없다가, 사용자가 늘어나는 순간 $O(n^2)$ 코드만 갑자기 응답 지연이나 타임아웃을 일으키는 경우가 실무에서 매우 흔합니다. 정렬 알고리즘으로 병합 정렬·퀵 정렬($O(n \log n)$)이 버블 정렬·삽입 정렬($O(n^2)$)을 대체해 표준 라이브러리에 채택된 이유가 여기에 있습니다.

## 4. 자료구조별 연산 복잡도 — 같은 연산도 구조에 따라 비용이 다르다

> **개요** 배열, 연결 리스트, 해시 테이블, 균형 이진 탐색 트리가 접근·탐색·삽입·삭제라는 같은 이름의 연산을 내부적으로 얼마나 다른 비용으로 처리하는지 비교합니다.

<u>**자료구조(Data Structure)**</u>는 데이터를 저장하고 조직하는 방식이며, 어떤 방식을 택하느냐가 곧 각 연산의 시간복잡도를 결정합니다. 이는 자료구조와 알고리즘을 함께 다루는 이유이기도 합니다 — 알고리즘의 복잡도는 그 알고리즘이 사용하는 자료구조의 특성에서 직접 도출됩니다.

![자료구조별 연산 복잡도 비교 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part1_개발기초/1-3_자료구조와_알고리즘_복잡도/assets/diagram2_structures_compare.svg)

> **해석** 배열은 메모리에 값이 연속으로 저장되어 있어 인덱스만 알면 주소를 계산해 즉시 접근($O(1)$)할 수 있지만, 중간에 값을 삽입·삭제하려면 뒤 요소를 전부 밀어야 해 $O(n)$이 걸립니다. 연결 리스트는 그 반대로, 노드의 위치를 이미 알고 있다면 앞뒤 포인터만 바꾸면 되어 삽입·삭제가 $O(1)$이지만, 특정 위치를 찾아가려면 처음부터 순서대로 따라가야 해 접근이 $O(n)$입니다. 해시 테이블은 키를 해시 함수로 변환해 저장 위치를 계산하므로 평균적으로 $O(1)$에 탐색·삽입·삭제가 가능하지만, 여러 키가 같은 위치로 몰리는 충돌이 심해지면 최악의 경우 $O(n)$까지 저하될 수 있습니다. 균형 이진 탐색 트리는 정렬 순서를 유지하면서도 트리의 높이를 $\log n$으로 유지하도록 균형을 잡아, 최악의 경우에도 $O(\log n)$을 보장합니다.

> **참고** 해시 테이블의 평균 $O(1)$은 해시 함수가 키를 고르게 분산시킨다는 전제에서 나옵니다. 이 전제가 실무에서 항상 성립하지는 않기 때문에, Java의 `HashMap`이나 Python의 `dict` 같은 표준 구현체는 충돌이 많은 버킷을 트리 구조로 전환하는 등 최악의 경우를 완화하는 별도 장치를 두고 있습니다.

> **팁** 자료구조를 고를 때 실무에서 던지는 질문은 "이 데이터에 어떤 연산을 얼마나 자주 할 것인가"입니다. 조회가 압도적으로 많고 삽입·삭제가 드물다면 배열이나 해시 테이블이 유리하고, 중간 삽입·삭제가 잦다면 연결 리스트나 트리 기반 구조가 유리합니다.

## 5. 실생활 예시 — 전화번호부에서 이름을 찾는 세 가지 방법

> **해석** 아래 표는 "저장된 전화번호부에서 특정 이름의 번호를 찾는다"는 동일한 문제를, 자료구조를 무엇으로 택했느냐에 따라 비교한 것입니다.

| 방식 | 자료구조 | 탐색 복잡도 | 실제 체감 |
|---|---|---|---|
| 정렬 안 된 목록을 처음부터 훑기 | 배열(정렬 없음) | $O(n)$ | 이름이 뒤쪽에 있으면 수천 명을 다 살펴봐야 함 |
| 이름 순으로 정렬된 종이 전화번호부에서 절반씩 넘기기 | 배열(정렬 + 이진 탐색) | $O(\log n)$ | 수십만 명이어도 20번 이내로 찾음 |
| 이름을 색인(해시)으로 바로 찾아가기 | 해시 테이블 | $O(1)$ 평균 | 명단 크기와 무관하게 즉시 찾음 |

> **팁** 실제 종이 전화번호부가 이름 순으로 정렬되어 있는 이유가 바로 이진 탐색을 가능하게 하기 위해서입니다. 정렬이라는 사전 투자(정렬 자체는 $O(n \log n)$) 덕분에, 이후 반복되는 탐색을 $O(n)$에서 $O(\log n)$으로 낮출 수 있습니다.

## 요약

> **핵심**
> - Big-O 표기법 $f(n) = O(g(n))$은 "$n$이 충분히 커지면 $f(n)$이 $g(n)$에 상수를 곱한 값보다 커지지 않는다"는 **점근적 상한**을 뜻하며, 하드웨어·언어와 무관하게 알고리즘의 증가 추세를 비교할 수 있게 해줍니다.
> - 복잡도 클래스는 $O(1) < O(\log n) < O(n) < O(n \log n) < O(n^2) < O(2^n)$ 순으로 증가하며, $n$이 커질수록 이 격차는 산술적이 아니라 **기하급수적으로** 벌어집니다.
> - 자료구조 선택은 곧 연산 복잡도의 선택입니다. **어떤 연산을 얼마나 자주 수행하는가**에 따라 배열·연결 리스트·해시 테이블·균형 이진 탐색 트리 중 유리한 구조가 달라집니다.
