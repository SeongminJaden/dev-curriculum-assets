# 1-4. 버전 관리와 협업 워크플로우 — 실습

> **참고** 이 실습은 제출물이 아니라, Git의 네 가지 영역과 브랜치·PR 워크플로우를 직접 손으로 경험해보는 과정입니다.

## 실습 1 — 네 가지 영역을 직접 오가보기

- [ ] 빈 폴더에 `git init`으로 저장소를 만들고, 파일 하나를 작성한 뒤 `git status`로 어느 영역에 있는지 확인해보세요.
- [ ] `git add`로 스테이징 영역에 올리고, `git diff --staged`로 무엇이 커밋될지 확인해보세요.
- [ ] `git commit`으로 로컬 저장소에 기록한 뒤, `git log`로 커밋 이력을 확인해보세요.
- [ ] GitHub에 원격 저장소를 만들고 `git push`로 올려보세요.

## 실습 2 — 브랜치 생성과 병합 충돌 재현하기

- [ ] `main` 브랜치에서 새 브랜치를 만들어 파일의 특정 줄을 수정하고 커밋하세요.
- [ ] `main` 브랜치로 돌아가 같은 파일의 같은 줄을 다르게 수정하고 커밋하세요.
- [ ] 두 브랜치를 병합해보고, 병합 충돌이 어떤 형태로 표시되는지 직접 확인한 뒤 충돌을 해결해보세요.
- [ ] 충돌을 해결하며 느낀 점을 한두 문장으로 적어보세요. 왜 Git이 자동으로 판단하지 않고 사람에게 넘겼을지 생각해보세요.

## 실습 3 — Pull Request 워크플로우 경험하기

- [ ] 팀원(또는 스터디 동료) 저장소에 브랜치를 만들어 작은 수정을 하고 Pull Request를 열어보세요.
- [ ] 상대방의 PR에 리뷰 코멘트를 하나 이상 남겨보세요.
- [ ] 승인 후 병합하고, 병합이 끝난 브랜치를 삭제해보세요.

## 실습 4 — 커밋 이력으로 되돌아가기

- [ ] 최근 커밋 3개 이상이 쌓인 저장소에서 `git log --oneline`으로 커밋 목록을 확인하세요.
- [ ] 특정 커밋 시점으로 되돌아가는 명령(`git checkout <커밋 해시>` 또는 `git revert`)을 실습해보고, 파일 상태가 어떻게 달라지는지 관찰하세요.
- [ ] 파일명 기반 백업 방식이었다면 같은 작업을 어떻게 해야 했을지 비교해서 적어보세요.

> **팁** 실습 중 실수로 원치 않는 상태가 되어도, 로컬 저장소에는 이전 커밋들이 그대로 남아 있으므로 대부분 되돌릴 수 있습니다. 두려워하지 말고 직접 브랜치를 만들고 병합하고 충돌을 내보는 경험 자체가 이 강의의 핵심입니다.
