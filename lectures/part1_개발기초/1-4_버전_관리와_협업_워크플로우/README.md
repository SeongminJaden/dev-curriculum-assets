# Part 1. 개발 기초

## 1-4. 버전 관리와 협업 워크플로우 — Git은 왜 단일 파일 백업 방식과 근본적으로 다른가

> **학습 목표**
> - Git이 스냅샷 기반 분산 버전 관리 시스템으로 설계된 배경과, 델타 기반 중앙집중형 시스템과의 구조적 차이를 설명할 수 있다.
> - 작업 디렉터리·스테이징 영역·로컬 저장소·원격 저장소 네 영역 사이에서 파일이 이동하는 흐름을 그릴 수 있다.
> - 브랜치와 커밋 그래프의 관계를 이해하고, Pull Request 기반 협업 워크플로우가 왜 표준이 되었는지 설명할 수 있다.

> **개요** `보고서_최종.docx`, `보고서_최종_진짜.docx`, `보고서_최종_진짜_수정2.docx`처럼 파일명으로 버전을 관리해본 경험은 누구에게나 있습니다. 이 강의는 이런 수동 백업 방식이 왜 협업에서 무너지는지, 그리고 리누스 토르발스(Linus Torvalds)가 2005년 설계한 Git이 이 문제를 어떤 데이터 모델로 풀었는지를 다룹니다.

---

## 1. 왜 파일 이름으로 버전을 관리하면 안 될까요?

> **Q.** 폴더에 `v1`, `v2`, `최종`처럼 이름을 붙여 저장하는 방식과 Git은 근본적으로 무엇이 다를까요?
> **A.** 파일명 백업은 "누가 무엇을, 왜 바꿨는지"에 대한 기록이 전혀 없고, 두 사람이 동시에 같은 파일을 고치면 나중에 저장한 사람의 작업이 먼저 사람의 작업을 덮어써 버립니다. 버전 관리 시스템(Version Control System, VCS)은 모든 변경 이력을 시간순으로 기록하고, 누가 무엇을 바꿨는지 추적하며, 두 사람의 동시 수정을 병합(merge)하는 규칙을 갖추고 있다는 점이 근본적으로 다릅니다. Git은 이 VCS 중에서도 <u>**분산 버전 관리 시스템(Distributed VCS)**</u>에 속하며, 중앙 서버 없이도 각자의 컴퓨터가 전체 이력을 완전히 갖고 있다는 점이 특징입니다.

<u>**Git**</u>은 2005년 리누스 토르발스가 리눅스 커널 개발에 쓰던 상용 버전 관리 시스템 BitKeeper의 라이선스 문제가 불거지자, 대규모 오픈소스 협업에 맞는 시스템을 직접 만들며 탄생했습니다. CVS·Subversion(SVN) 같은 이전 세대의 <u>**중앙집중형 버전 관리 시스템(Centralized VCS)**</u>은 중앙 서버 하나가 전체 이력을 갖고, 각자의 컴퓨터는 최신 버전 하나만 내려받는 구조였습니다. 중앙 서버가 멈추면 이력 조회조차 할 수 없었고, 리눅스 커널처럼 전 세계 수천 명이 동시에 기여하는 프로젝트에는 근본적인 병목이었습니다.

> **참고** Git 공식 문서(Pro Git, Scott Chacon & Ben Straub 저)는 이 차이를 "각 개발자의 컴퓨터가 원격 저장소의 완전한 백업本이 된다"고 설명합니다. 실제로 원격 저장소(GitHub 서버)가 통째로 사라져도, 최근에 `clone`한 사람의 컴퓨터에는 전체 커밋 이력이 그대로 남아 있습니다.

## 2. Git의 데이터 모델 — 왜 "차이"가 아니라 "스냅샷"을 저장할까요?

> **개요** Git이 파일의 변경분(diff)이 아니라 매 커밋마다 전체 상태의 스냅샷을 저장하는 이유와, 이것이 가능한 이유(내용 기반 해시 주소 지정)를 다룹니다.

CVS·SVN 같은 이전 VCS는 파일을 저장할 때 "이전 버전 대비 무엇이 바뀌었는가"라는 <u>**델타(delta, 변경분)**</u>만 기록했습니다. 그래서 특정 시점의 파일을 복원하려면 처음 버전부터 그 시점까지의 델타를 순서대로 다시 적용해야 했습니다.

Git은 이와 다르게, 커밋할 때마다 그 순간 프로젝트 전체 파일의 <u>**스냅샷(snapshot)**</u>을 저장합니다. 다만 바뀌지 않은 파일까지 매번 통째로 복사하지는 않습니다. 각 파일 내용은 SHA-1(최근 버전은 SHA-256으로 이전 중) 해시값을 주소로 삼아 저장되는데, 내용이 같은 파일은 해시값도 같으므로 실제로는 이전 커밋의 동일한 객체를 가리키기만 합니다. 즉 "바뀌지 않은 파일은 이전 스냅샷의 파일을 재사용하는 방식"으로 스냅샷 방식이면서도 저장 공간 낭비를 피합니다.

> **Q.** 스냅샷 방식이 델타 방식보다 나은 점은 정확히 무엇일까요?
> **A.** 특정 시점의 상태를 복원하는 속도입니다. 델타 방식은 원하는 시점까지 변경 이력을 순서대로 재생(replay)해야 하므로 오래된 버전일수록 복원이 느려집니다. 스냅샷 방식은 해당 커밋이 가리키는 파일 상태를 그대로 읽기만 하면 되므로, 커밋이 언제 만들어졌든 복원 속도가 거의 동일합니다. 이는 브랜치를 자유롭게 오가며 작업하는 것이 흔한 협업 환경에서 결정적인 이점입니다.

## 3. Git의 네 가지 영역 — 파일은 어떻게 이동하는가

> **개요** 하나의 파일 변경이 실제로 커밋되어 팀 전체와 공유되기까지, Git 내부에서 거치는 네 개의 영역과 그 사이를 옮기는 명령어를 다룹니다.

![Git의 네 가지 영역 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part1_개발기초/1-4_버전_관리와_협업_워크플로우/assets/diagram1_git_areas.svg)

1. <u>**작업 디렉터리(Working Directory)**</u> — 실제로 파일을 열어 편집하는 폴더입니다.
2. <u>**스테이징 영역(Staging Area, Index)**</u> — `git add`로 지정한, 다음 커밋에 포함될 변경분만 모아두는 임시 공간입니다.
3. <u>**로컬 저장소(Local Repository)**</u> — `git commit`으로 스테이징 영역의 내용이 스냅샷(커밋 객체)으로 영구히 기록되는 곳입니다.
4. <u>**원격 저장소(Remote Repository)**</u> — `git push`로 로컬 커밋을 올리는, 팀 전체가 공유하는 참조 지점(예: GitHub)입니다.

> **원칙** 스테이징 영역이 따로 존재하는 이유는 "수정한 모든 것"과 "이번 커밋에 포함할 것"을 분리하기 위함입니다. 예를 들어 한 파일에서 두 가지 다른 목적의 수정을 했을 때, `git add`로 그중 하나만 골라 커밋하고 나머지는 다음 커밋으로 미룰 수 있습니다. 이 덕분에 각 커밋은 "하나의 논리적인 변경"만 담을 수 있고, 이후 문제가 생겼을 때 어느 커밋이 원인인지 추적하기 쉬워집니다.

> **팁** `git commit` 전에 `git diff --staged`로 스테이징 영역에 무엇이 들어갔는지 확인하는 습관은, 의도치 않은 파일이 커밋에 섞여 들어가는 실수를 막아줍니다.

## 4. 브랜치와 커밋 그래프 — 브랜치는 왜 "가벼울까요"

> **개요** 브랜치가 실제로는 커밋 하나를 가리키는 포인터에 불과하다는 사실과, 이것이 왜 Git의 브랜치 생성·전환이 거의 즉각적인지를 설명합니다.

Git의 커밋은 이전 커밋을 가리키는 참조를 가지고 있어, 전체 커밋들이 하나의 그래프(정확히는 방향 비순환 그래프, DAG)를 이룹니다. <u>**브랜치(Branch)**</u>는 이 그래프 안에서 특정 커밋 하나를 가리키는 이름표에 지나지 않습니다.

![브랜치와 커밋 그래프 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part1_개발기초/1-4_버전_관리와_협업_워크플로우/assets/diagram2_branch_graph.svg)

> **해석** 위 그래프에서 `feature` 브랜치는 `c2` 커밋에서 갈라져 `f1`, `f2` 커밋을 쌓았고, 이후 `main` 브랜치에 `merge` 커밋으로 합쳐졌습니다. `main`이라는 이름표는 지금 가장 최근 커밋(`c4`)을 가리킬 뿐이며, 새 커밋을 만들면 이 이름표가 그 새 커밋으로 옮겨갈 뿐입니다. `HEAD`는 "지금 내가 작업 중인 브랜치가 어디인가"를 가리키는 특수한 포인터입니다.

> **Q.** 다른 도구에서는 브랜치 생성이 프로젝트 전체를 복사하는 무거운 작업인데, Git에서는 왜 즉각적일까요?
> **A.** 브랜치를 만드는 것이 파일을 복사하는 일이 아니라, 현재 커밋을 가리키는 41바이트짜리 이름표(참조 파일) 하나를 새로 만드는 일에 불과하기 때문입니다. 실제 파일 데이터는 이미 커밋 객체 안에 저장되어 있으므로 복사할 필요가 없습니다. 이 가벼움 덕분에 Git에서는 기능 하나마다, 버그 수정 하나마다 브랜치를 만들어 작업하는 것이 표준 관행이 되었습니다.

> **참고** 병합 시 두 브랜치가 같은 줄을 서로 다르게 고쳤다면 <u>**병합 충돌(Merge Conflict)**</u>이 발생합니다. Git은 어느 쪽을 남길지 자동으로 판단하지 못하므로, 개발자가 직접 어떤 내용을 남길지 결정해야 합니다. 충돌은 오류가 아니라, 두 사람의 의도가 겹쳤다는 사실을 시스템이 정확히 알려주는 안전장치입니다.

## 5. 협업 워크플로우 — Pull Request가 표준이 된 이유

> **개요** 팀이 하나의 원격 저장소를 공유할 때, 브랜치와 Pull Request(PR)를 조합한 워크플로우가 왜 코드 리뷰와 품질 관리의 표준 절차가 되었는지 다룹니다.

실무에서 팀은 보통 다음 순서로 협업합니다.

1. `main`(또는 `develop`) 브랜치에서 새 브랜치를 만들어 기능 단위로 작업합니다.
2. 작업이 끝나면 원격 저장소에 브랜치를 `push`하고, `main`으로 병합해 달라는 <u>**Pull Request(PR)**</u>를 엽니다.
3. 팀원이 PR의 변경 내용을 리뷰하고, 의견을 남기거나 승인합니다.
4. 승인되면 PR을 병합(merge)하고, 더 이상 필요 없는 브랜치는 삭제합니다.

> **원칙** PR 기반 워크플로우가 표준이 된 이유는 "코드가 `main`에 들어가기 전에 반드시 다른 사람의 눈을 거치게 강제"하기 때문입니다. 이는 1-1 강의에서 다룬 SDLC의 원칙과 같습니다 — 문제를 발견하는 시점을 앞당길수록 수정 비용이 줄어듭니다. 리뷰어가 코드가 병합되기 전에 문제를 발견하면, 병합되어 배포까지 된 뒤 발견하는 것보다 수정 비용이 훨씬 낮습니다.

> **팁** GitHub·GitLab 등에서 `main` 브랜치에 "PR과 최소 1명 이상의 승인 없이는 직접 커밋할 수 없다"는 보호 규칙(branch protection rule)을 설정하는 것이 실무에서 널리 쓰이는 이유도 같은 원칙에서 나옵니다.

## 6. 실생활 예시 — 조별 과제 문서를 세 가지 방식으로 공유하기

> **해석** 아래 표는 "여러 명이 함께 쓰는 발표 자료"를 관리하는 세 가지 방식을 비교한 것입니다.

| 방식 | 동시 수정 시 벌어지는 일 | 이력 추적 | 되돌리기 |
|---|---|---|---|
| 이메일로 파일 주고받기 | 마지막에 보낸 사람의 버전만 남고 나머지는 유실 | 불가능 | 불가능 |
| 파일명에 버전 표시(`v1`, `v2_최종`) | 동시 수정 시 이름 충돌, 수동 병합 필요 | 파일명으로만 대략 추정 | 이전 파일명으로 되돌아가야 함 |
| Git + 원격 저장소 | 각자 브랜치에서 작업 후 병합, 충돌 나면 명시적으로 표시 | 모든 변경에 작성자·시각·이유(커밋 메시지) 기록 | 특정 커밋으로 정확히 복원 가능 |

> **팁** 조별 과제처럼 비개발 문서라도 텍스트 기반(마크다운, 코드)으로 작성한다면 Git으로 관리할 수 있습니다. 실무에서도 설정 파일, 인프라 코드, 문서(README) 모두 같은 이유로 Git 저장소에 함께 넣어 관리합니다.

## 요약

> **핵심**
> - Git은 델타가 아니라 매 커밋마다 전체 상태의 <u>**스냅샷**</u>을 저장하며, 내용 기반 해시로 중복을 제거해 이전 시점으로의 복원 속도가 항상 일정합니다.
> - 파일은 <u>**작업 디렉터리 → 스테이징 영역 → 로컬 저장소 → 원격 저장소**</u> 순으로 이동하며, 스테이징 영역은 "이번 커밋에 무엇을 포함할지"를 정교하게 고를 수 있게 해줍니다.
> - 브랜치는 실제로는 **커밋을 가리키는 가벼운 이름표**에 불과하며, 이 가벼움 덕분에 기능 단위로 브랜치를 나누고 Pull Request로 리뷰한 뒤 병합하는 워크플로우가 실무 표준이 되었습니다.
