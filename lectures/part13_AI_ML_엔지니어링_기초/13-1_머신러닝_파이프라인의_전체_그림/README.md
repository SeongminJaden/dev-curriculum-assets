# Part 13. AI/ML 엔지니어링 기초

## 13-1. 머신러닝 파이프라인의 전체 그림 — 데이터 수집부터 서빙까지

> **학습 목표**
> - 머신러닝 파이프라인을 구성하는 6단계(데이터 수집, 전처리, 학습, 평가, 서빙, 모니터링)와 각 단계의 역할을 설명할 수 있다.
> - 이 6단계가 하나로 뭉쳐 있지 않고 독립적으로 관리되어야 하는 이유를 학습-서빙 스큐(training-serving skew) 개념으로 설명할 수 있다.
> - 데이터 드리프트가 무엇이며, 모니터링 단계가 왜 파이프라인을 다시 데이터 수집으로 되돌리는지 설명할 수 있다.

> **개요** 일반 소프트웨어는 배포가 끝나면 코드가 고정되지만, 머신러닝 모델은 배포 이후에도 세상이 계속 바뀌기 때문에 코드가 아니라 데이터 자체가 낡아갑니다. 이 강의는 모델이 데이터에서 태어나 서비스로 나가기까지의 6단계 파이프라인을 지도로 그리고, 왜 이 단계들을 각각 독립된 단위로 관리해야 하는지를 다룹니다.

---

## 1. 왜 "파이프라인"으로 봐야 할까요?

> **Q.** 모델 학습 코드 하나만 잘 짜면 되는 것 아닐까요? 왜 파이프라인 전체를 알아야 할까요?
> **A.** 실무에서 모델 학습 알고리즘 코드는 전체 시스템의 일부에 불과합니다. 2015년 구글 연구진(D. Sculley 등)이 발표한 논문 "Hidden Technical Debt in Machine Learning Systems"는, 실제로 운영되는 머신러닝 시스템에서 학습 코드를 둘러싼 데이터 수집·검증·특성 추출·서빙 인프라·모니터링 같은 주변 시스템이 훨씬 큰 비중을 차지한다는 점을 지적합니다. 모델 하나를 "잘 학습시키는 것"과 그 모델을 "안정적으로 서비스하는 것" 사이에는 큰 엔지니어링 격차가 있고, 이 격차를 메우는 것이 바로 파이프라인 설계입니다.

이 강의는 데이터가 모델이 되어 사용자에게 도달하기까지의 흐름을 하나의 지도로 그립니다. Part 1(1-1)에서 본 SDLC가 "코드"의 생명주기였다면, 이번 지도는 "데이터와 모델"의 생명주기입니다.

---

## 2. 머신러닝 파이프라인 6단계

> **개요** 데이터가 들어와서 실제 서비스에 반영되기까지 거치는 6단계와, 그 단계들이 다시 순환하는 구조를 살펴봅니다.

![머신러닝 파이프라인 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part13_AI_ML_엔지니어링_기초/13-1_머신러닝_파이프라인의_전체_그림/assets/diagram1_pipeline.svg)

1. **데이터 수집** — 로그, 데이터베이스, 센서, 사용자 행동 등 원천 데이터를 모읍니다.
2. **전처리** — 결측치를 처리하고, 이상치를 걸러내고, 모델이 이해할 수 있는 형태(특성, feature)로 변환합니다.
3. **학습** — 전처리된 데이터로 모델 파라미터를 최적화합니다.
4. **평가** — 학습에 쓰지 않은 데이터(검증·테스트 셋)로 모델 성능을 측정하고, 실제로 배포할 만한지 판단합니다.
5. **서빙** — 검증을 통과한 모델을 실제 요청을 받아 추론 결과를 돌려주는 서비스로 배포합니다.
6. **모니터링** — 서빙 중인 모델의 성능과 입력 데이터의 분포를 지속적으로 감시합니다.

> **참고** 1-1에서 다룬 SDLC와 비교하면, "데이터 수집·전처리"는 SDLC의 요구사항·설계에, "학습·평가"는 구현·테스트에, "서빙"은 배포에, "모니터링"은 운영·유지보수에 대응합니다. 다만 SDLC와 결정적으로 다른 지점이 하나 있습니다. SDLC의 운영 단계에서 발견되는 문제는 대부분 "코드"의 결함이지만, 머신러닝 파이프라인의 운영 단계에서 발견되는 문제는 코드가 아니라 <u>**세상이 변해서 데이터의 성격 자체가 달라졌기 때문**</u>에 발생하는 경우가 많습니다.

### 실생활 예시 — 쇼핑몰의 "함께 구매한 상품" 추천

> **해석** 아래 표는 "이 상품을 산 사람들이 함께 산 상품" 추천 기능이 파이프라인 6단계를 어떻게 통과하는지 보여줍니다.

| 단계 | 실제로 벌어지는 일 |
|---|---|
| 데이터 수집 | 사용자의 주문 내역, 장바구니 로그, 상품 클릭 이력을 수집 |
| 전처리 | 중복 주문 제거, 상품 ID를 학습 가능한 벡터로 변환 |
| 학습 | 함께 구매되는 패턴을 학습해 상품 간 연관도 점수를 계산하는 모델을 학습 |
| 평가 | 최근 한 달간의 실제 구매 데이터로 추천 정확도(예: 추천 상품이 실제로 함께 구매되었는가)를 검증 |
| 서빙 | 상품 상세 페이지에서 실시간으로 추천 목록을 반환하는 API로 배포 |
| 모니터링 | 신상품 출시 이후 추천 클릭률이 떨어지는 것을 발견해, 재학습 주기를 앞당김 |

## 3. 왜 각 단계가 독립적으로 관리되어야 할까요?

> **개요** 6단계를 하나의 스크립트로 뭉쳐서 처리하지 않고, 왜 각 단계를 독립적인 산출물과 책임 단위로 분리해야 하는지를 다룹니다.

> **Q.** 전처리와 서빙을 같은 코드로 처리하면 편할 텐데, 왜 굳이 분리할까요?
> **A.** 전처리 로직을 학습 코드와 서빙 코드에 각각 따로 구현하면, 두 코드가 미묘하게 달라지는 문제가 생깁니다. 이를 <u>**학습-서빙 스큐(training-serving skew)**</u>라고 부릅니다. 예를 들어 학습 때는 "가격을 천 원 단위로 반올림"하는 전처리를 거쳤는데, 서빙 코드에서 이 로직을 깜빡하고 원본 가격을 그대로 넣으면, 모델은 학습 때 한 번도 본 적 없는 입력 분포를 받게 됩니다. 이 오차는 코드 에러처럼 예외를 던지지 않고 조용히 성능만 떨어뜨리기 때문에 발견하기가 훨씬 어렵습니다. 전처리 로직을 파이프라인의 독립된 한 단계로 만들어 학습과 서빙이 <u>**동일한 코드**</u>를 공유하게 하는 것이 이 문제를 막는 가장 확실한 방법입니다.

> **참고** 각 단계를 분리하는 실무적인 이유는 또 있습니다.
> - 데이터 수집은 데이터 엔지니어링 팀이, 모델 학습은 리서치 팀이, 서빙 인프라는 백엔드·MLOps 팀이 서로 다른 주기로 작업하는 경우가 많습니다. 단계가 분리되어 있어야 각 팀이 서로의 진행을 막지 않고 독립적으로 배포할 수 있습니다.
> - 실패 지점이 단계마다 다릅니다. 데이터 수집 실패는 "데이터가 안 들어온다"이고, 서빙 실패는 "응답이 느리거나 오류가 난다"입니다. 하나로 뭉쳐 있으면 문제의 원인을 좁히는 데 훨씬 오래 걸립니다.

## 4. 데이터 드리프트 — 모델이 서비스에 나간 뒤에도 감시해야 하는 이유

> **개요** 학습 시점의 데이터 분포와 운영 시점의 데이터 분포가 시간이 지나며 달라지는 현상인 데이터 드리프트를 다룹니다.

일반 소프트웨어는 배포 이후 입력이 코드가 예상한 범위를 크게 벗어나지 않는다면 계속 올바르게 동작합니다. 하지만 머신러닝 모델은 <u>**학습 시점에 관찰한 데이터의 통계적 패턴**</u>을 학습한 것이므로, 세상이 변해 입력 데이터의 패턴 자체가 달라지면 코드는 그대로인데도 예측 성능이 떨어집니다. 이 현상을 <u>**데이터 드리프트(data drift)**</u>라고 부릅니다.

![데이터 드리프트 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part13_AI_ML_엔지니어링_기초/13-1_머신러닝_파이프라인의_전체_그림/assets/diagram2_drift.svg)

> **참고** 드리프트는 크게 두 갈래로 나눌 수 있습니다.
> - **입력 분포 드리프트(covariate drift)** — 입력 데이터의 분포 자체가 변하는 경우입니다. 예를 들어 서비스 이용자의 연령대가 시간이 지나며 젊은 층에서 중장년층으로 옮겨가는 경우입니다.
> - **개념 드리프트(concept drift)** — 입력과 정답 사이의 관계 자체가 변하는 경우입니다. 예를 들어 코로나19 이전과 이후 "재택근무 가능 여부"가 채용 추천 모델의 정답에 미치는 영향이 달라진 것이 여기 해당합니다.

> **Q.** 이 문제를 왜 평가 단계가 아니라 모니터링 단계가 처리하나요?
> **A.** 평가 단계는 배포 전에 "지금 이 모델이 충분히 좋은가"를 한 번 판단하는 단계입니다. 반면 데이터 드리프트는 <u>**배포된 이후 시간이 지나면서**</u> 서서히 나타나는 문제이므로, 배포 시점 한 번의 평가로는 잡아낼 수 없습니다. 그래서 모니터링 단계가 운영 중인 입력 데이터의 분포를 주기적으로 학습 시점 분포와 비교하고, 차이가 임계값을 넘으면 재학습을 트리거하는 역할을 맡습니다. 다이어그램 1의 점선 화살표가 나타내는 것이 바로 이 순환입니다.

## 요약

> **핵심**
> - 머신러닝 파이프라인은 데이터 수집 → 전처리 → 학습 → 평가 → 서빙 → 모니터링의 6단계이며, 모니터링에서 감지된 드리프트가 다시 데이터 수집으로 순환됩니다.
> - 각 단계를 독립적으로 관리하는 이유는 <u>**학습-서빙 스큐를 막고, 서로 다른 주기로 일하는 팀들이 독립적으로 배포할 수 있게 하며, 실패 지점을 빠르게 좁히기 위함**</u>입니다.
> - 데이터 드리프트는 코드가 아니라 데이터의 통계적 패턴이 배포 이후에도 계속 변하기 때문에 생기며, 일반 소프트웨어와 달리 머신러닝 시스템이 <u>**배포 이후에도 지속적인 감시**</u>가 필요한 근본적인 이유입니다.
