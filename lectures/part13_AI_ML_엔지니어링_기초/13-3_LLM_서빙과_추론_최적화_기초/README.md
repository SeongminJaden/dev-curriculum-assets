# Part 13. AI/ML 엔지니어링 기초

## 13-3. LLM 서빙과 추론 최적화 기초 — 모델이 클수록 서빙이 어려운 이유

> **학습 목표**
> - 파라미터 수가 메모리 사용량 및 연산량과 어떤 관계를 가지는지, FP16 기준 파라미터당 2바이트라는 수치로 설명할 수 있다.
> - KV 캐시가 무엇이고, 자기회귀 방식으로 토큰을 생성하는 언어 모델에서 왜 필수적인지 설명할 수 있다.
> - 양자화가 정밀도와 메모리 사이에서 만드는 트레이드오프를 설명할 수 있다.
> - 배치 추론이 서빙 처리량을 높이는 원리와 그 대가를 설명할 수 있다.

> **개요** 모델 학습은 한 번(또는 가끔) 수행하는 연산이지만, 서빙은 사용자 요청이 들어올 때마다 반복되는 연산입니다. 이 강의는 파라미터 수가 커질수록 이 반복 연산의 메모리·시간 비용이 어떻게 늘어나는지, 그리고 양자화와 배치 추론이 이 비용을 어떤 대가로 완화하는지를 다룹니다.

---

## 1. 왜 모델이 클수록 서빙이 어려울까요?

> **Q.** 모델의 정확도가 높아지는 건 좋은데, 왜 파라미터 수가 늘어나면 서빙이 어려워질까요?
> **A.** 모델을 서빙한다는 것은 매 요청마다 모든 파라미터를 동원해 연산을 수행하고, 그 파라미터들을 메모리(주로 GPU 메모리)에 올려둬야 한다는 뜻입니다. 파라미터 수가 늘어나면 <u>**메모리 사용량과 연산량이 함께 늘어나기 때문에**</u>, 같은 하드웨어로 감당할 수 있는 모델의 크기에는 한계가 있습니다. 학습은 한 번 오래 걸려도 되지만, 서빙은 사용자가 기다리는 동안 매번 이 비용을 다시 치러야 하므로 지연 시간과 처리량 모두에 직접 영향을 미칩니다.

---

## 2. 파라미터 수와 메모리 사용량

> **개요** 모델 하나를 GPU에 올리는 데 필요한 메모리가 파라미터 수와 정밀도(precision)에 따라 어떻게 정해지는지 계산해봅니다.

모델의 가중치는 컴퓨터 내부에서 정해진 비트 수의 부동소수점(또는 정수)으로 저장됩니다. 널리 쓰이는 정밀도별 파라미터당 바이트 수는 다음과 같습니다.

| 정밀도 | 파라미터당 바이트 수 |
|---|---|
| FP32(32비트 부동소수점) | 4바이트 |
| FP16 / BF16(16비트 부동소수점) | 2바이트 |
| INT8(8비트 정수) | 1바이트 |
| INT4(4비트 정수) | 0.5바이트 |

가중치만 저장하는 데 필요한 메모리는 다음과 같이 계산됩니다.

$$\text{메모리(바이트)} = \text{파라미터 수} \times \text{파라미터당 바이트 수}$$

예를 들어 파라미터 70억 개(7B) 모델을 FP16으로 저장하면 $7\times10^9 \times 2\text{바이트} = 14\times10^9\text{바이트} \approx 14\text{GB}$가 필요합니다.

![정밀도별 메모리 사용량 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part13_AI_ML_엔지니어링_기초/13-3_LLM_서빙과_추론_최적화_기초/assets/diagram1_quantization_memory.svg)

> **참고** 이는 가중치만 계산한 수치이며, 실제 서빙 시에는 추론 과정에서 생기는 중간 결과(활성화 값)와 다음 절에서 다룰 KV 캐시가 추가로 메모리를 차지합니다. 또한 GPU 한 장이 가진 메모리 용량은 유한하므로, 모델이 커질수록 GPU를 여러 장에 나누어 올리는 분산 서빙이 필요해집니다.

## 3. KV 캐시 — 매 토큰마다 처음부터 다시 계산하지 않는 이유

> **개요** 트랜스포머 기반 언어 모델이 토큰을 하나씩 생성할 때, 이전 계산 결과를 재사용하기 위해 저장해두는 KV 캐시의 원리를 다룹니다.

트랜스포머의 셀프 어텐션(self-attention)은 현재 토큰이 이전의 모든 토큰과 얼마나 관련 있는지를 계산하기 위해, 각 토큰마다 키(Key)와 값(Value) 벡터를 만듭니다. 언어 모델이 문장을 한 토큰씩 생성하는 <u>**자기회귀(autoregressive) 방식**</u>에서는, 새 토큰을 만들 때마다 이전에 생성된 모든 토큰의 키·값 벡터가 다시 필요합니다. 이 벡터들을 매번 처음부터 다시 계산한다면, 문장이 길어질수록 계산량이 급격히 늘어납니다.

<u>**KV 캐시(KV cache)**</u>는 한 번 계산한 키·값 벡터를 저장해두고, 다음 토큰을 생성할 때 재사용하는 장치입니다. 이렇게 하면 새로 추가된 토큰의 키·값만 계산하고, 이전 토큰들의 키·값은 캐시에서 그대로 가져와 재사용할 수 있습니다.

![프리필과 디코드, KV 캐시 다이어그램](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part13_AI_ML_엔지니어링_기초/13-3_LLM_서빙과_추론_최적화_기초/assets/diagram2_kv_cache.svg)

> **참고** 실제 서빙은 크게 두 단계로 나뉩니다.
> - **프리필(prefill)** — 사용자가 입력한 프롬프트 전체를 한 번에 처리하며, 모든 토큰의 키·값 벡터를 계산해 캐시에 채워 넣습니다.
> - **디코드(decode)** — 토큰을 한 개씩 생성하며, 매 단계마다 캐시에 있는 키·값을 재사용하고 새로 생성된 토큰의 키·값만 캐시에 추가합니다.

> **Q.** KV 캐시는 계산을 줄여주는데, 왜 "공짜"가 아닌가요?
> **A.** KV 캐시는 시퀀스 길이와 동시에 처리하는 요청(배치) 수에 비례해 커집니다. 대략 <u>**캐시 크기 ≈ 2(키, 값) × 레이어 수 × 시퀀스 길이 × 히든 차원 × 정밀도 바이트 수 × 배치 크기**</u>로 늘어나므로, 문장이 길어지거나 동시 요청이 많아질수록 GPU 메모리에서 KV 캐시가 차지하는 비중이 가중치 자체보다 커질 수 있습니다. 즉 계산량은 줄지만 <u>**메모리 사용량이라는 새로운 비용**</u>을 지불하는 것입니다.

## 4. 양자화 — 정밀도를 낮춰 메모리를 줄이기

> **개요** 파라미터를 더 적은 비트 수로 표현하는 양자화가 메모리·속도와 정확도 사이에서 만드는 트레이드오프를 다룹니다.

<u>**양자화(quantization)**</u>는 FP32나 FP16으로 저장된 파라미터를 INT8, INT4처럼 더 적은 비트 수의 값으로 변환하는 기법입니다. 위 표에서 보았듯, 정밀도를 절반으로 줄이면 메모리 사용량도 거의 절반으로 줄어듭니다. 메모리 대역폭이 줄어드는 만큼 데이터를 GPU 연산 장치로 옮기는 시간도 줄어들어, 추론 속도가 함께 빨라지는 경우가 많습니다.

> **참고** 양자화가 희생하는 것은 <u>**표현 가능한 값의 세밀함(정밀도)**</u>입니다. FP32는 매우 미세한 값 차이까지 구분해 표현할 수 있지만, INT4는 표현할 수 있는 값의 가짓수가 훨씬 적습니다. 이 과정에서 원래 가중치 값과 양자화된 값 사이에 오차(양자화 오차)가 생기고, 이 오차가 누적되면 모델의 출력 품질이 떨어질 수 있습니다. 실무에서는 이 손실을 줄이기 위해 양자화 이전에 대표 데이터로 값의 분포를 보정하는 보정(calibration) 과정이나, 학습 과정에 양자화를 포함시키는 양자화 인식 학습(Quantization-Aware Training) 같은 기법을 함께 사용합니다.

> **Q.** 그렇다면 항상 가장 낮은 정밀도(INT4)를 쓰는 것이 최선 아닐까요?
> **A.** 그렇지 않습니다. 정밀도를 낮출수록 메모리와 속도 이득은 커지지만, 정확도 손실 위험도 함께 커집니다. 실무에서는 서비스가 감당할 수 있는 정확도 저하의 한계 안에서, 메모리·속도 이득이 가장 큰 정밀도를 선택하는 <u>**트레이드오프 판단**</u>이 필요합니다.

## 5. 배치 추론 — 여러 요청을 함께 처리하기

> **개요** 동시에 들어온 여러 요청을 묶어 한 번에 처리하는 배치 추론이 GPU 활용률을 높이는 원리를 다룹니다.

GPU는 하나의 요청만 처리할 때는 연산 장치의 일부만 사용하고 나머지는 유휴 상태로 남는 경우가 많습니다. <u>**배치 추론(batch inference)**</u>은 비슷한 시점에 도착한 여러 요청을 하나의 배치로 묶어 동시에 연산함으로써, 같은 시간 동안 처리할 수 있는 요청 수(처리량, throughput)를 늘리는 방식입니다.

> **참고** 배치를 키우면 처리량은 늘지만, 개별 요청 입장에서는 다른 요청과 묶이기 위해 잠시 대기해야 하므로 <u>**지연 시간(latency)이 늘어날 수 있습니다**</u>. 이것이 배치 추론의 근본적인 트레이드오프입니다. 실제 LLM 서빙 시스템은 요청마다 도착 시점과 생성 길이가 다르다는 점을 고려해, 이미 진행 중인 배치에 새 요청을 동적으로 끼워 넣는 방식(연속 배치, continuous batching)으로 이 트레이드오프를 완화합니다.

### 실생활 예시 — 챗봇 응답이 혼잡한 시간대에 느려지는 이유

> **해석** 아래 표는 사용자가 체감하는 "챗봇이 느려졌다"는 현상이 지금까지 배운 개념과 어떻게 연결되는지 보여줍니다.

| 상황 | 서버 내부에서 벌어지는 일 |
|---|---|
| 평소 시간대 | 요청 수가 적어 배치 크기가 작고, 개별 요청이 곧바로 처리되어 지연 시간이 짧음 |
| 혼잡한 시간대 | 동시 요청이 몰려 배치가 가득 차고, 새 요청이 기존 배치에 합류하기 위해 대기하며 지연 시간이 늘어남 |
| 메모리 부족 상황 | 동시에 처리 중인 요청들의 KV 캐시가 GPU 메모리를 모두 소진해, 새 요청을 즉시 받지 못하고 대기열에 쌓임 |

## 요약

> **핵심**
> - 모델을 서빙하는 데 필요한 메모리는 <u>**파라미터 수 × 파라미터당 바이트 수**</u>로 계산되며, FP16 기준 파라미터당 2바이트입니다. 파라미터 수가 늘어날수록 서빙에 필요한 메모리와 연산량이 함께 늘어나 서빙이 어려워집니다.
> - KV 캐시는 자기회귀 생성에서 이전 토큰의 키·값 벡터를 재사용해 재계산을 막아주지만, <u>**시퀀스 길이와 배치 크기에 비례해 커지는 메모리 비용**</u>을 새로 발생시킵니다.
> - 양자화는 파라미터의 정밀도를 낮춰 메모리와 속도를 얻는 대신, <u>**표현 가능한 값의 세밀함(정확도)**</u>을 희생하는 트레이드오프입니다.
> - 배치 추론은 여러 요청을 묶어 GPU 활용률과 처리량을 높이지만, 그 대가로 개별 요청의 지연 시간이 늘어날 수 있습니다.
