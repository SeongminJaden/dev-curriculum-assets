# Part 4. 프로세서 아키텍처 심화 — x86 vs ARM & 통합 메모리 아키텍처

## 4-3. 통합 메모리 아키텍처(UMA)의 원리 — CPU와 GPU가 메모리를 공유하면 왜 복사 비용이 사라지는가?

> **학습 목표**
> - 전통적인 분리형(discrete) GPU 구조에서 CPU-GPU 간 데이터 복사가 왜, 어디서 발생하는지 설명할 수 있다.
> - 통합 메모리 아키텍처(UMA)가 이 복사 비용을 어떻게 근본적으로 제거하는지 설명할 수 있다.
> - Apple Silicon, NVIDIA Jetson 같은 사례를 통해 로봇·엣지 디바이스가 왜 UMA를 선호하는지 설명할 수 있다.

> **개요** CPU와 GPU가 나란히 있어도, 서로 다른 메모리를 갖고 있다면 데이터를 주고받을 때마다 "복사"라는 숨겨진 작업이 발생합니다. 이 강의는 이 복사가 왜 발생하고, 왜 비용이 크며, 메모리를 물리적으로 공유하는 UMA가 이 문제를 어떻게 해결하는지, 그리고 이것이 왜 로봇 같은 엣지 디바이스에 특히 중요한지를 다룹니다.

---

## 1. 왜 CPU와 GPU는 원래 메모리를 따로 썼을까요?

> **개요** 전통적인 분리형(discrete) GPU 구조와 그 메모리 구조를 살펴봅니다.

데스크톱 PC에 그래픽카드를 꽂아본 적이 있다면, 그 카드 자체에 큼직한 메모리 칩들이 붙어 있는 것을 본 적이 있을 것입니다. 이것이 GPU 전용 메모리인 VRAM(예: GDDR6, HBM)입니다. CPU는 메인보드에 꽂힌 시스템 RAM(DRAM)을 쓰고, GPU는 자기 카드 위의 VRAM을 씁니다. 이 둘은 PCIe(PCI Express)라는 버스로 연결되어 있습니다.

> **Q.** CPU와 GPU를 같은 컴퓨터 안에 두면서 왜 메모리는 따로 뒀을까요?
> **A.** GPU는 수천 개의 코어로 동시에 대량의 데이터를 처리하도록 설계되어 있어, 매우 넓은 대역폭(초당 전송량)의 메모리가 필요합니다. GDDR이나 HBM 같은 GPU 전용 메모리는 GPU 바로 옆에 배치되어 이 대역폭을 극대화하도록 설계되었습니다. 반면 시스템 RAM은 CPU의 접근 패턴(지연시간에 민감한 순차적 접근)에 맞춰 설계되어 있습니다. 각자의 용도에 최적화된 메모리를 따로 쓰는 것이 오랫동안 합리적인 선택이었습니다.

문제는 GPU로 무언가를 계산시키려면, 먼저 CPU 쪽 메모리에 있는 데이터를 GPU 쪽 VRAM으로 **복사**해야 하고, 계산이 끝나면 결과를 다시 CPU 쪽으로 복사해와야 한다는 점입니다. 이 복사는 PCIe 버스를 통해 이루어지는데, PCIe의 대역폭은 CPU나 GPU 내부, 혹은 메모리 칩 자체의 대역폭보다 훨씬 낮습니다. 게다가 이 복사는 그냥 데이터 전송이 아니라, 운영체제·드라이버 수준의 동기화(synchronization)까지 동반해 지연시간(latency)을 추가로 발생시킵니다.

![분리형(Discrete) GPU 구조와 통합 메모리 구조(UMA) 비교](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part4_프로세서_아키텍처_심화/4-3_통합_메모리_아키텍처의_원리/assets/diagram1_discrete_vs_uma.svg)

> **해석** 왼쪽 분리형 구조에서는 CPU의 RAM과 GPU의 VRAM이 물리적으로 분리되어 있고, 그 사이를 잇는 PCIe 버스를 통해 데이터가 "복사"되어야만 GPU가 그 데이터를 쓸 수 있습니다. 오른쪽 통합 메모리 구조에서는 CPU와 GPU가 하나의 SoC(System on Chip) 경계 안에서 물리적으로 같은 메모리 풀에 넓은 내부 버스로 연결되어 있어, 복사라는 단계 자체가 존재하지 않습니다.

## 2. UMA는 복사를 어떻게 없애는가

> **개요** 통합 메모리 아키텍처가 물리적으로 무엇을 공유하고, 그 결과 왜 복사 비용이 사라지는지 살펴봅니다.

통합 메모리 아키텍처(Unified Memory Architecture, UMA)의 핵심은 단순합니다. CPU와 GPU(그리고 신경망 전용 가속기 등)가 **같은 물리적 DRAM 칩 묶음을 하나의 주소 공간으로 공유**합니다. 시스템 RAM과 VRAM이라는 구분 자체가 없습니다.

- 데이터를 GPU에 "보내는" 행위는 더 이상 바이트를 복사하는 것이 아니라, 그 데이터가 있는 메모리 주소(포인터)를 GPU에 알려주는 것으로 끝납니다. 이를 **제로카피(zero-copy)**라고 부릅니다.
- 메모리 칩 자체도 SoC 다이 바로 옆(온 패키지, on-package)에 배치되어, PCIe 같은 외부 버스를 거치지 않고 매우 넓은 폭의 내부 버스로 직접 연결됩니다.

> **원칙** <u>**데이터를 옮기지 않고 그 자리에서 CPU와 GPU가 번갈아 접근하게 만드는 것**</u>이 UMA의 설계 원칙입니다. 컴퓨터 구조에서 데이터를 물리적으로 이동시키는 작업은 같은 양의 데이터를 그 자리에서 읽는 것보다 훨씬 많은 에너지와 시간을 소모한다는 것이 잘 알려진 사실이며, UMA는 이 이동 자체를 설계에서 제거하는 접근입니다.

> **참고** UMA라고 해서 모든 비용이 사라지는 것은 아닙니다. CPU와 GPU가 같은 메모리 영역을 동시에 건드릴 수 있으므로, 캐시 일관성(cache coherency)을 유지하고 접근 순서를 동기화하는 비용은 여전히 존재합니다. 다만 그 비용은 "칩 내부의 짧은 배선을 오가는" 수준이지, "카드 슬롯 밖 PCIe 커넥터를 왕복하는" 수준이 아닙니다. 사라지는 것은 정확히는 "물리적으로 먼 별도 메모리로의 대량 데이터 복사"입니다.

> **참고** 반대급부도 명확히 짚어야 합니다. UMA는 하나의 메모리 대역폭을 CPU와 GPU가 나눠 쓰므로, 완전히 독립된 고대역폭 VRAM(HBM 등)을 GPU 혼자 쓰는 분리형 구조에 비해 GPU 단독의 최대 처리량(순수 대규모 학습 등)에서는 불리할 수 있습니다. UMA의 이점은 "복사 제거로 인한 지연시간·에너지 절감과 메모리 중복 저장 방지"에 있지, 무조건적인 최대 성능 우위에 있는 것이 아닙니다.

## 3. 사례로 보는 UMA — Apple Silicon과 NVIDIA Jetson

> **개요** UMA 원리를 실제로 채택한 두 대표 사례를 통해 설계가 실제 제품으로 어떻게 이어지는지 살펴봅니다.

애플의 M 시리즈 칩(M1, M2, M3, M4 등)은 공식적으로 "통합 메모리 아키텍처"라는 용어를 제품 설명에 그대로 사용합니다. CPU 코어, GPU 코어, 신경망 연산 전용 유닛(Neural Engine)이 모두 하나의 LPDDR 메모리 풀을 공유합니다. 예를 들어 대규모 언어 모델의 가중치를 메모리에 한 번 올려두면, CPU가 전처리한 데이터를 GPU가 곧바로 같은 메모리 주소에서 읽어 추론에 사용할 수 있습니다. 별도의 VRAM으로 가중치를 복제해둘 필요가 없습니다.

NVIDIA의 Jetson 시리즈(Jetson Nano, Xavier, Orin 등)는 로봇·드론·산업 카메라 같은 임베디드/엣지 컴퓨팅 기기를 위한 모듈입니다. 데스크톱의 NVIDIA 그래픽카드와 달리, Jetson의 GPU는 CPU(ARM 코어)와 같은 SoC 안에 통합되어 있고, 같은 LPDDR/DDR 메모리를 공유합니다. 즉 데스크톱에서는 "CPU + 분리형 GPU"였던 조합이, Jetson에서는 "ARM CPU + 통합 GPU + 공유 메모리"라는 UMA 구조로 바뀌어 있습니다.

## 4. 로봇·엣지 디바이스는 왜 UMA를 선호할까요?

> **개요** 실시간 인지-제어 루프를 도는 로봇에게 UMA가 왜 특히 유리한 선택인지 설명합니다.

> **Q.** 서버실의 대형 GPU 클러스터는 여전히 분리형 GPU를 쓰는데, 왜 로봇이나 드론은 UMA 구조를 선호할까요?
> **A.** 로봇과 드론은 세 가지 제약을 동시에 짊어집니다. 배터리로 동작하므로 전력 예산이 빠듯하고, 카메라·라이다 센서 입력을 받아 제어 신호를 내보내기까지 정해진 시간(실시간성) 안에 끝내야 하며, 기체에 실어야 하므로 무게와 부피도 제한적입니다. 분리형 GPU 구조를 그대로 쓰면 센서 프레임마다 CPU 메모리에서 GPU VRAM으로 복사하는 지연시간이 실시간 제어 주기를 위협하고, 그 복사에 쓰이는 전력이 한정된 배터리를 갉아먹으며, 별도의 그래픽카드와 PCIe 슬롯은 기체에 실을 수 없을 만큼 부피와 무게를 차지합니다. UMA 기반 SoC는 이 세 가지 제약을 동시에 만족시킵니다.

로봇의 전형적인 인지-제어 루프를 생각해봅시다. 카메라가 프레임을 촬영하면, CPU가 이를 전처리(왜곡 보정, 크기 조정)하고, GPU 또는 신경망 가속기가 물체 인식이나 장애물 회피 추론을 수행한 뒤, 다시 CPU가 그 결과를 바탕으로 모터에 보낼 제어 신호를 계산합니다. 이 루프는 초당 수십 번씩 반복되어야 합니다.

![로봇의 실시간 인지-제어 루프와 UMA](https://raw.githubusercontent.com/SeongminJaden/dev-curriculum-assets/main/lectures/part4_프로세서_아키텍처_심화/4-3_통합_메모리_아키텍처의_원리/assets/diagram2_edge_robotics_loop.svg)

> **해석** 이 루프에서 카메라 데이터는 CPU와 GPU를 여러 차례 오갑니다. 분리형 GPU 구조라면 이 루프 한 바퀴마다 PCIe를 통한 복사가 여러 차례 끼어들어 지연시간과 소비 전력이 누적됩니다. UMA 구조에서는 같은 메모리 주소를 CPU와 GPU가 번갈아 참조하기만 하면 되므로, 루프 한 바퀴를 도는 데 필요한 지연시간과 에너지가 크게 줄어듭니다. 이것이 배터리 하나로 비행해야 하는 드론과, 실시간 제어 주기를 지켜야 하는 로봇 모두에게 UMA가 유리한 이유입니다.

> **팁** "로봇에는 왜 항상 ARM + UMA 조합의 임베디드 보드(Jetson 등)가 쓰이는가"라는 질문을 받으면, ISA(ARM, 저전력)와 메모리 구조(UMA, 복사 제거)라는 두 개의 독립적인 설계 축이 동시에 저전력·저지연을 만족시키기 위해 함께 선택된 결과라고 답하는 것이 정확합니다. 두 가지는 서로 다른 개념이지만, 엣지 디바이스라는 같은 제약 조건 아래에서 같은 방향을 가리키기 때문에 함께 쓰이는 경우가 많습니다.

> **핵심**
> - 전통적인 분리형(discrete) GPU 구조에서는 CPU의 RAM과 GPU의 VRAM이 물리적으로 분리되어 있어, GPU를 쓰려면 PCIe 버스를 통한 데이터 복사와 동기화 과정이 반드시 필요합니다.
> - 통합 메모리 아키텍처(UMA)는 CPU와 GPU가 <u>**하나의 물리적 메모리 풀을 같은 주소 공간으로 공유**</u>하게 하여, 데이터 이동을 포인터 전달로 대체하고 복사 단계 자체를 제거합니다.
> - Apple Silicon과 NVIDIA Jetson은 이 원리를 각각 소비자용 컴퓨터와 임베디드/엣지 컴퓨팅 영역에서 구현한 대표 사례입니다.
> - 로봇·드론 같은 엣지 디바이스는 제한된 전력, 실시간 제어 주기, 제한된 무게·부피라는 세 가지 제약을 동시에 만족해야 하므로, 복사 비용이 없는 UMA 구조가 특히 유리합니다.

## 요약

> **핵심**
> - 분리형 GPU는 CPU RAM과 GPU VRAM이 물리적으로 분리되어 있어 PCIe를 통한 **데이터 복사**가 반드시 필요하며, 이 복사가 지연시간과 전력을 소모합니다.
> - UMA는 CPU와 GPU가 **같은 물리적 메모리 풀을 하나의 주소 공간으로 공유**하게 하여, 복사를 포인터 전달로 대체합니다.
> - Apple Silicon과 NVIDIA Jetson은 각각 소비자용·엣지용 UMA 구현 사례이며, 로봇·드론은 전력·실시간성·무게 제약 때문에 이 구조를 특히 선호합니다.
